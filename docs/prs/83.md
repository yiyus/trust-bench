# PR notes: Issue #83 - Port difficulty-family problems to native APL implementations

## Summary

`backends_ext/apl/problems/` has only the seven canonical problems;
every study built on one of `trust_bench/problems/families/` (`large_
residual`, `ill_conditioning`, `robust_loss`, `scaling`, `dimensionality`)
produces zero real `trust-apl` data, since the harness has no
implementation to evaluate and returns `ERROR: Unknown problem_id` for
every parametrised id (e.g. `"scaling(s=10.0)"`).

Scoped to four families for this pass: `ill_conditioned`, `large_
residual`, `outliers`, `scaling`. All four are small, fixed 2-variable
problems parametrised by one scalar constant, structurally similar to
the seven canonical problems. `dimensionality(n)` needs arbitrary-size
vectors/matrices and real performance at n up to 1000, a materially
different engineering effort; split out to #90 and noted on #83.

This requires one new piece of shared infrastructure: today's
`NameFor` maps a fixed problem id string 1:1 to a function name. A
parametrised id like `"scaling(s=10.0)"` needs parsing into a family
name plus a numeric argument, then dispatch to a function taking
`(param, x)` rather than just `(x)`. Both dispatch mechanisms coexist:
the seven canonical problems still resolve via the existing `NameFor`;
anything `NameFor` doesn't recognise falls through to the new
parametrised-id parser.

Every ported problem's residual/Jacobian/Hessian and the new dispatch
mechanism were verified directly (`dyalogscript`) against the Python
reference before any test was written - see Design notes.

**Design notes**

*Every ported family's residual/Jacobian/Hessian verified numerically
against the Python reference before writing any test*, at the specific
parameter/probe-point combinations later used by the tests above (all
matched to `1e-9`/`1e-6` on the first attempt): `Scaling`,
`IllConditioned`, `LargeResidual` (40-point exponential-decay model,
column-stacked Jacobian via `⍉2 40⍴...`, and its non-Gauss-Newton
Hessian correction term), `Outliers` (20-point corrupted linear
regression).

*The parametrised-id dispatch mechanism* - `ParseParametrised`
(extracts the family name and numeric argument from a `"name(key=
value)"` string via `⍳`-based substring positions, `⍎`-parsed for the
numeric part so both plain and Python-scientific-notation formatting
round-trip correctly), `FamilyNameFor`/`FamilyHessianNameFor` (mirror
`NameFor`/`HessianNameFor` for the four family names), `ApplyParam`
(the existing `Apply`'s name-string-plus-`⍎` dispatch pattern,
generalised to also splice in a parameter value) - verified directly
end-to-end (`dyalogscript`) for `solve` (all three methods, bounds,
`finite-difference` mode) and `evaluate` mode, against all four
families, before any test was written.

*`ApplyParam` is monadic, not dyadic*, despite taking a "parameter"
conceptually to the function's left: it destructures a single 3-element
right-argument strand (`name param point←⍵`), exactly like `Apply`
destructures a 2-element one. A first attempt called it dyadically
(`param ApplyParam f ⍵`) from within `solve.dyalog`'s per-method
evaluator closures - `⍵` there is a 2-element strand `(f ⍵)`, not the
3-element one `ApplyParam` destructures, so it raised `LENGTH ERROR`
from `Min`'s very first internal call. Caught via direct `dyalogscript`
verification before this reached a test file.

*`solve.dyalog`'s per-method evaluator closures branch on `isParam`
directly at each `Apply`/`ApplyParam` call site*, rather than routing
through one shared indirection closure (e.g. an `Eval←{isParam:...}`
local variable called from within `lower`). The shared-closure version
was tried first and is logically equivalent - confirmed correct when
called directly, bypassing `Min` - but broke specifically when invoked
through `trust`'s `Min` operator (`LENGTH ERROR`/`RANK ERROR` on its
very first internal call, depending on which variant was tried).
Root cause not pinned down further once the direct-branching version
was confirmed to work correctly end-to-end against all four families,
all three methods, and bounds; recorded here as a real, verified
constraint on this dispatch mechanism's shape, not a stylistic choice.

outliers}.dyalog` (new): each family's dyadic residual/Jacobian
function (parameter on the left, point on the right) plus its paired
Hessian function, exactly the design verified in scratch during RED.

`backends_ext/apl/dispatch.dyalog`: adds `FamilyNameFor`/
`FamilyHessianNameFor` (mirroring `NameFor`/`HessianNameFor` for the
four family names), `ParseParametrised` (splits a `"name(key=value)"`
id into family name and numeric value via `⍳`-based substring
positions), and `ApplyParam` (monadic, mirrors `Apply`'s name-string-

canonical problems); if that's empty, parses the id via
`ParseParametrised` and dispatches through `FamilyNameFor`/
`ApplyParam` instead. An id that matches neither still reports
`Unknown problem_id`.

`backends_ext/apl/solve.dyalog`: same two-path resolution at the top;
every per-method evaluator closure (`lm`/`BFGS`/`trust-exact`) branches
on `isParam` at its own `Apply`/`ApplyParam` call site, per the
direct-branching constraint recorded in the RED-stage design notes.

`backends_ext/apl/run_harness.sh`: concatenates the four new problem
files alongside the existing seven.

No changes needed in `trust_bench/backends/apl_backend.py`: `solve()`
and `evaluate_problem()` already forward `problem.id` to the harness
verbatim, so a parametrised id reaches the harness exactly the same
way a canonical one does.

## Review response (implementation stage)

Major: `ParseParametrised`'s `⍎`-based numeric parsing assumed Python's
float formatting always produces a Dyalog-parseable literal. It
doesn't: Python renders small magnitudes in scientific notation with an
ASCII `-` in the exponent (`f"{1e-05}"` → `'1e-05'`), while Dyalog's
numeric literal syntax needs the high-minus `¯` for a negative exponent
(or a negative mantissa). `⍎` saw the ASCII `-` as an unrelated token
after `e` and raised `VALUE ERROR: Undefined name: e`, reported as a
generic `ERROR` status with no indication of the real cause. No
parameter value exercised by this issue's own tests or any existing
study currently triggers this (`kappa`/`s` top out at `1e8`, still
plain-decimal in Python), but it was a real crash waiting for the next
family or study parametrised by a small fraction, rate, or scale factor
near zero.

Fixed in `ParseParametrised` by translating every ASCII `-` to `¯` in
the extracted parameter substring before `⍎`
(`'¯'@('-'∘=)⊢text`): the only place a hyphen can appear in a numeric
literal is a leading negative sign or a negative exponent, both of
which need the same translation, so a blanket per-character replace is
correct and sufficient. Added `test_parses_a_parameter_value_in_
python_scientific_notation` (`tests/backends_ext/apl/test_harness.py`),
confirmed genuinely RED against the pre-fix code (`returncode == 1`)
before the fix, passing after it; also confirmed directly
(`dyalogscript`) that a negative mantissa (`"scaling(s=-5.0)"`) and a
small-magnitude fraction (`"outliers(fraction=1e-05)"`, which rounds to
zero corrupted points) both parse and solve correctly now.

Re-verified: the three affected test files - 33 passed (up from 32,
the one new test). Full suite `pytest -q` (dyalogscript available):
319 passed, 0 failed. `ruff check .`: clean.

# PR notes: Issue #19 - RunConfig and intent-level tolerance mapping (config.py)

## Summary

Introduces `RunConfig`, the reviewed design's type for `Backend.solve`'s
`config` parameter and `RunResult.config` (`docs/plans/trust-bench.md`,
Section 4.2/4.3), replacing the plain dicts every prior backend issue used
as a placeholder. Adds intent-level `tolerance`, mapped per method to
`SciPyBackend`'s native stopping parameters (`ftol`/`xtol`/`gtol`), per
Section 7's comparability methodology. Part of Epic #18 (Phase 5).

Implementation scope explicitly includes `trust_bench/core/result.py`:
`RunResult.from_dict` must reconstruct `config` back into a `RunConfig`
instance, the same way it already does for `provenance` (`EnvProvenance`)
and `timing` (`TimingStats`). This was surfaced during RED-stage review,
not in the original scoping; recorded here so it is not a mid-GREEN
surprise.

**Design notes: what `tolerance` maps to, per method**

Verified directly against `scipy.optimize.show_options` for every
`minimize` method (`least_squares` accepts `ftol`, `xtol`, `gtol`
unconditionally as top-level keyword arguments, documented in its own
signature, so no per-method lookup is needed there):

| Method | Native tolerance options |
|---|---|
| `lm`, `trf`, `dogbox` | `ftol`, `xtol`, `gtol` (all three, always) |
| `BFGS` | `gtol` |
| `L-BFGS-B` | `ftol`, `gtol` |
| `Newton-CG` | `xtol` |
| `trust-exact` | `gtol` |
| `trust-constr` | `gtol`, `xtol` |

A single intent-level `tolerance` is applied to every native parameter a
method exposes among `{ftol, xtol, gtol}`, rather than picking one and
leaving the others at scipy's own default. For `least_squares`, this
mirrors scipy's own behaviour (all three default to the same `1e-8`); for
`minimize`, it means e.g. `L-BFGS-B` gets both its cost-change and
gradient-norm criteria tightened together, since a single number cannot
express three independently-meaningful thresholds and the plan does not
ask it to.

`test_no_tolerance_config_leaves_scipys_own_defaults_in_place` (both

elsewhere in this codebase, e.g. the existing `loss` and `derivative_mode`

## Review response (RED stage)

Two Major findings, both fixed:

- `test_run_config_round_trips_through_json_serialisation` used a
  tuple-shaped `bounds` value (`[(0.0, 1.0)]`), which does not survive a
  JSON round trip (JSON has no tuple type, so it comes back as
  `[[0.0, 1.0]]`) under the trivial `cls(**data)` `from_dict` pattern this
  codebase already uses for `TimingStats`/`EnvProvenance`. Rather than add
  bounds-specific reconstruction logic with no precedent elsewhere (and no
  single canonical shape to reconstruct, since `test_scipy_least_squares.py`
  and `test_scipy_minimize.py` pass `bounds` in two different shapes),
  changed the test to use a JSON-native `bounds=[[0.0, 1.0]]`. `RunConfig.
  from_dict` stays as trivial as its siblings.
- `RunResult.from_dict` had no reconstruction step for `config`, so
  `test_run_result_round_trips_through_json_serialisation` would fail
  once `config` became a `RunConfig` instance instead of a plain dict
  (the restored value would stay a dict; dataclass equality would then
  distinguish it from the original `RunConfig`). This was a real
  implementation-scope gap, not a test bug: added `trust_bench/core/
  result.py` to this issue's stated scope (see Summary and the coverage
  table above) and will add the matching `data["config"] = RunConfig.
  from_dict(data["config"])` line during the GREEN cycle.

## GREEN cycle (implementation)

**Files changed**

- `trust_bench/core/config.py` (new): `RunConfig` frozen dataclass
  (`max_iter`, `tolerance`, `bounds`, `derivative_mode`, `loss`), with
  `to_dict`/`from_dict` following the same trivial pattern as
  `TimingStats`.
- `trust_bench/core/backend.py`: `Backend.solve`'s `config` parameter
  typed as `RunConfig` instead of `Any`, matching Section 4.2's design.
- `trust_bench/core/result.py`: `RunResult.config` typed as `RunConfig`
  instead of `Any`; `from_dict` gained `data["config"] = RunConfig.
  from_dict(data["config"])`, mirroring the existing `provenance`/`timing`
  reconstruction. `to_dict` needed no change: `dataclasses.asdict`
  already flattens nested dataclass fields recursively, the same way it
  already handled `provenance` and `timing` before this issue.
- `trust_bench/backends/scipy_backend.py`: `_solve_least_squares` and
  `_solve_minimize` switched from `config.get(...)` dict access to
  `config.<field>` attribute access. Added `_MINIMIZE_TOLERANCE_PARAMS`
  (the per-method table from the design notes above) and tolerance
  wiring: `least_squares` gets `ftol`/`xtol`/`gtol` all set to
  `config.tolerance` when it is not `None`; `minimize` gets the subset in
  `_MINIMIZE_TOLERANCE_PARAMS[method]` set via its `options` dict,
  alongside the pre-existing `maxiter` wiring (both now share one
  `options` dict, built conditionally, rather than `maxiter` alone
  deciding whether `options` is passed at all).
- `tests/backends/toy_backend.py`: `config["max_iter"]` changed to
  `config.max_iter` (test-only `Backend` implementation, updated to match
  the new contract, not itself under review).

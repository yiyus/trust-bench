# PR notes: Issue #26 - Study: bounded and constrained

## Summary

Adds the bounded/constrained study (`docs/plans/trust-bench.md`, Section
9 item 5): sweeps three scenarios (inactive bounds, active-at-boundary
bounds, an infeasible start) across SciPy's `trf`/`dogbox`, using
`trust_bench.problems.quadratic.PROBLEM` with custom bounds and starts
constructed directly in the study (this scenario type has no dedicated
entry in `problems/families/`; Section 3's repository layout does not
list one, since it is a per-config concern, not a per-problem
difficulty axis). Part of Epic #21 (Phase 6).

**Scope deferral, same as issue #25:** the acceptance criterion compares
against `trust`'s (the APL backend) Coleman-Li affine scaling, but that
backend does not exist yet (Epic #34, Phase 8, not started). This issue
implements the SciPy side of the sweep now; the Coleman-Li comparison is
added once Phase 8 lands.

**Design notes**

`quadratic.PROBLEM`'s unconstrained optimum is `(0, 0)`. A lower bound
of `0.5` on `x1` makes that infeasible, so the constrained optimum sits
exactly on the boundary at `(0.5, 0)`, giving a clean active-at-boundary
scenario without needing a new problem family:

- `inactive`: bounds `([-10, -10], [10, 10])`, start `(1, -1)`, comfortably
  inside; the box never binds.
- `active_at_boundary`: bounds `([0.5, -10], [10, 10])`, start `(1, -1)`
  (feasible); the constrained optimum is `(0.5, 0)`.
- `infeasible_start`: same bounds, start `(-5, -5)` (outside the box on
  `x1`).

**A finding worth disclosing, not a bug:** verified directly that
SciPy's `least_squares` (both `trf` and `dogbox`) raises `ValueError:
Initial guess is outside of provided bounds` for *any* `x0` outside the
box, including values only marginally outside (`x1=0.499` against a
`0.5` lower bound) — there is no silent clipping or projection, contrary
to what the `trf`-specific note in `least_squares`'s own docstring
("the initial guess might be slightly adjusted...") suggests; that note
turned out to refer to floating-point noise exactly on the boundary, not
genuinely infeasible starts. `test_infeasible_start_raises_a_clear_
error_for_both_methods` asserts this hard-rejection directly, rather
than expecting a solve to succeed.

## Review response (RED stage)

Major: `test_inactive_bounds_converge_to_the_unconstrained_optimum` and
`test_active_at_boundary_bounds_converge_exactly_to_the_boundary`
asserted exact equality (`== 0.0`, `== 0.5`). Verified directly:
`dogbox` reaches both targets exactly under `SciPyBackend`'s default
tolerances, but `trf` does not (`[2.176e-16, -2.176e-16]` and
`[0.5000000001164153, 0.0]`), only matching exactly at
`ftol=xtol=gtol=1e-15`, essentially machine epsilon. Changed both
assertions to `pytest.approx(..., abs=1e-6)`: comfortably covers both
methods' actual deviations (`~1e-16` and `~1e-10`) with margin to spare,
without depending on a SciPy version, BLAS backend, or platform detail
as fragile as `1e-15`.

## GREEN cycle (implementation)

**Files changed**

- `trust_bench/studies/bounded.py` (new): `sweep(scenarios=SCENARIOS,
  methods=METHODS, backends=BACKENDS)`, keyed by `(scenario, method,
  backend_name)`, returning either the `RunResult` or the raised
  `ValueError`.

**A latent bug caught before it ran:** the first implementation keyed
`outcomes` by `(scenario, method)` only, dropping `backend_name` from the
key despite the docstring already describing it as three-part. With one
backend registered this happens to work; with a second registered
backend it would silently overwrite one backend's result with the
other's instead of keeping both. Fixed before running any test, in the
same pass as writing the implementation; `tests/studies/test_bounded.py`
updated to match the corrected three-part key.

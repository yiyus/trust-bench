# PR notes: Issue #96 - Extend the loss vocabulary to reach trust's Tukey, Welsch and Fair losses

## Summary

`apl_backend.py`'s `_LOSS_TO_TRUST` currently maps only the five losses
scipy also has; trust's `Loss` namespace also implements `Tukey`,
`Welsch`, and `Fair` (each with its own default tuning constant, same
dyadic calling convention as the existing five), but `RunConfig` has no
established name for any of them. Even after #94 fixed `robust_loss.py`'s
method mismatch, the study could not exercise trust's own redescending
losses - only the subset scipy also offers, comparing against a
hand-rolled Python Tukey-biweight IRLS reference standing in for
`trust` itself.

Empirically confirmed before writing any test (`dyalogscript`):
mapping all three works mechanically for every one of trust's three
methods. But from the study's existing "standard" (zero) start,
neither `tukey` nor `welsch` demonstrates redescending robustness at
high contamination - both perform no better than scipy's own built-ins
there. Warm-starting from an arctan fit (exactly the technique
`robust_loss.py`'s own `irls_tukey` already uses, for the documented
reason that a redescending loss has no basin-of-attraction guarantee
from an arbitrary start) changes this: `welsch` then recovers the true
parameters almost exactly up to 45% contamination and only breaks down
near the 49% theoretical limit for a redescending M-estimator, closely
tracking the hand-rolled reference's own behaviour. `tukey` remains
erratic even warm-started (fails around 30% contamination), likely
because of its harder cutoff (`trust`'s own code comments describe
`Welsch` as "smoother", `Tukey` as a hard-cutoff re-descending
M-estimator). Confirmed directly with the user before committing to
this design: sweep `welsch`, warm-started, as the one trust-only loss
`robust_loss.py`'s own sweep exercises by default; map all three so
`tukey`/`fair` remain available directly via `RunConfig`/`APLBackend`.

**Design notes**

*Warm start is scoped to the one new trust-only sweep, not the
existing "standard"-start comparison.* `trust_loss_precision` is a new
function, separate from `scipy_loss_precision`, rather than a
conditional branch inside it: the existing five-loss comparison (every
loss on the "standard" start) is unaffected, and the warm-start
mechanism (an arctan fit via `scipy.optimize.least_squares`, computed
per fraction and registered as a new `"warm"` start on a `dataclasses.
replace`d `Problem`) is factored out of `irls_tukey`'s own inline
computation into a shared `_warm_start(problem)` helper, so both
functions compute it identically rather than duplicating the logic.

*`tukey` is mapped but not swept by default.* `_LOSS_TO_TRUST` gains
all three names (satisfying "`RunConfig.loss` accepts `tukey`,
`welsch`, and `fair`" and "`_LOSS_TO_TRUST` maps all three" literally),
but `TRUST_LOSSES = ["welsch"]` is the only one `trust_loss_precision`
sweeps by default, since only `welsch` demonstrates the acceptance
criterion's actual claim reliably at this problem's scale. `tukey`/
`fair` remain reachable directly via `RunConfig(loss="tukey")` against
`APLBackend`, covered by `test_apl_loss_vocabulary.py`.

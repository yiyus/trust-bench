# PR notes: Issue #127 - Bump trust to da5dfcc (later 8ba5494), wire up x_scale/status classification

## Summary

Bumps the vendored `trust` submodule to `8ba5494` and wires up everything it newly enables:

1. **`pscale`/`x_scale`**: `apl_backend.py` maps a fixed `RunConfig.x_scale` to `trust`'s new `pscale` config field for `lm`/`BFGS`; `x_scale="jac"` (no adaptive equivalent) and a fixed vector for `trust-exact` (a Hessian needs outer-product scaling on both axes, not a column scale) are both still rejected. Confirmed directly: at `scaling.make(1e8)`, `trust-apl`'s `lm` goes from `FAILED` (`cost_final≈2e16`) unscaled to `CONVERGED` (`cost_final≈1.9e-12`) with a fixed `x_scale=(1.0, 1e-8)` matching the problem's own known anisotropy.
2. **`scaling.py`**: `X_SCALES` gains `"fixed"` (a per-`scale` `(1.0, 1.0/scale)` value); `trust-apl`'s `BFGS` also gains a fixed `pscale`, exercised separately since scipy's own `BFGS` has no `x_scale` concept to compare against at all (matching `robust_loss.py`'s own precedent for a backend-specific capability).
3. **`solve.dyalog`'s status classification rewritten** using vendored `trust`'s own new `Result` namespace (`Result.MaxIterations`/`DampingSaturated`/`StalledByEscalation`/`StalledByPrecision`) instead of the ad-hoc `:If`/`:ElseIf` chain built up across #95/#115/#117/#119 - same observable behaviour (confirmed: 72 tests across every affected file pass unchanged, no test needed updating for this part), cleaner mechanism.
4. **`grad_norm_final` no longer costs an extra evaluation**: `Min.aplo` now returns the final gradient it already computes internally (`r.grad`), read directly instead of a separate `Apply`/`ApplyParam` probe at the final point.

**A real bug found and fixed upstream before landing**: `Result.StalledByPrecision` shipped (`da5dfcc`) with its guard condition inverted - `⍵.((tolr≤rel)∨1≥dnorm):0` bails out (returns false) exactly for the low-damping case ("precision loss at low damping") its own name and comment describe, meaning it could never fire for the case it was built for. Confirmed directly (`ill_conditioned(kappa=1e7)` `trust-exact`, `dnorm≈0.001`): returned `0` before the fix. Flagged and fixed upstream (`8ba5494`, `1≥dnorm` -> `1<dnorm`) before implementing anything against it here - re-verified the fix directly against the same case (now returns `1`) and against the `kappa=1e4` `BFGS` escalation case (unaffected, still `1`).

**Two upstream diffs the issue's own text didn't quite match**, both confirmed directly rather than assumed:
- `docs/methodology.md` had no existing prose describing a `scaling`/`scale=1e8` capability boundary to "correct" (the issue's text implied one existed) - a new section was added instead of a correction.
- The new `Result` namespace's field names (`tolc`/`toli`/`tolr`/`dmax`/`cost`/`iter`/`dnorm`/`rel`/`grad`) resolve against the *same* namespace `Min` returns to its caller (config and result fields are merged onto one object by `Min.aplo`'s own `CB`/`⎕VSET` mechanism) - confirmed directly rather than assumed, since this is what makes `Result.XXX r` callable at all from `solve.dyalog`'s own `r←lower Min(req.x0)cfg`.

**Design notes**

*Status classification collapses to a clean 4-branch chain*, not a reimplementation of the old heuristic logic in `Result` terms: `StalledByEscalation`/`StalledByPrecision` partition exhaustively on `dnorm>1` vs `dnorm≤1`, so "neither fires" is exactly the existing "genuine convergence via a small-gradient stall" case (`large_residual`/`outliers`, `Result.Converged`'s own strict `cost<tc` case) - no separate check needed. The bounded-request exclusion (an active-boundary optimum's unconstrained gradient is genuinely nonzero) is preserved unchanged, since `Result.StalledByPrecision`'s gradient check has the identical false-positive problem there as the prior harness-level heuristic did.

*`scaling.py`'s `BFGS`+fixed exercise respects the caller's own `x_scales` override* (only runs when `"fixed"` is actually requested), rather than being unconditional - avoids surprising a caller who explicitly restricts `x_scales` to `[None]`.

## GREEN cycle (implementation)

`trust_bench/backends/apl_backend.py`: `config.x_scale` handling replaced - a string value (`"jac"`) is rejected for every method (no adaptive equivalent); a fixed vector is rejected only for `trust-exact` (Hessian-scaling gap) and otherwise mapped to `request["pscale"]`.

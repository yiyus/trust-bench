# PR notes: Issue #94 - robust_loss.py still crashes against trust-apl

## Summary

`robust_loss.py`'s `scipy_loss_precision` hardcodes `_METHOD = "trf"`
for every backend. `trust-apl` has no `"trf"` method (only `"lm"`,
`"BFGS"`, `"trust-exact"`), so `run(problem, backend, "trf", ...)`
raises `ValueError: trust-apl has no method 'trf'` uncaught -
crashing `trust-bench report --backends scipy trust-apl` entirely
whenever `robust_loss` runs. Flagged as a known, deferred gap since
#78/#81 and confirmed still broken after #83/#84 landed (the problem
family itself is ported; only the method selection was never fixed).

`trust-apl`'s `"lm"` declares `losses=frozenset(_LOSS_TO_TRUST)`, the
same five losses (`linear`/`soft_l1`/`huber`/`cauchy`/`arctan`)
`SCIPY_LOSSES` already sweeps, so once the method mismatch is fixed
there is no further loss-vocabulary gap for this study to close (the
issue's own companion-issue reference is about `trust`'s
Tukey/Welsch/Fair losses, not reachable via `RunConfig.loss` at all
yet - out of scope here).

## RED cycle (test-first)

**Test surface**

- `tests/studies/test_robust_loss.py` (extended): `test_scipy_loss_
  precision_skips_a_backend_it_has_no_method_mapping_for`, a new local
  `_UnmappedBackend` stub (mirrors `bounded.py`'s own skip-guard test
  pattern) proving an unrecognised backend name is skipped rather than
  driving a `ValueError`-raising call. Also removed a stale comment
  claiming "the APL backend does not exist yet" (it has, since #78).
- `tests/studies/test_robust_loss_apl.py` (new, slow, `dyalogscript`-
  gated): `test_trust_apls_lm_is_exercised_for_every_loss_instead_of_
  raising` runs `scipy_loss_precision` with both backends at zero
  contamination and checks every loss recovers the true parameters via
  `trust-apl`.

**Confirmed RED**

Both fail against the unmodified `robust_loss.py`, reproducing the
issue's own exact error: `ValueError: trust-apl has no method 'trf'`,
raised from `_UnmappedBackend`'s stub `solve()` in the fast test (proof
the current code always attempts a call rather than skipping) and from
the real `APLBackend.solve` in the slow test. `make lint`: clean on
both files.

**Design notes**

Fix is a per-backend method mapping (`{"scipy": "trf", "trust-apl":
"lm"}`), not a capability-driven method search: `"trf"` is one of
several scipy methods that would technically satisfy the full loss
vocabulary (`"dogbox"` too), so an automatic "first method supporting
every swept loss" search would be ambiguous for scipy specifically:
picking one method explicitly per backend, matching the issue's own
framing ("picking a backend-appropriate method"), is more legible than
resolving that ambiguity implicitly. An unmapped backend name is
skipped, not an error: matches the established precedent (`ill_
conditioning.py`, `bounded.py`) of never crashing on a combination the
study doesn't know how to drive.

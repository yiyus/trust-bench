# PR notes: Issue #32 - Capability matrix derivation (capability_matrix.py)

## Summary

Adds `trust_bench/reporting/capability_matrix.py`: for every registered
backend and method, cross-references the *declared* `Capabilities`
(`bounds`, `analytic_hessian`) against a live *measured* probe run
against the trivial quadratic problem, and records whether the two
agree. Part of Epic #30 (Phase 7).

**Design notes**

Section 9.1's illustrative matrix has rows that don't all map onto a
real `MethodCapabilities` field ("Redescending losses", "Matrix-free /
large n" are qualitative, research-derived categories, not booleans
this codebase declares anywhere). Scoped this issue to the two fields
that are genuine `MethodCapabilities` booleans and have a clean,
programmatic way to *measure* them live, rather than force-fitting the
plan's exact illustrative rows:

- **`bounds`**: probed by solving the trivial quadratic with a box
  constraint and checking whether the constraint is actually respected
  (`CONVERGED` and `x_final[0] >= 0.5`) versus rejected (raises
  `ValueError`). `least_squares`-family methods and `minimize`-family
  methods take *different* bounds shapes (a tuple of two arrays vs. a
  list of `(lo, hi)` pairs); the probe uses whichever shape matches the
  method, the same distinction `bounded.py` (issue #26) and `test_scipy_
  minimize.py` (issue #17) already make.
- **`analytic_hessian`**: probed by solving the same problem (which has
  a real analytic Hessian) and checking `RunResult.n_heval > 0` —
  directly measuring whether the Hessian was actually called, not just
  whether the method nominally accepts one.

Verified directly against every method on `SciPyBackend`, before writing
any assertions: both probes agree with the declared value for all nine
methods (initially found `L-BFGS-B`/`trust-constr` appearing to
disagree on `bounds`, traced to the probe itself using the
`least_squares` bounds shape for every method rather than switching per
family — a bug in the calibration script, not the backend; fixed before
writing the test).

`test_derive_matrix_flags_a_declared_measured_mismatch` uses a small
`Backend` test double that wraps `SciPyBackend`'s `lm` but declares
`bounds=True` (a lie: `lm` genuinely rejects bounds), to prove the
agreement check actually distinguishes a mismatch from a match rather
than only ever seeing agreement.

## GREEN cycle (implementation)

**Files changed**

- `trust_bench/reporting/capability_matrix.py` (new): `derive_matrix(
  backends=BACKENDS, fields=FIELDS)`, keyed by `(backend, method,
  field)`, with `_measure_bounds`/`_measure_analytic_hessian` as the
  live probes.

**A test bug, not an implementation bug, found by running the tests:**
`test_derive_matrix_flags_a_declared_measured_mismatch` asserted `row["
declared"] is True`; `pandas` stores the `declared`/`measured`/`agrees`
columns as `numpy.bool_`, and `numpy.True_ is True` is `False` (a
different object, same truthiness). Fixed by comparing `bool(row[...])
is True` instead. `derive_matrix` itself was correct; the identity
check in the test was not.

**A pre-existing, unrelated flaky test, encountered and not fixed:**
`make coverage` intermittently fails `tests/studies/test_dimensionality.
py::test_dense_hessian_methods_cost_far_more_per_step_at_large_
dimension` (issue #28's per-step-cost timing ratio, `elapsed > 5x
reference_time`), reproduced identically on `main` with none of this
issue's changes applied. Out of scope for this issue (`capability_
matrix.py` never touches `dimensionality.py` or `scipy_backend.py`);
noted here rather than silently worked around, and worth its own issue
if it keeps recurring.

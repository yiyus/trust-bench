# PR notes: Issue #23 - Study: large residual (LM vs Newton)

## Summary

Adds the large-residual study (`docs/plans/trust-bench.md`, Section 9
item 2), ported from `prototype/large_residual.py` and
`prototype/run_experiment.py`. Two independent things, matching the
prototype's own split into "(A) local iteration" and "(B) production
solvers":

- The predicted Gauss-Newton spectral radius at the optimum vs. the
  measured linear rate of a pure, undamped Gauss-Newton iteration,
  including the divergence boundary once that radius exceeds 1.
- `grad_norm_final` and the basin-of-attraction rate from every
  registered (globalized, damped) backend across the same `rho` sweep,
  which converge throughout, including past the point where undamped
  Gauss-Newton diverges.

Part of Epic #21 (Phase 6).

`test_predicted_spectral_radius_increases_monotonically_with_rho` is not
a direct reading of any one acceptance criterion; it exists for Phase
6's own instruction ("assert shape... monotonic trends where expected")
and is cheap insurance against a sign error in the spectral-radius
calculation, which the other two rate-comparison tests (fixed `rho`
values) would not by themselves catch.

**Design notes**

Ported the prototype's key finding that the undamped iteration must
start *near* the optimum (`x_star * 1.02`, matching
`prototype/run_experiment.py`'s `x_near`), not from the problem's
registered `standard` start: the spectral-radius prediction is a *local*
asymptotic rate, and undamped Gauss-Newton has no global convergence
guarantee. Confirmed directly that starting from `standard` instead
produces a wildly non-monotonic trajectory unrelated to the predicted
rate, even for `rho` values where the local rate is comfortably below 1.

Verified directly (`gn_spectral_radius` against each `rho` in `RHOS`,
computed ad hoc) that the boundary crosses 1 between `rho=50` (radius
0.986) and `rho=70` (radius 1.071); `rho=100` (radius 1.139) is used for
the divergence test with comfortable margin. `rho=10` and `rho=30` were
checked to reproduce the predicted radius to within a fraction of a
percent when started near `x_star`, so the `rel=0.05` tolerance is not
load-bearing precision, only insurance against float noise.

`backend_results()` reuses `SciPyBackend`'s `lm` method from the
registered `standard` start (not the near-optimum point above) across
the same `RHOS` sweep: confirmed directly that it converges for every
`rho`, including past the undamped-divergence boundary, which is the
whole point of contrasting a damped, globalized solver against the raw
iteration. `dist_to_opt` at these settings ranges up to ~2e-4 across the
sweep (looser than the trivial problems in issue #22's baseline study),
so the basin tolerance here is `1e-3`, not `1e-6`.

**Verification**

- `make test`: 151 passed (146 pre-existing + 5 new).
- `make lint`: clean.
- `make coverage`: 99% overall; `trust_bench/studies/large_residual.py`
  at 98%, one accepted gap: the `e > cap` defensive branch in
  `undamped_gauss_newton_errors`. Checked every `rho` in `RHOS` plus
  several larger ones (200, 300): the exponential model's `exp(k*T)`
  term always overflows into a singular/non-finite Jacobian once the
  iteration is diverging, so `np.linalg.solve` raises `LinAlgError` and
  exits via the `except` branch well before any error value would cross
  `cap=1e6`. The guard stays as a documented invariant against a
  hypothetical finite-but-runaway sequence; it is not reachable by this
  problem family, not a missed test case.

## Summary

Adds `trust_bench.problems.families`: five `make(**params) -> Problem`
factories (Section 3's `large_residual.py`, `ill_conditioned.py`,
`outliers.py`, `scaling.py`, `dimensionality.py`). Part of Epic #7

The three `..._rejects_...` tests are not a literal reading of either
criterion; each guards a parameter-range assumption the family's own
"defining property" depends on (see design notes), following the same

assumption that is not enforced is a silent correctness gap, not a
convenience.

**Design notes**

All five families were derived and numerically verified (parity, known
optimum, and the monotonic sweep itself) in the same throwaway script used
for issue #9, extended for this issue; not part of this PR.

- **`large_residual`** is `prototype/large_residual.py`'s exponential-decay
  model, exposed as `make(rho)`. Unlike every canonical problem and the
  other four families, its optimum is *not* closed-form once `rho>0`: the
  two-parameter model cannot represent the added `rho*(t-0.5)` term, so the
  best-fit `(A,k)` shifts to partially absorb it, with no simple formula for
  where it lands. `make()` finds it the same way the prototype's
  `reference_optimum` does: `scipy.optimize.minimize` with the analytic
  Hessian, `method="trust-exact"`, to a tight `gtol`. This is why `scipy`
  moves from absent to a real runtime dependency in this PR, not a dev-time
  aid like `sympy` was for issue #9: `make()` needs it every time it runs,
  for any `rho`, not just to derive a handful of fixed cases once.
  Known-optimum checks for this family use `atol=1e-4` on `grad_star`
  (looser than every other family's `1e-6`), matching the numerical solve's
  own achieved precision (worst observed `5.6e-8`, see verification below).
  Defining property: `||r(x*)||`, the irreducible residual norm, which
  Section 6 and Section 9 item 2 already name as the axis this project
  studies `trust` against.
- **`ill_conditioned`** is a consistent (zero-residual) linear system
  `r(x) = A(kappa)@x - b`, `A(kappa) = diag([kappa, 1]) @ R` for a fixed
  30-degree rotation `R`. The rotation is not cosmetic: an axis-aligned
  diagonal `A` makes `x2`'s row trivially well-scaled regardless of
  `kappa`, which is a much weaker exercise of ill-conditioning than a dense
  matrix with the same singular values. `cond(A) = kappa` exactly, by
  construction (rotations preserve singular values), which is also why
  the monotonic-sweep test can assert exact equality (`rtol=1e-6`), not just
  monotonicity.
- **`outliers`** is ordinary linear regression (`y = m*t+c` fit to 20
  points) with a nested, deterministic corruption pattern: the first
  `round(fraction*20)` points get a fixed `+5.0` offset. Nested (each larger
  fraction's corrupted set is a superset of every smaller fraction's) so
  growing the fraction can only add contamination, never remove it. Still
  not sufficient for monotonicity in general: verified numerically that
  "distance of the least-squares fit from the true `(m0,c0)`" stops
  increasing past `fraction=0.5` (at 0.75 it is *closer* to the truth than
  at 0.5), because once a majority of points are corrupted, the fit starts
  explaining the corrupted majority as the new trend rather than being
  dragged away from it. `make()` raises `ValueError` for `fraction >= 0.5`
  rather than silently returning a `Problem` whose defining property
  would not hold: past that point "outlier" no longer describes a
  minority of the data, so the family's own premise stops applying, not
  just this implementation's test coverage of it.
- **`scaling`** is `r(x) = [x1-a, s*(x2-b)]`, `a=3.0`, `b=-2.0`, an already
  consistent, closed-form system. Defining property: the Hessian's
  diagonal ratio, exactly `s**2` by construction. The parity (AC1) sweep
  and the monotonic-sweep (AC2) test use *different* ranges for a
  methodological reason, not a family limitation: at `s=1e6` the Hessian
  spans 12 orders of magnitude between its two diagonal entries
  (`1` vs `1e12`), and a fixed-step-size central-difference Hessian check
  (this project's own FD verification method, `tests/problems/
  fd_stencils.py`) suffers catastrophic cancellation on the small entry
  under that step size, independent of whether the analytic Hessian is
  correct (verified directly: the *relative* error stays under `1e-10`
  throughout; only the FD probe breaks down at extreme scale disparity).
  AC1 is checked up to `s=1000` (comfortably inside the FD method's valid
  range); AC2, which compares the analytic Hessian to `s**2` directly with
  no FD involved, is checked up to `s=1e6`, matching Section 9 item 6's
  framing ("many orders of magnitude").
- **`dimensionality`** is the generalised Rosenbrock (pairs of residuals
  per two variables, `n` even), `x*=(1,...,1)`, `cost*=0`, matching Section
  9 item 7's `n = 2, 10, 100, 1000`. The FD-parity check (AC1) is restricted
  to `n ∈ {2, 10}`: `fd_hessian`'s stencil is `O(n^3)` (an `n x n` grid of
  entries, each needing `O(n)`-cost objective evaluations), impractical at
  `n=100` or `1000` in a unit test and not informative beyond what `n=2`
  (identical formula to canonical Rosenbrock, already verified in issue #9)
  and `n=10` already establish. The known-optimum invariant (AC1's other
  half) and the monotonic-`n` sweep (AC2) use only the analytic Jacobian,
  `O(n)`, so both run at the full `{2, 10, 100, 1000}` Section 9 asks for.
  `make()` rejects odd `n`, since the residual pairing requires it.
- **No tags, consistent with issue #9's precedent**: `ill_conditioned`
  could plausibly carry `"ill-conditioned"`, `large_residual` could carry
  `"large-residual"` (both appear in `Problem.tags`'s own example set,
  Section 4.1), but no issue has yet decided what studies filter tags on.
  Adding them here would be no more grounded than issue #9 declining to tag
  Powell's confirmed-singular Jacobian.
- **No aggregating registry**, same reasoning as issue #9: no acceptance
  criterion in this issue needs one.

## Review response

`docs/reviews/10.md`'s Minor finding: `test_families.py`'s
`_assert_parity`/`_assert_known_optimum` duplicated the same logic
`test_canonical.py` (issue #9) already had inline, and a third
problem-test file would have been tempted to copy it a third time.

Fixed by extracting both to `tests/problems/invariants.py`, alongside
`fd_stencils.py` (the FD helpers were already shared the same way).
`test_canonical.py` now calls `assert_parity`/`assert_known_optimum` with
its own tolerances (`atol=1e-6`/`atol=1e-2` for parity, `atol=1e-8` for the
known-optimum check, all unchanged from issue #9) instead of its previous
inline loops. One visible effect: `test_canonical.py`'s Jacobian-parity and
Hessian-parity checks, previously two separate parametrised test functions,
are now one (`assert_parity` checks both per point), so its collected test
count for the seven canonical problems drops from 21 to 14. Diagnostic
granularity (which of Jacobian/Hessian failed) is still visible in the
assertion failure itself, just not in the test node ID; this is the same
shape the families' tests already used, which is what the review pointed
at as the version to keep.

The two Notes (the redundant second assertion in
`test_dimensionality_grows_with_n`; the deliberate `n` scoping for
dimensionality's FD-parity check) require no action per the review itself.

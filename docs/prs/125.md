# PR notes: Issue #125 - results_to_dataframe collapses declared-unsupported, timeout, and genuine crash into one ERROR status

## Summary

`results_to_dataframe` maps any caught exception (a study's own sweep
catching a backend's declared-unsupported rejection, e.g. `scaling.py`'s
`x_scale="jac"` probe against `trust-apl`) to a bare `status="ERROR"`
with every metric field blanked - indistinguishable, in every generated
table and plot, from a genuine crash or the harness's own subprocess
timeout.

Two changes close this gap:

1. `RunResult` gains a `message` field, threaded through from each
   backend's own already-computed termination explanation
   (`scipy`'s `OptimizeResult.message`; `trust-apl`'s harness response,
   which already includes a `message` key - `NULL` normally, an actual
   explanation for any `ErrorResult`, including the timeout case - but
   was previously discarded entirely). This alone distinguishes a
   timeout (`"harness did not complete within 60s"`) from any other
   harness-reported `ERROR` (e.g. `"Unknown problem_id: ..."`) without
   inventing a new status.
2. `results_to_dataframe` relabels a caught exception's status from
   `"ERROR"` to `"UNSUPPORTED"` and keeps `str(exception)` as its
   `message`, since every existing `except ValueError` block in this
   codebase's studies (`scaling.py`, `bounded.py`) catches exactly a
   backend's own declared-unsupported/rejected-input rejection - an
   expected, passing outcome of the sweep's own probe, never a genuine
   uncaught crash (which would propagate, not land here).

## RED cycle (test-first)

**Test surface**

- `tests/unit/test_result.py` (extended): `RunResult.message` defaults
  to `None`; round-trips through JSON serialisation.
- `tests/reporting/test_tables.py` (extended): a caught exception's row
  gets `status="UNSUPPORTED"` with its message preserved and every other
  metric field blank; a completed `RunResult`'s own `message` passes
  through unchanged.
- `tests/reporting/test_cross_study.py` (extended): `_pivot_by_backend`'s
  missing-backend guard treats an `"UNSUPPORTED"`-only backend the same
  as an `"ERROR"`-only one.
- `tests/backends/test_scipy_least_squares.py` /
  `test_scipy_minimize.py` (extended): `RunResult.message` carries
  scipy's own non-empty termination explanation for both the
  least-squares and minimize code paths.
- `tests/backends/test_apl_backend_timeout.py` (extended): the
  subprocess-timeout result's `message` mentions the timeout
  specifically, not just a bare `ERROR` status.
- `tests/backends/test_apl_difficulty_families.py` (extended): an
  unrecognised `problem_id`'s `ERROR` result's `message` contains
  `"Unknown problem_id"`, distinguishing it from the timeout case above
  by content alone.

**Confirmed RED**

Against the unmodified `RunResult`/`results_to_dataframe`/
`_pivot_by_backend`: every new/changed assertion fails as expected -
`AttributeError: 'RunResult' object has no attribute 'message'` and
`TypeError: RunResult.__init__() got an unexpected keyword argument
'message'` (the field doesn't exist yet), `AssertionError: assert
'ERROR' == 'UNSUPPORTED'` (the relabelling hasn't happened),
`KeyError: 'message'` (the table column doesn't exist yet), and
`Failed: DID NOT RAISE ValueError` (`_pivot_by_backend`'s filter still
only excludes `"ERROR"`, not `"UNSUPPORTED"`) - 9 failures in the
Python-only run, plus 2 more (timeout message, unknown-`problem_id`
message) confirmed separately against the APL backend (dyalogscript
available). All 79 pre-existing tests in the same files still pass
unchanged.

`ruff check` on every changed test file: clean.

## GREEN cycle (implementation)

`trust_bench/core/result.py`: `RunResult` gains `message: str | None =
None` (last field, so every existing keyword-argument construction site
stays valid unchanged).

`trust_bench/backends/apl_backend.py`: `solve` passes
`message=response.get("message")` - the harness's own JSON response
already includes this key for every result (`NULL` normally, an actual
explanation for any `ErrorResult`), previously discarded entirely.

`trust_bench/backends/scipy_backend.py`: `solve` passes
`message=result.message` - scipy's `OptimizeResult` already computes a
human-readable termination explanation for every call, converged or
not.

`trust_bench/reporting/tables.py`: `results_to_dataframe`'s
exception-branch status relabelled `"ERROR"` -> `"UNSUPPORTED"`, with
`str(exception)` kept as the row's `message` instead of being discarded.
`NON_RESULT_STATUSES = frozenset({"ERROR", "UNSUPPORTED"})` exported for
the two downstream consumers below.

`trust_bench/reporting/cross_study.py`: `_pivot_by_backend`'s
missing-backend guard filters on `NON_RESULT_STATUSES` instead of
`"ERROR"` alone - an `"UNSUPPORTED"`-only backend needs the same
fail-fast treatment as an `"ERROR"`-only one, not silent inclusion now
that the label has changed.

`trust_bench/cli.py`: `_check_backend_coverage`'s own equivalent guard
gets the identical fix - the same silent-empty-result gap would
otherwise reopen there under the new label.

`docs/methodology.md`: new "Declared-unsupported, timeout, and
genuine-crash reporting" section; the existing `dimensionality`/`BFGS`/
`n=1000` passage updated to reflect that the row's `message` now
actually distinguishes the timeout from any other harness-reported
`ERROR`, rather than the previous "no message distinguishing it from a
genuine crash" gap this issue closes.

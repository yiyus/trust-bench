# PR notes: Issue #117 - solve.dyalog maps tolerance onto tolr, which controls stall detection, not precision

## Summary

`solve.dyalog` maps a single `RunConfig.tolerance` value onto both `cfg.tolc` and `cfg.tolr` uniformly. `docs/methodology.md` documented this as deliberate, reasoning that both of `Newton.aplo`'s termination conditions (cost-below-threshold via `tolc`, relative-change-stalled via `tolr`) plausibly relate to precision. That reasoning is wrong: `tolr` bounds how small the relative change between successive accepted iterates can get before the solver treats it as "no progress" and gives up. It is a stall-detection threshold, not a precision selector - loosening it does not trade precision for speed, it only lowers the bar for declaring a stall.

Confirmed directly, independent of any submodule version: at `tolerance=0.1`, `trust-apl`'s `lm` reports `STALLED` after 1 iteration (`dist_to_opt≈1.95`); at `tolerance=0.01` - a *tighter*, not looser, request - it reports `CONVERGED` after 25 iterations with far better precision (`dist_to_opt≈0.078`). A looser tolerance is producing a worse, backwards result.

This bug is currently latent in `main`'s vendored `trust` version (its conservative `dinc:5` default rarely triggers the failure mode at the tolerances this project's existing test suite happens to exercise); a separate, still-open investigation into bumping the vendored submodule further (to a version with a more aggressive `dinc:100` default) hit this same mapping bug and initially misdiagnosed it as a `trust` regression. It is not - it is this harness bug, which any future engine change that escalates damping more aggressively is liable to re-expose more severely.

**Design notes**

*The fix is `cfg.tolr` should stay at its tight native default (`⎕CT`) always; only `cfg.tolc` should follow `RunConfig.tolerance`.* Verified directly (raw `Min` calls, `tolc` loosened progressively from `0.3` down to `0.01` with `tolr` left tight): the solver reaches a substantially better point at every tolerance than the current buggy mapping does, and the improvement is monotonic in the tolerance direction a user would expect.

*`docs/methodology.md`'s "Tolerance mapping" section needs correcting*, not just the code: its `trust-apl` subsection describes the (wrong) uniform mapping as intentional, and its "Why an equal numeric value is not an equal test" subsection's empirical claim ("`trust-apl` degrades smoothly as `tolerance` loosens") was measured under the buggy mapping and needs re-measuring under the fix.

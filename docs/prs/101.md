# PR notes: Issue #101 - cli.py discards large_residual's basin-of-attraction rate

## Summary

Same shape of bug as #98 (`baseline`), in a different study: `_write_large_residual` calls `large_residual.backend_results()`, which returns `(results, rates)`, but discards `rates` entirely (`results, _ = large_residual.backend_results(...)`), so the basin-of-attraction rate this study already computes never reaches the report output.

Mirrors #98's fix exactly: write a second table, `large_residual_basin_rates.csv`, alongside the existing `large_residual.csv`.

## RED cycle (test-first)

**Test surface**

- `tests/integration/test_cli.py` (extended): `"large_residual_basin_rates.csv"` added to `_EXPECTED_TABLES`, covered by the existing full-pipeline smoke test. New dedicated test
  `test_run_report_writes_large_residuals_basin_of_attraction_rate_table`
  mirrors #98's `test_run_report_writes_baselines_basin_of_attraction_rate_table` exactly: runs the `large_residual` study alone and asserts the basin-rates table exists with the right columns (`rho`/`backend`/`basin_rate` - `rho`, not `problem_id`, since `large_residual` is keyed by `rho` rather than a fixed problem set), one row per swept `rho`, every rate in `[0, 1]`.

**Confirmed RED**

`test_run_report_writes_large_residuals_basin_of_attraction_rate_table` fails with `FileNotFoundError: large_residual_basin_rates.csv` - genuinely RED (the file is never written), not a test-file bug.

`ruff check tests/integration/test_cli.py`: clean.

**Design notes**

*A second table, not a merged one*, exactly matching #98's own precedent: `large_residual.csv`'s rows are keyed by `(rho, method, backend)` (one row per swept problem run), while the basin-rate table's rows are keyed by `(rho, backend)` alone (one row per `basin_rate` call across every start) - different grain, so a separate file avoids either duplicating rows or losing the per-method detail.

# PR notes: Issue #82 - bounded study never exercises trust-apl's bounded lm method

## Summary

`bounded.py`'s `sweep()` applies a fixed `METHODS = ["trf", "dogbox"]`
list to every backend, so `trust-apl`'s `lm` (the one method where it
actually declares `bounds=True`) is never called, and every `trust-apl`
row reads `ERROR`. The fixed list is also scipy-specific in a second
way: scipy itself declares `bounds=True` for `L-BFGS-B` and
`trust-constr` too, neither of which the study exercises either.

Confirmed directly (`SciPyBackend`, `quadratic.PROBLEM`) before writing
any test: `scipy`'s minimize-family bounds-capable methods
(`L-BFGS-B`, `trust-constr`) expect a different bounds representation
than its least-squares-family methods (`trf`/`dogbox`, and `trust-apl`'s
`lm`) - a list of per-dimension `(low, high)` pairs versus a `(lower,
upper)` array pair - already handled by this exact split in
`reporting/capability_matrix.py`'s `_LEAST_SQUARES_BOUNDS_PROBE`/
`_MINIMIZE_BOUNDS_PROBE`. Also confirmed: an infeasible start is not
universally rejected. scipy's `trf`/`dogbox` validate `x0` against
`bounds` and raise `ValueError`; scipy's `L-BFGS-B`/`trust-constr` and
`trust-apl`'s `lm` (Coleman-Li affine scaling) project it into the box
and converge regardless. Both are genuine, correct behaviours of their
respective methods, not a bug in either family.

## RED cycle (test-first)

**Test surface**

- `tests/studies/test_bounded.py` (rewritten): method selection is now
  derived per-backend from `capabilities().methods[...].bounds`, not a
  fixed list. `test_sweep_covers_every_scenario_and_every_bounds_
  capable_method` and `test_sweep_exercises_more_than_the_least_
  squares_family` cover this directly.
  `test_infeasible_start_is_rejected_by_least_squares_methods_and_
  projected_by_others` replaces the old assumption that every method
  raises for an infeasible start, splitting on the behaviour actually
  observed above.
  `test_sweep_skips_a_method_a_backend_does_not_declare_bounds_for`
  mirrors the skip-guard test pattern already used elsewhere
  (`ill_conditioning.py`/`dimensionality.py`).
  `test_inactive_bounds_converge_to_the_unconstrained_optimum` and
  `test_active_at_boundary_bounds_converge_exactly_to_the_boundary` are
  kept, generalised to run over every outcome rather than a fixed
  method list; tolerance loosened from `1e-6` to `1e-3` to accommodate
  `trust-constr`'s looser default convergence tolerance on this
  problem, matching the `atol=1e-3` already established for exactly
  this case in `test_scipy_minimize.py`.
- `tests/studies/test_bounded_apl.py` (new, slow, `dyalogscript`-gated):
  `test_trust_apls_bounded_lm_is_exercised_by_every_scenario_instead_
  of_reading_error` runs `sweep(backends=[SciPyBackend(), APLBackend()])`
  and checks `trust-apl`'s `lm` produces a real, `CONVERGED` row for
  every scenario - the acceptance criterion this issue exists for.

**Confirmed RED**

`tests/studies/test_bounded.py`: `test_sweep_covers_every_scenario_and_
every_bounds_capable_method` and `test_sweep_exercises_more_than_the_

`tests/studies/test_bounded_apl.py`: `KeyError: ('inactive', 'lm',

the current `sweep()` at all.

`make test`: 258 passed, 2 failed (the two structural tests above,

new/changed files.

**Design notes**

Method-to-bounds-format mapping is name-based
(`_LEAST_SQUARES_METHODS = frozenset({"lm", "trf", "dogbox"})`),
matching the existing precedent in `capability_matrix.py` rather than
inventing a new abstraction; no backend currently overloads one of
these names to mean something else.

Scope: this issue's acceptance criteria ask for every `bounds=True`
method, not only least-squares-family ones, so `L-BFGS-B`/
`trust-constr` are included rather than carved out to keep the change
minimal. The resulting behavioural split (some methods reject an
infeasible start, others project it) is treated as a real finding to
record precisely, not an inconvenience to average away.

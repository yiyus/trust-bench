# PR notes: Issue #43 - CLI compare command

Part of #41 (Epic: Phase 10, Longitudinal tooling).

## Acceptance criteria

- `trust-bench compare <baseline> <candidate>` reports both classifications with full provenance

Scope clarification (issue #43 comment): "reports" means the full generated
HTML report, with the relevant graphs, not text-only CLI output - the
comparison folds into the same `report.html` `trust-bench report --html`
already produces, alongside whatever study artefacts are present.

## Cycle 1: RED

Three surfaces, three test files, all driving out code that does not exist
yet:

`tests/reporting/test_compare.py` (extends the #42 file) drives out two new
functions in `trust_bench/reporting/compare.py`:

- `compare_with_provenance(baseline, candidate, ...)`: `compare()`'s own
  classification, plus every `EnvProvenance` field for both sides, prefixed
  `baseline_`/`candidate_` - the "full provenance" the acceptance criterion
  asks for. `compare()` itself (#42) is unchanged; this is additive.
- `classification_counts(compared, group_column="backend")`: counts of each
  classification per backend, feeding the report's graph.

`tests/reporting/test_html_report.py` (extends the #42/#135 file) drives out
a "Longitudinal comparison" section in `build_html_report`: rendered when
`compare.csv` (and, where present, `compare.png`) exist in the output
directory, carrying the full-provenance table and a graph, omitted (like
every other artefact) when absent.

`tests/integration/test_cli.py` (extends the existing file) drives out:

- A `compare` subparser: positional `baseline`/`candidate` paths,
  `--output-dir` (default `reports`), `--html`.
- `run_compare(baseline_path, candidate_path, output_dir, html=False)`:
  loads both `results/*.jsonl` files (`storage.load`), writes
  `compare.csv`/`compare.png`, and - only with `--html` - (re)writes
  `report.html` via the existing `build_html_report`/`save_html_report`,
  the same path `trust-bench report --html` uses, so a comparison run folds
  into whatever report already exists in that directory rather than
  building a second one.
- `main`'s `compare` branch wiring the above through the real CLI entry
  point end to end.

RED confirmed:

- `tests/reporting/test_compare.py` fails to collect:
  `ImportError: cannot import name 'classification_counts' from

- `tests/integration/test_cli.py` fails to collect:

No other test files were run.

## Cycle 1: GREEN

- `trust_bench/reporting/compare.py`: adds `compare_with_provenance` (calls
  `compare()`, then merges in every `EnvProvenance` field for both sides,
  prefixed `baseline_`/`candidate_`) and `classification_counts` (row
  counts per backend/classification, for the report's graph). `compare()`
  and `drift_summary()` (#42) are unchanged.
- `trust_bench/reporting/html_report.py`: adds a `TITLES`/`CAPTIONS` entry
  for `"compare"` and `_comparison_section(output_dir)`, following the same
  exists-or-omit convention as every other study section. `build_html_report`
  renders it, when present, right after the hero and before Tier 1 - a
  regression or drift is the first thing a reader should see.
- `trust_bench/cli.py`: adds a `compare` subparser (positional
  `baseline`/`candidate`, `--output-dir` default `reports`, `--html`) and
  `run_compare`, which loads both `results/*.jsonl` files, writes
  `compare.csv`/`compare.png` (via `plot_metric_by_category`, reused rather
  than adding a new plotting function), and - only with `--html` - folds the
  result into `report.html` via the same `build_html_report`/
  `save_html_report` path `trust-bench report --html` already uses.

`tests/reporting/test_compare.py`, `tests/reporting/test_html_report.py`,
`tests/integration/test_cli.py` pass (77/77 across the three files); `ruff
check` clean on every touched file.

## Review: approved with minor changes

`docs/reviews/43.md` flagged two minor findings, both applied before
committing:

- `compare.py`'s `compare_with_provenance` docstring named the issue number
  directly; reworded to state the invariant without it.
- `compare_with_provenance` joined baseline/candidate provenance back onto
  the classification by row position (`pd.concat(..., axis=1)`) rather than
  by `key_columns`; changed to an explicit `merge` on `key_columns`, so
  correctness no longer rests on `compare()`'s internal merge happening to
  preserve row order.
- `html_report.py`'s `_comparison_section` duplicated `_study_section`
  exactly for this input; removed, `build_html_report` now calls
  `_study_section(output_dir, "compare")` directly.

Full suite: 494 passed after the fixes.

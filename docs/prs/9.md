# PR notes: Issue #9 - Canonical problem set: Rosenbrock, Beale, Powell, Helical, ExpDec, Quadratic, Linear

## Summary

Adds `trust_bench.problems`: seven canonical `Problem` instances (Section 3's
`rosenbrock.py`, `beale.py`, `powell.py`, `helical.py`, `expdec.py`,
`quadratic.py`, `linear.py`), each with analytic residual, Jacobian, and
Hessian, a known zero-cost optimum, and probe points. Part of Epic #7
(Phase 2), see `docs/plans/trust-bench.md`, Section 5, Section 9 (item 1),
Section 10 (Phase 2).

**Design notes**

All seven problems, definitions, and correctness were worked out and
numerically verified (analytic vs. finite-difference, at every intended
probe point, plus the known-optimum invariants) in a throwaway script before
writing this test surface, since getting the calculus right by inspection
alone is not reliable enough to trust. `sympy` was used as a derivation aid
(not a dependency: nothing in `trust_bench` imports it). The verification
script is not part of this PR; it is scratch work, not a deliverable.

- **All seven as `kind="residuals"`.** Rosenbrock, Beale, Powell, and
  Helical are Moré-Garbow-Hillstrom least-squares test problems by
  construction. `ExpDec` (below) is naturally residuals-form too.
  `Quadratic`/`Linear` are this project's own simple sanity checks (Section
  9 item 1: "the floor everything must clear"); making them residuals-form
  too keeps the parity/known-optimum test harness uniform across all seven,
  with no need to also exercise the `"scalar"` kind's different
  residual/Jacobian/Hessian interpretation in this issue.
- **`ExpDec`** is the exponential-decay model from `prototype/
  large_residual.py` (`phi(t;A,k) = A*exp(k*t)`, `M=40` points, `A0=2.0`,
  `K0=1.3`) at `rho=0`: the irreducible-residual parameter that family
  sweeps in issue #10, giving a clean, consistent (zero-residual) canonical
  baseline here, distinct from the parametrised difficulty family that
  reuses the same geometry.
- **`Quadratic`** (`r(x)=[x1, 2x2]`) and **`Linear`** (`r(x)=Ax-b` for a
  consistent, full-column-rank 3x2 system with `x*=(1,2)`) are this
  project's own definitions: the plan names them but does not define them.
  Both are deliberately the simplest possible residuals-form problems.
- **Powell's Jacobian is singular at `x*`** (`J(x*)` has rank 2 of 4):
  confirmed numerically, matching `Problem.tags`'s own example
  (`"singular-jacobian"`, Section 4.1). Not added as a tag in this PR:
  neither acceptance criterion exercises tags, and tag *usage* (which
  studies filter on which tags) is not yet decided by any issue. Recorded
  here so the tag is not forgotten when that decision is made.
- **`far` start for Rosenbrock** (Section 9 item 1 mentions it by name):
  `(-5.0, -5.0)`. The plan does not pin exact coordinates; chosen only to
  be clearly outside the basin the `standard` start already covers. Not
  otherwise exercised by this issue's tests (no acceptance criterion
  requires solving from it, only Phase 6's baseline study will).
- **No aggregating registry.** Issue #8's `Registry` is not populated here:
  no acceptance criterion in this issue needs a shared catalog. Each
  problem module exports its `Problem` as a module-level `PROBLEM`, callers
  import directly (as the test file does); a caller that needs to enumerate
  "all canonical problems" builds the `Registry()` and registers into it
  when it first needs to, per issue #8's own design notes.
- **Finite-difference tolerances**: `1e-6` for the Jacobian check (5-point
  central difference, empirically accurate to ~1e-9 to 1e-11 across all
  seven problems and every probe point) and `1e-2` for the Hessian check
  (a plain central-difference second-derivative stencil, empirically
  accurate to ~1e-6 to ~2e-3, worst case Helical's curved `arctan`/`sqrt`
  terms). Both margins are comfortably tighter than what a real
  transcription error (wrong sign, wrong coefficient, swapped index) would
  produce, and comfortably looser than the FD scheme's own truncation
  error, verified empirically per problem before picking the numbers rather
  than guessed.

## Review response

`docs/reviews/9.md`'s Major finding: the three parametrised tests each
loop over `problem.probe_points`/`problem.optima` with the assertion inside
the loop body, so a problem module that ships with an empty list (a
plausible slip when hand-transcribing seven problems' worth of `Problem`
construction) would pass vacuously rather than failing.

Fixed by asserting non-emptiness (`assert problem.probe_points, ...` /
`assert problem.optima, ...`) before each loop, so an empty list now fails
explicitly with a message naming which problem is missing what, rather than
passing silently.

Also addressed the Minor finding: renamed `tests/problems/numerics.py` to
`tests/problems/fd_stencils.py`, a more specific name less likely to
collide with an unrelated module of the same name in a different test
directory that also lacks an `__init__.py`.

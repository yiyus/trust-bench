# PR notes: Issue #20 - Runner: (problem, backend, method, start, config) -> RunResult (runner.py)

## Summary

Adds `trust_bench.core.runner.run`, the single entry point Section 3's
repository layout and Epic #18's description name as tying a problem,
backend, method, start and config together into a `RunResult`. Every
acceptance criterion (`dist_to_opt`/`cost_gap`/`grad_norm_final`
populated, trace availability) is already satisfied by `Backend.solve`'s
existing contract (Phase 4, issues #15-17); this issue's tests exercise
that contract end-to-end through the named entry point future callers
(CLI, studies, reporting) will use, rather than reaching into a specific
backend's `solve` directly. Part of Epic #18 (Phase 5).

**Design notes**

Uses Rosenbrock (`trust_bench.problems.rosenbrock`), not the trivial
quadratic used throughout the Phase-4 backend tests, per the plan's own
Phase 5 test description ("end-to-end on Rosenbrock"); confirmed directly
that `SciPyBackend.solve("lm", "standard", max_iter=200)` converges to
`dist_to_opt == cost_gap == grad_norm_final == 0.0` exactly for this
problem, so the test asserts `< 1e-6` rather than exact equality, to
avoid coupling to floating-point exactness across environments.

No production backend captures a per-iterate trace yet (`SciPyBackend`
always returns `trace=None`), so the "captured when available" branch
has nothing real to exercise. `_StubBackendWithTrace`, a minimal
`Backend` test double local to this file, returns a hardcoded
`RunResult` with a populated `trace` purely to prove `run()` passes it
through unmodified; it is not a solver and does no computation,
deliberately kept separate from `tests/backends/toy_backend.py` (which
exists to exercise the Phase-4 contract, a different concern).

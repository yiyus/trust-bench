# PR notes: Issue #93 - bounded and capability_matrix send corrupted bounds to trust-apl's BFGS/trust-exact methods

## Summary

`bounded.py`'s `_bounds_for` and `capability_matrix.py`'s
`_measure_bounds` both pick between two bounds representations by
checking whether a method's name is one of scipy's own least-squares
names (`lm`/`trf`/`dogbox`, expecting a `(lower_array, upper_array)`
pair) or not (falling back to scipy's `minimize`-family per-dimension
tuple list). `APLBackend.solve` unconditionally does `lower, upper =
config.bounds` for every one of its own methods. `BFGS`/`trust-exact`
were added to `APLBackend` in #84, deliberately named to mirror
scipy's own vocabulary for the closest conceptual equivalent - but they
still use the least-squares-style format internally. Both call sites'
name-based routing misclassifies them and sends scipy's per-dimension
tuple list instead, which `APLBackend.solve` then unpacks as
`lower=(-10.0, 10.0)`, `upper=(-10.0, 10.0)` for the "inactive"
scenario - collapsing the feasible region to a single point.

Confirmed directly, reproducing the issue's own numbers exactly:
`APLBackend().solve(quadratic.PROBLEM, "BFGS", "scenario",
RunConfig(bounds=[(-10.0, 10.0), (-10.0, 10.0)]))` (the exact value
`bounded.py` passes for "inactive") returns `cost_final â‰ˆ 250.0`,
`x_final â‰ˆ [-10, 10]`.

**Design notes**

The fix is a routing fix, not a new capability: `_bounds_for`/
`_measure_bounds` both need to know which *backend* they're probing,
not just the method name, since the same method name (`"BFGS"`) means
a different bounds format depending on which backend declares it.
scipy is the one backend with a genuine per-method split (it wraps two
different underlying scipy call paths, `least_squares` and
`minimize`); trust-apl uses the least-squares-style format
universally, for every one of its own methods. The routing predicate
becomes: use the minimize-style format only when the backend actually
is scipy and the method isn't one of its own least-squares names;
default to the least-squares-style format otherwise - treating it as
the common case rather than special-casing every non-scipy backend
individually as more are added.

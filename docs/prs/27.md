# PR notes: Issue #27 - Study: variable scaling

## Summary

Adds the variable-scaling study (`docs/plans/trust-bench.md`, Section 9
item 6): sweeps `trust_bench.problems.families.scaling.make(s)`'s
parameter scale disparity, solved with and without SciPy's `x_scale`.
Part of Epic #21 (Phase 6).

**Scope deferral, same pattern as issues #25/#26/#27:** the acceptance
criterion also names `trust`'s adaptive diagonal scaling, but that
backend does not exist yet (Epic #34, Phase 8, not started). This issue
implements the SciPy `x_scale` side now; the `trust` comparison is added
once Phase 8 lands.

**Implementation scope explicitly includes `trust_bench/core/config.py`
and `trust_bench/backends/scipy_backend.py`:** `RunConfig` has no
`x_scale` field yet, so `least_squares`'s own `x_scale` parameter
(`None` or `'jac'`) cannot currently be set through the existing
backend contract. Adding it is the same shape of change issue #19 made
for `tolerance`: an `Any`-typed, `None`-default field forwarded straight
to `least_squares(x_scale=...)` when set, with no effect on `minimize`
(which has no `x_scale` parameter at all).

**Design notes**

`SCALES = [1.0, 10.0, 1e2, 1e4, 1e6, 1e8]`, `METHODS = ["lm", "trf",
"dogbox"]`, `X_SCALES = [None, "jac"]`. Verified directly against raw
`scipy.optimize.least_squares` (bypassing this repo's backend layer,
since `x_scale` wiring does not exist yet) across the full sweep before
writing any assertions.

**A genuine, counter-intuitive finding, not the result Section 9 item 6
seems to assume:** `scaling.make(s)`'s Hessian is exactly diagonal
(`r(x) = [x1-a, s*(x2-b)]`), so an *unscaled* Gauss-Newton/Newton step
already accounts for the scale disparity exactly — Newton's method is
affine-invariant for a problem this simple, and `x_scale=None` converges
in a small, constant number of function evaluations across the entire
sweep (`nfev` stays at 2 for `lm`, 4 for `trf`, 3-4 for `dogbox`,
regardless of `s`). Layering SciPy's adaptive `x_scale='jac'` on top of
an already-perfectly-conditioned Newton step costs *more* function
evaluations as `s` grows (`lm`: 2 -> 24; `trf`: 4 -> 22; `dogbox`: 3 ->
22, from `s=1` to `s=1e6`), not fewer. This does not mean adaptive
scaling is not useful in general (methods without curvature information,
e.g. gradient descent, are exactly where it earns its keep); it means
this particular problem family is the wrong vehicle for demonstrating a
*benefit*, and the honest, verified result is recorded rather than
reshaped to fit the expected narrative.

**A second, more severe finding at the extreme end (`s=1e8`):**
`trf`/`dogbox` with `x_scale='jac'` report `status=2`
(`xtol`-termination, `success=True`, mapped to `RunStatus.CONVERGED` by
this backend) while sitting at `dist_to_opt ≈ 3.6`, nowhere near the true
optimum `(3, -2)` — a silent false convergence, not a raised error or a
`FAILED` status. `lm` with `x_scale='jac'` does not exhibit this at the
same scale (`dist_to_opt == 0.0`); the failure is specific to
`trf`/`dogbox`'s scaled trust-region step at this level of disparity.
`test_jac_scaling_can_converge_to_the_wrong_answer_at_extreme_scale_
disparity` asserts on `dist_to_opt`, not `status`, precisely because
`status` alone would hide this.

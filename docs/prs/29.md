# PR notes: Issue #29 - Study: derivative source

## Summary

Adds the derivative-source study (`docs/plans/trust-bench.md`, Section 9
item 8): analytic vs finite-difference Jacobians compared on evaluation
count and precision, for the canonical problem set across every
`least_squares` method. Part of Epic #21 (Phase 6), the last child.

**Implementation scope explicitly includes `trust_bench/backends/
scipy_backend.py`:** while calibrating this study directly against
scipy, found that `RunResult.n_feval` (built from `result.nfev`)
massively undercounts the true number of residual evaluations whenever
`derivative_mode="finite-difference"` — `nfev` only counts calls the
outer algorithm makes itself, not the extra calls scipy's own
finite-difference Jacobian estimation makes internally. Verified
directly, `lm`/`trf`/`dogbox`, across the whole canonical set: `nfev`
under-reports by 2x to 9x depending on the problem's dimension (e.g.
`quadratic`: 2 reported vs. 7 real; `powell`: 37 reported vs. 336 real,
for `lm`). Since this issue's entire acceptance criterion is a
comparison of evaluation *counts*, reporting a systematically wrong
count would make the study's central claim false. This is squarely this
issue's problem to fix, not a pre-existing issue to merely disclose.

**Design notes**

Uses `trust_bench.problems.CANONICAL_PROBLEMS` (the same seven problems
issue #22's baseline study centralised) and `METHODS = ["lm", "trf",
"dogbox"]`: Section 9 item 8 says "Jacobian" specifically, which is
`least_squares` vocabulary (`minimize`'s gradient/Hessian are not
usually called a Jacobian), so `minimize` methods are out of scope here.

Verified directly, once `n_feval` was corrected in the scratch
calculation used to design this study (not yet in the implementation):
every one of the seven canonical problems, every method, needs strictly
more evaluations with a finite-difference Jacobian than with the
analytic one, and both modes still converge to a small `dist_to_opt`
(finite-difference is very slightly less precise, e.g. `powell`:
`2.6e-9` vs. `3.6e-11` for `lm`, still comfortably under the `1e-6`
tolerance used here).

**A related inaccuracy found, disclosed, and deliberately not fixed
here:** `minimize`-family methods (`BFGS`, `L-BFGS-B`, etc.) have a
different `n_feval` undercount in *analytic* mode specifically: this
backend's `_objective`/`_gradient` wrappers each independently call
`problem.residual`, so every scipy "function evaluation" (which calls
both `fun` and `jac`) triggers two residual calls, but scipy's reported
`nfev` only counts calls to `fun`. Verified directly (`BFGS`: 10 real
calls vs. 5 reported). This issue's acceptance criterion is about
`least_squares`/Jacobians specifically, not `minimize`; fixing this
adjacent, differently-shaped inaccuracy is out of scope here and left
for whichever future issue needs `minimize`'s eval counts to be
accurate.

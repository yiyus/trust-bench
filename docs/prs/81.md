# PR notes: Issue #81 - Two studies crash against a backend with partial method coverage

## Summary

`derivative_source.sweep` and `scaling.sweep` gain the same `if method
not in backend.capabilities().methods: continue` skip-guard already used
by `ill_conditioning.sweep`, so selecting `trust-apl` alongside `scipy`
no longer crashes `trust-bench report` on the first method it doesn't
support (`trf`/`dogbox`). Both studies already share a real method with
`trust-apl` (`"lm"`), so the fix produces a genuine, non-empty
comparison rather than papering over a gap.

Narrowed from the original four-study scope after discussion (see the
issue's own comments): `dimensionality.py`'s real fix is #84 (`trust`'s
BFGS/Newton trust-region methods, so it reports how far `trust` actually
scales rather than nothing); `robust_loss.py`'s real fix is picking a
backend-appropriate method rather than skipping `trust-apl` outright,
since `trust` already implements most of the same robust losses and the
only blocker is `_METHOD` being hardcoded to scipy's `"trf"`. Neither is
a defensive skip-guard problem, so neither belongs in this issue.

## RED cycle (test-first)

**Test surface**

- `tests/studies/test_derivative_source.py` (extended):
  `test_sweep_skips_a_method_a_backend_does_not_support`, using a new
  `_LMOnlyBackend` stub (wraps `SciPyBackend`, declares only `"lm"`).
- `tests/studies/test_scaling.py` (extended): same test name, same
  `_LMOnlyBackend` stub, scoped to `scaling.sweep`.

Both stubs mirror `test_ill_conditioning.py`'s existing
`_BFGSOnlyBackend` exactly: `solve()` delegates to a real `SciPyBackend`
without validating the method itself, so the test genuinely exercises
whether `sweep()` consults `capabilities()` before calling `solve`, not
whether some inner validation happens to catch it.

**Coverage table**

| Acceptance criterion | Test(s) |
|---|---|
| `derivative_source.py`/`scaling.py` skip an unsupported `(method, backend)` pair | the two `test_sweep_skips_a_method_a_backend_does_not_support` tests |
| follows `ill_conditioning.py`'s existing pattern | same stub-backend technique as `test_ill_conditioning.py`'s own skip test |
| `trust-bench report --backends scipy trust-apl` completes without a traceback for both studies | covered indirectly: once `sweep()` itself never calls `solve()` with an unsupported method, there is nothing left to raise |

**Confirmed RED**

Both new tests fail on real assertion mismatches, not crashes: `results`
contains extra `("rosenbrock", "trf"/"dogbox", ..., "lm-only")` /
`(1.0, "trf"/"dogbox", ..., "lm-only")` entries, because `_LMOnlyBackend.
solve` doesn't validate its own method argument (delegating straight to
a real `SciPyBackend`), so today's unfiltered `sweep()` happily calls it
for every method regardless of what `capabilities()` declares. `make
test`: 227 passed, 2 failed (the two new tests), 28 deselected. `make
lint`: clean.

## GREEN cycle (implementation)

**Files changed**

- `trust_bench/studies/derivative_source.py`: adds the
  `if method not in supported: continue` skip-guard, `ill_conditioning.
  py`'s exact pattern.
- `trust_bench/studies/scaling.py`: same skip-guard, plus a second fix
  found while verifying the real acceptance criterion (see below).

**Self-found gap, beyond the approved RED tests: `scaling` still crashed
against real `APLBackend` after the method-only skip**

Running the actual acceptance criterion (`trust-bench report --backends
scipy trust-apl`, not just the stub-backend tests) against real
`APLBackend` showed `scaling.sweep` still raising `ValueError: lm does
not support x_scale`. The method-level skip only excludes `trf`/
`dogbox`; `"lm"` *is* one of `trust-apl`'s supported methods, but
`x_scale` is a config field within that method the backend rejects
outright, and `MethodCapabilities` has no declarative field for
`x_scale` to check in advance the way it does for `bounds`/
`derivative_mode`. The approved RED tests didn't catch this because the
stub backend they use (`_LMOnlyBackend`, delegating to a real
`SciPyBackend`) doesn't reject `x_scale` either — real `SciPyBackend`
doesn't need to.

Added `_LMWithoutXScaleBackend` (mirrors `_LMOnlyBackend`, additionally
raises for any `x_scale`, matching `APLBackend`'s real behaviour
exactly) and
`test_sweep_records_a_raised_value_error_as_an_outcome_instead_of_crashing`,
confirmed RED (uncaught `ValueError`) before fixing. Fixed by wrapping
the `run(...)` call in `scaling.sweep` in `try/except ValueError`,
recording the caught exception as the outcome — `bounded.py`'s own
pattern, the right fit here since there is no declarative capability to
skip on ahead of time. `results_to_dataframe` already renders any
`BaseException` value as a visible `"ERROR"` row (since #37), so no
reporting-side change was needed.

This mirrors #33's own precedent (a coverage gap found and fixed mid-
GREEN, documented here rather than folded silently into the diff).

**Verification**

- `make test`: 230 passed, 28 deselected (3 new: the two RED-reviewed
  skip-guard tests plus the one self-found `x_scale` test).
- `make lint`: clean.
- Verified directly against real `APLBackend`, not just the stub tests:
  `derivative_source.sweep` on `rosenbrock` now returns two genuine
  `CONVERGED` `trust-apl` results (analytic and finite-difference).
  `scaling.sweep` no longer raises; its `trust-apl` rows are `ERROR`
  (the `scaling(s=...)` problem family isn't ported to APL yet — #83's
  known gap, not a crash) for `x_scale=None` and a caught `ValueError`
  for `x_scale="jac"`.
- Ran the actual acceptance criterion end-to-end:
  `run_report(only=["scaling", "derivative_source"], backends=["scipy",
  "trust-apl"])` completes without a traceback, and `trust-apl` appears
  in both output tables (satisfying #77's `_check_backend_coverage`,
  which only requires a backend's name to appear at least once, not
  that every row succeeds).

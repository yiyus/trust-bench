# PR notes: Issue #36 - backends_ext/apl/ harness

## Summary

Adds `backends_ext/apl/`: a small shell-invoked Dyalog harness that
loads `trust` (vendored as a pinned git submodule at
`backends_ext/apl/trust`), runs a problem defined natively in APL, and
writes a RunResult-shaped JSON payload to a file. Part of Epic #34
(Phase 8), first child after the environment spike (#35).

**Design notes**

*Harness input/output is a deliberate subset of `RunResult`, not the
full record.* The harness never sees a Python `Problem` object, so it
cannot compute `dist_to_opt`/`cost_gap` (need `problem.optima`) and has
no way to know `backend`/`provenance`/`harness_git_sha`/`timestamp` (all
adapter-side facts about the calling process, not the solve itself).
Input is `{problem_id, x0, max_iter?, tolerance?, loss?}`; output is
`{problem_id, status, message, x_final, cost_final, n_iter, n_feval,
n_jeval, n_heval, grad_norm_final}`. Issue #37's `apl_backend.py`
receives the full `Problem` (per the `Backend` ABC contract, same as
`SciPyBackend`) and is the natural place to compute the optimum-relative
fields and assemble the rest, exactly as `SciPyBackend.solve` already
does for `dist_to_opt`/`cost_gap` today.

*`trust` is vendored as a pinned git submodule* at
`backends_ext/apl/trust`, commit `7bbb08181bba8ec6d83dd005897f1fb82626cff0`
(v0.2.0) — the same commit exercised in #35's spike. Pinning makes
`backend_version` (Section 4.3's `EnvProvenance`) reproducible via the
submodule's own git sha, and avoids "wherever it happens to be cloned"
ambiguity for anyone checking this repo out fresh.

*One problem, not a dispatch table.* `backends_ext/apl/problems/
rosenbrock.dyalog` is the only native problem implementation; `Solve`
checks `problem_id` against the literal string `'rosenbrock'` rather
than through a lookup indirection. Dyalog dfns cannot hold an unapplied
function reference as an ordinary value — `{cond: SomeFunction ⋄ ⍬}`
is a `SYNTAX ERROR`, confirmed while building this — so a real
multi-problem dispatch table needs a different mechanism (most likely
name-based `⍎` dispatch) that issue #37 should design deliberately
against the full problem set, rather than something built prematurely
here against a set of one.

*The process exit code is a second, coarser signal alongside the JSON
`status` field: 0 whenever the harness produced a result at all
(`CONVERGED`, `MAX_ITER`, `FAILED`), 1 only for `ERROR` (malformed
input, unknown `problem_id`, or an uncaught APL condition).* The eventual
Python adapter (#37) still needs to read the output file and branch on
`status` regardless, since `MAX_ITER`/`FAILED` are legitimate solve
outcomes rather than harness failures, but a nonzero exit on `ERROR`
gives a cheap, standard "did the subprocess itself work" check without
opening the file.

*`n_feval` counts every residual/Jacobian evaluation, including the
one used to compute the final gradient.* `trust`'s `Min` does not
return the final residual/Jacobian pair, so getting `grad_norm_final`
needs one more evaluation at the accepted `p` after `Min` returns. That
call goes through the same counting wrapper as every other evaluation
during the solve, rather than calling the raw problem function directly
and leaving it uncounted — the same undercount class of bug fixed for
`SciPyBackend` in #29. (A dfn's plain `←` assignment to a name that
already exists in an enclosing scope creates a fresh local rather than
mutating the outer binding, confirmed while building this; only
compound assignment such as `+←` reaches through, which is why the
counter is incremented that way and the final residual/Jacobian pair is
read from the wrapper's own return value rather than a captured
variable.)

*Session initialisation via `dyalogscript`, not raw `dyalog -script`.*
Per #35's spike findings, `dyalog -script` never loads `⎕SE.Link`
regardless of `DYALOG_INITSESSION` set as a shell environment variable.
`dyalogscript` (the wrapper Dyalog ships at `scriptbin/dyalogscript`)
accepts `DYALOG_INITSESSION=1` as a leading positional argument instead,
and forwards it to `dyalog` as a native command-line parameter rather
than an environment variable — that combination does load `⎕SE.Link`
inside otherwise-clean `-script` execution (no banner, no line echo,
proper multi-line parsing). `run_harness.sh` concatenates the driver's
source files into one temporary script and runs it via `dyalogscript
DYALOG_INITSESSION=1 <script> <input> <output>`; the trailing arguments
are read inside the script via `2 ⎕NQ '.' 'GetCommandLineArgs'`.

Two follow-on problems surfaced and were resolved before settling on
this design:

- `⎕SE.Link.Create` sets up a filesystem watcher on the linked
  directory; under `-script`, that watcher intermittently kept the
  process alive past `⎕OFF`, observed as the harness hanging even
  though its output file had already been written correctly.
  `⎕SE.Link.Import` (a one-shot fix into a namespace, explicitly
  `opts.watch←'none'` in Dyalog's own `Import.aplf`) does not set up a
  watcher and removed the hang entirely across 38 repeated runs of
  every scenario below.
- `⎕FIX`, on both a `file://` URI and an inline vector of lines,
  reliably raised `DOMAIN ERROR: There were errors processing the
  script` in this environment even for a two-line textbook example,
  unrelated to anything in this harness's own code. `Solve` and `Run`
  (the two functions needing real `:Trap`/`:If` control structures,
  which are tradfn-only and unavailable inside dfns' `{}`) are defined
  with the classic `∇`-editor header form directly in their source
  files instead; `-script` parses that reliably as part of the whole
  file, unlike the piped-stdin session form tried earlier.

## GREEN cycle (implementation)

**Files changed**

- `backends_ext/apl/trust` (new): git submodule, pinned to
  `7bbb08181bba8ec6d83dd005897f1fb82626cff0` (v0.2.0).
- `backends_ext/apl/problems/rosenbrock.dyalog`, `null.dyalog`,
  `error_result.dyalog` (new): the Rosenbrock residual/Jacobian, a
  reusable JSON-null sentinel, and the `ErrorResult` constructor.
- `backends_ext/apl/solve.dyalog`, `run.dyalog` (new): `Solve` (problem
  dispatch, config defaulting, the `Min` call, status inference) and
  `Run` (reads the request file, traps any failure, writes the result
  file, sets the process exit code), both as `∇`-editor tradfns.
- `backends_ext/apl/run_harness.sh` (new): concatenates the source
  files above into one temporary script and runs it via `dyalogscript
  DYALOG_INITSESSION=1 <script> <input> <output>`.

  slow tests).
- `make lint`: clean.
- The 5 new tests, run directly against a real `dyalogscript`: 5/5

  ruled out along the way).

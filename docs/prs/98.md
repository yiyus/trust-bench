# PR notes: Issue #98 - cli.py never surfaces baseline's basin-of-attraction rate

## Summary

`baseline.py` already implements `basin_rates()`: per `docs/plans/
trust-bench.md` Section 9 item 1, it is "the first Tier-1 metric in
the plan that needs no `trace` at all", comparable across every
backend regardless of trace-instrumentation support. `cli.py`'s
`_write_baseline` only calls `standard_start_results()`; `basin_
rates()` has no caller anywhere in `cli.py` and never reaches
`reports/`. A wiring gap, not a missing capability - the metric and
its own tests already exist and work.

Noticed a parallel gap while looking at this: `_write_large_residual`
discards `large_residual.backend_results()`'s own basin-rate return
value (`results, _ = ...`) the same way. Filed as #101, out of scope
for this issue to keep the fix narrow.

## RED cycle (test-first)

**Test surface**

- `tests/integration/test_cli.py` (extended): `"baseline_basin_
  rates.csv"` added to `_EXPECTED_TABLES` (covered automatically by
  the existing `test_report_command_produces_every_milestone_
  artefact` smoke test). `test_run_report_writes_baselines_basin_of_
  attraction_rate_table` checks the new table's columns, row count
  (one per canonical problem), and that every rate is a valid
  probability.

**Confirmed RED**

Both fail against the unmodified `cli.py`: `test_run_report_writes_
baselines_basin_of_attraction_rate_table` with `FileNotFoundError`
(`baseline_basin_rates.csv` doesn't exist yet); `test_report_command_
produces_every_milestone_artefact` with `AssertionError:
baseline_basin_rates.csv` (missing from `_EXPECTED_TABLES`'s smoke
check). `make lint`: clean.

**Design notes**

New output file (`baseline_basin_rates.csv`), not appended to the
existing `baseline.csv`: `basin_rates()`'s return shape (one rate per
`(problem_id, backend)`, aggregated across every registered start) is
structurally different from `standard_start_results()`'s one-`RunResult`-
per-`(problem_id, backend)` table, and mixing them into one table
would mean either duplicating the rate across every problem's row or
inventing a sparse/nested column - a separate table matches the
existing precedent (`robust_loss.csv`'s own custom, non-`RunResult`
shape) more directly than either alternative.

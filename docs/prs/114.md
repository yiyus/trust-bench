# PR notes: Issue #114 - Flaky timing bound in test_evaluate_at_n_1000_completes_quickly

## Summary

`test_evaluate_at_n_1000_completes_quickly` compares a single wall-clock
measurement of `evaluate_problem` at `dimensionality`'s `n=1000` against
a fixed `10.0s` bound. Flagged during #90's implementation review: one
full-suite run observed an `11.99s` failure while isolated measurements
consistently showed `8-9.3s` - close enough that background system load
can tip a single measurement over the line.

No production code change: this is a test-only flakiness fix, matching
the acceptance criteria. Presented as a single pass rather than a
RED/GREEN split, mirroring #113's own precedent for the same category
of fix - there is no new coverage or bug being fixed in `trust_bench`
itself, only the existing test's own measurement being made robust.

## Investigation

Measured directly (this environment): 10 single-shot trials ranged
`2.0s` to `4.25s`, already a `2x` spread from scheduling noise alone with
no artificial load at all. Under 3 concurrent `yes` processes (matching
#113's own artificial-load methodology, a 16-core machine), single-shot
measurements ranged `2.86s` to `4.45s` - similar spread, confirming the
noise is process-launch jitter, not proportional to load in this range.

Taking the median of 3 repeated measurements (the same relative/
percentile-based fix #113 settled on for its own analogous flaky timing
assertion, which this issue's acceptance criteria explicitly named as
the approach to reuse "for consistency") kept the measured median in a
`2.3s-3.5s` band across 5 repeated trials, both idle and under artificial
load - comfortably tight, but this environment's own baseline is not a
safe universal threshold: the original issue's own baseline (`8-9.3s`)
came from a slower or more contended machine, and the fix must hold
there too, not just here.

The bound is loosened from `10.0s` to `30.0s` - `3x` the original
issue-reported worst case (`11.99s`) and comfortably below
`apl_backend.py`'s own `60s` harness subprocess timeout (a single
`evaluate_problem` call approaching half the harness's own hard timeout
is itself worth flagging, regardless of the specific multiple chosen).
The test's actual claim (the port must not become the dominant cost of
running the study, and must fail outright on an accidentally
quadratic-in-n-squared regression) is unweakened: a real complexity
regression at `n=1000` would blow past `30s` by a wide margin, not
linger just above it.

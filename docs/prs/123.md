# PR notes: Issue #123 - Add a parity scatter plot and a capability-frontier chart to the report

## Summary

The report's existing plots are all per-study, sweep-value line/bar charts with both backends' raw values overlaid - answering "what is the value at each point," not "how is trust-apl positioned relative to scipy" without the reader building that view by eye across eight separate figures. Adds two new, cross-study report artefacts:

- **Parity scatter** (`parity_scatter.csv`/`.png`): pools `baseline`/`typical`/`bounded` (the common, non-adversarial studies) into one log-log scatter, `dist_to_opt` on each axis, one axis per backend, with a `y = x` reference line. A tight cloud along the diagonal is the "these two backends agree" claim in one glance; a non-convergent row is drawn with a distinct marker rather than dropped.
- **Capability frontier** (`capability_frontier.csv`/`.png`): small multiples, one panel per existing difficulty sweep (`ill_conditioning`, `scaling`, `dimensionality`, `large_residual`, `robust_loss`), one line per backend, both axes log-scaled - the point is the transition where a backend's line departs from a flat baseline, not the value at any single x.

Also commented on this issue from #121's own cycle, flagging that the new `scalar_cost` study should be included in this pooled data set once both land - not done here (out of scope: `scalar_cost` isn't one of the "common, non-adversarial" studies this issue names, and folding it in cleanly needs its own design pass, left for a follow-up).

Both plots reuse data the studies already collect - reporting/aggregation only, no new experiments - with three exceptions the investigation surfaced, each confirmed directly before writing any test:

1. **`robust_loss` is excluded from the parity pool.** Its own precision tables report distance to the true (uncorrupted) parameters, not `dist_to_opt`/`status` - a different shape that doesn't pool with the other three studies.
2. **`bounded` needed a per-backend method mapping, not a shared method name.** scipy's own `"lm"` declares `bounds=False`; trust-apl's only bounds-capable method is its Coleman-Li-scaled `"lm"`. Pivoting on `(scenario, method)` directly finds zero matching pairs (confirmed directly: 0 rows survived). Mirrors `robust_loss.py`'s own already-established `_METHOD_FOR_BACKEND` mapping (`scipy: "trf"`, `trust-apl: "lm"`) instead of requiring the same method name.
3. **The capability-frontier chart's `robust_loss` panel needed a new, small aggregation, not reuse of `scipy_loss_precision`/`trust_loss_precision`.** Those two sweep disjoint loss vocabularies (scipy's five vs. trust's `"welsch"`) and share no common loss to compare directly. `"linear"` is the one loss every backend already declares support for; a small dedicated helper (`_robust_loss_linear_frame`) computes it directly via `run()`, matching `scipy_loss_precision`'s own shape. Caught a real modeling bug while validating this: the frame initially used `result.dist_to_opt` (distance to *this problem's own registered optimum*, which a `"linear"`-loss fit reaches reliably regardless of contamination - it isn't robust, it's just well-posed), producing a flat, meaningless line; fixed to measure distance to `outliers.TRUE_PARAMETERS`, the same quantity `robust_loss.py`'s own precision functions already use, which is what "does contamination degrade the fit" actually means.

**A fourth design problem, resolved at the CLI-selection level rather than inside the artefact itself:** `parity_scatter` is inherently a two-backend comparison (the pivot needs exactly two backend columns), but a plain `trust-bench report` (no `--backends` flag) defaults to scipy alone - meaning simply registering it in `STUDIES` would make the primary, documented CLI invocation crash outright. `capability_frontier` has no such requirement (each panel is just one line per whatever backends are present, informative even with one). Fixed by introducing `MULTI_BACKEND_STUDIES`, auto-excluded from the default "run everything" selection when fewer than two backends are selected, but still run (and still raising its own clear error) when explicitly named via `--only`.

**Design notes**

*A new `trust_bench/reporting/cross_study.py` module*, not folded into `plots.py` (which stays purely `df -> fig`, with zero knowledge of `trust_bench.studies.*`, preserving its existing synthetic-fixture-only test isolation) or bare in `cli.py`. Mirrors `capability_matrix.py`'s own `derive_matrix` (data) / `plot_capability_matrix` (rendering) split, extended to study-based aggregation.

*A `1e-16` display floor for an exact-zero metric*, reused identically in both `parity_frame` and `frontier_panels`: several problems reach `dist_to_opt == 0.0` or `grad_norm_final == 0.0` exactly (confirmed directly: scipy's `"lm"` on every swept `scale`), which a log axis can't plot. Not a precision claim - well below the smallest nonzero value measured anywhere in this project (`docs/methodology.md`'s own measured floor is `~1e-14`).

*Non-convergent rows are marked, not dropped*, in the parity scatter (a distinct `"x"` marker in the same colour as the group's regular points) - matching this project's existing convention (`plot_metric_vs_sweep`'s hollow-marker overlay, `plot_metric_by_category`'s hatching) of keeping an exception visible rather than silently excluding it.

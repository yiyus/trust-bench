# PR notes: Issue #113 - Flaky timing assertion in test_dense_hessian_methods_cost_far_more_per_step_at_large_dimension

## Summary

`test_dense_hessian_methods_cost_far_more_per_step_at_large_dimension` compares a single wall-clock measurement of a dense-Hessian method against a single measurement of `L-BFGS-B` at `n=1000`, asserting the dense method takes at least `5x` longer. The reference (`L-BFGS-B`) measurement is itself short (~0.1-0.2s) and noisy; measured directly, a single bad reference measurement can push the ratio as low as `5.07x` - just over the threshold, and the true cause of the intermittent failures reported in #28/#32.

No production code change: this is a test-only flakiness fix, matching the acceptance criteria ("this is a flakiness fix, not a redesign of what the test verifies"). Presented as a single pass rather than a RED/GREEN split, since there is no new coverage or bug being fixed in `trust_bench` itself - only the existing test's own measurement is being made robust.

## Investigation

Measured directly (single-shot): ratios across 5 trials ranged from `5.07x` to `43.5x` for the two `DENSE_METHODS` (`trust-exact`, `BFGS`) - the reference time itself varied `2x` between runs (`0.094s` to `0.20s`) purely from scheduling noise, since two iterations of a matrix-free method at `n=1000` is intrinsically a very short measurement.

Taking the median of `3` repeated measurements for both the reference and the dense-method timing raised the observed minimum ratio to `~5.8-6.0x` across the same trials; over 15 repeated trials with median-of-3, the true minimum observed was `~5.85x`. This directly matches the acceptance criteria's ask to investigate "a looser or relative/percentile-based timing check": median-of-repeats is the relative/percentile-based fix, and the ratio threshold is loosened from `5x` to `3x` to leave a comfortable margin below the `~5.8x` worst case measured, without weakening the test's actual claim (a `3x` per-step cost difference is still an unambiguous "costs far more").

## Verification

- `tests/studies/test_dimensionality.py::test_dense_hessian_methods_cost_far_more_per_step_at_large_dimension`: 5 consecutive clean runs.
- Repeated under artificial CPU load (3 concurrent `yes` processes on a 16-core machine): 3 consecutive clean runs, confirming the fix holds under contention, not just on an idle machine.
- `ruff check tests/studies/test_dimensionality.py`: clean.
- Full suite `pytest -q` (dyalogscript available): 382 passed, 0 failed, no regressions.

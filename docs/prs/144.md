# PR notes: Issue #144 - Revisit CLI options and defaults

Part of #41 (Epic: Phase 10, Longitudinal tooling), after #43 (CLI compare
command) landed.

## Context

The issue itself proposed no specific change, deferring the decision until
`trust-bench compare` existed. With #43 merged, a concrete gap surfaced:
`trust-bench report` never wrote `results/*.jsonl` at all - every study's
`_write_*` function in `cli.py` only produces CSV/PNG artefacts via
`results_to_dataframe`/`save_table`, so `trust-bench compare` had no real
producer of the files it consumes; the only way to get two `.jsonl` files
to diff was to hand-craft them, as the tests do. Section 8 of
`docs/plans/trust-bench.md` assumes `results/*.jsonl` exists, and #44 (CI
job, next in the epic) depends on "appends results to `results/*.jsonl`"
too.

Decision, taken with the user: fold jsonl persistence into this issue
rather than opening a separate one, since #144 is exactly "revisit CLI
defaults now that `compare` exists" and this is the concrete thing that
decision was waiting on.

## Design

- `trust_bench/core/runner.py` gets a `recording_results(path)` context
  manager, mirroring the existing `measuring_timing()`: every `run()` call
  made while it's active appends its `RunResult` to `path` via
  `storage.append`. `None` is a no-op, matching `measuring_timing`'s own
  shape. Every study already funnels through `run()`, so this needs no
  per-study change, the same reasoning that motivated `measuring_timing`.
- `run_report` gains a `results_dir="results"` parameter and wraps its
  study loop in `recording_results(...)` alongside the existing
  `measuring_timing()`. The file is `{output_dir}/{results_dir}/
  {harness_git_sha()}.jsonl` - colocated under `output_dir` like every
  other artefact (not a fixed path outside it, which would leak into the
  real working directory during tests and CI runs from different
  directories), and named by commit so repeated runs on the same commit
  accumulate (matching `storage.append`'s own "never overwrites"
  contract) while a new commit gets its own file, ready to diff against an
  older one with `trust-bench compare`.
- `results_dir=None` disables persistence entirely (no `results/`
  directory created).
- CLI: `report` gains `--no-results` (default off), wired to
  `results_dir=None` in `main`.

## Cycle 2: design

Agreed with the user. trust-bench's purpose is benchmarking `trust`, so a
scipy-only default was backwards; and now that `report` feeds a
longitudinal ledger that `compare` diffs, the everyday invocation wants to
be cheap and directory-oriented rather than a rare, comprehensive,
file-path-juggling one.

`trust-bench report [BASELINE_DIR ...]`
- Positional, optional: zero or more prior report output directories.
  Each one's `results/*.jsonl` files are pooled as the baseline; when any
  are given, this run's own results are also classified (regression/
  drift) against that pool and folded into the report, via the same
  machinery `compare` uses.
- `--scipy`: also run the scipy backend (off by default now - trust-bench
  benchmarks trust-apl by default).
- `--no-trust`: turn off the trust-apl backend (on by default) - e.g. for
  a scipy-only reference run.
- `--full`: also run the slow studies (`dimensionality`,
  `capability_frontier`) - off by default now, replacing `--skip-slow`.
- `--no-html`: skip writing `report.html` - on by default now, replacing
  `--html`.
- `--no-results`: skip appending to `results/*.jsonl` - on by default
  (unchanged from cycle 1).
- `--only`/`--skip`/`--output-dir`: unchanged.
- `--backends` is removed in favour of `--scipy`/`--no-trust` (and a
  future `--julia`).

`trust-bench compare BASELINE_DIR CANDIDATE_DIR`
- Kept as the explicit lower-level tool (e.g. for CI diffing two
  already-generated report directories without rerunning studies).
  Arguments change from raw jsonl file paths to directories, for
  consistency with `report`'s new baseline argument - both load via the
  same `_load_results_dir` helper ("pool every `results/*.jsonl` under
  this directory").

Also fixes a bug found while implementing this: `_select_studies` was
stripping `SLOW_STUDIES` even from an explicit `--only dimensionality`,
contradicting the project's existing rule for `MULTI_BACKEND_STUDIES` (an
explicit request is never silently dropped). `--full`'s default-skip
behaviour now only prunes the *default* selection, same as the existing
`MULTI_BACKEND_STUDIES` carve-out.

Posted to the issue (comment) for review to check against.

## Cycle 2 review: request changes

`docs/reviews/144.md` found a Critical bug: `compare.csv` explodes into
duplicate rows for any real report, reproduced directly with no synthetic
fixtures (`only=["baseline"]`). Root cause, in two parts:

1. `results/*.jsonl` genuinely contains duplicate keys from a single
   `run_report` call: `baseline.basin_rates()` re-probes the `"standard"`
   start already covered by `standard_start_results()`, both through
   `run()`, both recorded by `recording_results` (cycle 1) with no notion
   of "the" result for a key versus "every recording of it". This is
   expected, not a bug in `baseline.py` to fix here - cycle 1's own design
   says repeated runs on one commit accumulate by design, and any study
   could revisit a key the same way.
2. `compare_with_provenance`'s second, separate merge (for provenance)
   cross-joins duplicate keys on each side instead of aligning them: 2x2
   duplicated rows in `compare()` alone become 4, then 4x4=16 once
   `compare_with_provenance` re-merges the same un-deduplicated frames.

A second, Major finding: `_load_results_dir`'s hardcoded `"results"`
subdirectory ignores a caller's own `results_dir`, so `baselines` combined
with a non-default `results_dir` raises a misleading "no results found"
even though the run wrote real results elsewhere.

Fixes:

- `trust_bench/reporting/compare.py`: new `_dedupe_latest(df,
  key_columns)` collapses to one row per key, keeping the latest
  `timestamp` - `results/*.jsonl` is an append-only log, so "this key's
  result" means its most recent recording. Applied in both `compare()`
  (its own merge) and `compare_with_provenance()` (both its call into
  `compare()` and its own separate provenance merge), so each function is
  self-sufficient against duplicate-keyed input regardless of caller.
- `trust_bench/cli.py`: `run_report`'s baseline-comparison branch now
  loads its own candidate results from `results_path.parent` directly
  instead of `_load_results_dir(output_dir)`, so a custom `results_dir`
  is respected. `_load_results_dir` (for baseline directories, which are
  independent prior reports whose own `results_dir` isn't recorded
  anywhere) keeps assuming the `"results"` default, documented as such.

New regression tests: `tests/reporting/test_compare.py` (`compare()` and
`compare_with_provenance()` each collapse duplicate-keyed rows to one
before classifying, verified against the exact 2x2-duplicate shape that
produced 16 rows); `tests/integration/test_cli.py` (a real `only=
["baseline"]` `run_report`/`baselines` round trip asserts an exact
expected row count - the sum of each canonical problem's own registered
starts, not a bare "> 0" - plus a `results_dir="custom_results"` case).
Confirmed each new test fails against the pre-fix code (reverted the two
source files locally, reran) before restoring the fix.

`tests/reporting/` (whole directory), `tests/integration/test_cli.py`,
`tests/integration/test_runner.py`: 165 passed. `ruff check` clean. Full
suite runs once now, in the background, while this cycle is under review.

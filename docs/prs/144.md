# PR notes: Issue #144 - Revisit CLI options and defaults

Part of #41 (Epic: Phase 10, Longitudinal tooling), after #43 (CLI compare
command) landed.

## Context

The issue itself proposed no specific change, deferring the decision until
`trust-bench compare` existed. With #43 merged, a concrete gap surfaced:
`trust-bench report` never wrote `results/*.jsonl` at all - every study's
`_write_*` function in `cli.py` only produces CSV/PNG artefacts via
`results_to_dataframe`/`save_table`, so `trust-bench compare` had no real
producer of the files it consumes; the only way to get two `.jsonl` files
to diff was to hand-craft them, as the tests do. Section 8 of
`docs/plans/trust-bench.md` assumes `results/*.jsonl` exists, and #44 (CI
job, next in the epic) depends on "appends results to `results/*.jsonl`"
too.

Decision, taken with the user: fold jsonl persistence into this issue
rather than opening a separate one, since #144 is exactly "revisit CLI
defaults now that `compare` exists" and this is the concrete thing that
decision was waiting on.

## Design

- `trust_bench/core/runner.py` gets a `recording_results(path)` context
  manager, mirroring the existing `measuring_timing()`: every `run()` call
  made while it's active appends its `RunResult` to `path` via
  `storage.append`. `None` is a no-op, matching `measuring_timing`'s own
  shape. Every study already funnels through `run()`, so this needs no
  per-study change, the same reasoning that motivated `measuring_timing`.
- `run_report` gains a `results_dir="results"` parameter and wraps its
  study loop in `recording_results(...)` alongside the existing
  `measuring_timing()`. The file is `{output_dir}/{results_dir}/
  {harness_git_sha()}.jsonl` - colocated under `output_dir` like every
  other artefact (not a fixed path outside it, which would leak into the
  real working directory during tests and CI runs from different
  directories), and named by commit so repeated runs on the same commit
  accumulate (matching `storage.append`'s own "never overwrites"
  contract) while a new commit gets its own file, ready to diff against an
  older one with `trust-bench compare`.
- `results_dir=None` disables persistence entirely (no `results/`
  directory created).
- CLI: `report` gains `--no-results` (default off), wired to
  `results_dir=None` in `main`.

No other existing flag (`--only`/`--skip`/`--skip-slow`/`--backends`/
`--html`/`--output-dir`) changes: nothing else surfaced a concrete gap
once `compare` existed, and CLAUDE.md's minimal-scope convention argues
against touching them speculatively.

## Cycle 1: RED

`tests/integration/test_runner.py` drives out `recording_results`:

- A `run()` call outside the context writes nothing.
- Every `run()` call inside the context is appended to the given path.
- `recording_results(None)` is a no-op.

`tests/integration/test_cli.py` drives out `run_report`'s `results_dir`
parameter and the `--no-results` flag:

- `run_report` writes exactly one `results/*.jsonl` file by default,
  containing real `RunResult` rows.
- The file is named `{harness_git_sha()}.jsonl`.
- A second `run_report` call on the same (monkeypatched) sha appends
  rather than overwrites.
- `results_dir=None` disables persistence.
- `--no-results` defaults to `False`; passing it disables persistence
  through the real `main()` entry point.

RED confirmed:

- `tests/integration/test_runner.py` fails to collect:
  `ImportError: cannot import name 'recording_results' from
  'trust_bench.core.runner'`.
- `tests/integration/test_cli.py`'s new tests fail: `--no-results` is
  rejected by argparse (`unrecognized arguments`), and
  `monkeypatch.setattr("trust_bench.cli.harness_git_sha", ...)` fails
  (`cli.py` does not import it yet).

No other test files were run.

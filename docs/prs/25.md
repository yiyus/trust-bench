# PR notes: Issue #25 - Study: robust loss and outliers

## Summary

Adds the robust-loss/outliers study (`docs/plans/trust-bench.md`,
Section 9 item 4): sweeps the outlier fraction of
`trust_bench.problems.families.outliers.make(fraction)` and compares the
precision (distance from the fixed, uncorrupted `TRUE_PARAMETERS`, not
the L2-biased `Problem.optima`) of SciPy's five built-in `least_squares`
losses against a hand-rolled IRLS reference using Tukey's biweight, a
genuinely redescending psi-function (weight reaches exactly zero beyond
a cutoff, unlike SciPy's bounded-but-nonzero-influence losses). Part of
Epic #21 (Phase 6).

**Scope deferral, agreed with the user before writing any code:** the
acceptance criteria compare against `trust`'s (the APL backend)
redescending losses, but the APL backend does not exist yet (Epic #34,
Phase 8, not started, and sequenced after Phase 6 in the plan's own
ordering — Section 10's rationale is explicit that Phases 1-7 land before
either non-Python backend is touched). This issue implements the
SciPy-vs-IRLS comparison now; the `trust` comparison itself is deferred
until Phase 8 lands, at which point `trust`'s redescending losses become
the third comparison point the acceptance criteria describe, alongside
this IRLS reference (currently the second, standing in for it).

`test_outliers_true_parameters_are_the_fixed_underlying_trend_not_the_biased_l2_optimum`
pins a fact about the existing `outliers` family (issue #10) that every
other test in this file depends on: `Problem.optima` is the plain-L2 fit
to the *corrupted* data, which is itself dragged away from the true
trend, so precision here is measured against `TRUE_PARAMETERS`, not
`Problem.optima[0].x_star`.

**Design notes**

`FRACTIONS = [0.0, 0.1, 0.2, 0.3, 0.4, 0.45, 0.49]` (the family's own
constructor rejects `fraction >= 0.5`). Verified directly against
`SciPyBackend`'s `trf` method (the only least_squares method supporting
all five losses) across the full sweep before writing any assertions:

- `linear`/`soft_l1`/`huber`/`cauchy` are all dragged more than 7 units
  from `TRUE_PARAMETERS` by `fraction=0.4`, out of a trend whose own
  scale is `O(1)`.
- `arctan` degrades far more gracefully (`0.111` at `fraction=0.4`,
  `0.254` at `fraction=0.49`) since its psi-function also decays toward
  zero, just never reaching it exactly (soft, not hard, redescending) —
  it is the best of the five, but still clearly worse than the IRLS
  reference below.

**A genuine implementation obstacle, found and resolved before writing
the hand-rolled IRLS:** a Tukey-biweight IRLS started from an ordinary
least-squares fit fails outright from `fraction=0.2` for this family.
The corrupted points are the *first* `n_corrupt` points (lowest `t`),
which gives them high leverage on the slope; OLS is dragged badly enough
by `fraction=0.2` that its residuals no longer cleanly separate inliers
from outliers, so the very first re-weighting pass locks onto the wrong
group and never recovers. Warm-starting the same IRLS loop from SciPy's
`arctan`-loss fit instead (already a soft-robust estimate, confirmed
above) fixes this completely: exact recovery of `TRUE_PARAMETERS`
(`dist == 0.0`) at every fraction up to and including `0.45`, only
failing at `fraction=0.49`, consistent with the family's own documented
limit ("past a majority-corrupted dataset... no longer describes what
the family's own defining property assumes"). This is why
`test_irls_recovers_true_parameters_past_the_contamination_level_where_
every_scipy_loss_fails` uses `fraction=0.4`, not a value closer to the
family's `0.5` boundary.

## GREEN cycle (implementation)

**Files changed**

- `trust_bench/studies/robust_loss.py` (new): `scipy_loss_precision`,
  `irls_tukey`, `irls_precision`, and the `FRACTIONS`/`SCIPY_LOSSES`
  sweep constants.

**A bug found and fixed during this cycle:** the first implementation
of `irls_tukey` fell back to all-ones weights (a plain, unweighted
refit) whenever the residual scale dropped near zero, treating "no
spread left to reweight against" the same as "trust everyone equally".
Once the fit was already accurate, that unweighted refit pulled `beta`
straight back toward the corrupted points' influence, since it no
longer excluded them: `test_irls_recovers_true_parameters_past_the_
contamination_level_where_every_scipy_loss_fails` failed with
`irls_distance == 8.77` instead of the expected near-zero. Fixed by
breaking the loop on a near-zero scale instead of falling back to
uniform weights, keeping the last (already-converged) `beta`. Verified
the fix restores exact recovery (`dist == 0.0`) at every fraction up to
`0.45`, matching the ad hoc calibration in the design notes above.

# PR notes: Issue #121 - Add a scalar-cost, no-Jacobian problem to exercise BFGS on its own terms

## Summary

Every problem in the current battery is a nonlinear-least-squares fit (residual vector + Jacobian), so every BFGS-vs-LM comparison so far has BFGS solving the same problem with strictly less information than LM gets to use. Adds two genuine scalar-cost, no-Jacobian problems (`kind="scalar"`, `jacobian=None`) - a problem shape where BFGS is the natural, structurally-appropriate choice rather than a handicapped stand-in for LM:

- **`rastrigin`**: the classic multimodal global-optimisation test function, chosen deliberately over another MGH least-squares function (Himmelblau, Beale, etc. are all secretly sums of squares underneath). Standard start `(0.25, -0.25)` sits inside the global minimum's basin (confirmed directly: `(0.3, -0.4)` already lands in a neighbouring local minimum instead) - the point is a clean scalar landscape for BFGS, not a hard multimodal search.
- **`cauchy_mle`**: a Cauchy location-scale MLE fit to synthetic heavy-tailed data. Its negative log-likelihood has no natural sum-of-squares decomposition at all (unlike a Gaussian MLE, which reduces exactly to least squares) - the "small MLE fit" case the issue itself suggested, and it ties into the project's existing robust-statistics theme (Tukey/Welsch/Huber losses, outlier studies).

Two real code gaps surfaced, not just wiring:

1. `scipy_backend.py`'s `_objective`/`_gradient` always treat `problem.residual(x)` as a vector and square-sum it. Confirmed directly: handing either new problem to `BFGS`/`L-BFGS-B` crashes outright (`ValueError: matmul: Input operand 0 does not have enough dimensions`), since a scalar cost isn't a 1-D array.
2. Nothing currently distinguishes `kind="scalar"` from `kind="residuals"` when routing to `lm`/`trf`/`dogbox`. Confirmed directly: `lm` happens to raise (for an unrelated reason - "number of residuals is less than variables"), but `trf`/`dogbox` don't raise at all: `least_squares` silently treats the scalar cost as a 1-element residual and drives it toward zero, which only coincidentally lands on the right answer for `rastrigin` (whose true minimum cost is exactly `0`) - a real correctness landmine for any scalar problem whose minimum isn't at cost `0` (`cauchy_mle`'s is `â‰ˆ91`, and would be handled wrong silently). An explicit guard is needed, not reliance on scipy's incidental behaviour.

`trust-apl` is deliberately not wired in for these problems: its own BFGS engine always derives cost/gradient from a residual-and-Jacobian-bearing evaluator internally (`backends_ext/apl/solve.dyalog`), so genuinely supporting a Jacobian-free scalar objective there is real backend work, out of scope for this issue - matching the precedent set in #84 for other backend gaps. No code is needed to enforce this: since no APL-side evaluator exists for `rastrigin`/`cauchy_mle`, the harness's own `NameFor` lookup fails and reports `ERROR` gracefully if `trust-apl` is ever explicitly selected for this study.

Also commented on #123 (parity scatter / capability-frontier report plots) to flag that whichever study this becomes should be included in that issue's pooled data set once both land.

## RED cycle (test-first)

**Test surface**

- `tests/problems/test_scalar_cost.py` (new): both problems declare `kind="scalar"`/`jacobian=None`/`hessian=None`; `assert_known_optimum` (extended in `tests/problems/invariants.py` to check a finite-difference gradient for `kind="scalar"`, since there's no analytic Jacobian to build the residuals-kind check from) holds at each registered optimum.
- `tests/backends/test_scipy_scalar_cost.py` (new): `BFGS`/`L-BFGS-B` solve both problems to their known optimum; `lm`/`trf`/`dogbox` raise `ValueError` (matching `"scalar"`) for a `kind="scalar"` problem instead of crashing or silently misbehaving.
- `tests/studies/test_scalar_cost_study.py` (new): the new `scalar_cost` study's `sweep()` covers every `(problem, method, backend)` combination and converges on every problem.
- `tests/integration/test_cli.py` (extended): `"scalar_cost.csv"` added to `_EXPECTED_TABLES`.

**Confirmed RED**

`ruff check` on every new/changed file: clean.

**Design notes**

*A dedicated study, not reuse of `typical.py`/`derivative_source.py`.* `typical.py` iterates every method a backend declares, including `lm`/`trust-exact`/`Newton-CG`, none of which can run a Jacobian-free scalar objective; `derivative_source.py` is least-squares-only, the opposite of what's needed. `scalar_cost.py` mirrors the existing `sweep(problems=..., methods=..., backends=...)` shape (closest to `derivative_source.py`'s own structure) restricted to `["BFGS", "L-BFGS-B"]`.

*No plot*, matching `bounded`/`derivative_source`'s own precedent: two problems x two methods isn't sweep-shaped data a line/bar chart naturally represents.

*Test file renamed* (`tests/studies/test_scalar_cost.py` -> `test_scalar_cost_study.py`) to avoid a basename collision with `tests/problems/test_scalar_cost.py` - this project's test tree has no `__init__.py` packages, so pytest requires unique basenames across directories.

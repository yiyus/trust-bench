# PR notes: Issue #33 - CLI report command (cli.py)

## Summary

Adds `trust_bench/cli.py` with a `trust-bench report` subcommand: runs
every study built in Epic #21 (Phase 6) end to end and writes the
resulting comparison tables, one plot, and the capability matrix to
`reports/` (git-ignored, per Section 3's own repository layout comment).
This is Phase 7's milestone: "full Python-only pipeline is shippable
here." Part of Epic #30, the last child.

Adds a `[project.scripts]` entry point (`trust-bench = "trust_bench.
cli:main"`) so the command is genuinely installable and runnable by
name, not just importable.

Everything else is basic CLI usability, requested directly rather than
read from the acceptance criterion: `--help` at both the top level and
the `report` subcommand; `--only`/`--skip` to select which
studies/artefacts run, mutually exclusive with each other and validated
against the real registry (`choices=`, so a typo is an argparse error,
not a silent no-op); `--skip-slow` as a dedicated shortcut for the one
study (`dimensionality`) that is genuinely expensive. `_select_studies`
is a small, pure function separated from `run_report`'s side effects
specifically so this selection logic has fast, direct tests instead of
only ever being exercised through the slow, full end-to-end run.

**Design notes**

All eight Phase 6 studies are included, not a subset: `baseline`,
`large_residual`, `ill_conditioning`, `robust_loss`, `bounded`,
`scaling`, `dimensionality`, `derivative_source`, plus the capability
matrix (issue #32). Two studies don't fit `results_to_dataframe`'s
`RunResult`-only assumption as it stands (flagged as a known gap back
in issue #31's own PR notes, deferred there to "whichever future issue
... actually builds a report", i.e. this one):

- `bounded.sweep()` mixes `RunResult` and a raised `ValueError` in the
  same dict (an infeasible start is expected to raise). Extending
  `results_to_dataframe` itself to treat any `BaseException` value as an
  `"ERROR"` row (rather than writing a bespoke adapter just for this one
  study) fixes it generically, and is a small, backward-compatible
  addition: existing callers passing only `RunResult` values are
  unaffected.
- `robust_loss.scipy_loss_precision()`/`irls_precision()` return plain
  `float` distances, not `RunResult` at all — there is no status,
  eval count, or gradient norm to report, so `results_to_dataframe`
  genuinely does not apply. Built as a small, one-off `DataFrame`
  directly in `cli.py` instead of forcing a generic function to handle
  a shape it was never meant to.

**Runtime, disclosed rather than hidden:** `trust-bench report` calls
`dimensionality.sweep()` with its own real defaults (`n` up to `1000`,
`max_iter=200`, all five methods), the same computation issue #28's own
`slow`-marked tests already pay for; running every study together is
genuinely a multi-minute operation. This is treated as an accepted cost
of a deliberate, occasional batch command, not a `make test`-style
dev-loop cost: `test_report_command_produces_every_milestone_artefact`
is itself `@pytest.mark.slow`, so it only runs under `make coverage`/CI,
never under `make test`.

- `trust_bench/reporting/tables.py`: `results_to_dataframe` now treats
  a `BaseException` value as an `"ERROR"` row (status only, every
  metric field blank), covering `bounded.py` generically.
- `trust_bench/cli.py` (new): `STUDIES` (name -> a `_write_*(output_dir)`

  frozenset({"dimensionality"})`, `_select_studies(only, skip,

  `build_parser()`, `main(argv)`.
- `pyproject.toml`: `[project.scripts]` entry point (`trust-bench =
  "trust_bench.cli:main"`); `.gitignore`: `reports/`.

**One more genuinely testable branch, added rather than accepted as a
gap:** the first coverage run showed `_select_studies`'s own `unknown
study` `ValueError` uncovered — `argparse`'s `choices=` already rejects
an unknown name at the CLI boundary, but `_select_studies`/`run_report`
are also directly callable, bypassing that boundary entirely. Added
`test_select_studies_rejects_an_unknown_name_when_called_directly`
rather than disclaim it, the same judgement call issue #24 made for its
own genuinely-testable branch.

**Real smoke test, not just the automated suite:** installed the
package (`pip install -e .`) and ran `trust-bench --help`, `trust-bench
report --help`, and `trust-bench report --skip-slow` against a scratch
directory. Help text renders cleanly at both levels; the real run wrote
all eight non-slow artefacts, including `bounded.csv`'s two
`infeasible_start` rows as clean `status=ERROR` with blank metric
fields, not a crash.

**Design notes**

`build_html_report(output_dir)` scans `output_dir` for `*.csv`/`*.png`
after the selected studies have already written their own files,
rather than threading each study's `DataFrame`/`Figure` through
`run_report` to a separate bundling step. This keeps the two concerns
decoupled: `--html` bundles *whatever artefacts exist* in the directory,
so it automatically respects `--only`/`--skip`/`--skip-slow` (fewer
files on disk means a smaller bundle) with no extra plumbing, at the
cost of assuming the directory reflects the current run (a stale file
left over from an earlier run into the same directory would be bundled
too; not solved here, documented instead).

Plots are embedded as base64 `data:` URIs, matching the review
Artifact's own self-contained approach the user pointed to directly,
so `report.html` is portable on its own without needing the PNGs
alongside it. Tables are rendered inline via `pandas.DataFrame.to_html`.

**Confirmed failure**

**Design notes**

- **`load` does not round-trip through `RunResult`.** Section 8 asks for "a
  DataFrame matching the stored records", not a DataFrame of reconstructed
  `RunResult` objects; parsing each JSONL line into a plain dict and handing
  the list straight to `pd.DataFrame(...)` is both simpler and a more
  literal reading than reconstructing `RunResult` instances (via
  `RunResult.from_dict`) only to immediately flatten them back into the
  same dict shape via `to_dict`, a needless round-trip. One consequence:
  `status` lands in the DataFrame as the plain string `RunResult.to_dict`
  already serialises it to (`"CONVERGED"`), not a `RunStatus` member. This
  matches results/*.jsonl's own stated purpose (Section 8): diff-friendly,
  streamable, and loadable to pandas *on demand*, i.e. for inspection and
  comparison, not for reconstructing live objects.
- **No column flattening.** `pd.json_normalize` would expand nested fields
  (`provenance`, `timing`, `config`) into dotted column names
  (`provenance.backend_name`, etc.). Plain `pd.DataFrame(records)` instead
  keeps each nested field as a single object-dtype column holding the raw
  dict, matching "a DataFrame matching the stored records" at the
  granularity the records actually have (one JSON object per row), not a
  restructuring of them. Flattening is a reasonable thing to want later,
  but no acceptance criterion asks for it, and it is easy to add as a
  parameter without touching the append side at all.
- **`pandas` becomes a real dependency**, alongside `numpy`/`scipy`,
  since `load`'s entire purpose is producing a `pandas.DataFrame`.

**Anticipated implementation order**

1. `pyproject.toml`: add `pandas` to `[project] dependencies`.
2. `trust_bench/core/storage.py`: `append`, `load`.

## Review response

`docs/reviews/13.md`'s Minor finding: the design notes committed to "no
column flattening" as the intended reading of the acceptance criterion, but
`test_load_reconstructs_a_dataframe_matching_the_stored_records` only
asserted on scalar columns (`problem_id`, `cost_final`, `status`), so an
implementation that flattened nested fields with `pd.json_normalize`, or
dropped them, or serialised them to JSON strings, would have passed
unchanged.

Fixed by asserting on two nested fields directly:
`df.loc[0, "provenance"]["backend_name"]` (confirming `provenance` survives
as a dict, not a flattened set of columns or a string) and `df.loc[0,
"config"] == {"ftol": 1e-8}` (same, for the field with no dataclass type at
all). Both now fail against a flattening or string-serialising
implementation, closing the gap the review identified.

The Note (`pandas` not yet a declared dependency) requires no action per
the review itself: it is the expected RED-stage state, called out only so
the GREEN-cycle review checks it landed.

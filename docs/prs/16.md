# PR notes: Issue #16 - SciPy backend: least_squares methods (lm, trf, dogbox)

## Summary

Adds `trust_bench.backends.scipy_backend`: `SciPyBackend`, the first real
`Backend` implementation, covering `least_squares`'s three methods (`lm`,
`trf`, `dogbox`). Part of Epic #14 (Phase 4), see
`docs/plans/trust-bench.md`, Section 9 (item 5) and Section 10 (Phase 4).

**Design notes**

- **`SciPyBackend` is wired into `BACKENDS`, the generic contract suite from
  issue #15.** `docs/prs/15.md` stated real backends would be added to
  `BACKENDS` alongside the toy backend "going forward"; this is that point.
  `BACKENDS` moves out of `toy_backend.py` (which now only defines the toy
  backend itself) into a new `tests/backends/all_backends.py`, since it is
  no longer that one module's list to own once a second backend exists.
  Verified directly (not just assumed) that `SciPyBackend` satisfies
  issue #15's inherited contract before wiring it in: `least_squares`
  with `max_nfev=1` returns `status=0` (mapped to `MAX_ITER`) rather than
  raising, for all three methods, and the trivial `quadratic` problem
  converges in 2 evaluations regardless of a larger `max_iter`, so the
  "monotone across increasing `max_iter`" check holds (trivially, since
  the counts do not grow past convergence, but they do not decrease
  either, which is what that check actually asserts).
- **`lm` rejecting bounds is enforced by `scipy` itself, not by an
  explicit check in `solve`.** Verified directly: `least_squares(...,
  method="lm", bounds=(...))` raises `ValueError: Method 'lm' doesn't
  support bounds.` unprompted. `solve` passes `bounds` through to `scipy`
  whenever the caller's `config` provides it, for every method, with no
  per-method branch; the rejection the acceptance criterion asks for falls
  out of that pass-through rather than needing to be reimplemented
  independently of `scipy`'s own validation, which the tests confirm
  agrees with the declared `capabilities()` values (the actual point of
  "matching capabilities().methods").
- **`config` stays a plain `dict`** (`config["max_iter"]`, `config.get(
  "bounds")`), same forward-reference situation as issues #6, #12, and
  #15: `RunConfig` (issue #19) does not exist yet.
- **`harness_git_sha()` is a new function in `provenance.py`, not
  `scipy_backend.py`.** `RunResult.harness_git_sha` (Section 4.3) needs a
  real value now that a real backend produces real `RunResult`s (the toy
  backend used the literal string `"test"`, acceptable there since it is
  never shipped). No earlier issue's scope covered "what commit of
  trust-bench itself produced this result", but it is exactly the same
  kind of environment fact `provenance.py` already owns, and every future
  backend (Julia, APL) will need the identical value, not one specific to
  SciPy. Implemented as `subprocess.run(["git", "rev-parse", "HEAD"])`
  with a fallback to `"unknown"` on `OSError` (git not installed) or
  `subprocess.CalledProcessError` (not a git checkout), mirroring
  `_detect_cpu_model`'s own fallback pattern (issue #3); that fallback
  branch is not covered by a test, for the same reason
  `_detect_cpu_model`'s was not: this repository always has git available
  in CI and in development, so triggering it needs mocking a private-ish
  path rather than a real scenario.

**Anticipated implementation order**

1. `trust_bench/core/provenance.py`: add `harness_git_sha()`.
2. `trust_bench/backends/__init__.py`
3. `trust_bench/backends/scipy_backend.py`: `SciPyBackend`.
4. `tests/backends/all_backends.py`, moving `BACKENDS` out of
   `toy_backend.py`.

## Review response

`docs/reviews/16.md`'s implementation-stage review found a Major issue:
every method's `derivative_modes` declared `"finite-difference"`
support, but `solve` always passed `jac=problem.jacobian` with no way to
request FD and no fallback when `problem.jacobian` is `None`, the exact
case `Problem.jacobian`'s own docstring reserves for "FD only". No
problem in the repository currently has `jacobian=None`, so this had not
failed yet, but Section 9 item 8 ("Derivative source") treats
analytic-vs-FD as a first-class comparison this backend needs to
actually support, not just declare.

Fixed rather than narrowed: `solve` now reads `config.get(
"derivative_mode")`, and uses `jac="2-point"` whenever that equals
`"finite-difference"` *or* `problem.jacobian is None` (so a problem with
no analytic Jacobian gets FD automatically, without the caller having to
ask); otherwise `jac=problem.jacobian`, unchanged from before.

Two new tests, initially confirmed to fail against the unfixed code
(worth noting: a first version of the FD test only checked convergence,
which passed even before the fix, since an ignored `derivative_mode` key
just silently fell through to the analytic Jacobian and converged anyway
via the untouched path; rewritten to spy on `problem.jacobian` directly
and assert it is never called):

- `test_finite_difference_derivative_mode_never_calls_the_analytic_jacobian`:
  wraps `problem.jacobian` in a call-counting shim, requests
  `derivative_mode="finite-difference"` with the *original*, still-present
  analytic Jacobian, and asserts the shim is never called, for all three
  methods.
- `test_solve_falls_back_to_finite_difference_when_problem_has_no_analytic_jacobian`:
  `dataclasses.replace(PROBLEM, jacobian=None)`, confirms `solve` reaches
  `CONVERGED` rather than raising `scipy`'s own `jac must be '2-point',
  '3-point', 'cs' or callable` error.

Also verified directly: `lm` with `jac="2-point"` reports `njev=None`
(MINPACK's own internal FD routine, `lmdif`, does not expose a separate
Jacobian-evaluation count the way `lmder` does for an analytic Jacobian),
while `trf`/`dogbox` report a real count under FD just as they do under
analytic. `RunResult.n_jeval` is already `int | None` (Section 4.3), so
this needs no special-casing: `n_jeval=result.njev` already does the
right thing whether `njev` is a real count or `None`.

The Minor finding (`n_iter` reusing `n_feval`'s value with no documented
reason) is addressed by this note itself, restated in the implementation
comment: `least_squares`'s `OptimizeResult` has no per-iteration count
the way `minimize`'s does (no `nit`-equivalent field), so `n_iter` here
is `n_feval`, not an independent count.

The fix also changed how `grad_norm_final` is computed: `result.jac`
(scipy's own Jacobian at `x_final`, analytic or FD, whichever `solve`
actually used) replaces a fresh `problem.jacobian(x_final)` call, which
would otherwise call the analytic Jacobian regardless of what was
requested, defeating the fix's own guarantee, and would crash outright
once `problem.jacobian` can be `None`.

## Review response: addendum

A closer-check addendum to the same review round found a second Major
issue, the same root cause as the derivative-modes finding but live
today rather than latent: `trf`/`dogbox`'s declared `losses`
(`{"linear", "soft_l1", "huber", "cauchy", "arctan"}`) had no `config`
key reaching `least_squares`'s `loss=` parameter at all; `solve` always
ran with `scipy`'s own `"linear"` default regardless of what a caller
requested. Confirmed directly that `loss` is not cosmetic (`loss=
"huber"` takes a different number of evaluations than `"linear"` to
reach the same optimum on the trivial quadratic: 3 against 2) and that
`scipy` validates the `loss` name strictly (an unrecognised string raises
`ValueError` naming the actual accepted set), which is what makes the fix
verifiable without depending on evaluation counts.

Fixed the same way as `bounds`/`derivative_mode`: `loss=config.get(
"loss", "linear")` passed to `least_squares` unconditionally (the default
matches what every existing test already assumed, so nothing behind it
changes for a caller that never sets `loss`). `lm` rejecting a non-
`"linear"` loss is, like bounds, enforced by `scipy` itself
(`method='lm' supports only 'linear' loss function.`), not reimplemented.

Three new tests, the first two confirmed to fail against the pre-fix
code, the third a declarative check that was already passing (the
capability was correctly declared; only the plumbing behind it was
missing):

- `test_loss_config_reaches_scipy_for_bounded_methods`: an unrecognised
  loss name raises `ValueError`, which only happens if `solve` actually
  forwards it to `least_squares`; a `solve` that silently ignores
  `config["loss"]` would not raise here, exactly the bug this closes.
- `test_lm_rejects_a_non_linear_loss`: `loss="huber"` with `method="lm"`
  raises `ValueError`, matching `capabilities().methods["lm"].losses ==
  {"linear"}`.
- `test_capabilities_losses_match_which_methods_accept_non_linear_losses`:
  the declared values themselves.

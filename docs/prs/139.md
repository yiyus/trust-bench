# PR notes: Issue #139 - Persistent APL session: shared by study sweeps and the test suite, instead of one dyalogscript spawn per call

## Summary

Every `APLBackend.solve()`/`evaluate_problem()` call spawned a fresh
`dyalogscript` subprocess - 3-9s of interpreter startup, virtually all
overhead, not compute. Measured impact: ~15 minutes of pure spawn
overhead across 134 `trust-apl` rows in one real report run; ~3.3s/test
average across the ~90 APL-touching test functions in 18 files.

Replaced with one long-lived `dyalogscript` process per outer run (one
`trust-bench report` invocation, one pytest session), talking
newline-delimited JSON over stdin/stdout, synchronous
one-request-at-a-time - with no change to how any caller works: every
study's `sweep()`, every test file's `APLBackend()` construction, stays
exactly as it is today. Full design and the investigation behind every
transport decision: `docs/plans/persistent-apl-session.md`.

`run_harness.sh` (the existing one-shot, file-based interface) is left
completely untouched: `tests/backends_ext/apl/test_harness.py`'s own
20-test suite and this project's own established "confirm directly via
a raw `run_harness.sh` invocation" review convention both depend on it
working exactly as it always has. This change adds a new, parallel
entry point (`session.dyalog`/`run_session.sh`) rather than repurposing
the existing one; the two scripts share their combined-source-file list
via a new `backends_ext/apl/_sources.sh` fragment, so a new problem
family only needs registering once, not copied by hand into both.

## Investigation: two protocol bugs found and fixed before this worked correctly

Both confirmed directly with controlled, reproducible experiments -
not assumed from documentation, since none of this is documented
anywhere and the failure modes were each initially mysterious.

1. **`⍞` (APL's line-input primitive) silently mangles a long, piped,
   non-interactive line** past roughly 1-2KB - not a clean truncation,
   a genuine corruption (a 2000-character input line came back reporting
   length 583, not 2000 or any other consistent value). This broke the
   very first real exercise of the new session:
   `dimensionality(n=1000)`'s own request carries a 1000-element `x0`
   array. Fixed by never sending the request body itself over stdin:
   Python writes it to a small temp file and sends only that file's path
   (always short) as the stdin line; `session.dyalog` reads the path via
   `⍞` (safe) and the real request via `⎕NGET`, the same way the
   existing one-shot `run.dyalog` already does.
2. **`⎕←` wraps any single output past 32767 characters into several
   physical `\r`-terminated segments**, each one after the first padded
   with exactly 6 leading space characters that aren't part of the
   payload (confirmed byte-for-byte, no data loss - only where the `\r`
   lands and that padding). This is the *response* direction, where a
   file-based workaround isn't available - and it's worse than the first
   bug: an unhandled wrapped response didn't just fail its own request,
   it desynchronised every *subsequent* request in the same pytest
   process from message boundaries entirely, cascading into 20 unrelated
   test failures the first time it was hit
   (`dimensionality(n=1000)`'s own `evaluate_problem` response is a
   1000x1000 Hessian). Fixed by having `session.dyalog` announce the
   payload's own character length on its own short line first, so the
   Python reader knows exactly how many payload characters to expect and
   reassembles them correctly regardless of how many physical segments
   they arrive wrapped across.

An earlier, simpler version of this fix (RED-confirmed against the
unmodified per-call-spawn code, then implemented, then found broken by
these two bugs during the full-suite regression run) is not preserved
as a separate cycle here: both bugs were found and fixed within the same
implementation pass, before presenting anything for review, since
presenting a version known to corrupt large requests/responses would
not have been a meaningful review checkpoint.

## Other findings

- `_dyalog_version()` (called via `environment()` on every `solve()`)
  is its own ~3-7s subprocess spawn (`dyalog -version`), previously
  invisible under the old per-call harness cost. Left unfixed, it would
  have become the new dominant cost once the harness itself got fast,
  quietly defeating the whole point of this issue for the primary
  `solve()` path. Cached via `functools.lru_cache` - the version can't
  change within one process's lifetime.
- `_dyalog_version()` has a separate, pre-existing bug (unrelated to
  this issue, not fixed here): it searches `result.stdout` for the
  version string, but Dyalog writes that text to stderr, so it always
  silently returns `"unknown"`. Flagged for its own follow-up issue.

## Design (see docs/plans/persistent-apl-session.md for full detail)

- Module-level singleton session in `apl_backend.py`, lazily started,
  shared by every `APLBackend()` instance and `evaluate_problem()` call
  in the process - not a pytest fixture, which would need threading
  through 18+ test files and contradict "no change to any caller."
- `stdbuf -oL` (coreutils) forces line-buffered output: confirmed
  directly that dyalogscript fully buffers stdout when it's a pipe
  rather than a terminal, so without this a response never actually
  reaches the pipe until the buffer fills or the process exits.
- Reads happen in binary mode, splitting on `\r` (dyalogscript's actual
  line terminator) rather than Python's text-mode readline, which
  deadlocks trying to disambiguate a trailing bare `\r` from a possible
  `\r\n` while the interpreter is blocked waiting for the next request.
- Timeout (`select`-bounded) or a dead process: kill and discard the
  session so the *next* call gets a fresh one; the current call still
  returns today's familiar `_timeout_result(...)` shape. No automatic
  retry of the failed request.
- `atexit`-registered teardown (no `conftest.py`/pytest hooks exist in
  this project; none needed).

## Files changed

- New: `backends_ext/apl/session.dyalog`, `backends_ext/apl/run_session.sh`,
  `backends_ext/apl/_sources.sh`.
- `trust_bench/backends/apl_backend.py`: module-level session state,
  `_send_request`/`_read_response`/`_read_until_cr`, `atexit` teardown;
  `_dyalog_version` cached; `_run_harness`/`tempfile`-per-call/`_HARNESS`
  removed (replaced by the persistent path); `evaluate_problem`/`solve`
  call `_send_request` instead.
- `backends_ext/apl/run_harness.sh`: refactored to source the shared
  `_sources.sh` file list - behaviourally identical, verified via its
  own unchanged 19-test suite (`tests/backends_ext/apl/test_harness.py`).
- `tests/backends/test_apl_backend_timeout.py`: rewritten for the new
  transport (its old mechanism, monkeypatching `subprocess.run` to raise
  `TimeoutExpired`, doesn't apply to a persistent `Popen`+`select`
  design). Uses a harmless `cat` stub process in place of a real
  session for the timeout-shape assertions, preserving the file's
  original property of not requiring `dyalogscript` to be installed to
  run those two tests.
- `tests/backends/test_apl_session.py` (new): session reuse across
  independently-constructed `APLBackend()` instances and across
  `evaluate_problem`/`solve`; a killed session is transparently replaced
  by the next call.

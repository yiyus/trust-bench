# PR notes: Issue #128 - robust_loss study compares different loss tuning constants, not just different libraries

## Summary

`robust_loss.py`'s scipy-vs-trust-apl comparison for `huber`/`cauchy` doesn't
control for each library's tuning constant: trust's `Loss.apln`
(`backends_ext/apl/trust/APLSource/Loss.apln`) bakes in the textbook
~95%-asymptotic-efficiency constants (`huber=1.345`, `cauchy=2.385`,
`softl1=1`, `arctan=1`), while `scipy_backend.py` never passes `f_scale` to
`least_squares`, so every scipy run uses its own fixed default of `1.0`.
The measured precision gap in `robust_loss.csv` tracks this tuning-constant
mismatch, not a genuine robustness difference between the two libraries.

Adds `RunConfig.f_scale` (mirroring the existing `x_scale` field's shape)
and wires it through: `scipy_backend.py` forwards it to `least_squares`;
`apl_backend.py` rejects it explicitly for every method, since trust has
no per-request knob to override either its fixed per-loss constant or its
MAD-based auto-scaling; `robust_loss.py`'s `scipy_loss_precision` passes
trust's own constants as `f_scale` for the scipy backend only, leaving
trust-apl's calls unaffected.

## RED cycle (test-first)

**Test surface**

- `tests/backends/test_scipy_f_scale.py` (new): `f_scale` reaches
  `least_squares` (pinned against a direct scipy call using the same
  value); left `None`, scipy's own default is unaffected (regression
  coverage for the no-op case).
- `tests/backends/test_apl_f_scale.py` (new): every method (`lm`/`BFGS`/
  `trust-exact`) rejects an explicit `f_scale`.
- `tests/studies/test_robust_loss.py` (extended): `_F_SCALE_FOR_LOSS`
  pins trust's own constants exactly; `scipy_loss_precision`'s `huber`
  run (a loss where trust's constant, `1.345`, actually differs from
  scipy's default, `1.0` - `soft_l1`/`arctan` wouldn't distinguish
  "wired correctly" from "coincidentally already matching") matches a
  direct scipy call using the same `f_scale`.

# PR notes: Issue #8 - Problem dataclass and registry (problem.py, registry.py)

## Summary

Adds `trust_bench.core.problem` (the `Problem` and `Optimum` dataclasses)
and `trust_bench.core.registry` (a `Registry` class for lookup by id and
tag). Part of Epic #7 (Phase 2), see `docs/plans/trust-bench.md`,
Section 4.1 and Section 10 (Phase 2).

## RED cycle (test-first)

**Test surface**

`tests/unit/test_problem.py`:

- `test_problem_is_immutable`

`tests/unit/test_registry.py`:

- `test_get_returns_the_registered_problem_by_id`
- `test_by_tag_returns_the_registered_problems_with_that_tag`
- `test_register_rejects_a_problem_without_a_callable_residual`

**Coverage table**

| Acceptance criterion | Test(s) |
|---|---|
| registry lookup by `id` and by `tag` returns the expected `Problem` instances | `test_get_returns_the_registered_problem_by_id`, `test_by_tag_returns_the_registered_problems_with_that_tag` |
| registering a `Problem` without a `residual` callable is rejected | `test_register_rejects_a_problem_without_a_callable_residual` |

`test_problem_is_immutable` is not a literal reading of either criterion; it
checks Section 4.1's explicit `@dataclass(frozen=True)` contract, which
neither criterion covers directly.

**Confirmed failure**

```
ModuleNotFoundError: No module named 'trust_bench.core.problem'
```

Same "not implemented" collection-error class as issues #2, #3, #5, and #6.

**Design notes**

- Section 4.1 defines `Problem.optima: list[Optimum]` but never defines
  `Optimum` itself anywhere in the plan. Issue #8 is the natural place to
  define it, since `Problem`'s own field type references it and no other
  issue owns `problem.py`. Its comment ("known solutions: x*, cost*, basin
  notes") maps to `x_star`, `cost_star`, `basin_notes` (the last given a
  default of `""`, since not every known optimum needs a note).
- `residual`'s type is `Callable`, not `Callable | None` (Section 4.1), and
  dataclasses do not validate types at construction. AC2 says *registering*
  is rejected, not *constructing*, so the callable check lives in
  `Registry.register()`, not `Problem.__init__`: constructing a `Problem`
  with `residual=None` succeeds (a dataclass is just a data container),
  and only registering it raises `TypeError`.
- `Registry` is a class, not a module-level singleton dict, so tests can
  each create their own instance with no shared state (`CLAUDE.md`'s test
  independence rule). A single shared catalog that canonical problems
  register into (issue #9) is a `Registry()` instance exposed once that
  module needs it, not something issue #8 needs to pre-build speculatively.
- Fixture `Problem` construction is duplicated (not shared via
  `conftest.py`) across the two test files: it is a dozen lines used by
  four tests total, and every future test needing a `Problem` fixture will
  likely want a real, non-trivial one built from actual analytic functions
  (issue #9), not this dummy-scaffolding shape, so it is not clearly the
  same abstraction to share yet.

**Anticipated implementation order**

1. `trust_bench/core/problem.py`: `Optimum`, `Problem`.
2. `trust_bench/core/registry.py`: `Registry`.

## Review response

`docs/reviews/8.md`'s Minor finding: the callable-rejection test only
exercised `residual=None`, which an `is None` check (narrower than
`not callable(residual)`) would also satisfy, without forcing the intended
`callable()` check the design notes describe.

Fixed by parametrising
`test_register_rejects_a_problem_without_a_callable_residual` over
`[None, "not callable", 42]`, so the test now fails against an `is None`
check as soon as a non-`None`, non-callable value is registered, forcing
the broader `callable()` check the acceptance criterion's wording implies.

The Note (no test for `Registry.get()` on an unregistered id) is left as
is: the review itself marks it as requiring no action, since neither
acceptance criterion covers it.

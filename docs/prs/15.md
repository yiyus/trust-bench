**Design notes**

- **"Every registered backend" has no concrete backend to test against
  yet.** SciPy's backend is issues #16/#17, not built. Rather than defer
  the contract tests until a real backend exists (which would leave the
  contract itself unverified, and mean the SciPy backend's own PR would
  need to author its contract tests from scratch under time pressure), a
  minimal `Backend` implementation is written now, in `tests/`, purely to
  prove the contract itself is well-defined and testable. Real backends
  are added to `BACKENDS` alongside it going forward, not in place of it:
  a toy implementation deliberately kept simple continues to exercise the
  ABC boundary itself even once real backends also do.
- **`config` is a plain `dict`, not `RunConfig`.** Same forward-reference
  situation issues #6 and #12 already hit: `RunConfig` is issue #19's
  deliverable (Phase 5), not built yet. `Backend.solve`'s signature types
  it `Any`; the toy backend reads `config["max_iter"]` as its own ad-hoc,
  test-only convention, not a preview of `RunConfig`'s real shape.
- **"Monotone across iterations" is read as monotone in `max_iter` across
  separate calls, not a per-iteration trace.** `RunResult` has no field
  recording eval counts at each iteration, only their final totals
  (`n_feval`, `n_jeval`, `n_heval`) and, separately, `trace` for *iterates*
  (Section 4.3), not eval counts. The only way this criterion is testable
  against `RunResult`'s actual schema is by calling `solve` several times
  with increasing `max_iter` on a problem that does not converge within
  the smallest of them, and checking the reported final counts do not
  decrease. Verified numerically before writing the test: the toy
  backend's fixed step size (`0.3`) on the canonical `quadratic` problem
  from its `standard` start converges at exactly iteration 40 (`gtol=
  1e-6`), so `max_iter ∈ {5, 15, 30}` all stop early with strictly
  increasing eval counts (`5, 15, 30` exactly, one residual and one
  Jacobian evaluation per iteration), and `max_iter=45` reaches
  `CONVERGED` at 40 evaluations, giving the "well-formed result" test a
  genuine convergent case, not only a `MAX_ITER` one.
- **Capability-method consistency is checked at the method-name level
  only** (every declared method solves without an "unknown method"
  error; an undeclared name is rejected), not per-field (`bounds`,
  `analytic_hessian`). The toy backend has one method with no bounds
  support to contrast against, so a `bounds`-specific consistency check
  would have nothing real to verify yet; the SciPy backend (`lm` rejects
  bounds, `trf`/`dogbox` accept them, Section 4.2) is where that
  distinction first has two concrete methods to compare, and is better
  placed to add the stronger check when it does.

# PR notes: Issue #78 - APLBackend: support bounds and finite-difference derivatives

## Summary

Extends `backends_ext/apl/solve.dyalog` and `APLBackend` to genuinely
support box constraints (`trust`'s Coleman-Li interior-point method) and
finite-difference derivatives (`trust`'s `Eval` operator), instead of
declaring `bounds=False`/`derivative_modes={"analytic"}` and rejecting
anything else. Follow-up to #37.

  `test_solves_a_bounded_problem_and_stops_at_the_active_boundary`,

**Confirmed RED**

- The two harness-level tests fail on real assertions, not crashes:
  `bounds` is silently ignored today, so the "bounded" request converges
  to the true unconstrained optimum `(≈0, ≈0)` instead of the boundary
  `(0.5, 0)`; `derivative_mode` is silently ignored, so the
  finite-difference request uses the same `n_feval` (6) as the analytic
  one instead of more.
- The three backend-level tests fail because `APLBackend.solve` still
  raises `ValueError` for `bounds`/`derivative_mode="finite-difference"`
  (the #37 rejection logic), and `capabilities()` still declares
  `bounds=False`.
- `make test`: 219 passed, 26 deselected (up from 21; 5 new). `make
  lint`: clean.

**Design notes**

*Verified both mechanisms directly against `trust` before writing
assertions, not just Python's shape.* Ran `Quadratic Min` three ways via
`dyalogscript`: unconstrained (converges to `(≈0,≈0)`, cost `1.56e-16`);
bounded with `lower←0.5 ¯10` (converges to exactly `x1=0.5`, cost
`0.125`, matching `0.5×0.5²` — the constrained optimum sits exactly on
the boundary, as `bounded.py`'s own `_ACTIVE_BOUNDS` docstring already
states for this problem); and with the evaluation function returning
only the residual, no Jacobian, wrapped in a call counter (converges to
the same precision using `15` calls, vs. `6` for the analytic case,
since `Eval`'s finite-difference estimate needs extra evaluations per
Jacobian, not one). The RED assertions' tolerances and expected
directions come from these real numbers, not a guess at what "should"
happen.

*`quadratic`, not `rosenbrock`, for the bounds test.* Matches
`bounded.py`'s and `capability_matrix.py`'s own choice of probe problem
for exactly this reason: its unconstrained optimum `(0,0)` becomes
infeasible under a lower bound on `x1`, giving an unambiguous, exactly
computable constrained optimum to assert against, rather than a bound
that might or might not bind depending on the solve path.

*Not attempting to reuse `bounded.py`/`derivative_source.py` directly
with `backends=[..., APLBackend()]`.* Both studies hard-code SciPy
method names in their own `METHODS` (`bounded.py`: `["trf", "dogbox"]`;
`derivative_source.py`: `["lm", "trf", "dogbox"]`), neither of which is
`APLBackend`'s only method, `"lm"`, for the bounds case, and
`derivative_source.py` doesn't catch the `ValueError` `bounded.py` does
for an unsupported method. Running either with an unfiltered
`backends=` override would either silently test nothing for APL
(`bounded.py`, since `ValueError` is caught for `"trf"`/`"dogbox"`, which
were never APL's method anyway) or crash outright
(`derivative_source.py`, once it reaches `"trf"` for APL). Extending
those studies to run a genuine three-way (or eventually four-way, once
Julia lands) comparison is exactly the kind of thing #77 (CLI backend
selection) should make possible generically, rather than papering over
it here with one-off filtering.

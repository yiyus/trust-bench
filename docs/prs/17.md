# PR notes: Issue #17 - SciPy backend: minimize methods (BFGS, L-BFGS-B, Newton-CG, trust-*)

## Summary

Extends `SciPyBackend` (issue #16) with five `minimize`-based methods:
`BFGS`, `L-BFGS-B`, `Newton-CG`, `trust-exact`, `trust-constr`. Part of
Epic #14 (Phase 4), see `docs/plans/trust-bench.md`, Section 4.2 and
Section 10 (Phase 4).

**Coverage table**

| Acceptance criterion | Test(s) |
|---|---|
| each method solves a trivial quadratic to the known optimum within tolerance | `test_method_solves_the_trivial_quadratic_to_the_known_optimum` |
| only `L-BFGS-B`/`trust-constr` accept box constraints, matching `capabilities().methods` | `test_capabilities_bounds_flag_matches_which_methods_accept_bounds`, `test_bounded_methods_accept_and_respect_box_constraints`, `test_unbounded_methods_reject_box_constraints` |

The remaining tests (losses, derivative modes, finite-difference
behaviour) are not a reading of either criterion; they exist to avoid
repeating issue #16's review findings, both of which were the same
shape: a declared `Capabilities` field with no `solve()` behaviour behind
it. Verified this issue's equivalent claims directly against `scipy`
*before* writing them, rather than assume the same shape of gap would not
recur (see design notes).

**Confirmed failure**

```
ValueError: scipy has no method 'BFGS'
```
(and the same for `L-BFGS-B`, `Newton-CG`, `trust-exact`, `trust-constr`)

Same "not implemented" failure class as every prior issue, though this
time via a raised exception rather than a collection error, since
`SciPyBackend` already exists (issue #16); the new methods just are not
in `capabilities().methods` yet, so `solve()`'s own existing "unknown
method" guard is what actually fires. 14 of 19 tests in the new file fail
this way.

**A methodological caveat, disclosed rather than left for review to
catch:** the other 5 tests in the new file (`test_unbounded_methods_
reject_box_constraints`, `test_finite_difference_derivative_mode_is_
rejected_where_scipy_requires_an_analytic_jacobian`) currently pass, but
vacuously: they assert `pytest.raises(ValueError)`, and calling `solve`
with an unregistered method already raises `ValueError` (the same
"unknown method" guard), for the wrong reason, not because the
bounds/derivative-mode rejection logic these tests are actually meant to
pin down exists yet. This is the same shape of gap `docs/reviews/9.md`
flagged for an empty `probe_points` list passing a loop vacuously; the
fix is the same in spirit (the assertion needs the real mechanism behind
it, which the GREEN cycle provides), but there is no equivalent of
"assert the list is non-empty first" for "assert this exception came from
the right guard", so the honest thing is to name it here rather than
claim a clean RED.

**Design notes**

- **Method set**: the issue title's `trust-*` wildcard is read as two
  concrete variants, not every `trust-*` method `scipy` offers.
  `trust-exact` is included because the prototype's own methodology
  (`prototype/harness.py`'s `run_minimize(..., method="trust-exact")`)
  already uses it as the true-Newton comparator Section 9 item 2 studies;
  `trust-constr` is included because the acceptance criterion names it
  directly as a bounds-accepting method. `trust-ncg`/`trust-krylov` (also
  verified to converge on the trivial quadratic) are left out: no
  acceptance criterion or plan section currently asks for them, and
  adding methods nobody has asked for yet is exactly the shape of
  unrequested addition this project avoids.
- **`kind="residuals"`, not `"scalar"`, despite Section 4.1's own comment**
  (`Problem.kind`: "dispatch: LM/Newton vs BFGS", pairing BFGS with
  `"scalar"`). Every problem in the catalog today is `kind="residuals"`
  (issue #9's own decision), and these five methods are wired to solve
  those same problems via an adapter (`0.5*||r||^2` as the objective,
  `J^T r` as the gradient, `problem.hessian` passed through directly,
  since `Problem.hessian`'s own docstring already defines it as "full
  Hessian of 0.5||r||^2", matching `minimize`'s convention with no
  transformation needed). Declaring `"scalar"` would describe a pairing
  no problem in the catalog exercises; a genuinely `kind="scalar"`
  problem, when one exists, may need its own dispatch path in `solve`
  rather than reusing this adapter unchanged, since Section 4.1's comment
  for `Problem.residual` says a scalar-kind problem's `residual` already
  *is* the objective, needing no `0.5*||r||^2` wrapping at all.
- **`scipy.optimize.minimize` does not itself reject bounds for methods
  that cannot use them.** Verified directly: unlike `least_squares`
  (issue #16), passing `bounds=` to `BFGS`/`Newton-CG`/`trust-exact`
  produces a `RuntimeWarning: Method ... cannot handle bounds` and
  silently solves unconstrained, rather than raising. `solve` therefore
  checks `capabilities().methods[method].bounds` itself before calling
  `minimize`, the explicit check issue #16 did not need for
  `least_squares` because `scipy` already enforced it there.
- **`derivative_modes` differs per method, verified directly rather than
  assumed uniform** (learning applied directly from issue #16's Major
  finding): `BFGS`/`L-BFGS-B`/`trust-constr` accept `jac="2-point"`;
  `Newton-CG`/`trust-exact` raise `Jacobian is required for ... method`
  the moment finite differences are requested for the gradient, with no
  fallback. Section 7's "derivative source held constant" principle (a
  comparison fixes analytic-or-FD for every backend, never mixed) is read
  as also meaning *within* one `solve` call: `Newton-CG` can technically
  take an FD Hessian with an analytic gradient in isolation (verified),
  but `derivative_modes` does not expose that partial mix, since a
  half-FD, half-analytic run is not the "analytic" or "finite-difference"
  choice Section 7 means a comparison to fix.
- **`result.jac` is not the objective gradient for every method.**
  `trust-constr`'s `OptimizeResult.jac` is empty (`trust-constr` is a
  general nonlinear-constrained solver where `.jac` means the constraint
  Jacobian); its gradient is `.grad` instead. The other four methods use
  `.jac`. `grad_norm_final` reads whichever one is actually populated,
  the same reasoning as issue #16's fix to use `result.jac` over a fresh
  `problem.jacobian` call, extended to cover this method-specific naming
  difference too.
- **`n_iter=result.nit`, not `result.nfev`** (a deviation from issue #16's
  own `n_iter=n_feval` for `least_squares`, whose `OptimizeResult` has no
  iteration count): `minimize`'s `OptimizeResult` does expose `nit`, a
  real per-iteration count, so `n_iter` here is not a relabelled
  evaluation count.
- **MAX_ITER detection**: `minimize`'s `status` codes are not consistent
  across methods (verified: `trust-constr` uses a different scheme from
  the other four), so `status`/`success` alone cannot distinguish "hit
  the iteration limit" from "failed for some other reason" the way
  `least_squares`'s single numeric `status` could. Verified directly
  that `result.nit` reaches exactly the requested `max_iter` when the
  limit is what stopped the run, for all five methods; `MAX_ITER` is
  reported when `not result.success and result.nit >= max_iter`,
  `FAILED` otherwise.

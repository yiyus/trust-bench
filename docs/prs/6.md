# PR notes: Issue #6 - Basin-of-attraction rate aggregator

## Summary

Adds `basin_rate()` to `trust_bench.core.metrics`: the fraction of a set of
per-run outcomes whose distance to a known optimum is within tolerance.
Part of Epic #4 (Phase 1), see `docs/plans/trust-bench.md`, Section 6
(Tier 1) and Section 10 (Phase 1).

**Design notes: `RunResult` does not exist yet**

The issue text (and `docs/plans/trust-bench.md`, Section 10 Phase 1) says
"a set of `RunResult`s". `RunResult` is Phase 3's deliverable (issue #12),
deliberately scheduled after Phase 1 in Section 10's ordering rationale, so
it is not importable from this issue. Hard-depending on it here would
either block on issue #12 out of the plan's own stated order or require
inventing and later reconciling a stub.

`basin_rate()` instead takes a plain iterable of `dist_to_opt` values
(`float | None`, matching the field `RunResult.dist_to_opt` will carry per
Section 4.3): a run counts as having reached a known optimum when its
distance is `<= tol`, and `None` (a run with no known optimum to compare
against, e.g. a failed run) counts as not reached but still counts in the
denominator, since the rate is a fraction of every attempted start, not
only the ones that produced a distance. Once issue #12 lands, a caller
passes `[r.dist_to_opt for r in results]`; `metrics.py` gains no dependency
on `result.py` either way, matching the plan's "language-agnostic core"
principle (Section 2) that pure functions in `metrics.py` should not carry
cross-module type dependencies.

`tol` has no default: the natural scale of `dist_to_opt` varies per problem
(Section 9 item 6 sweeps parameters across many orders of magnitude), so a
fixed numeric default would silently misclassify runs on problems whose
scale it does not suit. Callers (the studies in Phase 6) choose a tolerance
appropriate to the problem being studied.

**Anticipated implementation order**

1. `trust_bench/core/metrics.py`: add `basin_rate()`.

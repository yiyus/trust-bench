# PR notes: Issue #130 - Include scalar_cost in the parity scatter and capability-frontier data

## Summary

Decision: `scalar_cost` belongs in neither `parity_frame` nor
`frontier_panels`, for two independent reasons, both confirmed directly
rather than assumed. No production code change - documentation only,
matching this issue's own acceptance-criteria fallback ("if not, close
with the reasoning documented", the same resolution `robust_loss` got
in #123 for the parity pool, for a different reason).

## Investigation

**Parity scatter.** Shape-wise, `scalar_cost` fits: its rows are already
`dist_to_opt`/`status` per `(problem_id, method, backend)`, the same
shape `baseline`/`typical`/`bounded` already pool - unlike `robust_loss`'s
own distance-to-true-parameters shape. The actual blocker is `trust-apl`
itself, confirmed directly: pooling `scalar_cost`'s real
`results_to_dataframe` output (`scipy`: `BFGS`/`L-BFGS-B` both converge
normally; `trust-apl`: only `BFGS` is attempted at all, since
`APLBackend` doesn't declare `"L-BFGS-B"`, and that one row reports
`ERROR`, `"Unknown problem_id"`, for both `rastrigin` and `cauchy_mle`)
and pivoting it the same way `parity_frame` pivots the other three
studies raises `_pivot_by_backend`'s own missing-backend guard
unconditionally: `ValueError: no results for backend(s) trust-apl`.
There is no `trust-apl` side to plot at all - not a partial coverage gap
`results_to_dataframe`'s `UNSUPPORTED`/`ERROR` labelling (#125) could
paper over, a total one.

**Capability frontier.** `scalar_cost` sweeps exactly two fixed problems
with no swept difficulty parameter at all, unlike every other panel
(`ill_conditioning`'s `kappa`, `scaling`'s `scale`, `dimensionality`'s
`n`, `large_residual`'s `rho`, `robust_loss`'s `fraction`) - there is no
natural x-axis for a small-multiples line here regardless of backend
coverage. This was anticipated directly in the issue's own acceptance
criteria and confirmed by inspection: `scalar_cost.sweep` has no
parameter analogous to the other five studies' own sweep argument.

Both exclusions are permanent under `trust-apl`'s current capability
set, not a reporting-layer gap to close by more wiring here: the parity
blocker specifically would need `trust-apl` to gain a genuine
Jacobian-free scalar evaluator first - the real, nontrivial capability
gap #121 already flagged and #136 worked around at the report level,
not something this issue's own reporting/aggregation scope can or
should resolve.

## Changes

`trust_bench/reporting/cross_study.py`: `parity_frame`'s and
`frontier_panels`'s own docstrings extended with the reasoning above,
matching how `robust_loss`'s parity exclusion is already documented
inline rather than only in `docs/methodology.md`.

`docs/methodology.md`: new "scalar_cost and the parity/frontier pool"
section, documenting both exclusions and the direct confirmation each
rests on.

No test changes: nothing in `cross_study.py`'s observable behaviour
changed, only its own explanatory comments.

# PR notes: Issue #138 - Instrument RunResult.timing: nothing actually measures Tier-3 performance yet

## Summary

`RunResult.timing`/`TimingStats` already exist and match the timing
policy `docs/plans/trust-bench.md` Section 7 specifies (warm-up runs, N
repetitions, pinned BLAS thread count, median+MAD), but both backends
hardcode `timing=None`. Wires up real measurement for both.

Design decisions confirmed with the user before implementation (not
specified by the issue or `RunConfig`):

- `threadpoolctl` added as a new dependency for pinning scipy/NumPy's
  BLAS thread count at runtime - the issue's own text already names it
  as the example mechanism.
- `warmup=1`, `n_reps=5` as fixed defaults (not exposed via
  `RunConfig`) - matches the timing policy's own wording ("warm-up
  run(s)") without adding new user-facing configuration surface the
  issue didn't ask for.

Per #139 landing first (explicitly anticipated by this issue's own
"Related" section: "if #139 lands first, use it here rather than
building a narrower one-off batching just for repetitions"), the N
repeated `trust-apl` measurement calls are now just N ordinary
`_send_request` round trips through the already-persistent session
(a few ms each) - no special batching mechanism needed for that part.
What #139 does *not* solve on its own is excluding IPC/JSON-transfer
overhead from the *reported number itself*: `solve.dyalog` measures
its own compute time via `⎕AI[2]` (confirmed directly: computation
time in milliseconds) around the `Min` call, and reports it back in
the response (`solve_ms`) - `apl_backend.py` reads that value rather
than wall-clocking the round trip from Python.

## RED cycle (test-first)

**Test surface**

- `tests/unit/test_metrics.py` (extended): a new `mad` (median absolute
  deviation, 1.4826-scaled - the same normal-consistent constant this
  project already uses in `robust_loss.py`'s `irls_tukey`) matches a
  hand-computed value and is zero for identical inputs.
- `tests/unit/test_timing.py` (new): a shared `summarize(samples, ...)`
  helper builds a real `TimingStats` from repeated timing samples;
  `WARMUP`/`N_REPS` match the confirmed defaults (1, 5).
- `tests/backends/test_scipy_timing.py` (new): `solve()` populates real
  `TimingStats` (not `None`); `threadpool_limits` is actually called
  with a thread-count limit, not just recorded after the fact; the
  returned solution is still genuine despite the repeated measurement
  calls (not a stale warm-up artifact).
- `tests/backends/test_apl_timing.py` (new, `dyalogscript`-gated):
  `solve()` populates real `TimingStats`; the reported median is no
  larger than a real external wall-clock measurement of the same
  warmed-up call (confirms IPC/transfer overhead isn't being counted as
  solve time); the returned solution is still genuine.

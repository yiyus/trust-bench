# PR notes: Issue #104 - Add a typical-problems study to baseline engine behaviour on non-adversarial fits

## Summary

Every existing study is either narrowly scoped to `"lm"` on classic
adversarial MGH test problems (`baseline`), or a difficulty family
built to probe one specific failure mode (`ill_conditioning`,
`dimensionality`, `large_residual`). There is no study answering how
the engines compare on an unremarkable, everyday fitting problem, run
through every method rather than just `lm`.

Confirmed directly the gap this closes matters: `trust-apl`'s
`trust-exact` diverges (`MAX_ITER`, wildly wrong `x_final`) from the
ordinary "standard" start on two of the four new problems
(`noisy_expdec`, `gaussian_peak`), because their true Hessian is
indefinite away from the optimum - while scipy's own `trust-exact`
converges cleanly on the exact same problems from the exact same
start (its trust-region safeguards handle an indefinite Hessian at the
starting point; `trust`'s own engine does not, at least not here).
Neither engine is "the reliable one" in general - this is a real,
measured capability difference the batch is meant to surface, not a
bug in this port.

Four problems, chosen from ordinary curve-fitting practice, not to
stress a specific failure mode:

- `noisy_expdec`: the existing `expdec` model with fixed Gaussian noise
  added to the target data, so the residual at the optimum isn't
  exactly zero.
- `logistic`: a 4-parameter (`A`, `B`, `C`, `D`) logistic/sigmoid
  dose-response curve over 9 log-spaced doses.
- `michaelis_menten`: 2-parameter (`Vmax`, `Km`) enzyme kinetics over 8
  substrate concentrations.
- `gaussian_peak`: a 3-parameter (amplitude, centre, width) Gaussian
  peak fit over 30 points.

Each gets a native APL implementation and cross-language parity tests,
following the pattern already established for the canonical problems.
A new `typical` study runs every problem through every method each
backend declares support for, from its own ordinary start, and is
wired into `cli.py`'s `STUDIES`/`report` command.

**Design notes**

*Every Jacobian/Hessian formula derived symbolically (`sympy`), not by
hand, then cross-checked against central finite differences before
writing any problem file* - the 4-parameter logistic model's full 4×4
Hessian in particular is involved enough (`cosh`/`sinh` terms from
differentiating `(x/C)^B`) that a hand-derivation error would have been
easy to miss without this. Caught and fixed one real sign error this
way (a `(mu-x)` vs `(x-mu)` mix-up in the Gaussian peak's
`∂²f/∂mu∂sigma` term) before it reached any file.

*Each problem's optimum is computed numerically (`scipy.optimize.
least_squares`) against noisy synthetic data, not closed-form* -
matching `large_residual.py`'s own established precedent for a problem
with no analytic optimum. `logistic`/`gaussian_peak` needed a tighter
explicit `ftol`/`xtol`/`gtol` than scipy's own default to bring the
gradient at the computed optimum within the test's tolerance, given
their parameter scale (`D` reaches ~100 for `logistic`).

*Fixed, reproducible noise* (`numpy.random.default_rng` with a fixed
seed, stable across numpy versions for a given seed) rather than a
literal noise array: transparent about how the data was generated,
and exactly reproducible.

*`typical.sweep()` has no skip-guard*, unlike `ill_conditioning.py`'s
or `bounded.py`'s: it iterates every method a backend's own
`capabilities().methods` declares directly, rather than filtering a
fixed candidate list against backend support - there is no "backend
doesn't support this method" case to skip, since the method list
itself comes from the backend.

## GREEN cycle (implementation)

`trust_bench/problems/{noisy_expdec,logistic,michaelis_menten,
gaussian_peak}.py` (new): each problem's residual/Jacobian/Hessian and
a numerically-computed optimum, per the design notes above.
`trust_bench/problems/__init__.py`: new `TYPICAL_PROBLEMS` list.

`backends_ext/apl/problems/{noisy_expdec,logistic,michaelis_menten,
gaussian_peak}.dyalog` (new): each problem's data embedded as a
literal vector, and dyadic-shaped residual/Jacobian/Hessian functions
mirroring the existing canonical problems' pattern. Verified against
the Python reference at every probe point via `dyalogscript` before
writing any test. `dispatch.dyalog`/`run_harness.sh`: the four new
names registered alongside the existing seven canonical problems (the
plain `NameFor` path, not the parametrised-family one, since these
problems take no external parameter).

`trust_bench/studies/typical.py` (new): `sweep()`, per the design
notes. `cli.py`: `_write_typical`, registered in `STUDIES`.

One RED-stage test-naming collision found and fixed during GREEN,
unrelated to any implementation logic: `tests/studies/test_typical.py`
and `tests/problems/test_typical.py` share a basename, which pytest's
default rootdir-based import mode rejects when neither test directory
has an `__init__.py` (`import file mismatch`). Renamed the studies
file to `tests/studies/test_typical_study.py`.

One test tolerance adjustment found and fixed during GREEN:
`logistic`/`gaussian_peak`'s computed optimum didn't quite satisfy the
known-optimum invariant's `atol=1e-6` (gradient `~1.7e-5`/`~1e-6`
respectively) - not a formula bug (parity against finite differences
already matched to `1e-6`/`1e-3`), but scipy's own default `least_
squares` tolerances leaving more numerical slack at these problems'
larger parameter scale. Tightened the optimum computation itself
(`ftol=xtol=gtol=1e-14`) rather than only loosening the test.

Verified: the three affected test files - 26 passed. Full suite
`pytest -q` (dyalogscript available): 364 passed, 0 failed, no
regressions. `ruff check .`: clean.

# PR notes: Issue #136 - trust-bench report crashes entirely when a study has zero rows for a selected backend

## Summary

`trust-bench report --backends trust-apl scipy` crashes the entire
report generation - not just `scalar_cost`, the one study affected -
because `_check_backend_coverage` treats any zero-coverage backend as
fatal. `trust-apl` has no evaluator wired up for `scalar_cost`'s
Jacobian-free scalar objectives at all (per #121/#130): a known,
permanent gap, not a bug matching the "genuine crash" symptom
`_check_backend_coverage`'s own docstring says it exists to catch.

`run_report`'s own per-study loop now catches the `ValueError`
`_check_backend_coverage` (or any other per-study coverage check) raises,
collects it against the study's name, and continues to the next study -
matching this issue's own first-preference proposal. If every selected
study fails this way, `run_report` still raises a clear aggregate error
(nothing to report at all is a genuine failure, not a partial success) -
preserving the existing single-study-selection crash behaviour
unchanged. `main()` prints a `Skipped <study>: <reason>` line per
skipped study before the usual "Report artefacts written to" line.

## Post-implementation-review fix

The implementation review found a Major gap: catching bare `ValueError`
in `run_report`'s per-study loop absorbs *any* `ValueError` a study's
write path raises, not just `_check_backend_coverage`'s - an unrelated
bug elsewhere (an unsupported `x_scale`/`loss`/`derivative_mode`
combination, an invalid problem-family parameter) would silently land in
`skipped` alongside genuine coverage gaps instead of crashing the report,
undermining `_check_backend_coverage`'s own reason for existing.

Fixed by introducing `CoverageError(ValueError)`, raised only by
`_check_backend_coverage`; `run_report`'s catch is narrowed to
`CoverageError` specifically. Existing tests expecting a plain
`ValueError` (`test_run_report_raises_clearly_when_a_study_does_not_
support_the_selected_backend`,
`test_run_report_raises_clearly_when_parity_scatter_is_explicitly_
selected_with_one_backend`) needed no changes, since `CoverageError`
*is* a `ValueError` and `parity_frame`'s own backend-count check was
never routed through `_check_backend_coverage` in the first place.

Added `test_run_report_does_not_absorb_an_unrelated_value_error_as_a_
coverage_gap`, mirroring the review's own verification: a broken study
writer raising a plain `ValueError` alongside a second, healthy study
selected in the same run - confirmed genuinely RED against the
bare-`ValueError` catch (silently absorbed, no exception raised) and
GREEN against the `CoverageError`-narrowed one. A single-study
selection alone doesn't distinguish "properly propagated" from
"absorbed, then re-raised once everything failed" - an earlier draft
of this test picked exactly that scenario and passed against both the
broken and fixed code, a false-positive caught by re-running it against
the pre-fix `cli.py` directly rather than trusting it on the fixed code
alone.

Verified: `tests/integration/test_cli.py -m "not slow"` - 36 passed
(the new absorption-guard test plus every previous one, unaffected).
`ruff check`: clean. Full suite queued next.

# PR notes: Issue #95 - Document trust's BFGS robustness gap; stop trust-bench's status classification from conflating a stall with convergence

## Summary

Two related but separate problems, both surfaced by the ill_conditioning
and dimensionality studies once `trust-apl` was wired into their sweeps
(#83/#84/#90):

- trust's own BFGS engine is genuinely less robust to ill-conditioning
  than scipy's BFGS - a real, measured capability gap worth recording.
- `backends_ext/apl/solve.dyalog`'s status classification reports
  `CONVERGED` for every termination that isn't `MAX_ITER`/`FAILED`, but
  `Newton.aplo`'s own termination check distinguishes two different
  "otherwise" conditions: `tc>c` (cost genuinely below tolerance, a real
  guarantee) and `(r>0)∧tr>r` (the relative-change metric stalled, which
  only means the last accepted step stopped changing much). The harness
  collapses both into `CONVERGED`, which is a trust-bench overclaim, not
  a behaviour of trust itself (trust's own API never asserts convergence
  - see `docs/methodology.md`'s existing tolerance-mapping section).

Confirmed directly (`ill_conditioning.sweep(backends=[SciPyBackend(),
APLBackend()])`): at kappa=1e4, `trust-apl BFGS` reports `CONVERGED` with
`cost_final≈1.1e6`; at kappa=1e7/1e8, `trust-apl trust-exact` reports
`CONVERGED` with `cost_final≈6.96` (`dist_to_opt≈3.73`), while scipy's
own BFGS/trust-exact stay near machine precision or genuinely report
`FAILED` in the same range. Both are the same underlying harness bug:
the relative-change criterion firing without the cost criterion also
holding.

**Design notes**

*A new `RunStatus.STALLED`, not repurposing `DIVERGED`.* `RunStatus`
already has an unused `DIVERGED` member, but "diverged" implies the
solver moved away from a solution; the relative-change metric firing
means the opposite - the solver stopped moving, without any guarantee
it's near the optimum. `STALLED` names the actual condition.

*Fix lives entirely in `solve.dyalog`'s status classification, not in
`Newton.aplo`.* `Newton.aplo` never claims convergence itself; it
returns `iter`/`cost`/`rel`/`dnorm`/`p` and lets the caller interpret
which of its four termination conditions fired. `solve.dyalog` already
has `r.cost` and `cfg.tolc` (the same tolerance value `Newton.aplo`
compares `c` against) available when it builds the response - the fix
is an explicit `r.cost<cfg.tolc` check before falling through to
`CONVERGED`, with the remaining case (relative-change fired without the
cost criterion) reported as `STALLED`.

*Documentation of the BFGS robustness gap belongs in
`docs/methodology.md`, alongside the existing tolerance-mapping
section*, since it is more capability-boundary context than a new
concern. Confirmed directly (see Summary): `trust-apl BFGS` degrades
from kappa=1e4 (cost in the millions, previously mislabelled
`CONVERGED`, will report `STALLED` once the classification fix lands)
through kappa=1e7/1e8 (`FAILED`, cost in the trillions), while scipy's
own BFGS stays at or near machine precision (`cost_final==0.0` from
kappa=1e6 onward) across the same range. `trust-apl trust-exact`, by
contrast, tracks scipy's own trust-exact closely through kappa=1e5-1e6
before the same relative-change-stall pattern appears at kappa=1e7/1e8
- the capability gap is BFGS-specific, not a general trust-apl
weakness. `dimensionality`'s own test coverage
(`tests/backends/test_apl_difficulty_families.py::
test_solve_bfgs_completes_without_a_timeout_at_a_practical_dimension`,
`::test_solve_trust_exact_converges_at_n_1000`) already pins the same
BFGS-specific pattern at increasing `n` (`MAX_ITER` at n=100, a subprocess
timeout at n=1000) against `trust-exact` remaining solid at n=1000; no
new test needed there, the documentation update cites this existing
coverage directly.

Out of scope, per the issue's own comment: whether `trust-exact`'s
indefinite-Hessian fragility (already pinned on `noisy_expdec`/
`gaussian_peak` in `tests/studies/test_typical_study_apl.py`, added in
#104) should widen this issue's scope beyond the `ill_conditioning`-
specific stress case. Left for a follow-up issue rather than decided
here, since the acceptance criteria for this issue are scoped to the
BFGS gap and the status-classification bug.

# PR notes: Issue #12 - RunResult, RunStatus, TimingStats dataclasses (result.py)

## Summary

Adds `trust_bench.core.result`: `RunStatus` (enum), `TimingStats`, and
`RunResult`, matching Section 4.3's field list. Part of Epic #11 (Phase 3),
see `docs/plans/trust-bench.md`, Section 4.3 and Section 10 (Phase 3).

## RED cycle (test-first)

`tests/unit/test_result.py`:

- `test_run_result_round_trips_with_no_timing_stats`
- `test_constructing_a_run_result_without_a_provenance_argument_fails`

**Coverage table**

| Acceptance criterion | Test(s) |
|---|---|
| `RunResult` serialises and deserialises without loss | `test_run_result_round_trips_through_json_serialisation`, `test_run_result_round_trips_with_no_timing_stats` |
| constructing a `RunResult` without `provenance` fails | `test_constructing_a_run_result_without_a_provenance_argument_fails`, `test_constructing_a_run_result_with_a_none_provenance_fails` |

The second test in each pair is not a literal reading of its criterion, but
guards a case the first test's fixture does not exercise: `timing` is the
only `Optional` field that is itself a nested dataclass (`TimingStats`), so
it is the one field a custom `from_dict` could plausibly mishandle when
`None`. `provenance=None` is the same "narrow implementation passes,
broader intent does not" gap `docs/reviews/8.md` flagged for `Problem.
residual`: omitting the argument already fails for free (`provenance` has
no default, so the dataclass-generated `__init__` requires it), but passing
`provenance=None` explicitly would not fail without an explicit check.

**Design notes**

- **`RunStatus` and `TimingStats` fill a gap**, the same shape as issue #8's
  `Optimum`: Section 4.3 uses both as field types (`status: RunStatus`,
  `timing: TimingStats | None`) but only gives `RunStatus`'s five member
  names as a field comment, and `TimingStats`'s fields as a comment
  ("median, MAD, n_reps, warmup, thread counts"), never a full definition.
  Issue #12's own title lists both alongside `RunResult`, so this is the
  issue that defines them.
- **`config: RunConfig` cannot be typed as written.** `RunConfig` is
  issue #19's deliverable (Phase 5), not built yet, the same forward
  reference issue #6 hit with `RunResult` itself. Typed as `Any` here;
  `result.py` does not import `config.py`, so there is nothing to
  reconcile when issue #19 lands, only a type annotation to tighten. The
  round-trip tests use a plain `dict` as a stand-in value, since `Any`
  imposes no runtime shape and the acceptance criterion is about
  `RunResult`'s own serialisation, not `RunConfig`'s.
- **`to_dict`/`from_dict`, not bare `dataclasses.asdict`.** `asdict`
  recurses into nested dataclass fields (`provenance`, `timing`)
  automatically, which is why `EnvProvenance.to_dict`/`TimingStats.to_dict`
  do not need to be called explicitly, but it leaves `status` as a
  `RunStatus` enum member, which `json.dumps` cannot serialise. `to_dict`
  overrides `status` to `.value` after `asdict`; `from_dict` reverses it
  and reconstructs `provenance`/`timing` from their nested dicts back into
  dataclass instances (guarding `timing is None`), since a round-tripped
  `RunResult` holding a plain dict where `provenance`/`timing` should be
  dataclass instances would fail equality against the original despite
  every underlying value matching.
- **`RunStatus` values are the member names as strings** (`CONVERGED =
  "CONVERGED"`, etc.), not auto-generated integers: the JSON on disk should
  read as the status it names, since `results/*.jsonl` (Section 8) is
  meant to be diff-friendly and human-inspectable.

**Anticipated implementation order**

1. `trust_bench/core/result.py`: `RunStatus`, `TimingStats`, `RunResult`.

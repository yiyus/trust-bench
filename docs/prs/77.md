# PR notes: Issue #77 - CLI: select which backends a report runs against

## Summary

Adds a `--backends` option to `trust-bench report`, and `AVAILABLE_BACKENDS`
(name to instance, keyed by each backend's own `.name`) plus
`_select_backends` to `cli.py`, mirroring the existing `--only`/`--skip`
study-selection pattern. Every `_write_*` study writer now takes a resolved
`backends` list and forwards it to its study function's own `backends=`
parameter, rather than always relying on the hard-coded production
`trust_bench.backends.BACKENDS`.

**Design notes**

*`_select_backends`, mirroring `_select_studies` exactly.* Same shape: a
small, pure function separated from `run_report`'s side effects, its own
direct tests instead of only being exercised through a full report run.
`_select_backends(None)` returns the production `BACKENDS` list itself
(not a copy or a re-derived equivalent), so passing `backends=None`
through to `run_report` is provably identical to today's behaviour, not
merely "close enough."

*No new attempt to make every study backend-agnostic.* `ill_conditioning`,
`scaling`, `dimensionality`, and `robust_loss`'s SciPy-loss sweep all
hard-code SciPy-specific method names (`"trust-exact"`, `"BFGS"`, etc.) in
their own `METHODS`; `APLBackend` only has `"lm"`. `baseline`,
`derivative_source` (`"lm"` is already one of its three methods), and
`capability_matrix` (probes each backend's own declared methods, never
assumes a shared list) already work against any registered backend
without change. For the rest, this issue does not rewrite any study's
method list to be backend-aware — see the Review response below for how
their failure mode for an unsupported backend is actually handled.

*`AVAILABLE_BACKENDS` lives in `cli.py`, not
`trust_bench/backends/__init__.py`.* It is a CLI-selection concern (which
names `--backends` accepts), not a change to what studies run against by
default (`BACKENDS`, untouched, still `[SciPyBackend()]`) — the same
separation #37 already established between "the production default" and
"a backend registered for direct/test use."

## Review response (RED stage)

Major: the RED test for "fails clearly, not a silent empty result" picked
`ill_conditioning` and asserted a `ValueError`, but `ill_conditioning.
sweep`'s own pre-existing `if method not in supported: continue` guard
means calling it with only `APLBackend` (none of whose methods overlap
`ill_conditioning.METHODS`) returns `{}` — zero rows, zero exceptions,
exactly the silent-empty-result the issue exists to prevent, not the
`ValueError` the test and the design notes both claimed. Confirmed
independently before making any change: `ill_conditioning.sweep(backends=
[APLBackend()])` does return `{}`; `bounded.sweep` (which has a similar
per-pair guard, but catches into the outcome dict rather than skipping)
returns 6 entries, each the caught `ValueError` itself, which
`results_to_dataframe` already renders as visible `"ERROR"` rows, not an
empty table.

Fixed generally, per the review's second suggested option: added
`_check_backend_coverage(df, backends, study)`, called from every
`_write_*` right after building its table, which raises if any selected
backend has zero rows in the `"backend"` column — regardless of whether
the underlying study function silently skips, catches into error rows, or
raises natively. This covers `ill_conditioning`'s actual failure mode
(caught by the new check) without touching its legitimate
skip-on-partial-support behaviour for genuinely mixed multi-backend
sweeps, and is uniform across all nine studies rather than special-cased
per study. The existing RED test needed no change: with the fix in place,
`ill_conditioning` + `trust-apl` now raises `ValueError` naming
`"ill_conditioning"` and `"trust-apl"`, satisfying
`pytest.raises(ValueError, match="trust-apl")` for the reason the issue
actually asks for, not the reason originally assumed.

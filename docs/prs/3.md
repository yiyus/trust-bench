# PR notes: Issue #3 - Environment provenance capture (provenance.py)

## Summary

Adds `trust_bench.core.provenance`: the `EnvProvenance` dataclass and a
`capture()` function that populates it from the running machine and
interpreter. Part of Epic #1 (Phase 0), see `docs/plans/trust-bench.md`,
Section 4.3 and Section 10 (Phase 0).

## RED cycle (test-first)

**Test surface**

`tests/unit/test_provenance.py`:

- `test_capture_returns_a_populated_env_provenance`
- `test_env_provenance_round_trips_through_json_serialisation`
- `test_machine_fingerprint_is_stable_across_calls`

**Coverage table**

| Acceptance criterion | Test(s) |
|---|---|
| `provenance.capture()` returns a populated `EnvProvenance` | `test_capture_returns_a_populated_env_provenance` |
| `EnvProvenance` round-trips through serialisation without loss | `test_env_provenance_round_trips_through_json_serialisation` |

`test_machine_fingerprint_is_stable_across_calls` is not a literal reading of
either acceptance criterion; it checks the contract Section 4.3's field
comment states directly ("stable hash to group runs by machine"), which
"populated" alone does not cover (a fingerprint could be populated and still
change on every call).

**Confirmed failure**

Both the new test module and the rest of the suite fail to collect:

```
ModuleNotFoundError: No module named 'trust_bench.core'
```

pytest treats a collection error as fatal for the run (`Interrupted: 1 error
during collection`), so the six existing `test_scaffolding.py` tests do not
get a chance to execute either. That is expected: `trust_bench.core` and
`provenance.py` do not exist yet, so import fails before any test body runs,
same class of failure ("not implemented") as issue #2's `ModuleNotFoundError`.

**Design notes**

- `EnvProvenance` is defined in `provenance.py`, not `result.py`, even
  though Section 4.3 introduces both together: `provenance.py` is the
  module that owns "environment capture" (Section 3's file-tree comment),
  and it is the natural producer of the type it returns. `result.py`
  (issue #12) will import `EnvProvenance` from here for `RunResult`.
- `backend_name`/`backend_version`/`blas_lapack` are backend-specific
  fields (Section 4.2's `Backend.environment()` also returns an
  `EnvProvenance`), but Phase 0 has no backend yet. `capture()` fills them
  with the core Python runtime's own identity (`backend_name="python"`,
  `backend_version` = the interpreter version) and detects BLAS/LAPACK via
  NumPy only if it happens to be installed, falling back to `"unknown"`
  rather than failing or leaving the field empty. `docs/reviews/trust-bench.md`
  already flags the two-producer merge rule (`capture()` vs
  `Backend.environment()`) as an open Minor finding; this issue does not
  resolve that, it only needs `capture()` itself to return a populated,
  round-trippable value.
- `to_dict`/`from_dict` round-trip through `dataclasses.asdict`/`cls(**data)`.
  The test goes via `json.dumps`/`json.loads` rather than the dict alone,
  since JSONL (Section 8) is the actual persistence format `RunResult` will
  use, and every field is a plain `str`/`int`, so JSON round-tripping is a
  strictly stronger check than a bare dict round-trip.

**Anticipated implementation order**

1. `trust_bench/core/__init__.py`
2. `trust_bench/core/provenance.py`: `EnvProvenance` dataclass,
   `to_dict`/`from_dict`, `capture()`.

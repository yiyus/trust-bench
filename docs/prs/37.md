# PR notes: Issue #37 - apl_backend.py: subprocess adapter, parity and contract tests

## Summary

Adds a Python-side `trust_bench.backends.apl_backend` module: `APLBackend`
(a `Backend` implementation wrapping #36's subprocess harness) and
`evaluate_problem(problem_id, x)` (a standalone residual/Jacobian probe
used by the cross-language parity tests). Extends the harness to cover
every canonical problem, not just `rosenbrock`. Part of Epic #34
(Phase 8), second child after #36.

**Design notes**

*`evaluate_problem` is a new, standalone function, not a method on
`APLBackend`.* Parity checks a raw residual/Jacobian at an arbitrary
point; it has nothing to do with running an optimiser and doesn't fit
the `Backend.solve(problem, method, start, config)` shape. Keeping it a
plain function in the same module avoids forcing the `Backend` ABC to
grow a probe-only method that only tests would ever call, while still
letting the parity tests import a single, obvious name.

*Cross-language Hessian parity (Section 5's third quantity) is
deliberately out of scope for this issue.* `trust`'s `Min`/`LM` never
computes the *full* Hessian of `0.5‖r‖²` — only the Gauss-Newton model
`JᵗJ`, which omits the residual-weighted second-order correction term
every Python `Problem.hessian` includes (`beale._hessian`,
`powell._hessian`, etc. all add it explicitly). Comparing `trust`'s
internal Hessian model against Python's true Hessian would only agree
at points where the residual is exactly zero, and would *correctly*
disagree everywhere else — asserting parity there would be asserting
the wrong thing, not filling a coverage gap. Residual and Jacobian are
the two quantities `trust` actually evaluates and the two this problem
family can make an honest claim about.

*Known risk: `helical`'s angle convention.* `trust`'s own `Test.aplf`
computes the helical valley's angle via a complex-number phase
(`12○` on `x1+ix2`, i.e. `atan2`-equivalent), while
`trust_bench.problems.helical._theta` computes it via a branching
`arctan(x2/x1)` with a manual `+0.5` correction when `x1≤0`. These are
almost certainly the same function, but "almost certainly" is exactly
what this parity test exists to pin down; `helical`'s `probe_points`
include two points with `x1<0` (`[-1,0,0]` and `[-0.5,0.3,0.2]`),
precisely where the two formulations would disagree if they weren't
equivalent. Flagged here rather than silently worked around in GREEN if
it turns out they aren't.

*The production `trust_bench.backends.BACKENDS` registry (used by every
study and the CLI) is untouched.* `APLBackend` is only added to the
test-only `tests/backends/all_backends.py` fixture. The harness supports
exactly the seven canonical problems; the difficulty-sweep studies
generate parametrised problem variants (`large_residual.make(rho=...)`,
etc.) that `apl_backend.py` has no way to serialise to the harness yet.
Wiring APL into the studies pipeline is future work once enough of that
gap is closed, not part of this issue's acceptance criteria.

*Method name: `"lm"`, matching cross-tool naming.* `trust`'s `Min`
auto-dispatches to LM for residual-form problems (there is no
user-selectable choice the way SciPy has `lm`/`trf`/`dogbox`), so
`capabilities()` exposes exactly one method. Lowercase `"lm"` matches
SciPy's own naming rather than the README's `LM`, so a future capability
matrix comparing method names across backends doesn't need a
case-insensitive join.

## GREEN cycle (implementation)

**Files changed**

- `backends_ext/apl/problems/{beale,powell,helical,expdec,quadratic,
  linear}.dyalog` (new): native APL problem implementations. `beale`,
  `powell`, and `helical` adapt `trust`'s own `Test.aplf` expressions
  (see the bug fix below); `expdec`, `quadratic`, and `linear` are new,
  matching `trust_bench`'s Python reference exactly (different, simpler
  parametrisations than `Test.aplf`'s own versions of the same names).
- `backends_ext/apl/dispatch.dyalog` (new): `NameFor` (problem id string
  to APL function name string) and `Apply` (`⍎`-based call by name).
  Replaces #36's hardcoded `'rosenbrock'` check in `Solve`.
- `backends_ext/apl/evaluate.dyalog` (new): `Evaluate`, the probe-only
  counterpart to `Solve` used by `mode: "evaluate"` requests.
- `backends_ext/apl/solve.dyalog`, `run.dyalog` (changed): `Solve` now
  dispatches via `NameFor`/`Apply`; `Run` reads `req.mode` (`:AndIf`,
  not `∧` — see the bug fix below) and calls `Evaluate` or `Solve`.
- `backends_ext/apl/run_harness.sh` (changed): concatenates the six new
  problem files plus `dispatch.dyalog`/`evaluate.dyalog` into the
  generated script.
- `trust_bench/backends/apl_backend.py` (new): `APLBackend` (`Backend`
  implementation) and `evaluate_problem(problem_id, x)`.

**Two bugs found and fixed while porting/wiring, beyond the tests
themselves**

- `trust`'s own `Test.aplf` Powell Jacobian has a wrong third-row,
  third-column entry: `(-2×b-2×c)`, which parses right-to-left as
  monadic negate of `2×(b-2×c)` (i.e. `-2u`), not `-4u` as the true
  derivative of `(x2-2x3)²` with respect to `x3` requires. Caught by
  running every problem's residual/Jacobian through both languages and
  diffing (the same check `test_apl_parity.py` runs, done first against
  a hand-built comparison script before writing the harness dispatch);
  `trust`'s own `t.Powell` test only checks convergence to the known
  optimum, which a wrong-but-still-descent-direction Jacobian entry
  doesn't necessarily break, so this had gone unnoticed. Fixed to
  `(¯4×b-2×c)` in `trust-bench`'s own `powell.dyalog`; `trust`'s
  vendored submodule is untouched (not this project's to fix).
- `Run`'s original mode check, `(2=⎕NC'req.mode')∧req.mode≡'evaluate'`,
  raised `VALUE ERROR: Undefined name: mode` for every `solve`-mode
  request (i.e. every request without a `mode` field at all, which is
  most of them): `∧` is a plain dyadic function, not a short-circuiting
  operator, so `req.mode≡'evaluate'` is evaluated unconditionally even
  when the existence check on its left is false. Fixed with `:AndIf`,
  which does short-circuit.
- Separately, discovered `⎕JSON` cannot export a genuine rank-2 array
  by default (`DOMAIN ERROR: JSON export: the right argument cannot be
  converted`, even for a bare matrix with no namespace involved).
  `Run`'s single `⎕JSON` call now uses `⍠'HighRank' 'Split'`, which
  handles any rank-2-or-higher field automatically; `Evaluate` assigns
  the Jacobian as the plain matrix `trust`'s own functions already
  return, with no manual reshaping needed at the call site.

**`n_jeval`/`n_heval`, resolving the RED-review finding**

`APLBackend.solve` sets `n_jeval` to the same value as `n_feval`: the
harness's `lower` wrapper (from #36) counts calls to a function that
always returns residual and Jacobian together, so every counted
evaluation is both. `n_heval` is `0` (or `None` alongside a `None`
`n_feval`, on `ERROR`) — LM never computes a true Hessian, only the
Gauss-Newton model, which is not a separate evaluation in `trust`'s own
accounting. `solve.dyalog` still emits `null` for both fields, unchanged
from #36 in that respect (its dispatch mechanism changed, but not its
output shape); the synthesis lives in the Python adapter, alongside
`dist_to_opt`/`cost_gap`, exactly where #36's own design notes already
pointed.

## Review response (GREEN stage)

Major: `APLBackend.solve` never referenced `config.loss`, `config.bounds`,
`config.derivative_mode`, or `config.x_scale`, unlike `SciPyBackend`, which
validates each against `capabilities()` and raises rather than silently
ignoring. `capabilities()` also declared `losses=frozenset({"l2"})`, a name
`RunConfig` never uses (its vocabulary is SciPy's own `"linear"`,
`"soft_l1"`, `"huber"`, `"cauchy"`, `"arctan"`, per `scipy_backend.py`'s
`_LEAST_SQUARES_LOSSES`) and that nothing ever sent to the harness anyway.

Fixed by making the backend actually behave like `SciPyBackend` for every
one of these fields:

- `config.loss` is validated against `caps.losses` and mapped through a new
  `_LOSS_TO_TRUST` table (`RunConfig`'s five SciPy-style names to `trust`'s
  `Loss` namespace: `"linear"→"L2"`, `"soft_l1"→"SoftL1"`, `"huber"→"Huber"`,
  `"cauchy"→"Cauchy"`, `"arctan"→"Arctan"`) and sent to the harness as an
  explicit `loss` field on every request. `capabilities()` now declares
  `losses=frozenset(_LOSS_TO_TRUST)` — the five names `RunConfig` actually
  uses, not `"l2"`. `trust` also implements `Tukey`/`Welsch`/`Fair`, which
  have no established `RunConfig`-side name; left unexposed rather than
  inventing one under review pressure.
- `config.bounds` and `config.x_scale`: the harness has no mechanism for
  either at all, so both now raise `ValueError` if set, rather than being
  silently dropped.
- `config.derivative_mode`: raises `ValueError` unless it is `None` or
  `"analytic"` (`caps.derivative_modes`), since the harness has no
  finite-difference path.

Verified directly: default `RunConfig` (loss `"linear"`) and an explicit
`"huber"` both still solve `rosenbrock` to the same converged result as
before; `"tukey"` (unsupported name), `bounds`, `x_scale`, and
`derivative_mode="finite-difference"` each raise `ValueError` with a message
naming the method and the unsupported field.

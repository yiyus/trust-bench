# PR notes: Issue #90 - Port dimensionality's generalised Rosenbrock to a native APL implementation

## Summary

`trust_bench.problems.families.dimensionality`'s generalised Rosenbrock
(split out from #83) needs an arbitrary-size APL port, unlike the four
small fixed-dimension families #83 already ported: residual/Jacobian/
Hessian construction that scales to n=1000, reusing #83's parametrised-
problem dispatch mechanism (`ParseParametrised`/`FamilyNameFor`/
`ApplyParam`) rather than inventing a new one.

Every part of the design was verified directly (`dyalogscript`) against
the Python reference before any test was written: correctness at
n=2/10/1000, and wall-clock cost at n=1000 (~7ms for residual+Jacobian,
~5ms for the Hessian - negligible next to `trust-exact`'s own
eigendecomposition-based subproblem solve at that size, the study's own
point).

**Design notes**

*Residual/Jacobian construction*: `oddpos`/`evenpos` (the 1-based odd/
even positions of an n-vector) select Python's `x[0::2]`/`x[1::2]`
directly with no off-by-one translation, since Python's 0-based
even/odd split maps exactly onto APL's 1-based odd/even split. The
Jacobian's three nonzero-entry families (diagonal, superdiagonal pair,
subdiagonal pair) are built by computing their row-major linear indices
directly (`(row-1)×n+col`) and scattering into a flat `n×n` vector
before reshaping, rather than any loop.

*A matrix does not support linear/ravel indexed assignment via a single
bracketed index list* - `h[diagIdx]+←...` on an already-reshaped `n n`
matrix raised `RANK ERROR`: a single index vector on a matrix selects
whole rows (axis 1), not ravel positions, unlike numpy's flat indexing.
Fixed by ravelling with `,h` before the scatter update and reshaping
back with `n n⍴` afterwards. Caught via direct `dyalogscript`
verification before writing any test, mirroring the `ApplyParam`-must-
be-monadic and `solve.dyalog`-direct-branching lessons from #83.

All formulas and this fix verified against the Python reference at
n=2, n=10 (all seven cells of the block-diagonal Jacobian/Hessian
pattern, not just the trivial n=2 case), and at n=1000 for both
correctness (`x_final`/`grad_norm_final` sampled) and timing.

## GREEN cycle (implementation)

`backends_ext/apl/problems/dimensionality.dyalog` (new): `Dimensionality`
(dyadic, `n` on the left, the point on the right) builds the residual
via boolean-mask odd/even position selection, and the Jacobian by
scattering its three nonzero-entry families into a flat `n²` vector via
their row-major linear indices before reshaping. `DimensionalityHessian`
computes `J^T J` and adds the per-pair correction term, ravelling before
its own scatter update (the `RANK ERROR` fix from the design notes).

`backends_ext/apl/dispatch.dyalog`: adds `"dimensionality"` to
`FamilyNameFor`/`FamilyHessianNameFor`. `backends_ext/apl/run_
harness.sh`: concatenates the new problem file. No changes needed to
`solve.dyalog`/`evaluate.dyalog` (already generic over any family
`#83` or this issue registers) or the Python side (`apl_backend.py`
already forwards `problem.id` verbatim).

Verified: the four affected test files - 40 passed (up from 33, the 7
new tests). Full suite `pytest -q` (dyalogscript available): 327
passed, 0 failed, no regressions. `ruff check .`: clean.

## Review response (implementation stage)

Critical: no test exercised a real `APLBackend.solve` at `n=1000` -
only construction cost (`evaluate_problem`) was measured there. Solving
`dimensionality(n=1000)` with `"BFGS"` genuinely does not complete
within `apl_backend.py`'s 60-second subprocess timeout (measured
directly: ~1.4s/iteration at `n=1000` versus ~0.04s/iteration at
`n=100`, and `max_iter=200` forces the full budget since `BFGS` is
expected not to converge at this dimension - `dimensionality.py`'s own
`test_bfgs_fails_to_converge_at_large_dimension` documents exactly this
phenomenon). Worse, the timeout was uncaught: `_run_harness` let
`subprocess.TimeoutExpired` propagate, crashing the whole `trust-bench
report` invocation rather than reporting one bad row.

`"trust-exact"`, also dense-Hessian, does not have this problem: measured
directly at `n=1000`, it converges in ~13s (33 iterations, far fewer
than BFGS needs and each one far cheaper). This is `trust`'s own `BFGS`
engine's per-iteration wall-clock cost at scale, not a defect in this
port's `Dimensionality`/`DimensionalityHessian` construction (confirmed
fast and correct independently).

Fixed in two parts, per direction to use a lower n for now rather than
raise the timeout or attempt to speed up `trust`'s own `BFGS` engine:

- `apl_backend.py`'s `_run_harness` now catches `subprocess.
  TimeoutExpired` and returns a clean `ERROR`-status result (matching
  `error_result.dyalog`'s own field set exactly), so a real timeout -
  at `n=1000` or anywhere else - is reported as one bad row rather than
  crashing the whole report. `tests/backends/test_apl_backend_
  timeout.py` (new, no `dyalogscript` dependency - a pure
  `subprocess.run` mock) proves this for both `solve()` and
  `evaluate_problem()`; confirmed genuinely RED against the pre-fix
  code first.
- Added `test_solve_trust_exact_converges_at_n_1000` (asserts real
  convergence, since it genuinely works at that size) and `test_solve_
  bfgs_completes_without_a_timeout_at_a_practical_dimension` (uses
  `n=100`, not `n=1000`, and asserts `MAX_ITER` - the scientifically
  expected outcome for a dense quasi-Newton method at that dimension,
  not an error - proving the port itself completes cleanly at a
  practical scale for `BFGS` without claiming `n=1000` is practical for
  it).

Not fixed here, and not this port's responsibility to fix: `trust`'s
`BFGS` engine's wall-clock cost at `n=1000`. `dimensionality.py`'s
study is unaffected by default (`trust-apl` is not in the default
`BACKENDS` list); a user who explicitly includes it alongside `BFGS` at
`n=1000` will now get a clean `ERROR` row instead of a crash, but not
real data for that cell.

Re-verified: `tests/backends/test_apl_difficulty_families.py` and
`tests/backends/test_apl_backend_timeout.py` - 12 passed. Full suite
`pytest -q` (dyalogscript available): 331 passed, 0 failed. `ruff
check .`: clean.

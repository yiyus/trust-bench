# PR notes: Issue #84 - APLBackend: expose BFGS and Newton methods

## Summary

`APLBackend` gains `"BFGS"` (scalar objective, `trust`'s BFGS-Hessian-model
trust-region engine) and `"trust-exact"` (analytic Hessian, `trust`'s true-Newton
trust-region engine — named to match scipy's own vocabulary for the closest
conceptual equivalent, not a 1:1 API match: both do an exact trust-region
subproblem solve against the real Hessian, unlike the CG-based `Newton-CG` or
Krylov-based `trust-krylov`). The harness routes to these via `trust`'s `BFGS`/
`Newton` operators, which requires porting each canonical problem's analytic
Hessian to APL — currently only residual/Jacobian are ported.

Narrowed scope, agreed before writing tests: `ill_conditioning.py` and
`dimensionality.py` sweep problem ids from `trust_bench/problems/families/`
(e.g. `"dimensionality(n=10)"`), not the seven canonical problems this issue
ports Hessians for, so both will keep returning `ERROR: Unknown problem_id`
for `trust-apl` even after this issue lands, until #83 (porting those
families) also lands — noted on both issues. This issue verifies what it
controls directly: the methods exist, are correctly wired, and produce real
data on the problems the harness already knows (`baseline`/
`derivative_source`-style studies over the seven canonical problems).
`dimensionality.py` still gets the missing skip-guard `ill_conditioning.py`
already has, so it stops crashing for methods it won't recognise even before
#83 lands.

**Design notes**

*Both new dispatch paths, and all seven Hessians, verified directly
against `trust`/Python via `dyalogscript` before writing any assertion.*
For Rosenbrock: a monadic evaluator returning `(cost, gradient)` as a
2-element result dispatches to `trust`'s BFGS engine; returning `(cost,
Hessian, gradient)` as a 3-element result dispatches to true Newton —
both via the same `Min` operator already used for LM, no ambivalent
(dyadic) calling convention needed at this level (that complexity is
internal to `trust`'s own `BFGS`/`Newton` operators). From the standard
MGH start, BFGS took 68 iterations, true Newton took 32 — a real,
verified difference (not assumed), giving `test_trust_exact_converges_
in_fewer_iterations_than_bfgs` a grounded assertion. All seven ported
Hessians were checked against the Python reference at every probe point
(22 points total) before being wired into `Solve`; all matched to
`1e-6` on the first attempt (no `#37`-style bug this time).

*`n_heval` is always a real integer now, never `null`, for any
successful solve — not just for `trust-exact`.* Before this issue, the
harness always emitted `null` and the Python adapter (`apl_backend.py`)
separately hardcoded `0` for a successful run. Now that `trust-exact`
genuinely counts Hessian evaluations, the harness tracks a real counter
(`hcalls`, incremented only for `trust-exact`) and reports it directly;
`apl_backend.py`'s own synthesis is removed in favour of passing
`response["n_heval"]` straight through. This changes the *wire* format
for `lm`/`BFGS` (`null` → `0`) but not the Python-level `RunResult`
(already `0` there, via the old synthesis) — updated the one existing
harness-level test that checked the raw wire value
(`test_solves_rosenbrock_from_the_standard_start_to_convergence`).
Chose `0`-not-`null` for consistency with `SciPyBackend`
(`getattr(result, "nhev", 0)`) and `ToyGradientDescentBackend`
(hardcoded `0`) — both already treat "this method has no Hessian" as
`0`, never `None`; reserving `null` only for the genuine `ERROR` case
keeps `n_heval >= 0` (already asserted by the existing contract tests)
true for every method, with no special-casing needed there.

*Bounds and `derivative_mode` validation, unchanged, now apply to all
three methods for free.* `#78`'s bounds/derivative_mode wiring in
`Solve` operates on `cfg` regardless of which evaluator `lower` is; the
`Bounded` operator wraps any Hessian-providing evaluator uniformly per
`trust`'s own docs. `test_bounds_work_for_the_trust_exact_method`
confirms this directly rather than assuming it from the docs: the same
active-boundary quadratic scenario from `#78`'s `lm` test, run through
`trust-exact`, converges to the identical `(0.5, ≈0)`.

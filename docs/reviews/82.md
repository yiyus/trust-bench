# Review: Issue #82 - bounded study never exercises trust-apl's bounded lm method

## Summary

Tests-only (RED) stage review. No issues found. The empirical claims in
the PR doc were verified directly: `SciPyBackend`'s `L-BFGS-B`/
`trust-constr` genuinely reject the current `bounded.py` scenario bounds
(a least-squares-shaped `(lower, upper)` array pair) with a bounds-format
error rather than a semantically infeasible-start rejection; once
reformatted to the minimize-family's `[(lo, hi), ...]` shape, both
methods project the infeasible start into the box and converge to the
same boundary point as `trf`/`dogbox`, exactly as claimed. `trust-apl`'s
`lm` was confirmed directly (`dyalogscript`) to converge for all three
scenarios, including `infeasible_start`. The RED test surface correctly
anticipates the bounds-format mapping this implies (design notes cite
`capability_matrix.py`'s existing per-family split as the precedent to
follow), and the two structural tests (`covers_every_scenario_and_
every_bounds_capable_method`, `exercises_more_than_the_least_squares_
family`) genuinely fail against the unmodified `bounded.py`.

## Implementation review

### Summary

`bounded.py`'s `sweep()` derives its method list per backend from
`capabilities().methods[...].bounds`, resolving exactly the finding
flagged during the tests-only review: `_bounds_for(method, active)`
picks the least-squares (`(lower, upper)` array pair) or minimize
(`[(lo, hi), ...]`) bounds shape by method name, mirroring
`capability_matrix.py`'s existing `_LEAST_SQUARES_BOUNDS_PROBE`/
`_MINIMIZE_BOUNDS_PROBE` split precisely. `SCENARIOS`' public shape
changes from `bounds`+`start` to `active`+`start`, since the concrete
bounds values are now method-dependent rather than fixed; `cli.py`'s
`_write_bounded` is unaffected, confirmed by reading it directly (calls
only `sweep(backends=backends)`). No issues found.

### Findings

No issues found.

# Review: Issue #22 - Study: baseline correctness and basin-of-attraction rate

## Summary

Tests-only (RED) stage review. `tests/studies/test_baseline.py` covers both
acceptance criteria: standard-start convergence across every canonical
problem and registered backend, and a per-problem/backend basin-of-attraction
rate that is genuinely evaluated across every registered start, not just
`standard`. The canonical set (`rosenbrock`, `beale`, `powell`, `helical`,
`expdec`, `quadratic`, `linear`) matches Section 9 item 1 of
`docs/plans/trust-bench.md` exactly, and the shape-only assertions (status,
rate in `[0, 1]`) match Phase 6's "assert shape, not exact numbers"
instruction. Verified directly: every canonical problem converges with
`SciPyBackend`'s `lm` method from every one of its registered starts,
including Rosenbrock's `far` start, so the spy-based third test is
necessary, not incidental, to prove `far` is actually exercised. Confirmed
the stated RED failure with the rest of the suite (143 tests) unaffected.
One minor note; otherwise no issues found.

## Findings

### Minor: third test couples to the internal name `baseline_module.run`

`test_basin_rate_evaluates_every_registered_start_not_only_standard`
monkeypatches `baseline_module.run` to spy on which `(problem, start)` pairs
`basin_rates()` evaluates. This is the only way to observe "was `far`
actually evaluated" given every canonical start converges (a black-box rate
of `1.0` cannot distinguish that from "only `standard` was ever run"), and
the design notes in `docs/prs/22.md` say so explicitly. The trade-off worth
recording: this pins `baseline.py` to importing the runner as a module-level
`run` symbol called directly; a later refactor that preserves behaviour
(e.g. wrapping the call, importing `runner` as a module and calling
`runner.run`) would break this test even though nothing observable changed.
No action needed now; worth keeping in mind if `baseline.py`'s internals
move during the GREEN cycle or a later phase.

Location: `tests/studies/test_baseline.py` (spy on `baseline_module.run`)

## Findings

### Minor: canonical problem list is duplicated between production and test code

`trust_bench/studies/baseline.py`'s `CANONICAL_PROBLEMS` and
`tests/problems/test_canonical.py`'s `PROBLEMS` list the same seven problems
independently. `docs/prs/22.md` justifies not importing the test module from
production, which is correct as far as it goes, but the reverse dependency
(a single canonical list in `trust_bench/problems/__init__.py`, imported by
both `baseline.py` and the test module) would remove the duplication without
creating a prod-depends-on-tests problem. Not blocking: the two lists are
easy to keep in sync while there are only seven problems and one contributor
per change, but worth revisiting if a future issue adds an eighth canonical
problem to only one side by mistake.

Location: `trust_bench/studies/baseline.py:7-15` vs `tests/problems/test_canonical.py:6-13`

# Review: Issue #37 - apl_backend.py: subprocess adapter, parity and contract tests

## Summary

Tests-only (RED) review; `docs/reviews/37.md` did not previously exist and
`trust_bench.backends.apl_backend` does not exist yet, confirmed directly:
`pytest -m "not slow" --continue-on-collection-errors` reports 211 passed,
9 deselected, 3 collection errors, exactly as `docs/prs/37.md` states. The
new parity test's cross-language claim about `helical`'s angle convention
checks out: verified independently with `dyalogscript` that `trust`'s
complex-phase `12○` formula and the Python reference's branching `arctan`
formula agree at every flagged probe point to floating-point precision. One
real gap needs resolving before GREEN: the existing, unchanged
`test_contract.py` requires `n_jeval`/`n_heval` to be non-negative integers
for every backend, but #36's harness always emits `null` for both, and
neither `docs/prs/37.md` nor the design notes say how `APLBackend` will
reconcile the two.

## Findings

### Major: `n_jeval`/`n_heval` contract requirement conflicts with #36's harness output

`tests/backends/test_contract.py::test_solve_returns_a_well_formed_run_result`
(existing, unchanged, now parametrized to run against `APLBackend` via
`all_backends.py`) asserts:

```python
assert result.n_jeval >= 0
assert result.n_heval >= 0
```

`RunResult.n_jeval`/`n_heval` are typed `int | None`, and issue #36's harness
(`backends_ext/apl/solve.dyalog:31-32`) always sets both to `NULL`
(`None` once through `⎕JSON`) — that design was deliberate for #36, whose own
`Solve` genuinely has no way to distinguish Jacobian-only calls from combined
residual+Jacobian calls. If `APLBackend.solve` passes the harness's output
through unchanged, `result.n_jeval >= 0` raises `TypeError: '>=' not
supported between instances of 'NoneType' and 'int'` the first time this
contract test runs against it, since `evaluate_problem`'s counting scheme in
#36 folds Jacobian evaluation into the same counter as residual evaluation
(`n_feval`).

Neither the acceptance criteria nor `docs/prs/37.md`'s design notes mention
this. Whatever the intended fix is (e.g. `APLBackend` synthesising
`n_jeval = n_feval` since `trust`'s LM evaluates the Jacobian at every
accepted step, or extending the harness's counters), it should be a stated
design decision before GREEN, not something discovered while making the
contract test pass.

Location: `tests/backends/test_contract.py:22-24`,
`backends_ext/apl/solve.dyalog:31-32`

## Implementation review (GREEN stage)

### Summary

`trust_bench.backends.apl_backend` (`APLBackend`, `evaluate_problem`) is now
implemented, along with native APL problem files for the six remaining
canonical problems, a name-based dispatch mechanism (`dispatch.dyalog`,
`Apply←{name point←⍵ ⋄ ⍎name,' point'}`), and an `evaluate.dyalog` counterpart
to `Solve`. Verified every one of the seven canonical problems'
residual/Jacobian output directly against the Python reference with
`dyalogscript`, independent of the test suite: all match to floating-point
precision, including the `Powell` Jacobian bug fix described in
`docs/prs/37.md` (confirmed `¯4×b-2×c`, not `trust`'s own `-2×b-2×c`, is the
correct third-row/third-column entry at both probe points). Ran the harness
directly end-to-end for both `solve` and `evaluate` modes against a non-
`rosenbrock` problem (`beale`, `helical`) to confirm the dispatch and
`⎕JSON⍠'HighRank' 'Split'` fixes work outside the test harness, not just
inside it. Full suite: 240/240 pass (228 prior + 12 new); `ruff check`: clean.
The RED-stage `n_jeval`/`n_heval` finding is resolved correctly. One new gap:
`APLBackend.solve` never forwards `config.loss`, `config.bounds`,
`config.derivative_mode`, or `config.x_scale` to the harness, and doesn't
validate them against its own declared `capabilities()` the way `SciPyBackend`
does for every one of these fields.

### Findings

#### Major: Capability-relevant `RunConfig` fields are silently ignored, unlike `SciPyBackend`

`APLBackend.solve` (`trust_bench/backends/apl_backend.py:91-133`) only ever
reads `config.max_iter` and `config.tolerance` from the passed `RunConfig`.
`config.loss`, `config.bounds`, `config.derivative_mode`, and `config.x_scale`
are never referenced, so a caller setting any of them has no effect at all,
silently.

`SciPyBackend` establishes the repo's actual convention for this: it validates
capability-relevant config against `capabilities()` and raises rather than
ignoring, e.g.

```python
# trust_bench/backends/scipy_backend.py:231-233
bounds = config.bounds
if bounds is not None and not caps.bounds:
    raise ValueError(f"{method} does not support bounds")
```

and similarly for `derivative_mode` (line 238). `APLBackend` has no equivalent
for any of the four fields it doesn't wire through. Two concrete
consequences:

- `capabilities()` declares `losses=frozenset({"l2"})`, but nothing in
  `solve()` ever sends a `loss` field to the harness at all — even if a
  caller passed exactly `config.loss = "l2"` (the one value the backend
  claims to support), the harness still just uses `trust`'s own default
  (`'L2'`) unconditionally. The declared capability is untested and
  unimplemented, not merely a case that hasn't come up.
- The naming doesn't match anyway: `SciPyBackend`'s `"lm"` method declares
  `losses=frozenset({"linear"})` (SciPy's own vocabulary — see
  `_LEAST_SQUARES_LOSSES = {"linear", "soft_l1", "huber", "cauchy",
  "arctan"}`), not `"l2"`. `trust`'s `Loss` namespace genuinely implements
  several of the same robust losses SciPy does (`Huber`, `Cauchy`, `SoftL1`,
  `Arctan`, confirmed in `backends_ext/apl/trust/APLSource/Loss.apln`), so
  restricting `capabilities()` to `{"l2"}` alone is also an understatement of
  what could be exposed, separate from the wiring gap.

Since `RunConfig.loss` defaults to `"linear"` and `trust`'s harness defaults
to `'L2'` (the same concept under a different name), the default path happens
to produce correct behaviour by coincidence, not because anything checks or
maps between the two vocabularies. This is worth resolving (or at minimum
documenting as an explicit scope cut, the way the difficulty-sweep and
multi-method deferrals already are) before any future study or CLI run
assumes setting `config.loss`/`config.bounds` on the APL backend does
anything.

Location: `trust_bench/backends/apl_backend.py:91-99`

#### Note: `n_jeval`/`n_heval` gap from the RED review is resolved correctly

`APLBackend.solve` sets `n_jeval = response["n_feval"]` and `n_heval = 0` (or
`None` alongside a `None` `n_feval` on `ERROR`), with a comment explaining why
(`trust`'s `Eval` convention returns residual and Jacobian from the same
call). This matches `trust`'s actual evaluation model and resolves the RED
finding. `solve.dyalog` itself is unchanged in this respect: the synthesis
correctly lives in the adapter, not the harness.

#### Note: An invalid `mode` value silently falls back to `solve`, not an error

`run.dyalog`'s `:If 2=⎕NC'req.mode' :AndIf req.mode≡'evaluate'` treats any
`mode` other than exactly `'evaluate'` (including a typo, e.g. `'Evaluate'`)
as "no mode", running `Solve` rather than raising. Not a problem in practice
since both Python call sites (`evaluate_problem`, `APLBackend.solve`) always
send an exact, correct value or omit the field entirely, but worth knowing if
`mode` is ever exposed to a less controlled caller later.

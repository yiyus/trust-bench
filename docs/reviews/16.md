# Review: Issue #16 - SciPy backend: least_squares methods (lm, trf, dogbox) (implementation)

## Summary

Implementation-stage review. `SciPyBackend` satisfies both acceptance
criteria: all three `least_squares` methods converge on the trivial
quadratic, and bounds handling is delegated to `scipy` itself rather than
reimplemented, exactly as the PR doc describes. `make test` (92 passed),
`make lint`, and `make coverage` (100% on `scipy_backend.py`) all
reproduce the PR doc's claims. One API-consistency gap and one
undocumented deviation from the design doc's iteration-definition rule
are worth fixing before merge; neither is exercised by the current test
suite, which is why they survived review of the tests-only stage.

## Findings

### Major: `capabilities()` advertises finite-difference support that `solve()` cannot deliver

Every method's `MethodCapabilities.derivative_modes` includes
`"finite-difference"`, but `solve()` always passes `jac=problem.jacobian`
to `least_squares` with no way to select finite-difference evaluation and
no fallback when `problem.jacobian` is `None` (the exact case
`Problem.jacobian`'s own docstring reserves for "FD only"). Confirmed
directly: `least_squares(fun, x0, jac=None, method=...)` raises
`ValueError: `jac` must be '2-point', '3-point', 'cs' or callable.` No
problem in the repo currently has `jacobian=None`, so this does not fail
today, but Section 9 (item 8, "Derivative source") of the design doc
treats analytic-vs-finite-difference as a first-class comparison axis
this backend is meant to support, and an FD-only problem is a foreseeable
near-term addition. Recommend either implementing the fallback (e.g.
`jac = problem.jacobian if problem.jacobian is not None else "2-point"`,
with a `config`-level switch to request FD even when an analytic
Jacobian exists) or narrowing `derivative_modes` to
`frozenset({"analytic"})` until that capability is real, so the declared
`Capabilities` never claims more than `solve()` can do.

Location: `trust_bench/backends/scipy_backend.py:36,43,50` (declared
capability); `trust_bench/backends/scipy_backend.py:63` (`jac` always
sourced from `problem.jacobian`).

### Minor: `n_iter` reuses `n_feval`'s value without documenting the deviation

`n_iter=result.nfev` (same value passed to `n_feval`) because
`least_squares`'s `OptimizeResult` has no `nit`-equivalent field, unlike
`minimize`. This is a defensible choice given what `scipy` exposes, but
Section 7 of the design doc ("Iteration definition") explicitly asks that
"how each backend counts iterations and evaluations" be recorded and
documented, precisely for cases like this where the definition deviates
from a literal per-iteration count. Neither the PR doc's GREEN cycle
notes nor a code comment states that this backend's `n_iter` is really a
function-evaluation count. Recommend a one-line note in the PR doc (or a
short comment at the assignment) stating that `least_squares` exposes no
true iteration count, so `n_iter` here is `n_feval`.

Location: `trust_bench/backends/scipy_backend.py:92`.

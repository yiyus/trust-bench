# Review: Issue #16 - SciPy backend: least_squares methods (lm, trf, dogbox)

## Summary

Tests-only (RED) stage review. The new test surface directly covers both
acceptance criteria: convergence to the known optimum for all three
`least_squares` methods, and the bounds/capabilities agreement between
`lm` (rejects bounds) and `trf`/`dogbox` (accept and respect them). The
`BACKENDS` move from `toy_backend.py` to `all_backends.py`, and the new
`harness_git_sha` test, are both scoped consistently with the PR doc's
stated rationale. All five affected test modules fail collection with the
expected `ModuleNotFoundError`/`ImportError`, confirming RED. No issues
found.

## Findings

None.

### Note: `harness_git_sha` test scope extends beyond the issue's stated acceptance criteria

`test_harness_git_sha_matches_git_rev_parse_head` is not a reading of
either acceptance criterion in issue #16; it exists because a real
backend now produces real `RunResult`s that need a real
`harness_git_sha`, per Section 4.3 of the design doc. The PR doc names
this explicitly as a genuinely new requirement rather than hiding it
inside the SciPy-specific test file. No action needed; recorded because
it is the kind of scope addition worth surfacing.

### Note: numeric semantics verified directly

The quadratic problem's residual is linear (`r = [x1, 2*x2]`), so its
constrained optimum under `x1 >= 0.5` is exactly `x1 = 0.5, x2 = 0`,
matching `test_bounded_methods_accept_and_respect_box_constraints`'s
assertions. The unconstrained optimum `[0, 0]` is also correct for
`test_method_solves_the_trivial_quadratic_to_the_known_optimum`. Verified
by inspection of `trust_bench/problems/quadratic.py`; both are ordinary
Python/NumPy, not APL, so no `dyalog-script` verification applies here.

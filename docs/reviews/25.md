# Review: Issue #25 - Study: robust loss and outliers

## Summary

Tests-only (RED) stage. `tests/studies/test_robust_loss.py` covers the
scoped-down acceptance criteria: a SciPy-loss-vs-fraction sweep as the
comparison baseline, and a hand-rolled IRLS reference recovering the true
parameters past the contamination level where every SciPy loss fails. The
scope deferral (comparing against a hand-rolled IRLS reference now, `trust`
itself once Epic #34/Phase 8 lands) is real: Epic #34 ("[Epic] Phase 8: APL
backend (trust)") exists and is open, and the plan's own phase ordering
(Section 10) sequences Phase 8 after Phase 6. Independently reproduced every
numerical claim in `docs/prs/25.md`: all four non-`arctan` SciPy losses are
dragged more than 7 units from `TRUE_PARAMETERS` by `fraction=0.4`; `arctan`
degrades far more gracefully (`0.111` at `0.4`, `0.254` at `0.49`); a Tukey
biweight IRLS warm-started from the `arctan` fit recovers `TRUE_PARAMETERS`
exactly at every fraction through `0.45`, failing only at `0.49`; and the
same IRLS warm-started from OLS instead fails from `fraction=0.2`, exactly
as the PR doc's "genuine implementation obstacle" note describes. The RED
failure and the "156 tests unaffected" claim both reproduce exactly. One
minor comment-hygiene finding; otherwise no issues found.

## Findings

### Minor: two comments in the new test file reference GitHub issue numbers directly

`tests/studies/test_robust_loss.py:11` ("Epic #34, Phase 8, not started")
and `:53` ("issue #10") name GitHub issue numbers directly in code comments.
Every sibling study test file (`test_baseline.py`, `test_large_residual.py`,
`test_ill_conditioning.py`) references only design-doc sections (e.g.
"Section 9 item 2"), consistent with the project convention that issue IDs
belong in PR docs, review docs, and commit messages, not source comments,
since they go stale on repo renames/migrations and force a context switch
for a reader who could otherwise understand the code on its own. Line 11's
content is otherwise a legitimate, load-bearing invariant (this study
deliberately doesn't yet compare against `trust`) and belongs in the
comment; only the epic number needs to go. Line 53 is a pointer to the
`outliers` family's own defining property, which the comment already
restates in prose immediately after the issue reference, so the number adds
nothing the surrounding text doesn't already say.

Recommendation: drop `Epic #34, Phase 8, not started` from line 11 (the
"the APL backend doesn't exist yet" clause already carries the invariant)
and drop `, issue #10` from line 53.

Location: `tests/studies/test_robust_loss.py:11`, `tests/studies/test_robust_loss.py:53`

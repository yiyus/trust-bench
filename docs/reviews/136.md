# Review: Issue #136 - Implementation

## Summary

Both RED-stage findings are fixed: the `#136` parenthetical is gone from `_LmOnlyBackend`'s docstring and the missing blank line before its class is added. `run_report` now collects per-study coverage failures into a `skipped` dict and continues, raising only when every selected study fails; `main()` prints a `Skipped <study>: <reason>` line per skip. All fast tests pass and the regression test is unaffected. One correctness gap: the per-study `try/except` catches bare `ValueError`, not specifically `_check_backend_coverage`'s, so an unrelated bug elsewhere in a study's write path would now be silently absorbed into `skipped` instead of crashing the report - undermining the exact safety net `_check_backend_coverage`'s own docstring says it exists for.

## Findings

### Major: bare `ValueError` catch can mask unrelated bugs as "known coverage gaps"

`run_report`'s new `try/except ValueError` around `STUDIES[name](output_dir, selected_backends)` catches any `ValueError` raised anywhere in that study's write path, not just `_check_backend_coverage`'s. Verified directly: patching a study's writer to raise an unrelated `ValueError` (`"unrelated bug: malformed input, not a coverage gap"`) causes `run_report` to silently record it in `skipped` and continue, exactly as if it were a real coverage gap. Several genuine, non-coverage `ValueError`s exist in the call graph a study can reach - `Backend.solve` raises one for an unsupported `x_scale`/`loss`/`derivative_mode` combination (`apl_backend.py`, `scipy_backend.py`), and the problem-family constructors raise one for invalid parameters (`dimensionality.py:12`, `ill_conditioned.py:17`, `scaling.py:15`, `outliers.py:25`). None of the current studies happen to trigger these today, but the catch has no way to distinguish "expected, permanent coverage gap" from "a real bug that happens to raise ValueError" - which is precisely the failure mode `_check_backend_coverage`'s own docstring was written to catch instead of letting pass silently.

Recommend narrowing the catch to a dedicated exception (e.g. a `CoverageError(ValueError)` raised only by `_check_backend_coverage`, caught specifically in `run_report`), so an unrelated `ValueError` from elsewhere in a study's write path still propagates and fails the report loudly.

Location: trust_bench/cli.py:283-286

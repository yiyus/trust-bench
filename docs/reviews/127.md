# Implementation review

## Summary

Reviewed `b92ff99` (the RED-review fix) and `5038962` (implementation) - the earlier `x_scale="jac"` coverage gap is closed, now parametrised over `lm`/`BFGS`/`trust-exact`. The `pscale` wiring, the `Result`-based status rewrite, and the `docs/methodology.md` write-up otherwise match the PR doc's description closely, and the architecture is sound (the rejection ordering in `apl_backend.py`, the `x_scales`-respecting `BFGS`+fixed exercise in `scaling.py`). However, direct verification of the new `pscale` capability surfaced a real correctness bug in `grad_norm_final` that no test in this cycle catches.

## Findings

### Critical: `grad_norm_final` is reported in `pscale`-transformed coordinates, not physical parameter space, whenever a fixed `x_scale` is supplied

`solve.dyalog`'s `gnorm←0.5*⍨+/r.grad×r.grad` (`= ||r.grad||`, the Euclidean norm) reads `Min.aplo`'s `r.grad` directly. `Min.aplo`'s internal solve operates on the `pscale`-reparameterised variable `y` (`x = pscale × y`); at the end it unscales the returned parameter estimate (`c.p×←c.pscale`) but never unscales `c.grad` - so `r.grad` is the gradient with respect to `y`, not `x`, whenever `pscale≠1` (i.e. whenever `RunConfig.x_scale` is a fixed vector). `apl_backend.py` then passes `response["grad_norm_final"]` straight through with no correction (`grad_norm_final=response["grad_norm_final"]`, `trust_bench/backends/apl_backend.py:207`).

Verified directly against `scaling.make(1e8)`/`lm` with `x_scale=(1.0, 1e-8)` (the exact case `test_a_fixed_x_scale_recovers_where_unscaled_fails_at_extreme_disparity` exercises):

- Reported `grad_norm_final`: `1.953992523340276e-06`.
- True physical-space `||J^T r||` at the same `x_final` (computed directly from the problem's own Python-side `residual`/`jacobian`): `199.84014443252818` - a five-order-of-magnitude difference on the exact case this PR's own tests certify as `CONVERGED`.
- Chain-ruling the physical gradient by `pscale` element-wise (`grad_y = grad_x * pscale`) gives `1.9984014443252818e-06` - matching the reported value to within rounding, confirming the field is genuinely returned in `y`-space, not a coincidence.
- Cross-checked `cost_final` at the same point against the Python-side cost function (`1.909e-12` reported vs `1.997e-12` computed, consistent to solver precision) to confirm this is a `grad`-specific defect, not a general parity mismatch between the Python and APL problem definitions.

Two concrete consequences:

1. `cli.py`'s already-shipped `_write_scaling` writes `scaling.csv`/`scaling.png` with unrestricted `x_scales` (`X_SCALES = [None, "jac", "fixed"]`) and plots `grad_norm_final` grouped by `x_scale` on a log axis (`trust_bench/cli.py:139-142`). The `"fixed"` line for `trust-apl` will show a gradient norm roughly `1/scale` smaller than the unscaled line at the same point, not because the fit is genuinely flatter there, but purely as a coordinate-scaling artefact - misleading on a chart whose entire purpose is a fair backend-vs-backend/scale-vs-scale reading.
2. `Result.StalledByPrecision`'s `gtol=1e-2` cutoff (documented in `docs/methodology.md` as calibrated against physical-space gradients: converged cases `≤1.6e-6`, indefinite-Hessian stalls `≥4.8`) is compared against this same unrescaled `r.grad` internally (`backends_ext/apl/solve.dyalog:99`). Any `pscale` whose magnitude pulls the transformed gradient across that threshold in either direction will misclassify `CONVERGED` vs `STALLED` for a run that would classify correctly at `pscale=1`. The one currently-tested case happens not to trigger this (the run is genuinely converged by other criteria too), but the threshold is no longer scale-invariant now that a caller can set `pscale≠1`.

No test in this cycle exercises `grad_norm_final`'s value under a fixed `x_scale` (both new tests check only `status`/`dist_to_opt`), so this passed the full suite undetected.

Fix needs to divide `r.grad` (or the harness's `gnorm`) by `pscale` element-wise before use, in both the `grad_norm_final` result field and (if `StalledByPrecision`'s heuristic is meant to stay physical-space-calibrated) the status-classification path. A test asserting `grad_norm_final` under a fixed `x_scale` approximates the same value computed at `x_scale=None` for an equivalent solution (or directly matches a physical-space gradient computed independently) would catch a regression here.

Location: `backends_ext/apl/trust/APLSource/Min.aplo` (root cause: `c.grad` never rescaled), `backends_ext/apl/solve.dyalog:88` (`gnorm`), `trust_bench/backends/apl_backend.py:207` (passthrough)

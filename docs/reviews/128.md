# Review: Issue #128 - robust_loss study compares different loss tuning constants, not just different libraries

## Summary

Implementation review. `RunConfig.f_scale` is added and wired through exactly
as described in the PR doc: `scipy_backend.py` forwards it to `least_squares`
only when set; `apl_backend.py` rejects it outright for every method, since
neither trust's fixed per-loss constant nor its MAD-based auto-scaling has a
per-request override; `robust_loss.py`'s `scipy_loss_precision` passes trust's
own constants as `f_scale` for the scipy backend only. `docs/methodology.md`
gets a new section documenting the mismatch and the fix, matching the
tolerance-mapping section's existing precedent for this category of
comparability issue. Full suite passes, diff is minimal, no issues found.

## Findings

No issues found.

### Note: fix addresses the root comparability gap, not a workaround

The issue's root cause is `scipy_loss_precision` never passing `f_scale` to
scipy while trust's `Loss.apln` bakes in its own non-1.0 constants for
`huber`/`cauchy`. The fix closes that gap directly (`_f_scale_for` supplies
trust's own constant on the scipy side) rather than adjusting the reported
numbers after the fact or silently normalising both backends to a shared
value trust itself doesn't use.

### Note: APL backend's blanket rejection is consistent with x_scale's precedent

`apl_backend.py`'s new `f_scale` check follows the same shape as the existing
`x_scale` rejection immediately above it: raise rather than silently drop a
parameter the backend has no native equivalent for. The comment explains why
(fixed per-loss constant plus call-by-call MAD auto-scaling, neither
overridable), matching this file's existing comment style.

### Note: methodology.md addition is well-evidenced

The new "Robust loss tuning constants" section states the mismatch, the
evidence it was the actual driver of the measured gap (gap size tracking the
constant ratio, not a general robustness difference), and a direct
re-verification post-fix (`cauchy` gap narrowing from `1.69` vs `7.86` to
`7.82` vs `7.86` at `fraction=0.3`). No narrative markers, no em-dashes, UK
spelling throughout.

`scipy_loss_precision`'s trust-vs-scipy comparison to the same tuning constant.

### Note: constants verified against Loss.apln

`_F_SCALE_FOR_LOSS`'s pinned values (`huber=1.345`, `cauchy=2.385`,
`soft_l1=1.0`, `arctan=1.0`) match `backends_ext/apl/trust/APLSource/Loss.apln`
exactly (`huber←1.345`, `cauchy←2.385`, `softl1←1`, `arctan←1`).

### Note: test design choices are sound

- `test_scipy_loss_precision_passes_trusts_tuning_constant_as_f_scale` uses
  `huber` specifically, where trust's constant (1.345) differs from scipy's
  default (1.0) - `soft_l1`/`arctan` would pass vacuously even if the wiring
  were broken, since trust and scipy already agree at 1.0 for those losses.
- `test_f_scale_left_none_keeps_scipys_own_default` is a genuine no-op
  regression check: it already passes today, confirming the field's mere
  addition to `RunConfig` won't perturb the untouched path.
- `test_every_method_rejects_f_scale` parametrises over all three APL methods
  (`lm`, `BFGS`, `trust-exact`), matching "every method" in the test name.

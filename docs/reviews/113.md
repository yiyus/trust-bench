# Review: Issue #113 - Flaky timing assertion in test_dense_hessian_methods_cost_far_more_per_step_at_large_dimension

## Summary

The fix replaces a single wall-clock measurement per side of the ratio with a median of three repeats, and loosens the threshold from `5x` to `3x` to leave margin below the `~5.8x` worst case measured directly under load. This is test-only, matches the stated root cause (a short, noisy reference measurement), and preserves the test's actual claim. No issues found.

## Findings

No issues found.

### Note: root cause and fix approach confirmed

The PR doc's own measurements (single-shot ratios as low as `5.07x`; median-of-3 raising the observed floor to `~5.8x` over 15 trials) directly explain the intermittent failures reported in #28/#32: the `L-BFGS-B` reference call at `n=1000, max_iter=2` is short enough (~0.1-0.2s) that scheduling noise alone can shift it by 2x between runs. Taking the median of repeats on both sides of the ratio is a relative/percentile-based fix as the acceptance criteria requested, not a threshold change alone; the `3x` threshold still leaves a comfortable margin below the measured floor while keeping the assertion meaningful (dense-Hessian methods still have to cost at least 3x more per step to pass).

### Note: comment hygiene

The new comment on `_SLOW_METHOD_TIME_RATIO` explains the *why* (noise in short single-shot measurements, the specific ratios measured, the margin chosen) rather than narrating the investigation. No GitHub issue numbers appear in the test file; issue references (#28, #32) are correctly confined to the PR doc.

### Note: reproduction verified independently

Re-ran `test_dense_hessian_methods_cost_far_more_per_step_at_large_dimension` three times independently after the fix: all three passed (11.65s, 11.84s, 13.19s), consistent with the PR doc's own repeated-run and load-testing verification.

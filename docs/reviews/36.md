# Review: Issue #36 - backends_ext/apl/ harness

## Summary

This is the implementation review (tests and implementation were both present
in the working tree, uncommitted, with no prior tests-only review recorded).
The harness is functionally correct: all five new tests pass against a real
`dyalogscript`, the full suite (228 tests) is green, and `ruff` is clean.
Verification with the `dyalog-script` skill confirms the Rosenbrock
residual/Jacobian values and the exit-code/status contract described in
`docs/prs/36.md` are accurate. Two issues need fixing before merge: a stray
42 MB Dyalog crash dump (`aplcore`) sitting in the working tree that must not
be committed, and an error-message path that silently discards the one
diagnostic detail a caller would need to debug a harness crash. A third
finding flags an uncommented, easily-misread APL construct that carries real
correctness weight for every future native problem implementation.

## Findings

### Critical: Untracked Dyalog crash dump in the working tree

`aplcore` (42 MB, untracked) is a Dyalog core-dump file, almost certainly left
over from a crashed `dyalogscript`/`⎕SE.Link` session during development (the
PR notes mention the `⎕SE.Link.Create` watcher hang that was worked around).
It must not be staged or committed. Add `aplcore` to `.gitignore` so a future
crash during this same workflow doesn't silently end up in a commit.

Location: `/workspace/aplcore` (repo root)

### Major: Uncaught-error path reports a generic category, not the diagnostic message

`run.dyalog`'s `:Trap 0 ... :Else res←ErrorResult ⎕EM ⎕EN` builds the `message`
field from `⎕EM ⎕EN`, which is the monadic *Event Message* function applied to
the error number — it returns only the generic category name (e.g.
`"DOMAIN ERROR"`), not the specific diagnostic. Verified directly:

```
$ bash backends_ext/apl/run_harness.sh <(echo 'not valid json{{{') /tmp/out.json
{"message":"DOMAIN ERROR", "status":"ERROR", ...}
```

The detail that would actually help a caller — e.g. `"DOMAIN ERROR: JSON
import: unexpected character at offset 0"` — is available via `⎕DM` and is
discarded. `test_reports_error_status_for_malformed_input` only asserts
`message is not None`, so this passes the test suite while providing none of
the diagnostic value the field exists for. This is also inconsistent with the
sibling `ErrorResult` call in `Solve` (`'Unknown problem_id: ',req.problem_id`),
which does carry a specific, useful message.

Location: `backends_ext/apl/run.dyalog:10`

### Major: Undocumented, easily-misread APL construct for the residual/Jacobian pair

`problems/rosenbrock.dyalog` is one line:

```apl
Rosenbrock←{p q←⍵ ⋄ ((10×q-×⍨p),1-p)[(-20×p)10 ⋄ ¯1 0]}
```

This is not bracket indexing despite the visual shape. Verified with
`dyalogscript`: the trailing `[... ⋄ ...]` is parsed as array notation (rows
separated by `⋄`, auto-padded, e.g. `V[99 ⋄ 2 3]` never raises `INDEX ERROR`
even though `99` is out of bounds for a 3-element `V`), and the resulting
matrix is stranded together with the untouched residual vector rather than
used to index it. The construct is correct (confirmed the residual and
Jacobian values match the analytic Rosenbrock formulas at `p q = ¯1.2 1.0`),
but nothing marks the trick as deliberate. On first read it looks like ordinary
indexing with a syntax mistake, which is exactly how it was initially
misread during this review, and the PR notes already flag that a second native
problem is expected to follow "issue #37 should design a dispatch table...
against the full problem set" — whoever writes that second problem needs to
either understand or reimplement this pattern from scratch with nothing to go
on.

Location: `backends_ext/apl/problems/rosenbrock.dyalog:1`

### Minor: A single `tolerance` field silently retunes an expert-only trust parameter

`solve.dyalog` sets both `cfg.tolc` and `cfg.tolr` from the one
caller-supplied `req.tolerance`. `trust`'s own README states: "With the
exception of `toli` and `tolc`, configuration parameters should generally be
modified only by expert users or in case of convergence problems" — `tolr` is
one of those parameters. Every caller that tightens `tolerance` for `tolc`
also retunes `tolr` as a side effect, contrary to `trust`'s guidance, with no
note in `docs/prs/36.md` explaining the choice. This may be a deliberate,
reasonable simplification for a harness that only exposes one tolerance knob,
but it should be recorded as a decision rather than left implicit.

Location: `backends_ext/apl/solve.dyalog:11-12`

### Note: `DIVERGED` and `FAILED` statuses are unreached by the test suite

Section 4.3's `RunStatus` enumerates `CONVERGED | MAX_ITER | FAILED |
DIVERGED | ERROR`. `Solve` can produce `CONVERGED`, `MAX_ITER`, or `ERROR`
directly; `FAILED` is reachable (`r.dnorm>r.dmax`) but untested, and
`DIVERGED` is never produced by this harness at all. Not necessarily wrong —
`FAILED` may be the harness's catch-all for what Python-side backends call
`DIVERGED` — but this mapping isn't documented anywhere.

### Note: `prototype/` is untracked and out of scope for this issue

`prototype/harness.py`, `prototype/large_residual.py`, and
`prototype/run_experiment.py` are untracked in the working tree alongside the
issue #36 files. The architecture doc describes `prototype/` as "the
validated throwaway experiment, for reference only" from an earlier phase, so
these are presumably intentional, but they have nothing to do with this
issue's acceptance criteria. Worth committing separately (or confirming they
already belong to a prior, unmerged commit) so this PR's diff stays scoped to
the harness.

Request changes: fix the crash dump and the `⎕EM`/`⎕DM` message before
merge; address or explicitly accept the `tolr` coupling and the uncommented
array-notation construct.

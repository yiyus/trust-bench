# Review: Issue #36 - backends_ext/apl/ harness

## Summary

This is the implementation review (tests and implementation were both present
in the working tree, uncommitted, with no prior tests-only review recorded).
The harness is functionally correct: all five new tests pass against a real
`dyalogscript`, the full suite (228 tests) is green, and `ruff` is clean.
Verification with the `dyalog-script` skill confirms the Rosenbrock
residual/Jacobian values and the exit-code/status contract described in
`docs/prs/36.md` are accurate. Two issues need fixing before merge: a stray
42 MB Dyalog crash dump (`aplcore`) sitting in the working tree that must not
be committed, and an error-message path that silently discards the one
diagnostic detail a caller would need to debug a harness crash.

## Findings

Location: `/workspace/aplcore` (repo root)

### Major: Uncaught-error path reports a generic category, not the diagnostic message

`run.dyalog`'s `:Trap 0 ... :Else resÔćÉErrorResult ÔÄĽEM ÔÄĽEN` builds the `message`
field from `ÔÄĽEM ÔÄĽEN`, which is the monadic *Event Message* function applied to

{"message":"DOMAIN ERROR", "status":"ERROR", ...}

discarded. `test_reports_error_status_for_malformed_input` only asserts
`message is not None`, so this passes the test suite while providing none of
the diagnostic value the field exists for. This is also inconsistent with the
sibling `ErrorResult` call in `Solve` (`'Unknown problem_id: ',req.problem_id`),
which does carry a specific, useful message.

Location: `backends_ext/apl/run.dyalog:10`

### Note: Rosenbrock's residual/Jacobian construction verified correct

`problems/rosenbrock.dyalog` builds the (residual, Jacobian) pair using
Dyalog v20 array notation (`Ôőä`-separated rows inside `[...]`, stranded with
the preceding array) rather than bracket indexing. Initially misread this as
bracket indexing during review; verified with `dyalogscript` that it is
standard array notation, correctly used, and confirmed the residual and
Jacobian values match the analytic Rosenbrock formulas at `p q = ┬»1.2 1.0`.
No change needed.

Location: `backends_ext/apl/problems/rosenbrock.dyalog:1`

### Note: `DIVERGED` and `FAILED` statuses are unreached by the test suite

Section 4.3's `RunStatus` enumerates `CONVERGED | MAX_ITER | FAILED |
DIVERGED | ERROR`. `Solve` can produce `CONVERGED`, `MAX_ITER`, or `ERROR`
directly; `FAILED` is reachable (`r.dnorm>r.dmax`) but untested, and
`DIVERGED` is never produced by this harness at all. Not necessarily wrong ÔÇö
`FAILED` may be the harness's catch-all for what Python-side backends call
`DIVERGED` ÔÇö but this mapping isn't documented anywhere.

### Note: `prototype/` is untracked and out of scope for this issue

`prototype/harness.py`, `prototype/large_residual.py`, and
`prototype/run_experiment.py` are untracked in the working tree alongside the
issue #36 files. The architecture doc describes `prototype/` as "the
validated throwaway experiment, for reference only" from an earlier phase, so
these are presumably intentional, but they have nothing to do with this
issue's acceptance criteria. Worth committing separately (or confirming they
already belong to a prior, unmerged commit) so this PR's diff stays scoped to
the harness.

# Review: Issue #121 - Implementation (GREEN)

## Summary

The fix is minimal and addresses root cause: `_objective` branches on `problem.kind`, using `residual(x)` directly as the cost for `kind="scalar"` with no sum-of-squares wrapping, and `solve()` raises explicitly for `kind="scalar"` routed to `lm`/`trf`/`dogbox` before scipy's own incidental (and, for `trf`/`dogbox`, silently wrong) behaviour can occur. `_gradient` needed no change: it is never reached for a `kind="scalar"` problem, since `problem.jacobian is None` already forces the finite-difference path. All 12 previously-RED tests now pass; the 4 structural tests are unaffected. No issues found.

## Findings

No issues found.

### Note: fix addresses root cause, not a workaround

The guard against `lm`/`trf`/`dogbox` is placed before `_solve_least_squares` is called at all, so no scipy call for a `kind="scalar"` problem reaches `least_squares` under any method - this rules out the `trf`/`dogbox` correctness landmine described in the PR doc (silently minimising the scalar cost as if it were a single residual) rather than just papering over the exception message for `lm`. `_objective`'s scalar branch reads `residual(x)` directly rather than reshaping/wrapping it, matching the model: a `kind="scalar"` problem's `residual()` already *is* the cost.

### Note: comment hygiene maintained in the implementation diff

Both new comments in `scipy_backend.py` explain the invariant (why `kind="scalar"` needs different handling, why `trf`/`dogbox` would otherwise be silently wrong) rather than narrating the fix. No issue-ID references. Commit message follows project convention (imperative subject, invariant-focused body).

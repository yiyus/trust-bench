## Findings

### Minor: Two tests each assert more than one fact

`test_ci_workflow_triggers_on_push_and_pull_request` asserts both the
`push:` and `pull_request:` triggers in one test, and
`test_ci_workflow_runs_test_lint_and_coverage_targets` loops over three
targets in one test. This departs from "one test, one behaviour" in
`CLAUDE.md`. Both read as intentional groupings around a single acceptance
criterion each ("triggers on every push and pull request", "runs the same
targets"), and pytest's assertion rewriting still identifies which trigger or
target failed, so diagnostic quality is not lost. Splitting into
one-assertion-per-test would align more strictly with the convention if that
matters more than the current test count.

Location: `tests/unit/test_scaffolding.py:38-47`

### Note: Trigger regex is unanchored to the `on:` block

`re.search(r"^\s*push\s*:", ...)` matches a `push:` key anywhere in the file,
not specifically nested under `on:`. For a fresh CI workflow this is not a
practical risk, but it means the test would still pass if `push:` appeared
in an unrelated part of the YAML (e.g. a step name or comment reused as a
key). Not worth a fix at this scaffolding stage.

Location: `tests/unit/test_scaffolding.py:40-41`

### Findings

#### Minor: `prototype/` is untracked, so the new ruff exclude has no effect outside the author's own working copy

`pyproject.toml`'s `tool.ruff.exclude = ["prototype"]` is justified in the PR
body as keeping the throwaway experiment `docs/plans/trust-bench.md` Section
1.1 describes out of lint. `docs/plans/trust-bench.md` Section 3 lists
`prototype/` as part of the repository layout ("the validated throwaway
experiment, for reference only"), implying it belongs in the tree. In fact
`prototype/` has never been committed (`git log --all -- prototype/` returns
nothing) and is not listed in `.gitignore`; it shows as untracked in
`git status`. On a fresh checkout, including the one CI runs against,
`prototype/` does not exist, so this exclude is currently dead configuration
there: `ruff check .` would pass without it. The seven violations the PR body
reports (verified directly: `E702` multiple-statements-per-line in
`prototype/run_experiment.py`, plus an unused import) are real, but only
reproducible on the machine where `prototype/` happens to sit on disk.

This is not a regression this PR introduces, `prototype/` was already
untracked before this change, but this PR is the first to add configuration
that assumes its presence, so it is a reasonable place to resolve the
inconsistency: either commit `prototype/` (making the exclude meaningful for
every clone and matching the design doc's repo layout) or add it to
`.gitignore` (making its absence from git explicit and correct, and the ruff
exclude then redundant with the default VCS-ignore handling).

Location: `pyproject.toml:20`; `docs/plans/trust-bench.md` Section 3.

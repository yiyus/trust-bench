# Implementation review

## Summary

`compare_with_provenance`/`classification_counts` in
`trust_bench/reporting/compare.py`, a `"compare"` section in
`build_html_report`, and a `compare` subcommand/`run_compare` in `cli.py`
correctly deliver the acceptance criterion as clarified by the issue's own
scope comment: `trust-bench compare` folds a full-provenance regression/drift
table and a classification-counts graph into the same `report.html` that
`trust-bench report --html` produces. All 77 targeted tests pass, `ruff` is
clean. Two findings: a source comment names the issue number directly
(comment-hygiene rule), and `_comparison_section` duplicates
`_study_section` exactly rather than reusing it. Neither blocks merge.

## Findings

### Minor: comment names the issue number directly

`compare.py:86` reads "what `trust-bench compare` writes and reports (#43's
acceptance criterion)". The project convention (and this repo's own
`crev` checklist) keeps issue IDs out of source comments - they belong in
PR docs, review docs, and commit trailers, not in code that outlives the
issue tracker. Reword to state the invariant directly, e.g. "a regression
or drift needs its full environment to be attributable, not only the
`machine_fingerprint`/`backend_version` `compare()` itself needs for
`drift_summary`'s grouping" (the rest of the docstring already says this
without needing the issue reference at all).

Location: `trust_bench/reporting/compare.py:86`

### Minor: `_comparison_section` duplicates `_study_section` rather than calling it

`_comparison_section(output_dir)` (html_report.py:232-238) checks for
`compare.csv`, renders the table, and appends the plot - exactly what
`_study_section(output_dir, name)` already does when `AUX_TABLES.get(name,
[])` is empty, which it is for `"compare"` (`AUX_TABLES` has no `"compare"`
entry). `build_html_report` could call `_study_section(output_dir,
"compare")` directly and `_comparison_section` could be deleted; the two
functions are byte-for-byte equivalent for this input. Not a correctness
issue, but it is duplicate logic that will need updating twice if
`_study_section`'s rendering ever changes (timing formatting, cell
rendering, etc.).

Location: `trust_bench/reporting/html_report.py:232-238`

### Note: `compare_with_provenance`'s second merge is redundant but not unsound

`compare_with_provenance` (compare.py:80-97) calls `compare()` (which merges
baseline/candidate once internally) and then merges the same two frames on
the same `key_columns` again to pull out provenance, concatenating the two
results by position (`pd.concat(..., axis=1)`) rather than by key.
Independently verified with a two-row, differently-ordered baseline/
candidate pair that this stays correctly aligned: pandas' inner merge is a
deterministic function of its inputs, so both merges produce identical row
order given identical baseline/candidate frames and `key_columns`. Not a
bug, but it computes the join twice and its correctness rests on that
determinism rather than an explicit shared key - a comment noting the
positional-alignment assumption (or merging provenance directly into
`classified` on `key_columns` instead of concatenating by position) would
make this more robust to future changes to `compare()`'s internals.

Location: `trust_bench/reporting/compare.py:80-97`

### Note: architectural conformity and comment hygiene otherwise clean

`run_compare` reuses `storage.load`, `save_table`, `plot_metric_by_category`,
`build_html_report`/`save_html_report` rather than introducing parallel
code paths, matching `run_report`'s existing structure. The `compare`
subparser and `main` wiring follow the same pattern as the `report`
subcommand. `_comparison_section` is placed first in `build_html_report`
(right after the hero, before Tier 1), matching the PR doc's stated
rationale ("a regression or drift is the first thing a reader should see").
No other GitHub issue references found in any touched source file.

## Verification

`_comparison_section` duplication is worth cleaning up in this cycle too,
but neither blocks shipping the behaviour itself, which is correct and well
tested.

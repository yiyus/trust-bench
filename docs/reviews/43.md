### Note: test design follows established conventions closely

`test_html_report.py`'s new `_write_compare` fixture stringifies
`changed_tier1_metrics` (`"['dist_to_opt']"`) rather than passing a real
list, correctly modelling what a CSV round trip through `save_table`/
`pd.read_csv` actually produces, rather than a synthetic in-memory shape that
wouldn't occur in practice. The new "compare" section is exercised with the
same present/absent-artefact pattern every other study section already
uses (`test_no_timing_caveat_is_shown_when_no_study_measured_timing` and
neighbours), so it slots into `build_html_report`'s existing conditional
structure rather than inventing a new one.

`test_compare_with_provenance_includes_every_baseline_and_candidate_provenance_field`
iterates `EnvProvenance.__dataclass_fields__` rather than hardcoding a field
list, so it stays correct if `EnvProvenance` gains or loses a field later.

The CLI tests (`test_compare_command_parses_baseline_and_candidate_paths`,
`test_compare_command_accepts_output_dir_and_html_flag`) test `build_parser`
argument parsing separately from `run_compare`'s file-writing behaviour and
from `main`'s end-to-end wiring, mirroring the existing `report` command's
own test structure in the same file.

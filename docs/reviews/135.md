# Review: Issue #135 - Restructure the HTML report: headline artifacts first, themed grouping, status pills

## Summary

Tests-only (RED) stage review. The new/rewritten test files define the
contract for a new `trust_bench.reporting.colors` module (fixed backend/status
colours), fixed colours threaded through the three plot functions, five new
`timing_*` columns in `results_to_dataframe`, and a full rewrite of
`test_html_report.py` replacing the old naive glob-based assertions with
tests for the three-tier structure, theme headings, captions, status pills,
the backend legend, and the timing display/caveat. RED is genuinely
confirmed. Two test comments cite GitHub issue numbers, against the
project's own explicit rule; everything else checks out cleanly against the
design doc and the existing `tables.py`/`plots.py` code shape.

## Findings

### Major: two test comments cite GitHub issue numbers

CLAUDE.md: "No GitHub issue IDs in source code or test comments... reasoning
by issue number forces the reader to context-switch instead of understanding
the code on its own." Two instances:

- `tests/reporting/test_colors.py:34`: `# UNSUPPORTED (#125) is a
  declared-unsupported rejection, an expected outcome of a study's own
  probe...`. The parenthetical issue number adds nothing the rest of the
  sentence doesn't already say; the reasoning already stands on its own
  (confirmed against `tables.py`'s own docstring, which makes the same point
  with no issue reference at all).
- `tests/reporting/test_tables.py:129`: `# measure_timing defaults to False
  (#138): most results have no timing at all...`. Same fix: state the
  invariant, drop the reference.

Location: `tests/reporting/test_colors.py:34`; `tests/reporting/test_tables.py:129`

### Note: theme groupings and captions verified against docs/plans/trust-bench.md Section 9

The test fixture's study-to-theme assignment (`baseline` under "Common
problems", `large_residual` under "Difficulty axes", `robust_loss` under
"Special capabilities") matches the issue's own proposed grouping. The
caption fragments the tests assert on ("floor every backend must clear" for
baseline, "Gauss-Newton" for large_residual) are close paraphrases of
Section 9's own item 1 ("the floor everything must clear") and item 2
("predicted vs measured Gauss-Newton failure boundary") - consistent with
the issue's own instruction to adapt the design doc's existing descriptions
rather than write new ones from scratch.

### Note: NON_RESULT_STATUSES / UNSUPPORTED handling is consistent with the existing implementation

`test_a_declared_unsupported_rejections_timing_columns_are_also_blank` and
`test_status_color_treats_unsupported_as_neutral_not_a_failure` build on
`tables.py`'s existing `NON_RESULT_STATUSES = frozenset({"ERROR",
"UNSUPPORTED"})` and its documented rationale (an exception row gets every
metric field blanked, `message` kept) - the new `timing_*` columns are a
natural, minimal extension of that existing shape, not a new pattern.

### Note: fixed-colour contract is a genuinely new requirement, correctly RED

`trust_bench/reporting/plots.py` has no fixed colour scheme today (colours
come from matplotlib's own default cycling); the three new
`test_plots.py` tests asserting a shared `colors.backend_color()` per
backend, consistent across `plot_metric_vs_sweep`, `plot_metric_by_category`,
and `plot_capability_frontier` regardless of group order, correctly capture
the issue's stated problem ("matplotlib's own default colour cycling
currently assigns backend colours inconsistently across different plots").

## Verification

- APL semantics: not applicable - no APL expressions in this test surface.

  - `tests/reporting/test_colors.py`, `test_plots.py`, `test_html_report.py`:
    fail at collection with `ModuleNotFoundError`/`ImportError` on

  - `tests/reporting/test_tables.py`: 4 failed (`KeyError` on the new

    `docs/prs/135.md`'s own account exactly.

## Recommendation

Approve with minor changes (drop the two issue-number references from test
comments before committing).

# Implementation review (GREEN)

## Summary

`trust_bench/reporting/colors.py` (new), and the `plots.py`/`tables.py`/
`html_report.py` changes wire it through, matching the RED-stage contract.
The two issue-number test comments flagged in round 1 are fixed. The
implementation is architecturally sound: fixed backend colours resolve
correctly through both the single-column and multi-column groupby paths in
`plots.py`, the timing-column flattening in `tables.py` reuses the existing
`NON_RESULT_STATUSES`/exception-blanking shape rather than inventing a new
one, and `html_report.py`'s `AUX_TABLES` correctly identifies the two
existing basin-rate CSVs (`baseline_basin_rates.csv`,
`large_residual_basin_rates.csv`) that `cli.py` already writes, nesting them
under their parent study instead of leaving them as orphaned Tier-2 entries.
One inaccuracy in `docs/prs/135.md`'s own test count.

## Findings

### Fixed: issue-number references removed from test comments

Both instances flagged in round 1
(`tests/reporting/test_colors.py:34`, `tests/reporting/test_tables.py:129`)
are gone; each comment now states the invariant directly. Confirmed by
reading the current file contents.

### Note: backend-colour resolution verified across both groupby shapes

`_backend_from_group` in `plots.py` handles a bare `"backend"` group
(`plot_metric_vs_sweep`'s tuple-free case) and a multi-column group (where
pandas `groupby` yields tuple keys) identically to how
`plot_metric_by_category` reconstructs a list from its own `"/"`-joined
`_group` label (`df[group].astype(str).agg("/".join, axis=1)` on write,
`group_name.split("/")` on read) - the join/split pair is consistent, so a
backend column embedded anywhere in a multi-column group still resolves to
the correct fixed colour.

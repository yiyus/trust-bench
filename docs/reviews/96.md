# Review: Issue #96 - Extend the loss vocabulary to reach trust's Tukey, Welsch and Fair losses

## Summary

Tests-only (RED) stage review. All three new/extended test files fail for the stated reason (missing loss mapping, missing new functions), and the underlying empirical/design claims check out against the actual `trust` APL source. No issues found.

## Findings

### Note: reproduction confirmed exactly

Ran `tests/backends/test_apl_loss_vocabulary.py` directly: all 4 tests fail against the unmodified `apl_backend.py` - the capabilities check with a plain `AssertionError`, and each of the three new losses with `ValueError: lm does not support loss=<name>`, raised from `apl_backend.py:152`. `tests/studies/test_robust_loss.py` and `tests/studies/test_robust_loss_apl.py` both fail to collect with `ImportError: cannot import name 'TRUST_LOSSES' from 'trust_bench.studies.robust_loss'`, since neither `TRUST_LOSSES` nor `trust_loss_precision` exist yet. All match the PR doc precisely.

### Note: Tukey/Welsch/Fair genuinely exist in trust, with the expected calling convention

Confirmed directly in `backends_ext/apl/trust/APLSource/Loss.apln:16,20,23`: `Tukey`, `Welsch`, and `Fair` are implemented as dyadic functions with the same calling convention as the five losses already mapped (`L2`/`SoftL1`/`Huber`/`Cauchy`/`Arctan`, `apl_backend.py:28-32`), and `backends_ext/apl/trust/README.md:117` lists all eight as valid `loss` values. The existing `_LOSS_TO_TRUST` comment (`apl_backend.py:23-26`) already flagged this exact gap before this issue's test surface was added, so the fix target is well-established, not speculative.

### Note: design rationale is sound

`trust_loss_precision` as a separate function (rather than a conditional branch inside `scipy_loss_precision`) keeps the existing five-loss, "standard"-start comparison untouched while isolating the new warm-start mechanism `welsch` needs. Sweeping `welsch` by default while still mapping `tukey`/`fair` (reachable directly via `RunConfig`, covered by `test_apl_loss_vocabulary.py`) is a defensible scope: the PR doc documents the empirical basis (`welsch` warm-started recovers to 45% contamination and only breaks down near the 49% theoretical limit; `tukey` remains erratic even warm-started) as confirmed with the user before committing to it. This is exactly the kind of judgment call this project's per-issue workflow expects to be surfaced and agreed before the RED commit, and it has been.

### Note: test design matches established precedent

`test_trust_loss_precision_skips_a_backend_it_has_no_method_mapping_for` and `test_trust_loss_precision_is_empty_when_no_backend_supports_it` both reuse the existing `_UnmappedBackend` stub and mirror `scipy_loss_precision`'s own skip-guard test exactly, rather than inventing a new pattern. `test_scipy_declares_none_of_trusts_redescending_losses` is a regression guard against `SciPyBackend`'s vocabulary accidentally growing to include a name it has no real equivalent for.

# Implementation review

## Summary

The fix (`7e83e71`) maps all three losses in `_LOSS_TO_TRUST`, adds `trust_loss_precision`/`TRUST_LOSSES`/a shared `_warm_start` helper to `robust_loss.py`, and wires the new rows into `cli.py`. All 16 tests across the three affected files pass, and the empirical claims from the tests-only design notes (welsch's exact recovery through 45% contamination, breakdown at 49%) were independently re-verified against the real harness. One self-found correction during GREEN - splitting the RED-stage `tukey` test once its basin-of-attraction fragility showed up against the real harness rather than only in the `outliers` sweep - is handled honestly rather than papered over. No issues found.

## Findings

### Note: fix addresses the root cause, not a workaround

The three new losses are added to `_LOSS_TO_TRUST` directly, so `lm`'s declared `losses` capability (derived from this dict's keys) picks them up automatically with no separate capability-declaration change needed - confirming the tests-only review's read that this really was a pure vocabulary gap, not a deeper capability-modelling problem. `trust_loss_precision` skips a `(backend, loss)` pair the backend doesn't declare support for, not just an unmapped backend name (`robust_loss.py:76-79`), a strictly more careful guard than `scipy_loss_precision`'s existing one, appropriate since this function is the first to sweep a loss not every mapped backend necessarily supports.

Location: `trust_bench/backends/apl_backend.py:23-33`, `trust_bench/studies/robust_loss.py:30-90`

### Note: empirical claims independently re-verified

Ran `trust_loss_precision(fractions=FRACTIONS, backends=[APLBackend()])` directly: `welsch` recovers to `~1e-10`-`1e-15` distance through 45% contamination and jumps to `9.36` at 49%, matching the PR doc's cited figures exactly. Separately confirmed `tukey`'s basin-of-attraction fragility directly: `APLBackend().solve(quadratic.PROBLEM, "lm", "standard", RunConfig(loss="tukey"))` returns `status=MAX_ITER`, `x_final == x0 == [1, -1]` (unmoved), while `welsch`/`fair` both converge to nearly `(0, 0)` from the same start - confirming the split test's premise (`tukey` can only honestly assert dispatch, not convergence) is a real property of the loss, not an implementation bug being hidden.

### Note: code quality

`_warm_start` is factored out of `irls_tukey`'s own inline computation into a shared helper used by both `irls_tukey` and `trust_loss_precision`, removing a duplication the tests-only review's design notes had already flagged as the intended shape - correctly delivered. `TRUST_LOSSES`'s module-level comment states the invariant (why `welsch` alone is swept by default) rather than narrating the investigation, and the new `test_lm_dispatches_tukey_without_raising` comment explains the underlying mechanism (a hard cutoff giving zero weight, hence zero gradient) rather than just describing the assertion. No GitHub issue references in any new source or test comment; no emojis or em-dashes.

### Note: honest handling of a self-found gap during GREEN

The RED-stage test asserted `CONVERGED` for `tukey` too; running it against the real harness during implementation surfaced `MAX_ITER` instead, a genuine basin-of-attraction property of Tukey's hard cutoff rather than a bug to fix. The commit splits the test into two rather than loosening the original assertion silently or forcing convergence artificially, and the PR doc documents exactly what was found and why the split is the correct response - matching this project's TDD conventions for a test that turns out to encode an incorrect expectation.

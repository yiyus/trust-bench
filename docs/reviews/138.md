# Review: Issue #138 - Instrument RunResult.timing: nothing actually measures Tier-3 performance yet

## Summary

Tests-only (RED) stage review. The new/extended test files define the
expected behaviour for a shared `summarize()` timing helper, a `mad()`
metric, and real `TimingStats` population in both `SciPyBackend` and
`APLBackend`. RED is genuinely confirmed: `trust_bench.core.timing` does not
exist and `mad` is not exported from `trust_bench.core.metrics`, matching
the errors recorded in `docs/prs/138.md`. No issues found.

## Findings

No issues found.

### Note: design and defaults verified against docs/plans/trust-bench.md

`WARMUP == 1` and `N_REPS == 5` are asserted directly against the fixed
constants rather than hardcoded per-test, and the Section 7 timing policy
("warm-up run(s) before measurement, N repetitions, pinned BLAS thread
count, report median + MAD") is satisfied by the shape of `TimingStats`
already in `result.py:19-24`. The `mad()` hand-computed test value
(`values=[1,2,3,4,100]`, median=3.0, absolute deviations `[2,1,0,1,97]`,
median of those =1.0, scaled by 1.4826) checks out arithmetically and
matches `robust_loss.py:124`'s existing scale computation, so the new
metric is consistent with the project's established convention rather than
introducing a second one.

### Note: APL timing test correctly gated and consistent with existing convention

`test_apl_timing.py`'s `pytestmark` (`pytest.mark.slow` +
`skipif(shutil.which("dyalogscript") is None, ...)`) matches every other
`tests/backends/test_apl_*.py` file. The IPC-overhead assertion
(`result.timing.median <= external_elapsed`) compares the median of N
internal repetitions against one external wall-clock wrap of a full
warmed-up `solve()` call (which itself performs the internal repetitions
plus IPC round trips) - a safe, non-flaky comparison rather than a tight
race, since the reported median should be well under the external figure
by construction if the fix is implemented correctly.

## Verification

- APL semantics: no inline APL expressions in these tests to check with
  `dyalog-script`; `test_apl_timing.py` only exercises `APLBackend.solve()`
  end-to-end (gated on `dyalogscript` availability, consistent with the
  rest of the suite).
- Tests: RED confirmed. Running the four new/changed test files against
  unmodified `main` produces `ImportError: cannot import name 'mad' from
  'trust_bench.core.metrics'` and `ModuleNotFoundError: No module named
  'trust_bench.core.timing'` across all four files, exactly as recorded in
  `docs/prs/138.md`.
- `ruff check` on all four new/changed test files: clean.

## Recommendation

Approve

# Implementation review (GREEN)

## Summary

Both backends now populate `RunResult.timing` with real `TimingStats`, matching
the RED-stage tests and the timing policy in `docs/plans/trust-bench.md`
Section 7. The scipy path pins a single BLAS thread via `threadpoolctl` and
times `N_REPS` repetitions with `time.perf_counter()`; the APL path reads a new
`solve_ms` field that `solve.dyalog` computes from `⎕AI[2]` deltas around the
`Min` call, avoiding IPC/JSON-transfer overhead in the reported number.
Verified `⎕AI[2]` directly against the Dyalog docs (`docsearch -r 268`): it is
cumulative compute time for the session in milliseconds, so a before/after
delta is a correct measure of CPU time spent in the wrapped call. One comment
hygiene issue recurs in two files.

## Findings

### Major: source comments cite GitHub issue numbers, against the project's own explicit rule

CLAUDE.md states plainly: "No GitHub issue IDs in source code or test
comments... Never write `// fixes #42`, `// see issue 123`... reasoning by
issue number forces the reader to context-switch instead of understanding the
code on its own." Two places do this:

- `backends_ext/apl/solve.dyalog:90`: `⍝ ... overhead (#138).` - the
  parenthetical adds nothing a reader can't already get from the surrounding
  three lines of prose; drop it.
- `trust_bench/backends/apl_backend.py:362` and `:364`: "a few ms each, per
  #139" and "What #139 doesn't solve on its own". Both should state the
  invariant directly - the persistent session removes interpreter-startup
  cost from each repetition, but still leaves IPC/JSON-transfer cost, which is
  why the measurement happens inside `solve.dyalog` rather than by
  wall-clocking `_send_request` - without naming the issue.

Location: `backends_ext/apl/solve.dyalog:90`; `trust_bench/backends/apl_backend.py:362,364`

## Verification

- Tests: `tests/unit/test_metrics.py`, `tests/unit/test_timing.py`,
  `tests/backends/test_scipy_timing.py`, `tests/backends/test_scipy_least_squares.py`,
  `tests/backends/test_scipy_minimize.py`, `tests/backends/test_scipy_f_scale.py`,
  `tests/backends/test_scipy_scalar_cost.py` - 93 passed.
  `tests/backends/test_apl_timing.py`, `tests/backends/test_apl_session.py`
  (`dyalogscript` available in this environment) - 6 passed, including the
  IPC-overhead assertion (`result.timing.median <= external_elapsed`).
  Full suite already running in another session; not repeated here.
- `ruff check` on `trust_bench/` and `tests/`: clean.
- APL semantics: confirmed `⎕AI[2]` directly (`docsearch -r 268`, and a live
  `dyalogscript` check) is compute time in milliseconds for the session, so
  the `t0`/`solveMs` delta in `solve.dyalog:90-92` correctly isolates the
  `Min` call's own CPU time from request parsing and response assembly.
- Edge case checked, not covered by a test: in `apl_backend.py`'s
  `solve()`, if a measured repetition (not the warm-up) returns `ERROR` after
  earlier repetitions already succeeded, the loop breaks and the final
  `response` used to build `RunResult` is the error response, discarding the
  earlier successful solves rather than reporting the result they found. This
  matches the pre-existing single-call convention (`response` was always used
  as-is regardless of status) and a numerical solve erroring intermittently on
  identical repeated inputs would itself be a surprising, notable event - not
  blocking, but worth a regression test if this is ever observed in practice.

## Recommendation

Approve with minor changes (drop the two issue-number references)

# Implementation review, round 2 (post-redesign)

## Summary

Two commits landed since the previous round: `c5dd481` drops the two
issue-number comments flagged above (confirmed fixed, verbatim), and
`0630228` makes the `WARMUP+N_REPS` measurement opt-in via a new
`RunConfig.measure_timing: bool = False`, in response to a full-suite
regression (`docs/prs/138.md`'s "Post-GREEN redesign" section: a run that
`#139` had brought to ~16 minutes was still running past 40 before being
stopped). The performance problem and its diagnosis are real and well
documented. The fix, however, leaves the issue's own acceptance criterion
unmet for every real invocation path in the harness.

## Findings

### Critical: no real caller ever sets `measure_timing=True` - every study, report, and CLI run still gets `timing=None`

`RunConfig.measure_timing` defaults to `False`, and `solve()` in both
backends only measures anything when it is `True`. Grepping the non-test
tree for `measure_timing` turns up exactly three lines: the field
definition and its two `if not config.measure_timing` checks. Every
`RunConfig(...)` construction in `trust_bench/studies/*.py`,
`trust_bench/reporting/*.py`, and `trust_bench/cli.py`'s `report`
subcommand (the only CLI command that runs studies end-to-end) omits it,
so every one of them keeps getting `timing=None` - identical to the state
before this issue, not the "real `TimingStats`, not `None`" the issue's
acceptance criteria asks for.

`docs/methodology.md:420` and `docs/prs/138.md:136` both assert "a caller
that explicitly wants real `TimingStats` (report generation) sets
`measure_timing=True`" as if this were already done; it is not - no line
in `reporting/` or `studies/` does this. The Phase 10 epic (`#41`-`#44`)
that this issue exists to unblock needs real numbers flowing out of an
actual `trust-bench report` run, not just out of a `RunConfig` a test file
happens to construct directly.

The regression that motivated the opt-in is legitimate and the right
constraint to design around, but the fix stopped one step short: nothing
threads `measure_timing=True` through to the report-generation call
sites (or exposes it as a `--measure-timing`/similar CLI flag that those
call sites read), so the feature this issue asks for is reachable only
from test code.

Location: `trust_bench/core/config.py:20`; no call site in
`trust_bench/studies/`, `trust_bench/reporting/`, or `trust_bench/cli.py`
sets it.

### Fixed: issue-number references removed from source comments

Both instances flagged in round 1 (`backends_ext/apl/solve.dyalog:90`,
`trust_bench/backends/apl_backend.py:362,364`) are gone; the surrounding
prose was rewritten to state the invariant directly rather than deleting
context. Confirmed by reading the current file contents.

## Verification

- Tests: not re-run per instruction; `docs/prs/138.md` records the
  post-redesign suites passing (94 scipy, 4 APL) and this round's `git
  show` of the two commits confirms the diffs match that description.
- `ruff check`: not re-run this round; no changes to test files' import
  structure that would newly trigger it.
- Confirmed by direct grep that `measure_timing` has zero production call
  sites outside the two backend files' own default-check and the test
  files that pass it explicitly.

## Recommendation

Request changes - wire `measure_timing=True` (or an equivalent CLI/report
opt-in) into at least the report-generation path so a real `trust-bench
report` run actually produces `TimingStats`, not just backend unit tests.

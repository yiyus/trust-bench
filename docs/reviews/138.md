# Review: Issue #138 - Instrument RunResult.timing: nothing actually measures Tier-3 performance yet

## Summary

Tests-only (RED) stage review. The new/extended test files define the
expected behaviour for a shared `summarize()` timing helper, a `mad()`
metric, and real `TimingStats` population in both `SciPyBackend` and
`APLBackend`. RED is genuinely confirmed: `trust_bench.core.timing` does not
exist and `mad` is not exported from `trust_bench.core.metrics`, matching
the errors recorded in `docs/prs/138.md`. No issues found.

## Findings

No issues found.

### Note: design and defaults verified against docs/plans/trust-bench.md

`WARMUP == 1` and `N_REPS == 5` are asserted directly against the fixed
constants rather than hardcoded per-test, and the Section 7 timing policy
("warm-up run(s) before measurement, N repetitions, pinned BLAS thread
count, report median + MAD") is satisfied by the shape of `TimingStats`
already in `result.py:19-24`. The `mad()` hand-computed test value
(`values=[1,2,3,4,100]`, median=3.0, absolute deviations `[2,1,0,1,97]`,
median of those =1.0, scaled by 1.4826) checks out arithmetically and
matches `robust_loss.py:124`'s existing scale computation, so the new
metric is consistent with the project's established convention rather than
introducing a second one.

### Note: APL timing test correctly gated and consistent with existing convention

`test_apl_timing.py`'s `pytestmark` (`pytest.mark.slow` +
`skipif(shutil.which("dyalogscript") is None, ...)`) matches every other
`tests/backends/test_apl_*.py` file. The IPC-overhead assertion
(`result.timing.median <= external_elapsed`) compares the median of N
internal repetitions against one external wall-clock wrap of a full
warmed-up `solve()` call (which itself performs the internal repetitions
plus IPC round trips) - a safe, non-flaky comparison rather than a tight
race, since the reported median should be well under the external figure
by construction if the fix is implemented correctly.

## Verification

- APL semantics: no inline APL expressions in these tests to check with
  `dyalog-script`; `test_apl_timing.py` only exercises `APLBackend.solve()`
  end-to-end (gated on `dyalogscript` availability, consistent with the
  rest of the suite).

  'trust_bench.core.metrics'` and `ModuleNotFoundError: No module named
  'trust_bench.core.timing'` across all four files, exactly as recorded in
  `docs/prs/138.md`.
- `ruff check` on all four new/changed test files: clean.

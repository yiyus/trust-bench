# Review: Issue #26 - Study: bounded and constrained

## Summary

Tests-only (RED) stage. `tests/studies/test_bounded.py` covers the
scoped-down acceptance criterion (inactive, active-at-boundary, and
infeasible-start scenarios for SciPy's `trf`/`dogbox`, `trust`'s
Coleman-Li comparison deferred to Phase 8, consistent with issue #25's
precedent and the same, still-open Epic #34). The infeasible-start
`ValueError` claim and the three scenario/bounds constructions were
independently reproduced and check out. One finding: the exact
floating-point equality assertions in two of the four tests are not
reliably satisfied by `trf` under SciPy's default tolerances, and hold
for `trf` at all only at `ftol=xtol=gtol=1e-15`, one order of magnitude
above machine epsilon. This is a genuine risk to the GREEN cycle, not a
hypothetical one: reproduced directly against the real backend and
problem.

## Findings

### Major: exact-equality assertions are not satisfied by `trf` under default tolerances

`test_inactive_bounds_converge_to_the_unconstrained_optimum` asserts
`outcome.x_final[0] == 0.0` and `outcome.x_final[1] == 0.0` for every
`(name, method)` pair, and `test_active_at_boundary_bounds_converge_
exactly_to_the_boundary` asserts exact equality to `0.5`/`0.0`. Verified
directly via `SciPyBackend.solve` with `RunConfig(max_iter=200,
bounds=...)` (no `tolerance` override, matching the pattern every other
study in this repo uses for its default sweep config):

- `dogbox` hits both targets exactly (`[0.0, 0.0]` and `[0.5, 0.0]`) in
  both scenarios.
- `trf` does not: `inactive` gives `[2.176e-16, -2.176e-16]`, and
  `active_at_boundary` gives `[0.5000000001164153, 0.0]`, under default
  tolerances. `trf` only reaches the exact target at `ftol=xtol=gtol=
  1e-15`, essentially machine epsilon, one order of magnitude tighter
  than `1e-14` (confirmed: `1e-14` still leaves the `inactive` scenario's
  residual at `2.176e-16`).

If `sweep()`'s `RunConfig` does not happen to set every tolerance to
`1e-15`, both tests will fail on `trf` as written. Even if it does, an
assertion that only holds one order of magnitude above machine epsilon
is fragile across SciPy versions, BLAS backends, and platforms, and is
liable to flip on an unrelated dependency bump. Recommend either
`pytest.approx` with an explicit, documented tolerance (matching the
project's `rel=0.05` precedent in issue #23, or a tighter absolute bound
appropriate to a quadratic problem), or asserting `abs(x_final[i]) <
tol` directly; either way, the choice of tolerance and the corresponding
`RunConfig` should be justified in the design notes, the way issue #23's
tolerance choice was.

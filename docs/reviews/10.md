# Review: Issue #10 - Difficulty families: large_residual, ill_conditioned, outliers, scaling, dimensionality

## Summary

Tests-only (RED) stage review. `tests/problems/test_families.py` covers both
acceptance criteria across all five families: a parity-and-known-optimum
test per family (AC1) and a monotonic-sweep test of each family's defining
property (AC2), plus three parameter-range rejection guards following the
non-emptiness-guard precedent from `docs/reviews/9.md`. The RED failure
(`ImportError: cannot import name 'dimensionality'`) is confirmed. The
analytically-tractable defining-property claims (scaling, ill_conditioned)
were independently verified. One Minor observation on test-helper
duplication; no blocking issues.

## Findings

### Minor: parity/known-optimum assertions are duplicated across the two problem-test files

`_assert_parity` and `_assert_known_optimum` in `test_families.py:16-33`
re-implement the same analytic-vs-FD and known-optimum checks that
`test_canonical.py:26-53` (from issue #9) performs inline. The two now carry
the same logic in two places, and a future third problem-test file would be
tempted to copy it a third time. The families' version generalises the
canonical version by making the tolerances configurable, so it is close to a
strict superset.

Recommendation: consider promoting these two helpers to a shared location
next to `fd_stencils.py` (the FD helpers are already shared this way) and
having `test_canonical.py` call them with its default tolerances. Not
blocking, and reasonable to defer if the tolerance-parameterisation is
expected to keep diverging; raising it now because this is the first point
at which the duplication actually exists rather than being hypothetical.

Location: `tests/problems/test_families.py:16-33`,
`tests/problems/test_canonical.py:26-53`

### Note: the second assertion in `test_dimensionality_grows_with_n` is implied by the first

`test_dimensionality_grows_with_n` (`test_families.py:162-165`) asserts
`sizes == DIMENSIONALITY_NS_WIDE` and then `_increasing(sizes)`. Since
`DIMENSIONALITY_NS_WIDE` (`[2, 10, 100, 1000]`) is itself strictly
increasing, the second assertion can never fail when the first passes. It is
harmless (it documents intent) but redundant. No action required.

Location: `tests/problems/test_families.py:162-165`

### Note: FD-parity for dimensionality is deliberately scoped to n in {2, 10}

AC1's FD-parity half runs only at `n in {2, 10}`, while the known-optimum
half and the AC2 sweep run at the full `{2, 10, 100, 1000}`. The PR doc
justifies this: `fd_hessian` is O(n^3) and impractical at n=100/1000 in a
unit test, n=2 is formula-identical to canonical Rosenbrock (already FD-
verified in issue #9), and n=10 exercises the general pairing. This is a
documented, sound scoping decision, not a coverage gap. No action required.

Location: `tests/problems/test_families.py:145-159`

  tractable: `scaling`'s Hessian diagonal ratio is exactly `s**2`
  (`J = diag(1, s)`, `H = J^T J = diag(1, s^2)`), strictly increasing in s;
  `ill_conditioned`'s `cond(diag([kappa, 1]) @ R) == kappa` exactly for
  `kappa >= 1` (rotations preserve singular values). Confirmed the
  `kappa=0.5` case yields `cond = 2.0 = 1/kappa`, so
  `test_ill_conditioned_rejects_a_condition_number_below_one` guards a real
  premise (the `cond == kappa` identity fails below 1), not a convenience.
- Rejection guards: all three (`ill_conditioned` kappa<1, `outliers`
  fraction>=0.5, `dimensionality` odd n) guard a documented assumption the
  family's defining property depends on, consistent with the precedent set
  in `docs/reviews/9.md`.
- Tolerance handling: `_assert_parity`'s `jacobian_tol`/`hessian_tol` use the
  `None`-then-`or dict(...)` idiom, avoiding the mutable-default-argument
  trap. The relaxed `large_residual` known-optimum tolerance (`atol=1e-4`)
  is documented as matching the numerical solve's own achieved precision,
  since that family's optimum is not closed-form for `rho > 0`.
- Numerical monotonicity for `large_residual` and `outliers`: not
  independently reproducible at this stage (both depend on the not-yet-
  existing implementation, and `large_residual` needs a numerical solve);
  the PR doc states both were verified in the throwaway derivation script,
  to be re-confirmed against these tests once the implementation lands.
- Comment hygiene: clean. No GitHub issue/PR IDs in `test_families.py`.
- Tests: RED confirmed. `pytest tests/problems/test_families.py` fails at
  collection with `ImportError: cannot import name 'dimensionality' from
  'trust_bench.problems.families'`, the same failure class as prior issues.

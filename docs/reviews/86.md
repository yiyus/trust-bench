# Review: Issue #86 - Document and test tolerance comparability between scipy and trust-apl

## Summary

Tests-only (RED) stage review. The empirical claims in the PR doc were
verified directly against the real `APLBackend`/`SciPyBackend`
(`dyalogscript` available in this environment): scipy's `lm` reaches
`dist_to_opt == 0.0` on rosenbrock at every tested tolerance
(`1e-2`..`1e-14`), while `trust-apl` degrades smoothly from `0.078` at
`1e-2` to a floor of `1.55e-7` from `1e-10` onward — figures match the
PR doc to the reported precision. The termination-criterion description
attributed to `Newton.aplo` (minimum of relative parameter-space change
and relative cost change, `tolc`/`tolr`) also checks out by reading
`Newton.aplo`'s `T`/`C` functions directly. The `tolerance_comparability`
module genuinely does not exist yet (`ModuleNotFoundError`), and
`test_apl_tolerance.py`'s single test is honestly documented as passing
unchanged (pre-existing #78 wiring, not new behaviour). One hygiene
issue: a test comment references the issue number directly.

## Findings

### Minor: GitHub issue ID referenced directly in a test comment

`tests/studies/test_tolerance_comparability.py:36` reads "The confound
`#86` exists to make visible, not to eliminate: ...". Project convention
(and this workflow's own comment-hygiene check) keeps issue IDs out of
source and test comments — they go in PR docs, review docs, and commit
messages instead, since the reasoning should stand on its own without
forcing a context-switch to the tracker, and the comment survives issue
renumbering/migration. Rephrase to state the invariant directly, e.g.

# Review: Issue #148 - Reduce the slow test suite to a minute or two

## Summary

This isn't a traditional red/green cycle - as `docs/prs/148.md` itself notes,
there's no new production behaviour, only the test suite's own structure and
coverage strategy changing, all in one working-tree state (nothing committed
yet on this branch beyond the merge base). Reviewed as a single pass rather
than split into tests-only/implementation stages. The three changes
(deleting `test_harness.py`, trimming `test_dimensionality.py` off n=1000,
replacing `test_cli.py`'s real report-generation tests with a fixture-based
one) are each well-reasoned and independently verified not to lose real
coverage. One clear gap: the issue's own acceptance criterion ("`pytest -q`
... completes on the order of one to two minutes") is not met - the full
suite still takes just over six minutes in this environment, consistent with
the ~5.5 minutes `docs/prs/148.md` itself reports and openly attributes to
scope explicitly excluded from this issue.

## Findings

### Major: the full suite does not meet the issue's own "one to two minutes" acceptance criterion

The issue states plainly: "`pytest -q` (the full, unfiltered suite)
completes on the order of one to two minutes." Measured directly in this
environment: `471 passed in 368.94s (0:06:08)` - almost exactly matching
`docs/prs/148.md`'s own "Full suite: 471 passed in 5:31" (the small
difference is consistent with this run overlapping another background test
run). Both figures are three to six times the stated target.

This isn't a surprise the PR doc hides - its own "Non-goals" section says so
directly: "The other ~10 `slow`-marked APL test files... are untouched...
They account for most of the ~5.5 minutes the full suite still takes." The
improvement over the 30-55 minute starting point is real and substantial,
and the reasoning for leaving those ~10 files out of this issue's scope is
sound (they exercise real APL solves across many parametrised cases, a
different, larger piece of work). But the issue's acceptance criteria, as
written, is a specific number, not "an improvement" - either the acceptance
criterion needs updating to reflect what's actually achievable within this
issue's stated scope (e.g. "under ten minutes", or explicitly deferring the
remaining ~10 files to a follow-up issue), or those files need to be brought
into scope here. Recommend closing this gap explicitly - by revising the
issue text/acceptance criteria before merging, or filing the remaining ~10
files as an immediate follow-up issue that this one references - rather than
merging silently short of its own stated goal.

Location: issue #148's acceptance criteria; `docs/prs/148.md`'s "Non-goals"
section already names the cause.

### Major: one test comment cites a GitHub issue number

CLAUDE.md: "No GitHub issue IDs in source code or test comments." One
instance:

- `tests/integration/test_cli.py:485`: `# ... scalar_cost is correctly
  absent (see #130): the fixture was generated with two backends, and
  scalar_cost has a known, permanent trust-apl coverage gap.` The
  parenthetical adds nothing beyond what the rest of the sentence already
  states; drop it and state the invariant directly.

Location: `tests/integration/test_cli.py:485`

### Note: test_harness.py removal verified not to lose real coverage

Read `run.dyalog` (the one-shot driver `test_harness.py` exercised via
`run_harness.sh`) and `session.dyalog` (the persistent driver
`apl_backend.py` uses today) directly: both call the exact same `Solve
req`/`Evaluate req` functions, differing only in the outer request-handling
loop (one-shot file I/O vs a persistent stdin/stdout loop). The
schema-level assertions `test_harness.py` made (e.g. `n_heval == 0` for
`lm`, `n_heval > 0` for `trust-exact`, convergence, `x_final`/`cost_final`
values) are already covered through the persistent-session path by
`tests/backends/test_apl_bfgs_newton.py` and its siblings. The only thing
genuinely lost is direct testing of `run_harness.sh`/`run.dyalog` as a
standalone artefact, which the issue explicitly accepts (kept as an
unused-by-tests manual debugging path) and no other file references
(confirmed by grep - the sole other `test_harness`/`run_harness` hit,
`tests/unit/test_provenance.py:29`'s `test_harness_git_sha_matches_...`, is
an unrelated name collision on `harness_git_sha`).

### Note: dimensionality trim and fixture provenance verified directly

`tests/studies/test_dimensionality.py`'s new `n_values=[n for n in N_VALUES
if n != 1000]` pattern is exercised alongside a new, separate
no-solve assertion (`test_n_values_includes_a_large_dimension`) that
`N_VALUES` itself still reaches 1000 - so the "goes up to n=1000" claim
stays protected even though no test solves there anymore. `tests/fixtures/
report/`'s 22 files are byte-identical to `docs/prs/135/report/` (`diff -rq`,
excluding the correctly-omitted `report.html`), confirming the "copied from
the already-approved #135 snapshot" claim directly rather than trusting it.

## Verification

- Tests: run directly, not just taken from `docs/prs/148.md`.
  - `tests/studies/test_dimensionality.py`: 6 passed in ~4s (exact match to
    the PR doc).
  - `tests/integration/test_cli.py`: 42 passed in ~26s (exact match).
  - `pytest -q -m "not slow"` (what `make test` runs): 384 passed, 87
    deselected in 195s - count matches the PR doc's 384 exactly; wall-clock
    is slower here (195s vs the doc's ~63s), most likely environment/CPU
    differences or contention from running both this and the full-suite
    verification concurrently, not a correctness concern.
  - `pytest -q` (full, unfiltered): 471 passed in 368.94s (6:08) - count
    matches the PR doc's 471 exactly; timing is close to the doc's own 5:31,
    both several times over the issue's stated one-to-two-minute target (see
    Major finding above).
- `ruff check` on all changed files: clean.
- No dangling references to the deleted `test_harness.py` or `run_harness.sh`
  anywhere else in the tree (confirmed by grep).

## Recommendation

Request changes - resolve the acceptance-criteria gap explicitly (update the
issue/criteria to match what this scope can deliver, or file a follow-up for
the remaining ~10 slow APL files) and drop the one issue-number reference
from `tests/integration/test_cli.py:485` before merging.

# Review: Issue #139 - Persistent APL session shared by study sweeps and the test suite

## Summary

Single-commit change (`f7994df`) replacing per-call `dyalogscript` subprocess spawns with one long-lived session process, shared transparently across every `APLBackend()` instance and `evaluate_problem()` call via a module-level singleton. Verified the design against `docs/plans/persistent-apl-session.md`, re-ran the new session tests, the full APL-touching suite (99 tests, real `dyalogscript`), and independently reproduced both protocol-bug fixes described in the PR doc (the `⍞` long-line corruption workaround and the `⎕←` 32767-character wrap handling, the latter directly against a 1000x1000 Hessian). No correctness issues found.

## Findings

### Note: `_dyalog_version()`'s stdout/stderr bug is genuinely pre-existing, correctly left unfixed

`dyalog -version` writes its version banner to stderr, not stdout (confirmed directly), so `_dyalog_version()`'s `re.search` against `result.stdout` always falls through to `"unknown"`. Confirmed this function predates this issue (`_dyalog_version` already existed; this commit only adds the `@lru_cache` decorator). Correctly scoped out of this PR and flagged for its own follow-up, as stated in `docs/prs/139.md`.

Location: `trust_bench/backends/apl_backend.py:236-247`

### Minor: `run_harness.sh`'s untouched-suite claim is stronger evidence than presented

The PR doc says the `_sources.sh` refactor is "behaviourally identical" and cites the harness's 19-test suite as confirmation; independently re-ran `tests/backends_ext/apl/test_harness.py` standalone (19 passed) to confirm this holds outside the combined `-k apl` run too. No action needed, this is already correctly claimed, noting it here only because it was worth verifying in isolation rather than trusting the combined-run number.

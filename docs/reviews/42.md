## Findings

### Minor: `config` fixture uses a non-existent `RunConfig` field instead of a `RunConfig` instance

`_run_result`'s default `config={"ftol": 1e-8}` (test_compare.py:40) passes a
plain dict with an `ftol` key, but `RunConfig` (trust_bench/core/config.py)
has no `ftol` field; its tolerance field is `tolerance`. This is harmless
here because `storage.load` returns raw JSON records into a DataFrame
without reconstructing `RunResult`/`RunConfig` (storage.py's `load` docstring
says nested fields "stay as dicts"), so nothing round-trips through
`RunConfig.from_dict`. It also diverges from `test_tables.py`'s
`_run_result`, which passes a genuine `config=RunConfig()`. Not worth
blocking RED on, but worth using `RunConfig()` (or `RunConfig(tolerance=1e-8)`)
in a later cycle for consistency with the sibling fixture, in case a future
test on this module needs `config` to survive a real `from_dict` round trip.

# Implementation review

## Summary

`trust_bench/reporting/compare.py` merges baseline/candidate result tables on
`problem_id`/`backend`/`method`/`start` and classifies each matched row
`regression` (any Tier-1 metric or `status` changed beyond `tier1_tol`),
`drift` (only `timing.median` changed), or `stable`, exactly matching Section
6/8 of `docs/plans/trust-bench.md` and Phase 10's plan. `drift_summary`
groups drifted rows by `machine_fingerprint`/`backend_version`. All 8 tests
pass, `ruff` is clean on both files, and the full suite (479 passed) is
green. CLI wiring (`trust-bench compare`) is correctly out of scope here:
it is issue #43, a separate child of the #41 epic. No blocking issues; two
minor observations below.

## Findings

### Minor: `_numeric_changed` also does the string-equality fallback for `status`, but its name only advertises numeric handling

`_numeric_changed` (compare.py:17-23) is applied to every `TIER1_METRICS`
entry, including `status`, which is a string (`"CONVERGED"`,`"STALLED"`).
The `try/except (TypeError, ValueError)` correctly falls back to `!=` for
non-numeric values, so the status-change test passes, but the function name
undersells what it actually does. A reader skimming for "how does a status
change get caught" would not find it under a name that says "numeric".
Consider `_metric_changed` or a one-line note that it also covers the
non-numeric `status` comparison via the fallback branch.

Location: `trust_bench/reporting/compare.py:17`

### Note: unmatched rows are silently dropped by the inner merge

`compare()`'s merge (compare.py:38) is an inner join: a `(problem_id,
backend, method, start)` combination present in only one of baseline or
candidate (e.g. a problem added or removed between runs) produces no row and
no signal, rather than an explicit "only in baseline"/"only in candidate"
classification. Not a defect against this issue's acceptance criteria (both
tests and the issue only require synthetic Tier-1/Tier-3 changes on matched
rows), and not something #43's CLI wiring necessarily needs either, but
worth keeping in mind if a future `trust-bench compare` run silently drops
a genuinely interesting case (a problem that stopped running entirely).

Location: `trust_bench/reporting/compare.py:38`

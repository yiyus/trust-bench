# Review: Issue #144 - Revisit CLI options and defaults

## Summary

Tests-only stage. The issue itself proposed no specific change and
deferred until `compare` (#43) existed; `docs/prs/144.md` documents the
concrete gap that surfaced once it did (`trust-bench report` never wrote
`results/*.jsonl`, so `compare` had nothing real to diff) and the decision,
taken with the user, to fold that persistence work into this issue rather
than opening a new one. `tests/integration/test_runner.py` drives out a
`recording_results(path)` context manager mirroring `measuring_timing()`;
`tests/integration/test_cli.py` drives out `run_report`'s `results_dir`
parameter and a `--no-results` CLI flag. RED confirmed directly, matching
`docs/prs/144.md`. No issues found.

## Findings

### Note: RED confirmed exactly as documented

- `tests/integration/test_runner.py` fails to collect: `ImportError:
  cannot import name 'recording_results' from 'trust_bench.core.runner'`.
- `tests/integration/test_cli.py`: 7 new tests fail for the two reasons the
  PR doc states - `--no-results` rejected by argparse
  (`unrecognized arguments: --no-results`) and
  `monkeypatch.setattr("trust_bench.cli.harness_git_sha", ...)` raising
  `AttributeError` because `cli.py` doesn't import `harness_git_sha` yet -
  plus `run_report() got an unexpected keyword argument 'results_dir'` and
  a plain missing-directory assertion failure for the remaining two tests.

### Note: scope decision is properly documented and consistent with the design doc

Folding `results/*.jsonl` persistence into a "revisit CLI defaults" issue is
a real scope expansion beyond the issue's literal text, but `docs/prs/144.md`
states the reasoning plainly (the gap `compare` exposed, the decision taken
with the user, why it belongs here rather than a new issue) rather than
silently smuggling it in. It also directly serves Section 8 of
`docs/plans/trust-bench.md` ("Results are appended to `results/*.jsonl`")
and unblocks #44 (the CI job), which the design doc says depends on the
same thing.

### Note: test design mirrors the established `measuring_timing` precedent

`recording_results`'s three tests (no-op outside the context, appends every
`run()` call inside it, `None` is a no-op) match `measuring_timing`'s own
existing test shape one-for-one, and reuse the same `_StubBackendWithTrace`
fixture, which returns a fully-formed `RunResult` (real provenance
included) rather than a bare stub - so `recording_results`'s eventual
`storage.append` call has something genuine to serialise, not a fixture
that would only coincidentally work. The `results_dir=None`
disables-persistence test and the `--no-results` flag test both follow
`compare`'s own established default-off/opt-out shape from #43's `--html`.

Deciding to leave every other flag (`--only`/`--skip`/`--skip-slow`/
`--backends`/`--html`/`--output-dir`) untouched, as the PR doc states, is
consistent with the issue's own text ("No specific change is proposed
here") and this project's minimal-scope convention: nothing else surfaced
a concrete gap once `compare` existed.

### Note: comment hygiene

No GitHub issue references in either changed test file.

# Implementation review

## Summary

`recording_results(path)` in `trust_bench/core/runner.py` mirrors
`measuring_timing()`'s existing contextvar-backed shape exactly, and
`run()` appends to the active path (a no-op when none is set or `path is
None`). `run_report` wires it up with `results_dir="results"` (default),
building `{output_dir}/{results_dir}/{harness_git_sha()}.jsonl`; `--no-results`
maps to `results_dir=None` in `main`. This delivers Section 8's "results are
appended to `results/*.jsonl`, nothing overwritten" and unblocks `compare`
(#43) with a real producer. All targeted tests pass (146 across
`test_runner.py`, `test_cli.py`, `tests/reporting/`), `ruff` clean. No
issues found.

## Findings

### Note: architectural conformity and code quality

`recording_results` is structurally identical to `measuring_timing`
(contextvar, `set`/`try`/`finally reset`), so a reader who already
understands one immediately understands the other - no new pattern
introduced for a problem this codebase already has a working answer to.
`run()`'s new behaviour is a strict addition (existing signature and
return value unchanged); the append happens after `backend.solve` returns,
so a raising backend never produces a partial/malformed record.

`results_path = output_dir / results_dir / f"{harness_git_sha()}.jsonl" if
results_dir is not None else None` (cli.py) short-circuits correctly:
`harness_git_sha()` (a subprocess call) only runs when persistence is
actually enabled, which is why `--no-results` needs no `harness_git_sha`
monkeypatch in its own test. `/` binds tighter than the conditional
expression here, so this parses as intended despite reading densely at a
glance.

### Note: comment hygiene

No GitHub issue references in either changed source file - `runner.py`'s
new docstring cites "Section 8" of the design doc, not an issue number,
improving on the pattern flagged in the #43 implementation review.

### Note: minimal scope maintained

Only `runner.py` and `cli.py` changed, both directly required by the
persistence gap `docs/prs/144.md` describes. Every other CLI flag (`--only`/
`--skip`/`--skip-slow`/`--backends`/`--html`/`--output-dir`) is untouched,
consistent with the issue's own text and the RED-stage review's note that
nothing else surfaced a concrete gap once `compare` existed.

## Verification

- Tests: pass. `tests/integration/test_runner.py`, `tests/integration/
  test_cli.py`, `tests/reporting/`: 146 passed.
- Linters: `ruff check` on `trust_bench/core/runner.py` and
  `trust_bench/cli.py` - clean.
- Full suite: not run in this review (deferred to the PR-opening session,
  consistent with the #43 review).

# Cycle 2 review (tests and implementation, single pass)

## Summary

Cycle 2 redesigns the CLI around a directories-based model (`trust-bench
benchmarks trust-apl by default; `--scipy`/`--no-trust` replace
`--backends`; `--full`/`--no-html`/`--no-results` replace `--skip-slow`/
`--html`; `report` gains an optional `BASELINE_DIR...` positional that
folds a regression/drift comparison into the report; `compare`'s arguments
become directories). The design was posted to the issue for review and
matches what shipped. RED and GREEN both happened in this working tree
without an intervening stop, so tests and implementation are reviewed
together here, as `docs/reviews/104.md` did for a similar situation.

`_select_studies`'s bug fix (an explicit `--only <slow study>` no longer
silently stripped) is correct and well-reasoned. However, independently
exercising the new baseline-comparison feature with the simplest possible
real study (`only=["baseline"]`, no synthetic fixtures) surfaces a Critical
correctness bug: `compare.csv` contains heavily duplicated rows - a 16-row
cross-join explosion for a single logical `(problem, backend, method,
start)` comparison - because `results/*.jsonl` already contains duplicate
keys from a single `run_report` call, and `compare_with_provenance`'s
key-based merge multiplies rather than aligns those duplicates. None of
the new tests catch this because none assert on row count or duplication,
only on file existence. A second, narrower issue: `_load_results_dir` in
the baseline-comparison path ignores a caller's custom `results_dir`.

## Findings

### Critical: `compare.csv` explodes into duplicate rows for any real report, because `results/*.jsonl` already has duplicate keys and `compare_with_provenance`'s merge multiplies them

Reproduced directly, no synthetic fixtures:

```python
run_report(baseline_dir, only=["baseline"], use_trust=False, use_scipy=True, html=False)
run_report(candidate_dir, only=["baseline"], use_trust=False, use_scipy=True, html=False,
           baselines=[baseline_dir])
```

`compare.csv` has 113 rows for what should be 7 problems x ~1 row each.
`beale`/`scipy`/`lm`/`standard` alone appears 16 times, all classified
`drift`. Root cause, traced in two independent steps:

1. **`results/*.jsonl` already has duplicate keys from a single
   `run_report` call.** `baseline.basin_rates()` (`trust_bench/studies/
   baseline.py:24-36`) iterates `problem.starts`, which includes
   `"standard"` - the same start `standard_start_results()`
   (`baseline.py:12-21`) already ran. Both calls go through `run()`, and
   `recording_results` (this issue's own cycle 1) appends every `run()`
   call unconditionally, with no notion of "this is the one canonical
   result for this key" versus "this is an internal probe that happens to
   share a key with one already recorded." A single `run_report(only=
   ["baseline"])` call therefore writes 2 rows for `(problem, scipy, lm,
   "standard")` for 6 of 7 canonical problems, confirmed directly against
   the real `results/*.jsonl` file it writes.
2. **`compare_with_provenance`'s merge multiplies duplicate keys instead
   of aligning them.** With 2 duplicate baseline rows and 2 duplicate
   candidate rows sharing a key, `compare()`'s own inner merge
   (compare.py:38) already produces the expected-but-easy-to-miss 2x2=4
   rows for that key (confirmed: `compare()` alone returns 29 rows, not 7,
   for this scenario). `compare_with_provenance` (compare.py:93-102) then
   independently re-merges `baseline`/`candidate` to build a `provenance`
   table (also 4 rows for that key) and joins it onto `classified` via
   `classified.merge(provenance, on=list(key_columns))` - a second,
   non-unique-key merge, which cross-joins the 4 already-duplicated
   `classified` rows against the 4 already-duplicated `provenance` rows:
   4x4=16. The docstring's stated rationale for this design ("joined back
   onto the classification explicitly by `key_columns`, not by row
   position, so this stays correct regardless of how `compare()`'s own
   internal merge orders its output") is true only when `key_columns` are
   unique per side; it silently compounds duplication instead when they
   aren't, which - per point 1 above, and per this project's own design
   ("repeated runs on the same commit accumulate", `docs/prs/144.md`'s
   cycle 1 section) - is an expected, not exceptional, occurrence.

Neither `test_run_report_writes_a_comparison_against_a_given_baseline_
directory` nor `test_run_report_pools_multiple_baseline_directories`
(tests/integration/test_cli.py) catches this: both only assert that
`compare.csv`/`compare.png` exist or that the row count is `> 0`, not that
it is the expected count or free of duplication.

This corrupts the feature's headline output for the ordinary case of
running `trust-bench report` against a study that computes a basin rate
(currently only `baseline`, but the pattern - reusing an already-recorded
start for a further sweep - is one other studies could adopt) or against
any baseline directory whose report was generated by running `trust-bench
report` more than once on the same commit (the behaviour cycle 1 explicitly
designed for: "repeated runs on the same commit accumulate"). A `compare`
run's `classification_counts` graph and row-level table would both
overstate how many rows regressed or drifted, in some cases severely.

Recommend fixing the root cause rather than patching the symptom:
`basin_rates` recomputing the already-known `"standard"` distance is
itself a correctness-independent inefficiency worth removing (reuse
`standard_start_results`'s own value instead of re-running `run()` for a
start already covered) - but that alone does not make `compare_with_
provenance`/`compare()` safe against duplicate keys in general, since
duplicate keys are also an expected consequence of running the same
commit's report twice, by this issue's own cycle 1 design. The merge logic
itself needs to either deduplicate before joining (e.g. keep the latest
row per `key_columns` when pooling multiple files or repeated appends) or
make explicit what should happen when a key isn't unique on one or both
sides, rather than silently cross-joining.

Location: `trust_bench/reporting/compare.py:93-102`,
`trust_bench/studies/baseline.py:24-36`

### Major: `_load_results_dir`'s hardcoded `"results"` subdirectory ignores a caller's custom `results_dir`

`run_report`'s `baselines` path calls `_load_results_dir(output_dir)`
(cli.py:365) to load this run's own candidate results back for comparison.
`_load_results_dir` (cli.py:278-288) always globs `directory / "results"`,
not `directory / results_dir`. Reproduced directly:

```python
run_report(candidate_dir, only=["baseline"], use_trust=False, use_scipy=True,
           html=False, results_dir="custom_results", baselines=[baseline_dir])
# -> ValueError: no results/*.jsonl found under .../candidate_report
```

even though the run genuinely wrote `custom_results/<sha>.jsonl` - the
error message is actively misleading about what went wrong. The current
CLI never exposes a way to pass a non-default `results_dir` alongside
`baselines` (only `None` via `--no-results`, which already raises for a
different, correct reason), so this isn't reachable through `main()`
today, but `run_report` is itself a tested, public function, and the
mismatch is a real latent inconsistency between two parameters that look
designed to compose. Either thread `results_dir` through to
`_load_results_dir`, or derive the candidate load from `results_path.parent`
directly instead of re-deriving a hardcoded literal.

Location: `trust_bench/cli.py:278-288`, `trust_bench/cli.py:363-366`

### Note: `_select_studies`'s explicit-`--only`-of-a-slow-study fix is correct

`selected -= SLOW_STUDIES` is now gated on `only is None` (cli.py:256),
matching the existing `MULTI_BACKEND_STUDIES` carve-out's own pattern
(`only is None and n_backends < 2`) one line below. An explicit `--only
dimensionality` now runs it even without `--full`, consistent with the
project's stated rule that an explicit request is never silently dropped.

### Note: architectural conformity and comment hygiene otherwise clean

`_write_compare_artefacts` is correctly shared between `run_report`'s
baseline path and `run_compare`, avoiding two divergent copies of the same
CSV/PNG-writing logic. `build_parser`/`main` wiring for the new flags
(`--full`, `--scipy`, `--no-trust`, `--no-html`, `--no-results`, the
`BASELINE_DIR...` positional) is consistent with the existing `report`/
`compare` subcommand structure, and the consolidated `main()`-wiring spy
test is a reasonable way to pin every flag's mapping in one place rather
than one test per flag. No GitHub issue references in any touched source
file. `README.md`'s rewritten usage section accurately reflects the new
defaults and adds the previously-undocumented `compare` example.

## Verification

- Tests: pass as reported. `tests/integration/test_cli.py`,
  `tests/integration/test_runner.py`, `tests/reporting/` (`-m "not slow"`):
  155 passed, 6 deselected. The Critical finding above is a gap in test
  *coverage* (no test asserts row-count/duplication correctness), not a
  test failure - every test as written passes.
- Linters: `ruff check` on all touched source files - clean.
- Full suite: not run in this review (deferred to the PR-opening session).

## Final recommendation (Cycle 2)

Request changes: the duplicate-row explosion in `compare.csv` corrupts the
feature's core output for realistic usage (not a contrived edge case) and
should be fixed, with a regression test asserting `compare()`/
`compare_with_provenance` produce exactly one row per matched key even
when `results/*.jsonl` contains duplicate-keyed rows, before this cycle
ships. The `results_dir`/`_load_results_dir` mismatch should be fixed
alongside it. Everything else in this cycle - the CLI redesign, the
`_select_studies` bug fix, the README update - is sound and well-tested.

## Findings

### Fixed: duplicate-row explosion in `compare.csv`

`_dedupe_latest(df, key_columns)` (compare.py:26-37) collapses each side to
its latest row per key (by `timestamp`) before matching, applied in both
`compare()` (compare.py:53-54) and, separately, in `compare_with_provenance`
(compare.py:113-114) ahead of both its call into `compare()` and its own
second provenance merge - so each function is defensively correct against
duplicate-keyed input on its own, not reliant on a caller pre-deduplicating.
"Keep the latest recording" is the right semantics for an append-only log,
matching cycle 1's own stated model (repeated runs on one commit
accumulate; the newest recording of a key is authoritative). Re-ran the
original reproduction directly:

```python
run_report(baseline_dir, only=["baseline"], use_trust=False, use_scipy=True, html=False)
run_report(candidate_dir, only=["baseline"], use_trust=False, use_scipy=True, html=False,
           baselines=[baseline_dir])
```

`compare.csv` now has 8 rows, exactly one per `(problem_id, backend,
method, start)` (7 canonical problems, `rosenbrock` alone contributing
both `standard` and `far`) - confirmed directly, not just via the new
tests. `test_compare_collapses_duplicate_keyed_rows_to_the_latest_before_
classifying` and `test_compare_with_provenance_also_collapses_duplicate_
keyed_rows` (test_compare.py) construct the exact 2-vs-1 and 2-vs-2
duplicate shapes that previously produced 4 and 16 rows respectively, and
assert the classification reflects the *latest* row's value, not an
earlier duplicate's - correctly exercising `keep="last"` rather than just
"some deduplication happened."
`test_run_report_writes_exactly_one_comparison_row_per_key_despite_
baselines_own_duplicate_start` (test_cli.py) exercises the real,
non-synthetic `basin_rates` scenario end to end and asserts the exact
expected count.

### Fixed: `results_dir`/`_load_results_dir` mismatch

`run_report`'s baseline-comparison branch (cli.py:369-375) now loads its
own candidate results via `results_path.parent.glob("*.jsonl")` directly,
respecting whatever `results_dir` was actually used, instead of
re-deriving a hardcoded `"results"` literal. `_load_results_dir` (for
*baseline* directories) correctly keeps assuming the `"results"` default
and now documents why: an independent prior report's own `results_dir`
choice isn't recorded anywhere a caller could look up. Re-ran the original
reproduction directly - `results_dir="custom_results"` combined with
`baselines=[...]` now succeeds and writes `custom_results/*.jsonl` rather
than raising. `test_run_report_can_compare_against_a_baseline_with_a_
custom_results_dir` pins this, including the negative assertion that
`results/` (the old hardcoded default) is *not* created.

# Review: Issue #144 - Revisit CLI options and defaults

## Summary

Tests-only stage. The issue itself proposed no specific change and
deferred until `compare` (#43) existed; `docs/prs/144.md` documents the
concrete gap that surfaced once it did (`trust-bench report` never wrote
`results/*.jsonl`, so `compare` had nothing real to diff) and the decision,
taken with the user, to fold that persistence work into this issue rather
than opening a new one. `tests/integration/test_runner.py` drives out a
`recording_results(path)` context manager mirroring `measuring_timing()`;
`tests/integration/test_cli.py` drives out `run_report`'s `results_dir`
parameter and a `--no-results` CLI flag. RED confirmed directly, matching
`docs/prs/144.md`. No issues found.

## Findings

### Note: RED confirmed exactly as documented

- `tests/integration/test_runner.py` fails to collect: `ImportError:
  cannot import name 'recording_results' from 'trust_bench.core.runner'`.
- `tests/integration/test_cli.py`: 7 new tests fail for the two reasons the
  PR doc states - `--no-results` rejected by argparse
  (`unrecognized arguments: --no-results`) and
  `monkeypatch.setattr("trust_bench.cli.harness_git_sha", ...)` raising
  `AttributeError` because `cli.py` doesn't import `harness_git_sha` yet -
  plus `run_report() got an unexpected keyword argument 'results_dir'` and
  a plain missing-directory assertion failure for the remaining two tests.

### Note: scope decision is properly documented and consistent with the design doc

Folding `results/*.jsonl` persistence into a "revisit CLI defaults" issue is
a real scope expansion beyond the issue's literal text, but `docs/prs/144.md`
states the reasoning plainly (the gap `compare` exposed, the decision taken
with the user, why it belongs here rather than a new issue) rather than
silently smuggling it in. It also directly serves Section 8 of
`docs/plans/trust-bench.md` ("Results are appended to `results/*.jsonl`")
and unblocks #44 (the CI job), which the design doc says depends on the
same thing.

### Note: test design mirrors the established `measuring_timing` precedent

`recording_results`'s three tests (no-op outside the context, appends every
`run()` call inside it, `None` is a no-op) match `measuring_timing`'s own
existing test shape one-for-one, and reuse the same `_StubBackendWithTrace`
fixture, which returns a fully-formed `RunResult` (real provenance
included) rather than a bare stub - so `recording_results`'s eventual
`storage.append` call has something genuine to serialise, not a fixture
that would only coincidentally work. The `results_dir=None`
disables-persistence test and the `--no-results` flag test both follow
`compare`'s own established default-off/opt-out shape from #43's `--html`.

Deciding to leave every other flag (`--only`/`--skip`/`--skip-slow`/
`--backends`/`--html`/`--output-dir`) untouched, as the PR doc states, is
consistent with the issue's own text ("No specific change is proposed
here") and this project's minimal-scope convention: nothing else surfaced
a concrete gap once `compare` existed.

### Note: comment hygiene

No GitHub issue references in either changed test file.

# Implementation review

## Summary

`recording_results(path)` in `trust_bench/core/runner.py` mirrors
`measuring_timing()`'s existing contextvar-backed shape exactly, and
`run()` appends to the active path (a no-op when none is set or `path is
None`). `run_report` wires it up with `results_dir="results"` (default),
building `{output_dir}/{results_dir}/{harness_git_sha()}.jsonl`; `--no-results`
maps to `results_dir=None` in `main`. This delivers Section 8's "results are
appended to `results/*.jsonl`, nothing overwritten" and unblocks `compare`
(#43) with a real producer. All targeted tests pass (146 across
`test_runner.py`, `test_cli.py`, `tests/reporting/`), `ruff` clean. No
issues found.

## Findings

### Note: architectural conformity and code quality

`recording_results` is structurally identical to `measuring_timing`
(contextvar, `set`/`try`/`finally reset`), so a reader who already
understands one immediately understands the other - no new pattern
introduced for a problem this codebase already has a working answer to.
`run()`'s new behaviour is a strict addition (existing signature and
return value unchanged); the append happens after `backend.solve` returns,
so a raising backend never produces a partial/malformed record.

`results_path = output_dir / results_dir / f"{harness_git_sha()}.jsonl" if
results_dir is not None else None` (cli.py) short-circuits correctly:
`harness_git_sha()` (a subprocess call) only runs when persistence is
actually enabled, which is why `--no-results` needs no `harness_git_sha`
monkeypatch in its own test. `/` binds tighter than the conditional
expression here, so this parses as intended despite reading densely at a
glance.

### Note: comment hygiene

No GitHub issue references in either changed source file - `runner.py`'s
new docstring cites "Section 8" of the design doc, not an issue number,
improving on the pattern flagged in the #43 implementation review.

### Note: minimal scope maintained

Only `runner.py` and `cli.py` changed, both directly required by the
persistence gap `docs/prs/144.md` describes. Every other CLI flag (`--only`/
`--skip`/`--skip-slow`/`--backends`/`--html`/`--output-dir`) is untouched,
consistent with the issue's own text and the RED-stage review's note that
nothing else surfaced a concrete gap once `compare` existed.

## Verification

- Tests: pass. `tests/integration/test_runner.py`, `tests/integration/
  test_cli.py`, `tests/reporting/`: 146 passed.
- Linters: `ruff check` on `trust_bench/core/runner.py` and
  `trust_bench/cli.py` - clean.
- Full suite: not run in this review (deferred to the PR-opening session,
  consistent with the #43 review).

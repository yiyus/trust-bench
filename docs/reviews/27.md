## Findings

### Major (fixed): `x_scale='jac'` was a no-op for method `'lm'`, breaking the nfev-comparison test

`test_jac_scaling_needs_more_function_evaluations_at_large_scale_disparity`
originally included `'lm'` in its method loop. Per SciPy's own docstring,
method `'lm'`'s default (`x_scale=None`) already *is* `x_scale='jac'`;
there is no genuinely unscaled mode for it through this parameter.
Verified directly against the real problem family: `x_scale=None` and
`x_scale='jac'` gave identical `nfev` for `'lm'` at every scale in
`SCALES`, so `jac_scaled.n_feval > unscaled.n_feval` was false
(`17 == 17`) at `s=_LARGE_DISPARITY=1e6`, not close-but-fragile.

Fixed by scoping the test to `sweep(scales=[_LARGE_DISPARITY],
methods=["trf", "dogbox"])`, mirroring how
`test_jac_scaling_can_converge_to_the_wrong_answer_at_extreme_scale_disparity`
already scopes itself, with the reasoning recorded as a code comment
(`tests/studies/test_scaling.py:30-33`). The design notes' misattributed
`s=1e8` `nfev` figure for `lm` (previously described as the `s=1e6`
value) has also been corrected, and `lm` dropped from that paragraph's
before/after comparison since there is nothing to contrast for it.

Location: `tests/studies/test_scaling.py:23-40`

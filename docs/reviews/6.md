### Findings

No issues found.

### Note: `d <= tol` degrades sensibly for a NaN distance, though this is untested

`distances_to_opt` is typed as `float | None` in the docstring; a `float`
`nan` (as opposed to `None`) is not one of the two documented cases.
Checked directly: `d <= tol` is `False` for `nan` in Python, so a NaN
distance is excluded from the numerator but still counted in the
denominator, the same treatment as `None`. This is the same conservative
default the empty-set guard uses elsewhere in this module, and matches
"reached" only being true for an actual, finite comparison, so no change is
needed. Flagging only because it is a plausible input (a failed run that
records `nan` rather than `None`) with no test pinning the behaviour down.

Location: `trust_bench/core/metrics.py:55`

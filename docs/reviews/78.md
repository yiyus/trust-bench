# Review: Issue #78 - APLBackend: support bounds and finite-difference derivatives

## Summary

Tests-only (RED) review; `docs/reviews/78.md` did not previously exist.
`backends_ext/apl/solve.dyalog` and `APLBackend.solve` still reject bounds
and `derivative_mode="finite-difference"` as of this commit (the #37
follow-up rejection logic), so this is genuinely pre-implementation. RED is
confirmed directly: 5 new tests fail (2 harness-level on real assertion
mismatches, 3 backend-level on the still-present `ValueError`s), and the
full `not slow` suite is otherwise unaffected (219 passed, 26 deselected, up
from 21). Both quantitative claims underpinning the new tests were verified
independently against `trust` with `dyalogscript`, not taken on faith from
`docs/prs/78.md`: the bounded quadratic genuinely converges to `x1=0.5`,
`cost=0.125`, and withholding the analytic Jacobian genuinely costs more
evaluations than providing it. No issues found.

## Findings

No issues found. The RED test surface is well-targeted, its quantitative
expectations are independently verified against `trust` itself rather than
guessed, and it correctly demonstrates the *current* silent-ignore/reject
behaviour is a real defect (not merely an absent feature) before any GREEN
implementation exists.

## Implementation review (GREEN stage)

### Summary

`backends_ext/apl/solve.dyalog` and `APLBackend` now genuinely wire bounds
and finite-difference derivatives through to `trust`, instead of rejecting
them. All 10 relevant tests pass (the 5 from this issue plus the 5
pre-existing APL harness tests, unaffected); full suite 245/245 (240 prior +
5 new), lint clean. Verified the mechanism directly, independent of the test
suite: `cfg.lower`/`cfg.upper` are recognised by `trust`'s `Min` purely by
their presence on the config namespace (no separate bounded/unbounded call
shape), and withholding the Jacobian from `lower` in `fd` mode correctly
forces `trust`'s own `Eval` to estimate it, exactly as intended. Also checked
the untested combination (bounds and finite-difference together) directly
against the harness: it composes correctly, converging to the same
constrained optimum with more evaluations than bounds alone. No issues
found.

### Findings

No issues found.

- `solve.dyalog`'s `fd` branch (`lower‚Üź{calls+‚Üź1 ‚čĄ result‚ÜźApply f ‚ćĶ ‚čĄ
  fd:1‚äÉresult ‚čĄ result}`) and the final-gradient computation
  (`finalpair‚ÜźApply f r.p`, bypassing `lower`) are correct for a subtle
  reason worth confirming rather than assuming: in `fd` mode, `lower`'s
  return shape changes from a `(residual jacobian)` pair to a residual-only
  vector, so if the final gradient computation went through `lower` as
  before, `2‚äÉfinalpair` would index-error against a bare vector. Computing
  the final gradient via a direct, uncounted-by-`lower` call to the full
  analytic pair (with `calls` incremented manually alongside it) sidesteps
  this correctly and keeps `grad_norm_final` reporting the true analytic
  optimality residual in both modes, matching `docs/prs/78.md`'s stated
  intent.
- `cfg.lower‚Üź1‚äÉreq.bounds ‚čĄ cfg.upper‚Üź2‚äÉreq.bounds` relies on `trust`'s
  `Min` auto-detecting boundedness from whether `lower`/`upper` exist as
  variables on the config namespace passed in (confirmed directly in
  `Min.aplo`: bounds default to infinity and are only overridden `‚ć£(‚ą®/(2‚ÜĎnb)
  ‚ąäc.‚éēNL¬Į2)`, i.e. only if defined). No separate 5-element `Y` argument or
  bounded/unbounded branch is needed in `Solve`; the existing `lower Min
  (req.x0) cfg` call shape already handles both.

- Lint: pass (`ruff check` on `apl_backend.py`, `solve.dyalog`'s touched

- APL semantics / end-to-end: ran `run_harness.sh` directly (outside pytest)
  for `quadratic` in four configurations and confirmed the actual

  - analytic, unconstrained: `n_feval=6`, converges to `(‚Čą0, ‚Čą0)`,

# Review: Issue #78 - APLBackend: support bounds and finite-difference derivatives

## Summary

Tests-only (RED) review; `docs/reviews/78.md` did not previously exist.
`backends_ext/apl/solve.dyalog` and `APLBackend.solve` still reject bounds
and `derivative_mode="finite-difference"` as of this commit (the #37
follow-up rejection logic), so this is genuinely pre-implementation. RED is
confirmed directly: 5 new tests fail (2 harness-level on real assertion
mismatches, 3 backend-level on the still-present `ValueError`s), and the
full `not slow` suite is otherwise unaffected (219 passed, 26 deselected, up
from 21). Both quantitative claims underpinning the new tests were verified
independently against `trust` with `dyalogscript`, not taken on faith from
`docs/prs/78.md`: the bounded quadratic genuinely converges to `x1=0.5`,
`cost=0.125`, and withholding the analytic Jacobian genuinely costs more
evaluations than providing it. No issues found.

## Findings

No issues found. The RED test surface is well-targeted, its quantitative
expectations are independently verified against `trust` itself rather than
guessed, and it correctly demonstrates the *current* silent-ignore/reject
behaviour is a real defect (not merely an absent feature) before any GREEN
implementation exists.

## Verification

- Tests: RED confirmed.
  `pytest tests/backends_ext/apl/test_harness.py tests/backends/test_apl_bounds_and_derivative_mode.py`:
  5 failed, 5 passed. The 2 harness-level failures are genuine value
  mismatches, not crashes: the bounded request converges to
  `x_final[0] ≈ 7.9e-9` (the true unconstrained optimum) instead of the
  expected `0.5`, and the finite-difference request uses `n_feval == 6`,
  identical to the analytic run, instead of more. The 3 backend-level
  failures are the still-present `ValueError`s from `APLBackend.solve`
  (`"lm does not support bounds"`, `"...does not support
  derivative_mode='finite-difference'"`) and the still-`False`
  `capabilities().bounds`. `pytest -m "not slow"
  --continue-on-collection-errors`: 219 passed, 26 deselected (0 collection
  errors), matching `docs/prs/78.md` exactly.
- Lint: pass (`ruff check` on both new/changed test files).
- APL semantics: verified directly with `dyalogscript` against `trust`
  itself (not the harness), independent of `docs/prs/78.md`'s stated
  numbers:
  - Unconstrained `Quadratic Min (1.0 ¯1.0)`: converges to `(≈0, ≈0)`,
    `cost ≈ 1.56e-16`.
  - Bounded (`cfg.lower←0.5 ¯10.0 ⋄ cfg.upper←10.0 10.0`): converges to
    exactly `x1 = 0.5`, `cost = 0.125` (matching `0.5×0.5²`), confirming
    the constrained optimum genuinely sits on the boundary as
    `bounded.py`'s own `_ACTIVE_BOUNDS` claims for this problem.
  - Residual-only evaluation function (no Jacobian returned, forcing
    `trust`'s own `Eval` to estimate it by finite difference): converges to
    the same precision using `15` calls versus `5` for the analytic
    Min-only call (`solve.dyalog`'s own `final←lower r.p` adds one more call
    on top of whatever `Min` uses, consistent with `docs/prs/78.md`'s
    6-vs-15 `n_feval` figures once that extra call is counted). Confirms
    finite-difference mode is unambiguously more expensive, not a fragile
    tie the RED assertion could flake on.
  - `bounds` request shape (`[[0.5, -10.0], [10.0, 10.0]]`, i.e. a
    lower-list/upper-list pair) matches `trust_bench/studies/bounded.py`'s
    own `_ACTIVE_BOUNDS = ([0.5, -10.0], [10.0, 10.0])` convention exactly,
    as `docs/prs/78.md` claims.

Approve.

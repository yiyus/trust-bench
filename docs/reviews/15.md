## Findings

### Note: Capability-method consistency check is name-level only

`test_capabilities_methods_are_consistent_with_what_solve_accepts` checks
that every declared method name solves and an undeclared name is rejected,
but not that per-field capabilities (`bounds`, `analytic_hessian`) match
`solve`'s actual behaviour. The PR doc already documents this as a
deliberate deferral to the SciPy backend (#16/#17), where `lm` vs
`trf`/`dogbox` gives two concrete methods with a real `bounds` distinction to
check against. The toy backend has one method with no bounds support, so
there is nothing to contrast yet. Agreed scope for this stage; flagging only
so the SciPy backend's own review checks that the stronger, per-field
version actually lands there.

Location: `tests/backends/test_contract.py:54-59`

### Note: `prototype/` is untracked but out of scope

`prototype/harness.py` and related files are untracked scratch work (a
scipy trust-region comparison harness), unrelated to the backend contract
under review, and already excluded from ruff (`pyproject.toml`
`[tool.ruff] exclude`). Not part of this PR's diff; no action needed here.

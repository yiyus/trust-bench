# Review: Issue #12 - RunResult, RunStatus, TimingStats dataclasses (result.py)

## Summary

Tests-only (RED) stage review. `tests/unit/test_result.py` covers both
acceptance criteria: lossless JSON round-tripping (with and without
`timing`) and rejection of `RunResult` construction without a valid
`provenance`. The RED failure
(`ModuleNotFoundError: No module named 'trust_bench.core.result'`) is
confirmed. Field lists in the test fixture match Section 4.3 of
`docs/plans/trust-bench.md` exactly, and the `provenance=None` case
correctly closes the same "omitting an argument passes for free, passing
`None` explicitly does not" gap flagged in `docs/reviews/8.md`. No issues
found.

## Findings

No issues found.

### Note: no round-trip coverage for `None` values on the other `Optional` scalar fields

`dist_to_opt`, `cost_gap`, `grad_norm_final`, `n_iter`, `n_feval`,
`n_jeval`, `n_heval`, and `trace` are all `Optional` per Section 4.3, but
only `timing=None` is exercised. The PR doc's rationale for singling out
`timing` (it is the only `Optional` field that is itself a nested
dataclass, so it is the only one a custom `from_dict` could plausibly
mishandle) is sound: plain scalars and lists round-trip through
`json.dumps`/`json.loads` as `None`/`null` without any custom handling, so
there is no equivalent failure mode to guard against. Not blocking; noting
only so the narrower scope is a deliberate reading of the criterion, not an
oversight.

## Summary

`trust_bench/core/result.py` implements `RunStatus`, `TimingStats`, and
`RunResult` exactly as anticipated in the RED-stage design notes: field
order matches Section 4.3 field-for-field, `config` is typed `Any` pending
issue #19's `RunConfig`, and `to_dict`/`from_dict` override `status` to/from

otherwise unreached (`RunResult.to_dict()`'s `dataclasses.asdict` recurses
into nested dataclasses directly, bypassing their own `to_dict()`), giving
`TimingStats` the same standalone round-trip test `EnvProvenance` already
has. All 74 tests pass, lint is clean, coverage on `result.py` is 100%. No
issues found.

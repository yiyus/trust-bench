# Review: Issue #93 - bounded and capability_matrix send corrupted bounds to trust-apl's BFGS/trust-exact methods

## Summary

Tests-only (RED) stage review. Both new/extended tests reproduce the issue's own numbers exactly and fail for the stated reason, not for an unrelated one. No issues found.

## Findings

### Note: reproduction confirmed exactly, including the corrupted x_final values

Ran both test files directly against the current `main`-derived tree (dyalogscript available). `test_trust_apls_bfgs_and_trust_exact_converge_to_the_same_constrained_optimum_as_lm` fails at `inactive/BFGS` with `status=FAILED`, `x_final=[-9.99999999999999, 10.00000000000001]`, `cost_final=250.0000000000003` - matching the issue body and `docs/prs/93.md` to the last decimal. Both `test_bounds_measurement_agrees_with_declaration_for_trust_apls_bfgs_and_trust_exact` parametrizations fail with `measured=False` against `declared=True`, reproducing the exact `capability_matrix.csv` disagreement the issue cites.

Traced the root cause directly in `backends_ext/apl/solve.dyalog:75-78`: `cfg.lower`/`cfg.upper` are assigned from `req.bounds` unconditionally, for `lm`, `BFGS`, and `trust-exact` alike, confirming trust-apl's own array-pair format is method-agnostic on the APL side. `bounded.py`'s `_bounds_for` and `capability_matrix.py`'s `_measure_bounds` both route by method name only (`_LEAST_SQUARES_METHODS = {"lm", "trf", "dogbox"}`), so `BFGS`/`trust-exact` fall into the `minimize`-style per-dimension branch regardless of backend. This confirms the PR doc's root-cause analysis.

### Note: pre-existing `lm` test is unaffected, as expected

`test_trust_apls_bounded_lm_is_exercised_by_every_scenario_instead_of_reading_error` (from #82) passes unchanged, confirming `lm`'s bounds routing was never broken and this bug is scoped to `BFGS`/`trust-exact` alone, consistent with the issue's framing (these two methods were added afterward in #84).

### Note: test correctness

Verified the constrained optimum reasoning: `quadratic.PROBLEM`'s unconstrained minimum is `(0, 0)`; the "inactive" scenario's box `[-10, 10] x [-10, 10]` contains it, and the "active" scenarios' `x1 >= 0.5` lower bound makes it infeasible, pinning the constrained optimum to the boundary `x1 = 0.5`. Comparing `BFGS`/`trust-exact` against `lm`'s already-correct `x_final` (rather than a hardcoded expected value) is the right invariant to assert, since it ties correctness to a method already known to work rather than duplicating the optimum's closed form. No emojis, no GitHub issue references, in the new test comments.

# Implementation review

## Summary

The fix (`c70a1b6`) routes bounds format by backend rather than method name alone, exactly matching the RED-stage design notes and this review's own root-cause finding. `_bounds_for` and `_measure_bounds` both now default to trust-apl's least-squares-style array pair, and only fall back to scipy's per-dimension format when `backend.name == "scipy"` and the method isn't one of scipy's own least-squares names. All four previously-RED tests now pass, the pre-existing `lm` test is unaffected, and the full suite (334 tests) passes with no regressions. No issues found.

## Findings

### Note: fix addresses the root cause, not a workaround

The routing predicate is inverted at both call sites: least-squares-style is now the default, minimize-style the exception carved out only for `backend.name == "scipy"`. This directly matches what the tests-only review traced in `backends_ext/apl/solve.dyalog:75-78` - trust-apl's array-pair unpacking is unconditional across all its methods, so a fix that special-cases scipy rather than trust-apl is the correct shape, not an accidental one that happens to pass the tests. `BACKENDS`/backend registry (`trust_bench/backends/__init__.py`) currently contains only `SciPyBackend` and `APLBackend`, so the `backend.name == "scipy"` check is exhaustive for both branches; no other backend is silently misrouted.

Location: `trust_bench/studies/bounded.py:32-39`, `trust_bench/reporting/capability_matrix.py:15-27`

### Note: comment quality

The new comments explain the invariant (why scipy alone needs the per-method split) rather than narrating the fix, and don't reference the issue number. Consistent with the tests-only stage's comments.

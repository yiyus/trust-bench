# Review: Issue #77 - CLI: select which backends a report runs against

## Summary

Tests-only (RED) review; `docs/reviews/77.md` did not previously exist.
`AVAILABLE_BACKENDS`/`_select_backends`/`--backends` don't exist in
`trust_bench.cli` yet, confirmed directly: `pytest -m "not slow"
--continue-on-collection-errors` reports 200 passed, 25 deselected, 1
collection error, matching `docs/prs/77.md` exactly. One finding requires
resolving before GREEN: the RED test for the second acceptance criterion
("selecting a backend that a study doesn't support fails clearly... not a
silent empty result") uses `ill_conditioning` as its example and asserts a
`ValueError`, but `ill_conditioning.sweep` has its own pre-existing
`if method not in supported: continue` guard that silently skips
unsupported (method, backend) pairs. Called directly against
`APLBackend()`, it returns an empty dict — zero rows, zero exceptions —
which is exactly the "silent empty result" the issue exists to prevent, not
the `ValueError` the RED test and `docs/prs/77.md`'s design notes both
claim will happen.

## Findings

### Major: The chosen RED test for "fails clearly, not silently" exercises a study that fails silently

`docs/prs/77.md`'s design notes state: "Selecting `trust-apl` for one of
these studies raises `ValueError` from deep inside `Backend.solve`'s own
`method not in capabilities().methods` check... which is squarely what
'fails clearly... not a silent empty result' asks for." The corresponding
test,
`test_run_report_raises_clearly_when_a_study_does_not_support_the_selected_backend`,
picks `ill_conditioning` specifically for this:

```python
with pytest.raises(ValueError, match="trust-apl"):
    run_report(tmp_path, only=["ill_conditioning"], backends=["trust-apl"])
```

Verified directly, independent of the test suite:

```python
>>> from trust_bench.backends.apl_backend import APLBackend
>>> from trust_bench.studies import ill_conditioning
>>> ill_conditioning.sweep(backends=[APLBackend()])
{}
```

`ill_conditioning.py:22-24` (`for method in methods: if method not in
supported: continue`) is a pre-existing, deliberate guard, per its own
docstring ("Skips a (method, backend) pair the backend does not support") —
sensible for a genuinely mixed multi-backend sweep where one backend
supports a subset of the swept methods and the rest should still be
reported. But for a *single*-backend selection where that backend supports
none of the study's methods (exactly what `--backends trust-apl` against
`ill_conditioning` produces, since APL's only method is `"lm"` and
`ill_conditioning.METHODS = ["trust-exact", "Newton-CG", "BFGS"]`), the
result is a genuinely empty `dict`, and `results_to_dataframe(...)` on it
is a genuinely empty, zero-row `DataFrame` — confirmed directly, no
exception raised anywhere in the chain. `scaling.sweep` and
`dimensionality.sweep` (checked directly the same way) do raise `ValueError`
uncaught for `APLBackend`, so the design notes' general claim holds for
*those* studies; `ill_conditioning` is the one counterexample, and it's the
one the test happens to use.

This isn't a fragile assertion needing an adjusted match string — it's the
acceptance criterion's own explicit failure mode ("not a silent empty
result"), already latent in the codebase, going undetected because
`ill_conditioning.sweep`'s default `backends=BACKENDS` has only ever been
`[SciPyBackend()]`, which trivially supports all three swept methods. Two
reasonable fixes, not mutually exclusive:

- Pick a different study for this specific test (`scaling` or
  `dimensionality`, both verified to raise `ValueError` uncaught for
  `APLBackend`) — cheapest, but leaves `ill_conditioning` (and any future
  study with a similar per-pair skip) able to silently produce an empty
  report table under `--backends`, contrary to the acceptance criterion as
  a general property.
- Have `run_report` (or each `_write_*` writer) check whether a study
  produced any rows at all for the selected backend(s) and raise clearly if
  not, generalising across every study regardless of whether that study's
  own sweep function happens to raise internally. This matches the
  acceptance criterion's wording ("fails clearly... not a silent empty
  result") as a blanket guarantee rather than an incidental side effect of
  a particular study's method list, and doesn't require touching
  `ill_conditioning`'s legitimate skip-on-partial-support behaviour for
  genuinely mixed multi-backend sweeps.

Location: `tests/integration/test_cli.py:173-177`,
`trust_bench/studies/ill_conditioning.py:22-24`

## Verification

- Tests: RED confirmed. `pytest -m "not slow"
  --continue-on-collection-errors`: 200 passed, 25 deselected, 1 collection
  error (`test_cli.py`, `ImportError: cannot import name
  'AVAILABLE_BACKENDS'`), matching `docs/prs/77.md` exactly.
- Lint: pass (`ruff check` on the changed test file).
- Architecture: confirmed every study function already accepts a
  `backends=BACKENDS` keyword (`baseline`, `large_residual`,
  `ill_conditioning`, `bounded`, `scaling`, `dimensionality`,
  `derivative_source`, `capability_matrix`, plus `robust_loss`'s
  SciPy-loss sweep specifically), so the design notes' claim that no
  individual study module needs changing for `cli.py` to forward a
  resolved `backends` list through is accurate.
- APL semantics: not applicable; no APL expressions in this test surface.
  Verified the Python-side behavioural claims above directly against
  `trust_bench.studies.{ill_conditioning,scaling,dimensionality}` and
  `APLBackend`, independent of the test suite.

## Recommendation

Request changes: resolve the `ill_conditioning`/silent-empty-result
mismatch (swap the test's chosen study, add a generic empty-result check,
or both) before GREEN.

# Review: Issue #17 - SciPy backend: minimize methods (BFGS, L-BFGS-B, Newton-CG, trust-*)

## Summary

Tests-only (RED) stage review. The test surface covers both acceptance
criteria directly, and goes further: it adds losses/derivative-mode tests
specifically to close the two Major gaps flagged in issue #16's
implementation review (`docs/reviews/16.md`), verifying the intended
fix's shape before writing it rather than repeating the same
declared-capability-without-behaviour pattern. Every scipy-semantics
claim in the PR doc's design notes (bounds handled by warning rather
than exception, `Newton-CG`/`trust-exact` rejecting finite differences,
`trust-constr`'s gradient living in `.grad` rather than `.jac`, per-method
convergence on the bounded quadratic) was independently reproduced
against `scipy` directly rather than taken on trust. 14 of 19 tests fail
collection/execution as expected; the PR doc's own disclosure of 5
vacuously-passing tests (failing for the "unknown method" guard rather
than the real rejection logic not yet written) was verified and found
accurate. No issues found.

## Findings

None.

### Note: vacuous-pass disclosure verified accurate

`test_unbounded_methods_reject_box_constraints` (`BFGS`, `Newton-CG`,
`trust-exact`) and
`test_finite_difference_derivative_mode_is_rejected_where_scipy_requires_an_analytic_jacobian`
(`Newton-CG`, `trust-exact`) pass today, confirmed by running the suite,
but only because `solve`'s existing "unknown method" guard raises
`ValueError` for methods not yet in `capabilities().methods` — the same
exception type the tests assert, for an unrelated reason. The PR doc
names this explicitly (citing `docs/reviews/9.md`'s prior finding of the
same shape) instead of presenting a clean RED. This is exactly the right
call: a false clean RED here would have been misleading, since GREEN
would need to re-verify these five pass for the *right* reason once the
real bounds/derivative-mode checks exist, not just continue passing.

### Note: intentional deviation from Section 4.1's illustrative `kind` pairing, documented

The design doc's comment on `Problem.kind` reads `# dispatch: LM/Newton
vs BFGS`, suggesting BFGS-family methods pair with `kind="scalar"`. The
PR doc declares `kind="residuals"` for all five methods instead, with a
direct justification: every problem in the catalog today is
`kind="residuals"` (issue #9), the adapter wraps `0.5*||r||^2` unchanged
for that kind, and a `kind="scalar"` problem would need a different
adapter (`residual` already being the objective, per Section 4.1's own
comment on `Problem.residual`) rather than reusing this one. This is a
documented, reasoned deviation, not an unexplained one.

### Note: method-set scoping justified against the plan and the prototype

`trust-ncg`/`trust-krylov` were verified to converge on the trivial
quadratic but excluded because no acceptance criterion or plan section
currently asks for them. `trust-exact` is included on `prototype/harness.py`'s
existing use of it as a true-Newton comparator (Section 9 item 2), not
invented for this issue. Reasoning is sound and avoids scope creep beyond
what issue #17 or the referenced design sections request.

## Verification

- Tests: 14 of 19 fail as expected (`ValueError: scipy has no method '<name>'`), matching the PR doc's count exactly; the remaining 5 pass vacuously as disclosed, confirmed by name.
- Linters: `ruff check` passes on `tests/backends/test_scipy_minimize.py`.
- scipy semantics: independently reproduced every claim in the design notes — bounds silently ignored with a `RuntimeWarning` (not raised) for `BFGS`/`Newton-CG`/`trust-exact`; `jac="2-point"` accepted by `BFGS`/`L-BFGS-B`/`trust-constr` and rejected with `ValueError: Jacobian is required for ...` by `Newton-CG`/`trust-exact`; `trust-constr`'s `.jac` is empty and `.grad` holds the real gradient, the other four methods populate `.jac`; bounded convergence to `x1=0.5` for `L-BFGS-B`/`trust-constr` matches the test's `atol=1e-6`.

Approve

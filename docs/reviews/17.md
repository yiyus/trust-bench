# Review: Issue #17 - SciPy backend: minimize methods (BFGS, L-BFGS-B, Newton-CG, trust-*)

## Summary

Tests-only (RED) stage review. The test surface covers both acceptance
criteria directly, and goes further: it adds losses/derivative-mode tests
specifically to close the two Major gaps flagged in issue #16's
implementation review (`docs/reviews/16.md`), verifying the intended
fix's shape before writing it rather than repeating the same
declared-capability-without-behaviour pattern. Every scipy-semantics
claim in the PR doc's design notes (bounds handled by warning rather
than exception, `Newton-CG`/`trust-exact` rejecting finite differences,
`trust-constr`'s gradient living in `.grad` rather than `.jac`, per-method
convergence on the bounded quadratic) was independently reproduced
against `scipy` directly rather than taken on trust. 14 of 19 tests fail
collection/execution as expected; the PR doc's own disclosure of 5
vacuously-passing tests (failing for the "unknown method" guard rather
than the real rejection logic not yet written) was verified and found
accurate. No issues found.

## Findings

None.

### Note: vacuous-pass disclosure verified accurate

`test_unbounded_methods_reject_box_constraints` (`BFGS`, `Newton-CG`,
`trust-exact`) and
`test_finite_difference_derivative_mode_is_rejected_where_scipy_requires_an_analytic_jacobian`
(`Newton-CG`, `trust-exact`) pass today, confirmed by running the suite,
but only because `solve`'s existing "unknown method" guard raises
`ValueError` for methods not yet in `capabilities().methods` — the same
exception type the tests assert, for an unrelated reason. The PR doc
names this explicitly (citing `docs/reviews/9.md`'s prior finding of the
same shape) instead of presenting a clean RED. This is exactly the right
call: a false clean RED here would have been misleading, since GREEN
would need to re-verify these five pass for the *right* reason once the
real bounds/derivative-mode checks exist, not just continue passing.

### Note: intentional deviation from Section 4.1's illustrative `kind` pairing, documented

The design doc's comment on `Problem.kind` reads `# dispatch: LM/Newton
vs BFGS`, suggesting BFGS-family methods pair with `kind="scalar"`. The
PR doc declares `kind="residuals"` for all five methods instead, with a
direct justification: every problem in the catalog today is
`kind="residuals"` (issue #9), the adapter wraps `0.5*||r||^2` unchanged
for that kind, and a `kind="scalar"` problem would need a different
adapter (`residual` already being the objective, per Section 4.1's own
comment on `Problem.residual`) rather than reusing this one. This is a
documented, reasoned deviation, not an unexplained one.

### Note: method-set scoping justified against the plan and the prototype

`trust-ncg`/`trust-krylov` were verified to converge on the trivial
quadratic but excluded because no acceptance criterion or plan section
currently asks for them. `trust-exact` is included on `prototype/harness.py`'s
existing use of it as a true-Newton comparator (Section 9 item 2), not
invented for this issue. Reasoning is sound and avoids scope creep beyond
what issue #17 or the referenced design sections request.

## Verification

- Tests: 14 of 19 fail as expected (`ValueError: scipy has no method '<name>'`), matching the PR doc's count exactly; the remaining 5 pass vacuously as disclosed, confirmed by name.
- Linters: `ruff check` passes on `tests/backends/test_scipy_minimize.py`.
- scipy semantics: independently reproduced every claim in the design notes — bounds silently ignored with a `RuntimeWarning` (not raised) for `BFGS`/`Newton-CG`/`trust-exact`; `jac="2-point"` accepted by `BFGS`/`L-BFGS-B`/`trust-constr` and rejected with `ValueError: Jacobian is required for ...` by `Newton-CG`/`trust-exact`; `trust-constr`'s `.jac` is empty and `.grad` holds the real gradient, the other four methods populate `.jac`; bounded convergence to `x1=0.5` for `L-BFGS-B`/`trust-constr` matches the test's `atol=1e-6`.

## Recommendation

# Review: Issue #17 - SciPy backend: minimize methods (BFGS, L-BFGS-B, Newton-CG, trust-*) (implementation)

## Summary

Implementation-stage review. `SciPyBackend.solve` now dispatches to
`scipy.optimize.minimize` for the five new methods via `_solve_minimize`,
sharing `RunResult` assembly with the existing `_solve_least_squares`
path. Both acceptance criteria are met, and the PR doc's stated goal of
closing issue #16's two Major findings (declared `Capabilities` fields
with no `solve()` behaviour behind them) is genuinely achieved: `loss`
and `derivative_mode` are now read from `config` and independently
verified to reach `scipy`, not just declared. `make test` (119 passed),
`make lint`, and `make coverage` (99% on `scipy_backend.py`) all
reproduce the PR doc's figures exactly, and the two GREEN-cycle surprises
disclosed in the PR doc (the `trust-constr` Hessian-update-strategy
requirement, the looser `trust-constr` tolerance) were independently
reproduced. One gap of the same shape as issue #16's findings survived
into this implementation, in a part of the design space the new tests
do not reach: `problem.hessian is None` for the three Hessian-using
methods.

## Findings

### Major: Hessian source ignores `problem.hessian is None`, unlike the parallel Jacobian handling in the same method

`_solve_minimize` derives `hess` from `use_fd` (the *gradient*
derivative-mode flag), not from whether `problem.hessian` is actually
available: `kwargs["hess"] = BFGSHessianUpdate() if use_fd else
problem.hessian`. When `use_fd` is `False` and `problem.hessian` is
`None`, this passes `hess=None` through to `minimize`. Confirmed
directly: `trust-exact` raises `ValueError: Hessian matrix is required
for trust region exact minimization.`; `Newton-CG` and `trust-constr` do
not raise, but silently solve without ever calling a Hessian
(`nhev == 0` in both cases, against `nhev == 3` for the same problem with
an analytic Hessian), despite both declaring `analytic_hessian=True`.
This is the identical pattern issue #16's review flagged twice
(`derivative_modes`/`losses` declared without the `config` plumbing
behind them) and this PR's own design notes describe deliberately
avoiding for the *gradient* case (`problem.jacobian is None` correctly
triggers `use_fd` a few lines above). No problem in the catalog
currently has `hessian=None`, so nothing fails today, but the same
`Callable | None` type on `Problem.hessian` that motivated the
FD-fallback test for `problem.jacobian` applies equally here, and no
test in `test_scipy_minimize.py` exercises it. Recommend mirroring the
Jacobian handling: `use_fd_hessian = problem.hessian is None` (or a
`config`-driven Hessian derivative-mode, if one is intended), routed to
the same `BFGSHessianUpdate()` already used for the FD-gradient case, so
`trust-exact` never receives `hess=None` and `Newton-CG`/`trust-constr`
never silently drop the Hessian a caller's `capabilities()` lookup told
them was used.

Location: `trust_bench/backends/scipy_backend.py:205-210`.

## Verification

- Tests: pass. `make test` reports 119 passed, matching the PR doc (19 new, 100 from prior issues unaffected).
- Linters: `make lint` passes (`ruff check .`).
- Coverage: `make coverage` reproduces the PR doc's figures exactly: 99% on `scipy_backend.py` (91 statements, 1 missed — `_map_minimize_status`'s `FAILED` fallback, the same accepted-gap shape as prior issues' defensive branches).
- Fixes for issue #16's findings verified directly, not just read off the diff: `loss="huber"` vs the default on `trf` changes `n_feval` (3 vs 2); a `derivative_mode="finite-difference"` `least_squares` call never invokes a counting-wrapped analytic Jacobian and still converges.
- Both GREEN-cycle surprises in the PR doc reproduced independently: `hess="2-point"` on `trust-constr` under FD-gradient raises the quasi-Newton-strategy error the PR doc quotes; `trust-constr`'s bounded solution error (~6.4e-5) is comfortably inside the loosened `atol=1e-3` while `L-BFGS-B`'s stays exact, and the unrelaxed `>= 0.5 - 1e-6` lower-bound assertion still holds since `trust-constr`'s deviation lands on the feasible side.

Approve with minor changes

# Review: Issue #45 - Bootstrap project plan and pipeline tooling

## Summary

This PR checks in the project's own pipeline tooling (`CLAUDE.md`, `PROCESS.md`, the two
slash commands, four hooks, three skills, the statusline, `settings.json`) plus the
already-reviewed `docs/plans/trust-bench.md` and `docs/reviews/trust-bench.md`. There is no
test surface: this is documentation and configuration, so the review was run as a single
implementation-style pass rather than split into RED/GREEN stages.

The writing-style rules the PR itself introduces are followed correctly throughout
(no em-dashes, no emojis outside the rule text's own bad examples, no first-person narrative
in the docs the rule scopes to). The commands, hooks, and skills are internally consistent
with the workflow `PROCESS.md` describes. Two Major findings came out of exercising the
hooks directly rather than reading them: `block-dangerous-bash.sh` blocks legitimate
commands that merely mention a protected path or a secret-file extension as plain text, and
the audit log the PR describes as gitignored is not actually covered by any `.gitignore` in
the repository. Both are mechanical fixes; neither is a design problem.

## Findings

### Major: `block-dangerous-bash.sh` blocks commands that only mention a protected path or secret-file name as text

Blocks 10 and 11 each grep the *entire* command string twice: once for a protected-path or
secret-file pattern, once for a write-op or reader-command token, and block when both
patterns appear anywhere in the string. Neither check confirms the two matches refer to the
same argument, so a command that writes to an unrelated location while merely mentioning a
protected path in a string, or that reads an unrelated file while mentioning `.env` in a
comment or description, gets blocked.

Reproduced directly against the shipped script (not hypothetical):

```
$ .claude/hooks/block-dangerous-bash.sh <<<'{"tool_name":"Bash","tool_input":{"command":"echo \"see .claude/hooks/audit-log.sh for details\" > /tmp/notes.txt"}}'
BLOCKED. Reason: Command writes to a protected path...
```

The command only writes to `/tmp/notes.txt`; `.claude/hooks/audit-log.sh` appears solely as
descriptive text inside the `echo` argument.

```
$ .claude/hooks/block-dangerous-bash.sh <<<'{"tool_name":"Bash","tool_input":{"command":"grep -n \"missing .env file\" README.md"}}'
BLOCKED. Reason: Command reads a secret file...
```

The command only reads `README.md`; `.env` appears solely inside the search string. This
class of false positive is not a corner case for this project: writing documentation about
the hooks themselves, or grepping for the literal string `.env`, are both things the pipeline
will need to do routinely (this review needed to, to verify the hook's behaviour). `git
add`/`sed -i` style writes to protected paths and `cat`/`grep` style reads of secrets are
worth blocking; blocking on unrelated co-occurrence of the same words is not.

`protect-paths.sh` and `protect-reads.sh` do not have this problem: they read
`tool_input.file_path` as a structured field, so the pattern only ever matches the actual
target. Blocks 10 and 11 exist because Bash has no structured target, but the fix is the
same shape of problem: anchor the match to the token immediately following a redirect/reader
command (or the command's actual arguments) rather than grepping the raw string twice.

Location: `.claude/hooks/block-dangerous-bash.sh:162-187` (blocks 10 and 11).

### Major: `.claude/audit.log` is not actually covered by a `.gitignore`, contradicting the PR's own claims

The commit message states "`.claude/audit.log` is left untracked: it is a per-session
runtime audit trail... not project configuration," `audit-log.sh`'s header comment says
"(gitignored, add it to .gitignore)," and `.claude/hooks/README.md` says "The file is
gitignored by default." None of this is true in the shipped state:

```
$ git check-ignore -v .claude/audit.log
(no output, exit 1 -> not ignored)
$ git status --short .claude/audit.log
?? .claude/audit.log
```

There is no `.gitignore` anywhere in the repository (only a template copy at
`.devcontainer/kit/.gitignore`, which is dev-container scaffolding outside the git tree and
already contains the right entries: `.claude/audit.log`, `CLAUDE.local.md`, etc.). Today
`.claude/audit.log` is untracked only because nothing has staged it yet; an explicit `git add
.claude/audit.log` is not blocked by any hook rule (block 3a only catches bulk/glob adds), so
a future session could commit the audit trail with no guard rail catching it. This PR is the
place to add the root `.gitignore`; the kit template already has the content ready to copy.

Location: repository root (missing `.gitignore`); claims at `.claude/hooks/README.md:12`,
`.claude/hooks/audit-log.sh:7`, and the commit message for `2240712`.

### Minor: `dyalog-docsearch` skill uses US spelling in three places

`CLAUDE.md`'s UK-English rule is followed carefully everywhere else added by this PR
(`CLAUDE.md`, `PROCESS.md`, `docs/plans/trust-bench.md`), but
`.claude/skills/dyalog-docsearch/SKILL.md` has "Summarize" (twice) and "behavior" (heading),
where the rest of the kit uses "Summarise"/"behaviour".

Location: `.claude/skills/dyalog-docsearch/SKILL.md:15,37,63`.

### Minor: `settings.json`'s `permissions.deny` list does not fully mirror `protect-reads.sh`

`.claude/README.md` and `.claude/hooks/README.md` describe `permissions.deny` as
"defence-in-depth" that "enforces the same paths" as `protect-reads.sh`. The two lists have
drifted: `protect-reads.sh` also blocks `.pgpass`, `id_ecdsa`, `id_dsa`, `*.key`, `*.p12`, and
`*.pfx`, none of which appear in `settings.json`'s deny list. Not independently exploitable,
since the `PreToolUse` hook still blocks these reads regardless of the `permissions.deny`
gap, but the two lists are documented as parallel and are not, which will mislead whoever
next edits one without the other.

Location: `.claude/settings.json:24-41` vs `.claude/hooks/protect-reads.sh:46-60`.

### Note: hook false positive triggered mid-review

While drafting the reproduction steps for the first finding above, an unrelated shell command
in this review session (a `cat > file <<EOF` listing changed filenames, one of which was a
hook path) was itself blocked by the same bug. No action is needed beyond the fix already
recommended; recorded here because it is direct evidence the failure mode is not rare.

## Verification

- Tests: not run. No test suite exists for this documentation/configuration-only change.
  `bash -n` was run against all five shipped shell scripts (`audit-log.sh`,
  `block-dangerous-bash.sh`, `protect-paths.sh`, `protect-reads.sh`, `statusline.sh`); all
  parse cleanly. `shellcheck` is not installed in this environment, so no linter pass was
  possible beyond that.
- The two Major findings were verified by invoking the shipped hook scripts directly with
  crafted JSON fixtures, not by inspection alone.
- Writing-style conventions (em-dashes, emojis, first-person narrative, UK spelling) were
  checked with `grep` across every file this PR adds; only the two Minor spelling
  inconsistencies above survived manual review of the matches (the rest were either the rule
  text's own illustrative bad examples or module names such as `scipy.optimize`).
- Cross-file architectural consistency (`.claude/README.md`, `.claude/commands/README.md`,
  `.claude/hooks/README.md`, `PROCESS.md`, `CLAUDE.md`, and the command/skill/hook files
  themselves) was read end to end; no contradictions found beyond the `permissions.deny` drift
  noted above.

## Recommendation

Request changes. Both Major findings are mechanical fixes (anchor the two grep checks in
`block-dangerous-bash.sh` to the actual write/read target; add the missing root
`.gitignore`), but they affect the guardrail tooling this entire pipeline depends on going
forward, and the false-positive hook bug is already reproducible in ordinary use rather than
a theoretical edge case.

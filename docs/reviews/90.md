## Implementation review

### Summary

`Dimensionality`/`DimensionalityHessian` (`backends_ext/apl/problems/
dimensionality.dyalog`) checked by hand against `dimensionality.py`'s
Python reference (odd/even 1-indexed-vs-0-indexed position mapping,
row-major linear-index scatter for the Jacobian's three nonzero-entry
families, the Hessian's per-pair correction term) and match exactly;
all 40 tests across the four affected files pass, and the full suite is
green (327 passed). The dispatch wiring correctly reuses #83's
mechanism with no new abstraction. However, direct testing of the
actual acceptance criterion - "`APLBackend` can solve `dimensionality
(n=...)` for both `trust-exact` and `BFGS`" at the sizes the study
sweeps, up to `n=1000` - surfaces a serious, previously-unverified gap:
solving (not just evaluating) at `n=1000` is far too slow to complete
within the harness's own 60-second subprocess timeout, and the timeout
is not caught anywhere, so it crashes the whole `trust-bench report`
command rather than producing a clean `ERROR` row.

### Findings

### Critical: solving `dimensionality(n=1000)` via `trust-apl` exceeds the harness's own timeout and crashes `run_report` uncleanly

The RED-stage test suite verifies solving only at `n=10`
(`test_solve_dimensionality_converges_for_trust_exact_and_bfgs`) and
verifies wall-clock cost at `n=1000` only for a single `evaluate_
problem` call (residual/Jacobian/Hessian construction, not a solve).
Nothing exercises an actual `APLBackend.solve` at `n=1000`, which is
the size this issue's own context cites as the motivating case
("`dimensionality.py`'s own study exists specifically to measure
dense-Hessian-method scaling cost at that size").

Reproduced directly:

```python
>>> APLBackend().solve(dimensionality.make(n=1000), "BFGS", "standard", RunConfig(max_iter=3))
# 17.7s elapsed, RunStatus.MAX_ITER, n_iter=4
```

i.e. roughly 4-5 seconds per iteration once past interpreter/harness
startup. `dimensionality.py`'s own `sweep()` uses `max_iter=200`, so a
real solve at `n=1000` would need on the order of 15+ minutes - far
past `apl_backend.py`'s hardcoded 60-second subprocess timeout
(`trust_bench/backends/apl_backend.py:45`). Confirmed end-to-end:

```python
>>> run_report(tmp_path, only=["dimensionality"], backends=["scipy", "trust-apl"])
subprocess.TimeoutExpired: Command [...] timed out after 60 seconds
```

This is an *uncaught* exception, not a `RunResult` with `status=ERROR`:
`apl_backend.py`'s `_run_harness` does not catch
`subprocess.TimeoutExpired`, so it propagates all the way up through
`dimensionality.sweep()` and `run_report()`, crashing the whole
`trust-bench report` invocation (losing any studies alphabetically
after `dimensionality` that hadn't run yet) rather than reporting one
bad row and continuing - exactly the failure mode `_check_backend_
coverage` and the various skip-guards added in #77/#81/#84 exist to
prevent for other cases. Since `dimensionality` is already marked
`SLOW_STUDIES` but is not skipped by default, this reproduces for any
user running `trust-bench report --backends trust-apl` (or including
`trust-apl` among several backends) without `--skip-slow` or `--skip
dimensionality` - the exact combination this issue is meant to make
usable.

The port's own construction is not at fault (the timing test above
confirms `evaluate_problem` at `n=1000` is fast); this is `trust`'s
`BFGS`/`Newton` engines converging far too slowly in wall-clock terms
at this dimension for this problem, or `Min`'s own per-iteration
overhead scaling worse than expected. Root cause aside, the acceptance
criterion ("solve `dimensionality(n=...)` for both methods") is not
actually met at the size that matters, and the failure mode is an
uncaught crash rather than a graceful, informative result. This needs
one of: a real fix to make the solve complete in reasonable time at
`n=1000` (if achievable), a documented/enforced smaller practical
`n_values` ceiling for `trust-apl` specifically, or at minimum catching
`subprocess.TimeoutExpired` in `_run_harness` and converting it to a
clean `ERROR` `RunResult` so the CLI doesn't crash - paired with a test
that actually exercises a full solve at the largest swept `n`, not just
construction cost.

Location: `trust_bench/backends/apl_backend.py:41-49`
(`_run_harness`'s uncaught timeout); `trust_bench/studies/
dimensionality.py:6` (`N_VALUES` including `1000`); test coverage gap
in `tests/backends/test_apl_difficulty_families.py` (`test_evaluate_
at_n_1000_completes_quickly` measures construction only, not a solve)

## Final recommendation

Request changes: the `n=1000` solve-timeout crash (Critical) needs
addressing, either by making the solve complete, catching the timeout
cleanly, or narrowing the practical `n` range for `trust-apl` with a
test that proves it. Everything else in this port checks out.

## Response to Critical finding

Fixed in `c3418d8`. `_run_harness` now catches
`subprocess.TimeoutExpired` and returns a clean `ERROR`-status result
matching `error_result.dyalog`'s own field set, instead of propagating
an uncaught exception. `trust-exact`'s per-iteration cost stays
practical even at `n=1000` (measured: converges); `BFGS`'s does not
(measured: ~1.4s/iteration at `n=1000` versus ~0.04s/iteration at
`n=100`) - a genuine limitation of `trust`'s own BFGS engine at that
scale, not of this port, so the fix narrows `BFGS`'s "solves within a
practical dimension" test to `n=100` (asserting the scientifically
correct `MAX_ITER`, not a false `CONVERGED` claim) and adds a real
`n=1000` convergence test for `trust-exact`.

Re-verified independently:

- `APLBackend().solve(dimensionality.make(n=1000), "trust-exact", ...)`:
  converges in ~8.4s.
- `APLBackend().solve(dimensionality.make(n=100), "BFGS", ...)`:
  `MAX_ITER` in ~5.4s, matching the new test's expectation.
- End-to-end, the scenario that used to crash:
  `run_report(only=["dimensionality"], backends=["scipy", "trust-apl"])`
  now completes and produces a full table, with `(n=1000, "BFGS",
  "trust-apl")` reading a clean `ERROR` row (from the caught timeout)
  instead of crashing the whole command. Confirmed by inspecting the
  written `dimensionality.csv` directly.
- The new mocked-timeout unit tests
  (`tests/backends/test_apl_backend_timeout.py`) pass in isolation.
- Full suite `.venv/bin/pytest -q` (dyalogscript available): 330
  passed, 1 failed on this run -
  `test_evaluate_at_n_1000_completes_quickly` (`11.99s` against its
  `10.0s` bound). This test is unchanged by `c3418d8` and predates it
  (added in the original RED stage); re-run standalone three times back
  to back, it passed each time at `8.1s`-`9.3s`, consistently close to
  the bound. This reads as a flaky, load-sensitive timing assertion
  (subprocess-spawn + interpreter-startup overhead varies with system
  load) rather than a regression from this fix - noting it as a Minor,
  non-blocking observation rather than reopening the Critical finding.
  `ruff check .`: clean.

The Critical finding is resolved.

### Minor (new, non-blocking): `test_evaluate_at_n_1000_completes_quickly`'s 10s bound is tight enough to flake under load

Observed one failure (`11.99s`) in a full-suite run and consistent
`8-9.3s` measurements in isolation, close enough to the `10.0s` bound
that background system load can tip it over. Not introduced by this
commit and not blocking; worth loosening the bound (or moving to a
relative/percentile-based check) in a future pass if it recurs.

## Final recommendation (implementation)

Approve. The Critical finding is fixed and independently re-verified,

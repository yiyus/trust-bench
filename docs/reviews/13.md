# Review: Issue #13 - Append-only JSONL result store and pandas loader (storage.py)

## Summary

Tests-only (RED) stage review. `tests/unit/test_storage.py` covers both
acceptance criteria: `test_appending_never_overwrites_an_existing_record`
appends two `RunResult`s to the same path and checks both lines survive in
order, and `test_load_reconstructs_a_dataframe_matching_the_stored_records`
checks that three scalar columns (`problem_id`, `cost_final`, `status`)
round-trip through `append`/`load`. The RED failure (`ModuleNotFoundError:
No module named 'trust_bench.core.storage'`) is confirmed. One Minor gap:
the PR doc's design notes commit to "no column flattening" (plain
`pd.DataFrame(records)`, not `pd.json_normalize`) as a deliberate reading
of the acceptance criterion, but no test forces that distinction over an
implementation that flattens nested fields.

## Findings

### Minor: no test distinguishes flat nested-field columns from `json_normalize`-style flattening

The PR doc's design notes argue that "a DataFrame matching the stored
records" means each JSONL line becomes one row with nested fields
(`provenance`, `timing`, `config`) kept as single object-dtype columns
holding the raw dict, not expanded into dotted columns
(`provenance.backend_name`, etc.) via `pd.json_normalize`. This is a
reasonable reading, but `test_load_reconstructs_a_dataframe_matching_the_
stored_records` only asserts on `problem_id`, `cost_final`, and `status`,
none of which are nested. An implementation that flattens nested fields
(or one that drops them, or one that leaves them as JSON strings instead
of dicts) would pass the test unchanged, so the acceptance criterion as
tested does not actually pin down the documented design decision.

Recommendation: add an assertion on a nested field, e.g. `df.loc[0,
"provenance"]["backend_name"] == results[0].provenance.backend_name` (or
equivalent for `config`), so a flattening or string-serialising
implementation fails the test rather than silently satisfying it.

Location: `tests/unit/test_storage.py:58-63`

### Note: `pandas` is not yet a declared dependency

`pyproject.toml`'s `[project] dependencies` still lists only `numpy` and
`scipy`; the PR doc's anticipated implementation order correctly lists
adding `pandas` as step 1 of the GREEN cycle, so this is expected at the
RED stage and not a gap in the tests themselves. Flagging only so the
GREEN-cycle review checks it landed.

Location: `pyproject.toml`

## Implementation review

## Summary

`trust_bench/core/storage.py` implements `append(result, path)` (opens in
`"a"` mode, one JSON line per call, creates the file if absent) and
`load(path)` (parses each line into a plain dict and hands the list
straight to `pd.DataFrame`, no `RunResult`/`json_normalize` round-trip),
matching the RED-stage design notes. The tests-only review's Minor finding
was addressed directly in the RED commit: the load test now asserts on
`df.loc[0, "provenance"]["backend_name"]` and `df.loc[0, "config"]`,
closing the gap that let a flattening or string-serialising implementation
pass unchanged. `pandas` was added to `[project] dependencies` as
anticipated. All 76 tests pass, lint is clean, coverage on `storage.py` is
100%. No issues found.

No issues found.

### Note: design rationale is sound

`bounded`'s exclusion is consistent with its data being categorical (scenario x method) rather than a natural sweep, matching the issue's own explicit scope note.

`test_plot_metric_vs_sweep_marks_non_converged_points_distinctly`'s expectation (2 main lines + 1 hollow-marker overlay line for the single `MAX_ITER` point) directly encodes the design notes' "draw the full line through every point, then overlay a hollow marker for the non-`CONVERGED` ones" approach, rather than the alternative (splitting into solid/dashed segments) the PR doc explicitly considered and rejected. `test_plot_metric_vs_sweep_draws_no_overlay_when_status_col_is_not_given` correctly guards that `status_col`'s absence doesn't change existing behaviour, protecting `large_residual`'s current plot call from a silent regression.

## Findings

### Note: fix addresses the root cause, not a workaround

Each acceptance-criterion plot call matches its stated spec precisely: `ill_conditioning` (`dist_to_opt` vs `kappa`, log-log, grouped by `method`+`backend`, `status_col="status"`), `dimensionality` (`dist_to_opt` vs `n`, log-x only), `scaling` (`grad_norm_final` vs `scale`, log-log, grouped by `method`+`x_scale`+`backend`), `robust_loss` (`distance` vs `fraction`, grouped by `loss`+`backend`), and `capability_matrix` (the new heatmap). `_write_bounded`/`_write_derivative_source` are confirmed untouched, matching the issue's explicit scope. `html_report.py` needed no changes: confirmed directly that `build_html_report` already globs every `.png` in the output directory (`html_report.py:39-44`), so a plot reaching `report.html` really is a direct consequence of `cli.py` writing it to the same directory, not a separate wiring path that could silently miss one of the five.

Location: `trust_bench/cli.py:70-150`, `trust_bench/reporting/plots.py`

### Note: end-to-end verification

Ran `run_report(tmp, only=["ill_conditioning", "dimensionality", "scaling", "robust_loss", "capability_matrix"])` directly: all five `.png` files are written, each a real non-empty image (30-54 KB), alongside their `.csv` tables.

### Note: `_capability_category`'s three-way branch is correct

Verified the logic directly: `agree` when `declared == measured` (either both `True` or both `False`, since `row["agrees"]` is exactly that equality); `declared-only` when `declared=True, measured=False`; the remaining case (`declared=False, measured=True`, the rarer/more surprising direction the design notes call out) correctly falls through to `disagree`. Matches the three categories in the tests-only design notes and the issue's acceptance criteria literally.

### Note: code quality and comment hygiene

`plot_metric_vs_sweep`'s new `subset.sort_values(x)` (added during GREEN, not present in the RED-stage test surface) is a correct robustness addition: `groupby` doesn't guarantee row order within a group, and an unsorted line would zigzag rather than trace a clean sweep - worth having even though no RED-stage test exercised it directly, since every new plot call groups data assembled from dict/JSON round-trips with no guaranteed order. `_group_label`, `_capability_category`, and `plot_capability_matrix`'s docstring all state the invariant/behaviour directly, with no narration of the implementation process and no GitHub issue references. No emojis or em-dashes.

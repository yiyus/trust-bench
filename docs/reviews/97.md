## Summary

Tests-only (RED) stage review. The new tests fail for the stated reason (missing `plot_capability_matrix`, missing `plot_metric_vs_sweep` keyword arguments, missing plot files), and the underlying design claims check out directly against the current source and real data. No issues found.

## Findings

### Note: reproduction confirmed exactly

`tests/reporting/test_plots.py` fails to collect with `ImportError: cannot import name 'plot_capability_matrix' from 'trust_bench.reporting.plots'`, matching the PR doc. Confirmed `trust_bench/reporting/plots.py` has no `status_col` parameter and no multi-column group-label joining (`plot_metric_vs_sweep`'s current `label=str(name)` would render a tuple repr like `"('a', 'p')"` for a list `group`, not `"a/p"`) - both gaps the new tests target.

Confirmed the CLI-level claim directly (without running the full slow smoke test, matching the PR doc's own stated approach): `run_report(tmp, only=["ill_conditioning"])` against the unmodified `cli.py` writes only `ill_conditioning.csv`, no `.png`. `cli.py` currently calls `save_figure`/`plot_metric_vs_sweep` exactly once, for `large_residual` only (`cli.py:65-66`) - confirming the other four studies plus the capability matrix have no plot output yet.

### Note: design rationale is sound

Verified `derive_matrix()`'s real shape matches what `test_plot_capability_matrix_draws_a_cell_per_field_and_backend_method_pair` asserts: 2 fields x 9 (backend, method) pairs for the default scipy-only backend set. Also ran it with both registered backends (`scipy`, `trust-apl`): 24 rows, all `agrees=True` - matching the design notes' claim that the categoriser currently sees an all-agree matrix, a reasonable state to confirm the categoriser doesn't misclassify the common case before adding the disagreement-detecting categories.

The three-category scheme (`agree`/`declared-only`/`disagree`) matches the issue's acceptance criteria literally, and is a different shape from Section 9.1's own mockup (per-language feature columns with yes/no cells) - a deliberate, documented adaptation to `derive_matrix()`'s actual per-(backend, method, field) row shape rather than an attempt to reproduce the mockup exactly, which is the right call since the mockup predates `capability_matrix.py`'s real output shape.

`bounded`'s exclusion is consistent with its data being categorical (scenario x method) rather than a natural sweep, matching the issue's own explicit scope note.

### Note: test design for the new plot_metric_vs_sweep behaviour is well-targeted

`test_plot_metric_vs_sweep_marks_non_converged_points_distinctly`'s expectation (2 main lines + 1 hollow-marker overlay line for the single `MAX_ITER` point) directly encodes the design notes' "draw the full line through every point, then overlay a hollow marker for the non-`CONVERGED` ones" approach, rather than the alternative (splitting into solid/dashed segments) the PR doc explicitly considered and rejected. `test_plot_metric_vs_sweep_draws_no_overlay_when_status_col_is_not_given` correctly guards that `status_col`'s absence doesn't change existing behaviour, protecting `large_residual`'s current plot call from a silent regression.

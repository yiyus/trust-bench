# Review: Issue #19 - RunConfig and intent-level tolerance mapping (config.py)

## Summary

Tests-only (RED) stage review. `docs/reviews/19.md` did not exist yet, and
`trust_bench.core.config` does not exist, confirming this is the first
review pass. The test suite is well-scoped against the acceptance criteria:
the per-method tolerance-to-native-parameter mapping in `test_scipy_
minimize.py`'s `TOLERANCE_PARAMS` was independently verified against
`scipy.optimize.show_options` for every method (`BFGS`, `L-BFGS-B`,
`Newton-CG`, `trust-exact`, `trust-constr`) and matches exactly; `least_
squares` accepting `ftol`/`xtol`/`gtol` unconditionally was confirmed
against its signature. Collection was run directly: 5 files fail to
collect with `ModuleNotFoundError: No module named 'trust_bench.core.
config'` as claimed, and the other 72 tests are unaffected, matching the
PR doc's "Confirmed failure" section exactly.

Two gaps surfaced by running the new round-trip test against a minimal
stand-in `RunConfig` and against the existing (unmodified) `RunResult.
from_dict`: one is a genuine defect in how the new config round-trip test
is written, the other is an implementation-scope gap the PR doc does not
mention. Both are laid out below with reproductions.

## Findings

### Major: `RunConfig` round-trip test cannot pass under the codebase's established `from_dict` pattern

`test_run_config_round_trips_through_json_serialisation` constructs
`RunConfig(bounds=[(0.0, 1.0)], ...)` and asserts equality after a
`to_dict` / JSON / `from_dict` round trip. `TimingStats.from_dict` and
`EnvProvenance.from_dict` (the two existing precedents in this codebase)
both use the trivial `cls(**data)` pattern with no field-specific
reconstruction. Reproduced directly: with that same trivial pattern
applied to `RunConfig`, `bounds` survives the JSON hop as `[[0.0, 1.0]]`
(list of lists) rather than `[(0.0, 1.0)]` (list of tuples), and the
round-trip assertion fails, since `[0.0, 1.0] != (0.0, 1.0)`.

Making this test pass forces `RunConfig.from_dict` to special-case
`bounds` reconstruction (rebuilding tuples from JSON's lists), a pattern
with no precedent elsewhere in the codebase. That special-casing is also
fragile against `bounds`' actual shape in practice: `test_scipy_least_
squares.py` uses a tuple of two array-likes (`([0.5, -np.inf], [np.inf,
np.inf])`), while `test_scipy_minimize.py` uses a list of per-parameter
pairs (`[(0.5, np.inf), (-np.inf, np.inf)]`). A `from_dict` written to
satisfy this one round-trip test's specific tuple placement will not
generalise across both shapes.

Recommend rewriting the round-trip test's `bounds` value as a JSON-native
structure (e.g. `[[0.0, 1.0]]`), which sidesteps the tuple/list asymmetry
entirely and keeps `RunConfig.from_dict` as trivial as its siblings, or,
if tuple-shaped `bounds` must round-trip exactly, say so explicitly and
scope the necessary `from_dict` logic now rather than leaving it to be
discovered as a GREEN-phase test failure.

Location: `tests/unit/test_config.py:16-21`.

### Major: `RunResult.from_dict` does not reconstruct `config` as `RunConfig`, and this is outside the PR's stated scope

`_run_result_kwargs` in `tests/unit/test_result.py` now builds `config=
RunConfig(tolerance=1e-8)` instead of a plain dict. `RunResult.to_dict`
uses `dataclasses.asdict`, which flattens the nested `RunConfig` into a
plain dict; `RunResult.from_dict` (`trust_bench/core/result.py`) already
reconstructs `provenance` and `timing` back into their dataclass types
but has no equivalent step for `config`. Reproduced directly against the
unmodified `result.py`: `test_run_result_round_trips_through_json_
serialisation` fails once `config` is a `RunConfig` instead of a dict,
because the restored `config` stays a plain `dict` while the original is
a `RunConfig` instance, and dataclass equality does not treat the two as
equal.

This was passing before this PR (when `config` was just a dict, and dict
round-trips through JSON are already lossless), so the test change is
correct and exposes a real gap. The gap is that `trust_bench/core/
result.py` needs a matching change, and neither the issue title ("config.
py") nor the PR doc's summary or coverage table mention `result.py` as
part of this issue's implementation scope. Recommend adding `data["config
"] = RunConfig.from_dict(data["config"])` to `RunResult.from_dict` during
the GREEN cycle and updating the PR doc's file list accordingly, so the
scope is explicit before implementation rather than discovered mid-GREEN.

Location: `trust_bench/core/result.py` (`RunResult.from_dict`);
exercised by `tests/unit/test_result.py:45-50`.

### Note: `bounds` shape is inconsistent across call sites, but this predates the issue

`RunConfig.bounds` is passed through unvalidated to both `least_squares`
(tuple-of-arrays convention) and `minimize` (list-of-pairs convention) in
the existing, unmodified `scipy_backend.py`. This inconsistency is
inherited from PR #17 and is not introduced by this issue; it only
matters here because it is the reason the round-trip test above cannot
pick one canonical tuple/list shape and have it generalise. No action
needed on this issue; flagged for context only.

## Verification

- Tests: not run (tests-only stage; the 5 test files that reference
  `trust_bench.core.config` fail to collect as expected under TDD). The
  other 72 tests in the suite were run and pass, confirming no
  regressions from the migration of existing dict-configs to `RunConfig`
  in `test_contract.py` and `toy_backend.py`.
- APL semantics: not applicable; no APL expressions appear in this test
  surface. The `scipy` tolerance-to-native-parameter mapping in `test_
  scipy_minimize.py`'s `TOLERANCE_PARAMS` was independently checked
  against `scipy.optimize.show_options` for every method and matches
  exactly; `least_squares` accepting `ftol`/`xtol`/`gtol` unconditionally
  was confirmed against its signature.

## Recommendation

Request changes: fix the `bounds` value in `test_run_config_round_trips_
through_json_serialisation` (or explicitly scope the `from_dict`
reconstruction it requires), and add `trust_bench/core/result.py`'s
`from_dict` fix to this issue's stated implementation scope before the
tests are committed.

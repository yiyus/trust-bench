## Implementation review (GREEN)

## Summary

The `report` command's core deliverable is complete and solid: `trust_bench/cli.py`
implements `STUDIES`/`SLOW_STUDIES`/`_select_studies`/`run_report`/`build_parser`/`main`
exactly as scoped, and `results_to_dataframe`'s `BaseException`-as-`"ERROR"-row` extension is
a small, backward-compatible, correctly targeted fix for `bounded.py`'s mixed result dict
(confirmed directly: `bounded.csv`'s `infeasible_start` rows come out as clean
`status=ERROR` with blank metric fields). However, the working tree currently also contains
a second, later RED cycle (`tests/reporting/test_html_report.py`, plus four new/extended
assertions in `test_cli.py` for a `--html` bundling flag) that has **not** gone through the
project's own RED to GREEN workflow: no tests-only review or approval preceded it, and it
has no implementation yet. Concretely, this means the full suite does not currently pass:
`pytest -m "not slow"` aborts with a collection error
(`ModuleNotFoundError: No module named 'trust_bench.reporting.html_report'`), and three
further assertions in `test_cli.py` fail once that error is worked around. This is a process
and completeness gap, not a defect in the reviewed report-command code itself.

## Findings

### Major: unreviewed, unimplemented `--html` scope-creep breaks the full test suite

`docs/prs/33.md`'s "Visual review" section records that a bundling feature (`--html`) was
requested *after* the report command was already implemented and visually approved, and
its own "RED cycle: `--html` bundle" section was added directly to this PR without a stop
for tests-only review, contrary to the project workflow ("STOP -> Present tests for review
-> Wait for approval" before any implementation begins). As a result:

- `pytest -m "not slow"` (i.e. `make test`) does not run to completion on the current tree:
  it aborts with `ModuleNotFoundError: No module named 'trust_bench.reporting.html_report'`
  during collection of `tests/reporting/test_html_report.py`.
- With that file excluded, three further assertions fail:
  `test_report_command_html_flag_defaults_to_false`,
  `test_report_command_can_request_an_html_bundle`, and
  `test_run_report_writes_an_html_bundle_when_requested`
  (`AttributeError`/`SystemExit`/`TypeError`, all because `--html` and `html=` do not exist
  yet on `build_parser`/`run_report`).
- `test_report_command_produces_every_milestone_artefact` was also extended to require
  `report.html`, which does not exist yet either, though this specific test is
  `@pytest.mark.slow` and was not re-run here (see Verification).

The report-command code that *is* implemented is not at fault; the issue is that this
implementation review was asked to run against a tree that mixes a finished cycle with an
unreviewed, half-started one. Recommend either: (a) split the `--html` addition into its own
RED submission, get it reviewed and approved on its own before implementing, then bring it
back for its own implementation review, or (b) finish `--html`'s GREEN cycle before
resubmitting this issue for review, so the full suite passes end to end as the project's
Definition of Done requires.

Location: `tests/reporting/test_html_report.py` (new, uncollectable),
`tests/integration/test_cli.py:108-149` (new/extended `--html` assertions)

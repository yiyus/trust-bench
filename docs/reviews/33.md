## Tests-only review: `--html` bundle (RED)

## Summary

`tests/reporting/test_html_report.py` defines the expected surface for
`trust_bench.reporting.html_report` (`build_html_report`, `save_html_report`); `--html`,
`test_run_report_writes_an_html_bundle_when_requested`, and the extended
`test_report_command_produces_every_milestone_artefact` extend `test_cli.py` to wire the
flag through `build_parser`/`run_report`/`main`. The design (scan `output_dir` for
`*.csv`/`*.png` after studies have already written their files, rather than threading
`DataFrame`/`Figure` objects through a separate bundling step) is a reasonable, disclosed
tradeoff: it stays decoupled from `--only`/`--skip`/`--skip-slow` for free, at the cost of
potentially bundling a stale file left over from an earlier run into the same directory,
which the design notes state plainly rather than leaving implicit. Confirmed the RED
failure exactly: `test_html_report.py` fails to collect
(`ModuleNotFoundError: No module named 'trust_bench.reporting.html_report'`), and three of
the four new/extended `test_cli.py` assertions fail for the matching reason
(`AttributeError`/`SystemExit`/`TypeError` on the not-yet-existing `--html`/`html=`
surface); the fourth (`test_run_report_does_not_write_an_html_bundle_by_default`) passes
vacuously since nothing writes `report.html` either way yet, as the PR doc itself notes.
213 other tests are unaffected. Two minor test-coverage gaps noted below; neither blocks
proceeding to GREEN.

## Findings

### Minor: no test exercises `build_html_report` with an empty or non-report directory

All three `test_html_report.py` cases pass a directory containing exactly one relevant file
(one CSV, one PNG, or a hand-written HTML string). There is no test for an empty directory
(no `*.csv`/`*.png` at all — does it return valid, if minimal, HTML, or raise?) or for a
directory containing unrelated files (should be silently ignored, per the "scans for
`*.csv`/`*.png`" design, but that filtering is untested).

Location: `tests/reporting/test_html_report.py`

### Minor: the realistic multi-table, multi-plot case is only exercised via the slow end-to-end test

No fast test calls `build_html_report` against a directory with several tables and a plot
together (the actual shape `trust-bench report --html` will produce: nine tables, one
plot). That combination is only exercised through
`test_report_command_produces_every_milestone_artefact`, which is `@pytest.mark.slow` and
so does not run under `make test`. A regression in how multiple files are combined (e.g.
ordering, one file's HTML clobbering another's) would only surface in CI/`make coverage`,
not in the fast dev loop.

Location: `tests/reporting/test_html_report.py`, `tests/integration/test_cli.py:129-149`

## Verification

- Tests: fail as expected (RED).
  `ModuleNotFoundError: No module named 'trust_bench.reporting.html_report'` on the new
  test file. Excluding that file, `test_report_command_html_flag_defaults_to_false`,
  `test_report_command_can_request_an_html_bundle`, and
  `test_run_report_writes_an_html_bundle_when_requested` fail as described in
  `docs/prs/33.md`; `test_run_report_does_not_write_an_html_bundle_by_default` passes
  vacuously. 213 other tests (7 deselected as `slow`) are unaffected.
- APL semantics: not applicable, no APL expressions in this test surface.
- Design conformity: matches Section 10's Phase 7 framing (still `cli.py`/`reporting/*`,
  no new deliverable outside that scope); the directory-scan approach is a disclosed
  tradeoff, not a hidden gap.

## Recommendation

Approve with minor changes (optional: add empty-directory and multi-file coverage before or
during GREEN)

# Review: Issue #5 - Convergence order and rate estimators

## Summary

Tests-only (RED) stage review. `tests/unit/test_metrics.py` covers all three
acceptance criteria in the issue with fixtures whose behaviour was verified
independently (not just re-derived from the prototype). The RED failure
(`ModuleNotFoundError: No module named 'trust_bench.core.metrics'`) is
confirmed by running the suite. No issues found.

## Findings

No issues found.

### Note: `rate()`'s own too-few-points guard has no dedicated test

The PR doc states this explicitly: issue #5's acceptance criteria require the
NaN guard only for `order`, not `rate`. That matches the issue text as
fetched from GitHub, which lists only three criteria and none mentions a
`rate` guard. No action needed at this stage; flagging only so it isn't
mistaken for an oversight if raised later.

Location: `tests/unit/test_metrics.py`

## Verification

- APL semantics: not applicable; this module is pure Python/NumPy, not APL.
- Fixture numerics: independently re-implemented the prototype's
  `convergence_order`/`linear_rate` formulas in plain Python and confirmed:
  - order fixture (`e_0=0.9`, 20 squarings): 9 points survive the `1e-12`
    floor, computed order = 2.0.
  - NaN fixture (`e_0=1e-3`, 10 squarings): 2 points survive the floor,
    computed order = NaN.
  - rate fixture (`e_0=1.0`, `r=0.7`, 60 multiplications): 61 points survive,
    computed rate = 0.7.
  - Confirmed the PR doc's claim that the merged four-point guard is
    equivalent to the prototype's two separate checks: the per-step order
    loop always yields exactly `len(e) - 2` values, so `len(e) < 4` implies
    fewer than 2 order values under the original two-check logic.
- Tests: RED confirmed. `pytest tests/unit/test_metrics.py` fails at
  collection with `ModuleNotFoundError: No module named
  'trust_bench.core.metrics'`, the same failure class as issues #2 and #3.

## Recommendation

Approve.

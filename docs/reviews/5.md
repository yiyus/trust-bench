## Implementation review

### Summary

`order()` and `rate()` are vectorised NumPy implementations of the RED-stage
contract. All three unit tests pass, the full suite (12 tests) shows no
regressions, and lint is clean. One robustness gap survives from the
prototype-to-NumPy port: `order()`'s vectorised division no longer guards
against a zero log-ratio in the denominator, so a plateau in the error trace
can inject `inf` into the values the median is taken over.

### Findings

### Major: `order()` can silently divide by zero when consecutive errors are equal

`trust_bench/core/metrics.py:16` computes
`orders = log_ratios[1:] / log_ratios[:-1]` with no guard on the
denominator. The ported-from prototype (`prototype/harness.py:99-103`)
explicitly filtered each per-step order to `abs(d0) > 1e-14` before
including it, precisely to exclude a zero log-ratio (two consecutive errors
equal). The vectorised version drops that filter.

Reproduced directly: feeding `order()` a trace with one repeated value
(`[1.0, 0.5, 0.5, 0.25, 0.125, 0.0625, 0.03, 0.015]`) raises
`RuntimeWarning: divide by zero encountered in divide` and returns
`0.972...`, versus `1.0` from a guarded re-implementation of the same
formula, a value the median only partially recovers from because one `inf`
in a length-6 array does not tip the median. A trace with more plateaus
would corrupt the result further, and the eventual output carries no signal
that it happened; the `RuntimeWarning` is easy to miss once solver runs are
non-interactive.

This matters beyond the current unit tests because Phase 4/5 of the plan
(`docs/plans/trust-bench.md`, Section 10) feed real optimiser traces into
these functions, where a stalled step producing two numerically identical
error values is plausible, not a contrived edge case.

Recommendation: filter `log_ratios` (or the intermediate `orders`) to finite
values with `abs(denominator) > <floor>` before taking the median, matching
the prototype's per-step guard, or use `np.errstate` plus an explicit
`np.isfinite` mask so the result degrades to NaN rather than a
warning-and-silently-off-by-some-amount value.

Location: `trust_bench/core/metrics.py:11-17`

### Note: no equivalent gap in `rate()`

`rate()`'s division (`e[1:] / e[:-1]`) has `e` filtered to `e > floor`
beforehand, so the denominator is always positive and this class of issue
does not apply there.

## Fix verification (PR #49)

### Summary

PR #49 fixes the Major finding above and closes issue #5. `order()` now
masks the log-ratio denominators to `abs(denominator) > 1e-14` (the
prototype's own threshold, named `_MIN_LOG_RATIO`) before dividing, so the
division that used to raise `RuntimeWarning` or inject `inf` into the
median never happens. A second guard (`valid.sum() < 2`) returns NaN when
too few order estimates survive the mask, the equivalent of the existing
too-few-points guard but for a trace with enough points overall and too
many plateaus among them. Two regression tests cover both branches. No
issues found.

### Findings

No issues found.

### Verification

- Reproduced the review's exact plateau fixture
  (`[1.0, 0.5, 0.5, 0.25, 0.125, 0.0625, 0.03, 0.015]`) independently, with
  warnings promoted to errors: no `RuntimeWarning`, `order()` returns
  `1.0`, matching the guarded re-implementation used in the original
  review.
- Checked the `valid.sum() < 2` boundary directly
  (`[1.0, 0.5, 0.5, 0.5]`): returns NaN, no warning.
- Checked a case the fixtures don't cover: a plateau mid-sequence inside an
  otherwise-quadratic trace (`e_0=0.9`, one duplicated value). Returns
  `1.9999999999999998`, correctly excluding only the affected step rather
  than degrading the whole estimate.
- Tests: pass. Full suite: 14 passed, 0 failed (`pytest tests/`); the two
  new tests (`test_order_ignores_a_plateau_in_the_error_sequence`,
  `test_order_returns_nan_when_plateaus_leave_too_few_valid_estimates`)
  both wrap the call in `warnings.simplefilter("error")`, so either would
  fail if the `RuntimeWarning` regressed.
- Lint: pass (`ruff check .`: all checks passed).
- Comment hygiene: clean. No GitHub issue/PR IDs in `metrics.py` or
  `test_metrics.py`; the new docstring states the plateau invariant
  directly (`abs(denominators) > _MIN_LOG_RATIO`... "since a near-zero
  denominator makes the ratio meaningless, not merely noisy") rather than
  narrating the fix.

### Recommendation

Approve.

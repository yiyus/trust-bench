## Findings

### Minor: a missing (category, group) combination is hatched even though it has no bar to hatch

`plot_metric_by_category`'s hatching loop compares every reindexed `status_col` value against `"CONVERGED"`, including positions where the reindex introduced a row that never existed in `df` (a `NaN` value there, since `float("nan") != "CONVERGED"` is `True` in Python). Reproduced directly with a synthetic ragged dataframe: a `(category, group)` combination absent from the input still gets `bar.set_hatch("//")` applied, even though its bar height is `NaN` and never actually renders. Harmless in practice (matplotlib doesn't draw a `NaN`-height bar, hatch or not) and doesn't affect `typical.csv`'s own data, which has no ragged groups (`typical.sweep()` runs the full problem x backend x method cross product, so every `(method, backend)` group has an entry for every `problem_id`). Worth an explicit `pd.notna(status)` guard if this function is ever reused for data that genuinely has missing combinations, but not blocking here.

Location: `trust_bench/reporting/plots.py:70-77`

### Note: fix addresses the root cause, not a workaround

`plot_metric_by_category` is a new function built for `typical`'s genuinely different (categorical, non-sweep) data shape, not a strained reuse of `plot_metric_vs_sweep`. Its hatching convention mirrors `plot_metric_vs_sweep`'s existing non-`CONVERGED` marker exactly (same `status_col` parameter name, same "flag without dropping the data point" philosophy), keeping the two functions' conventions consistent rather than introducing a divergent one.

Location: `trust_bench/reporting/plots.py:56-89`, `trust_bench/cli.py:158-165`

Additionally constructed a synthetic dataframe with a genuinely ragged group (present for one category, absent for another) and confirmed no crash, consistent with the design notes' claim that this was checked before writing the implementation (see the Minor finding above for the one cosmetic side effect of that reindex approach).

### Note: design rationale is sound

`typical`'s data (one row per `(problem_id, method, backend)`, no continuous sweep axis) genuinely doesn't fit `plot_metric_vs_sweep`'s line-per-group model - a grouped bar chart, one cluster per `problem_id` and one bar per `(method, backend)` within each cluster, is the right shape for a small, fixed category set, distinct from `bounded`'s scenario x method categorical data (excluded from #97 for the same underlying reason: no natural sweep-plot). The non-`CONVERGED` hatching, mirroring `plot_metric_vs_sweep`'s hollow-marker overlay convention rather than inventing an unrelated visual language, keeps the two plot functions' conventions consistent.

The log-y-axis choice for `dist_to_opt` is justified by real, previously-confirmed data (from #104's review cycle): values range from near-zero for a converged fit to the billions for `trust-apl`'s `trust-exact` diverging on `noisy_expdec`/`gaussian_peak` - a linear scale would make every converged bar visually indistinguishable from zero next to a diverged one.

### Note: test design is well-targeted

`test_plot_metric_by_category_draws_one_bar_cluster_per_category`'s expected bar/label counts (4 bars, 2 unique `(method, backend)` labels across 2 categories) match `category_df`'s fixture shape exactly. The hatching tests correctly isolate the `status_col`-present and `status_col`-absent cases, mirroring the equivalent pair already established for `plot_metric_vs_sweep`'s overlay behaviour in #97's own test surface.

# Review: Issue #125 - results_to_dataframe collapses declared-unsupported, timeout, and genuine crash into one ERROR status

## Summary

Tests-only (RED) stage review. The extended test files define the expected
behaviour correctly: `RunResult` gains a `message` field that defaults to
`None` and round-trips through JSON; `results_to_dataframe` relabels a caught
exception's row to `"UNSUPPORTED"` while keeping its message and blanking the
other metric fields; `_pivot_by_backend`'s missing-backend guard treats
`"UNSUPPORTED"` the same as `"ERROR"`; both backends surface their own
already-computed termination message rather than discarding it. Confirmed RED
against the unmodified source, with failures matching the PR doc's stated
count and errors. No issues found.

## Findings

No issues found.

### Note: test design correctly distinguishes the three collapsed cases

- `test_a_declared_unsupported_rejection_is_labelled_unsupported_not_error`
  exercises exactly the case the issue describes (a caught `ValueError` from
  a backend's declared-unsupported rejection), not a generic exception.
- `test_apl_backend_timeout.py`'s and `test_apl_difficulty_families.py`'s
  additions assert on message *content* (`"did not complete"` vs `"Unknown
  problem_id"`), which is exactly what distinguishes a timeout from another
  harness-reported `ERROR` - the issue's core complaint is that both currently
  read identically.
- `test_pivot_by_backend_treats_an_unsupported_row_the_same_as_an_error_row`
  targets `_pivot_by_backend`'s existing `status != "ERROR"` filter
  (`trust_bench/reporting/cross_study.py:45`), confirmed to not yet exclude
  `"UNSUPPORTED"` - the right regression point for the second change.

### Note: harness-side message plumbing already exists, confirmed by inspection

`backends_ext/apl/error_result.dyalog` and `solve.dyalog`/`evaluate.dyalog`
already set a `message` key on every harness response (`NULL` on success, an
explanation on `ErrorResult`) - `apl_backend.py` currently reads
`response["message"]` only for `evaluate_problem`'s failure path and
discards it everywhere else, matching the PR doc's description of the fix as
threading through an existing value rather than inventing new plumbing.

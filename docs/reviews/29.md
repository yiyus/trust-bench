# Review: Issue #29 - Study: derivative source

## Summary

Tests-only (RED) stage. This issue also fixes a real `n_feval`
undercount for `least_squares` in finite-difference mode, disclosed as
in-scope because the study's entire acceptance criterion is an
evaluation-count comparison. Both RED failures and the "179 tests
unaffected" claim reproduce exactly, as does the disclosed-but-deferred
`minimize`-family undercount (`BFGS`: confirmed 10 real calls vs. 5
reported). Three findings, two of them blocking: `powell` (Moré-Garbow-
Hillstrom #13, "Powell singular") structurally cannot reach the
precision test's `1e-6` threshold with `trf`/`dogbox` in either
derivative mode (`dist_to_opt` plateaus at `~6e-4` regardless of
tolerance); a second, separate `n_feval` undercount exists in both
derivative modes (not just finite-difference), undisclosed and untested
by the new regression test; and one figure in the design notes doesn't
match what's actually measured.

## Findings

### Major: `test_both_derivative_modes_converge_precisely_for_every_problem_and_method` will fail for `powell`/`trf`/`dogbox`

`powell` is Moré-Garbow-Hillstrom #13 ("Powell singular function"),
whose defining property is a Jacobian that is singular at the true
optimum. Verified directly against `SciPyBackend`: `trf` and `dogbox`
both converge (`status=CONVERGED`) to `dist_to_opt ~= 5.988e-4` for
`powell`, in *both* derivative modes, and tightening `RunConfig.tolerance`
to `1e-12` does not help (`~3.7e-5` at best, still three orders of
magnitude above the test's threshold). `lm` does not share this: it
reaches `3.6e-11` (analytic) and `2.6e-9` (finite-difference) on the same
problem. `test_both_derivative_modes_converge_precisely_for_every_
problem_and_method`'s `assert result.dist_to_opt < _PRECISION_TOL`
(`1e-6`) loops over every `(problem, method, mode, backend)` combination
unconditionally, so this will fail for `powell`/`trf` and
`powell`/`dogbox` in both modes as soon as `sweep()` exists, independent
of anything this issue's own fix changes.

This is a structural property of the problem, not a tunable precision
choice: MGH #13 is a classic test case specifically because trust-region
methods that rely on a well-conditioned Jacobian near the optimum
plateau well short of machine precision on it. Recommend either loosening
`_PRECISION_TOL` for this one problem (e.g. a per-problem tolerance map,
the same shape choice several other studies in this repo already make
for method-specific expectations), or scoping the "converges precisely"
test to the methods/problems that actually do, with `powell`/`trf`/
`dogbox`'s looser, still-legitimate convergence covered by a separate,
correctly-thresholded assertion.

Location: `tests/studies/test_derivative_source.py:17-23`

### Major: a second `n_feval` undercount exists in both derivative modes, undisclosed and untested

`_solve_least_squares` (`trust_bench/backends/scipy_backend.py:209`)
calls `problem.residual(result.x)` again after `least_squares` returns,
to compute `grad_final`. This is a genuine extra residual evaluation
that scipy's own `nfev` never counts, since it happens outside
`least_squares` entirely. Verified directly across every canonical
problem, every method, in *analytic* mode (no finite-difference
involved): real residual calls are always exactly `reported nfev + 1`
(e.g. `quadratic`: `2` reported vs. `3` real; `powell`/`lm`: `37`
reported vs. `38` real). This is present in every solve, independent of
`derivative_mode`, so it also contributes to the finite-difference
undercount this issue is already fixing, but is a distinct root cause
from the one described in `docs/prs/29.md` (extra Jacobian-estimation
calls), and is not eliminated by fixing that one alone.

The new regression test,
`test_n_feval_counts_every_residual_call_including_finite_difference_
jacobian_steps`, only exercises `derivative_mode="finite-difference"`,
so a fix that patches the FD-specific undercount without also
addressing this redundant call would leave `n_feval` silently off by one
in *every* measurement this study reports, including the analytic
baseline it compares against. Given the PR doc's own stated reasoning
for why the FD undercount is in-scope ("reporting a systematically
wrong count would make the study's central claim false"), the same
reasoning applies here.

`least_squares`'s return value already carries the residual at
`x_final` as `result.fun` (verified directly: identical to
`problem.residual(result.x)` for a solved problem), so the fix is to use
`result.fun` directly rather than recomputing it, which eliminates the
extra call rather than merely counting it correctly. Recommend either
that fix, or at minimum extending the new regression test to also
assert the exact count in analytic mode, matching the FD-mode test's
rigour.

Location: `trust_bench/backends/scipy_backend.py:209`

### Minor: a design-notes figure does not match the measured value

`docs/prs/29.md`'s design notes state "`powell`: 37 reported vs. 336
real, for `lm`" for finite-difference mode. Verified directly
(`RunConfig(max_iter=100, derivative_mode="finite-difference")`,
matching the regression test's own config): the real call count (`336`)
is correct, but the reported `n_feval` for `powell`/`lm` in
finite-difference mode is `67`, not `37` (likely a transposition with
`powell`/`lm`'s *analytic*-mode `nfev`, which genuinely is `37`, per the
Major finding above). This doesn't affect any assertion (the tests
compare values dynamically, not against this hardcoded figure), but the
design notes should record what was actually measured.

Location: `docs/prs/29.md` (design notes paragraph on the `n_feval` fix)

## Verification

- APL semantics: not applicable (pure Python/numpy/SciPy study, no APL
  expressions).
- RED failure (backend regression test) confirmed exactly:
  `AssertionError: assert 2 == 7` for `quadratic`, all three
  `least_squares` methods.
- RED failure (new study) confirmed:
  `ModuleNotFoundError: No module named 'trust_bench.studies.derivative_source'`.
- Full suite minus the 3 new n_feval test instances and the new file:
  179 passed, matching the PR doc's "176 (3 slow) unaffected" claim.
- FD-vs-analytic real call counts independently reproduced across all
  seven canonical problems, all three `least_squares` methods: every
  combination needs strictly more real residual calls in
  finite-difference mode than analytic, with undercount ratios in the
  claimed 2x-9x range (measured 2.48x-5.08x locally).
- `powell`/`trf`/`dogbox` precision plateau (Major finding) confirmed
  directly, including with `tolerance` tightened to `1e-12`; `lm` does
  not share it on the same problem.

  directly: `BFGS` on `quadratic`, analytic mode: `10` real residual
  calls vs. `5` reported `nfev`, exactly as stated.
- Second `n_feval` undercount (Major finding) confirmed directly: every
  canonical problem, every `least_squares` method, analytic mode: real
  calls are exactly `reported + 1`; `result.fun` verified to already
  hold the correct residual, making the extra call unnecessary rather
  than merely uncounted.
- Design-notes figure discrepancy (Minor finding) confirmed directly:
  `powell`/`lm` finite-difference `nfev` is `67`, not `37`.
- Tests: not run (tests-only stage; failures/collection errors recorded
  above are the expected RED state).

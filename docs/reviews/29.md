# Review: Issue #29 - Study: derivative source

## Summary

Tests-only (RED) stage. This issue also fixes a real `n_feval`
undercount for `least_squares` in finite-difference mode, disclosed as
in-scope because the study's entire acceptance criterion is an
evaluation-count comparison. Both RED failures and the "179 tests
unaffected" claim reproduce exactly, as does the disclosed-but-deferred
`minimize`-family undercount (`BFGS`: confirmed 10 real calls vs. 5
reported). Three findings were raised and have since been fixed: a
precision-threshold test that would have failed for `powell`/`trf`/
`dogbox` (a structural property of that MGH test problem, not a
tunable choice); a second, previously undisclosed `n_feval` undercount
present in both derivative modes; and a design-notes figure that didn't
match what was measured. No outstanding issues.

## Findings

### Major (fixed): `test_both_derivative_modes_converge_precisely_for_every_problem_and_method` would have failed for `powell`/`trf`/`dogbox`

`powell` is MorĂ©-Garbow-Hillstrom #13 ("Powell singular function"),
whose defining property is a Jacobian that is singular at the true
optimum. Verified directly: `trf` and `dogbox` both converge
(`status=CONVERGED`) to `dist_to_opt ~= 5.988e-4` for `powell`, in both
derivative modes, and tightening `RunConfig.tolerance` to `1e-12` does
not help (`~3.7e-5` at best). `lm` does not share this on the same
problem (`~1e-9` or better). The original, unconditional `assert
result.dist_to_opt < _PRECISION_TOL` (`1e-6`) across every `(problem,
method, mode, backend)` combination would have failed for
`powell`/`trf`/`dogbox` in both modes.

Fixed with a per-problem tolerance override,
`_PRECISION_TOL_OVERRIDES = {"powell": 1e-3}`
(`tests/studies/test_derivative_source.py:6-18`), justified with the
same structural reasoning verified above, rather than dropping `trf`/
`dogbox` from the sweep: keeps every method covered for every problem,
which reads closer to the acceptance criterion's "for the same problem
set" than excluding a method would. Re-verified directly: `1e-3`
comfortably covers the measured `~6e-4` plateau in both modes.

Location: `tests/studies/test_derivative_source.py:6-18,29-35`

### Major (fixed): a second `n_feval` undercount existed in both derivative modes, previously undisclosed and untested

`_solve_least_squares` recomputed `problem.residual(result.x)` after
`least_squares` returned, purely to derive `grad_norm_final` â€” a
genuine extra residual call that scipy's own `nfev` never counted
either, present regardless of `derivative_mode`. Verified directly
across every canonical problem, every method, in analytic mode: real
residual calls were exactly `reported nfev + 1` (e.g. `quadratic`: `2`
reported vs. `3` real). This was a distinct root cause from the
finite-difference-specific undercount the PR doc originally described,
and the original regression test only covered finite-difference mode,
so it would have survived that fix silently.

Fixed (implied by the new regression test's expectations; the actual
source change lands in the GREEN cycle) by using `least_squares`'s own
`result.fun`, already verified identical to recomputing the residual,
rather than calling `problem.residual` again. Extended coverage with
`test_n_feval_does_not_count_an_extra_call_made_only_to_report_grad_
norm_final` (`tests/backends/test_scipy_least_squares.py`), covering
analytic mode with the same rigour the finite-difference test already
had. Both tests now fail together against the current (unfixed)
implementation, as expected for this stage.

Location: `tests/backends/test_scipy_least_squares.py` (new test),
`trust_bench/backends/scipy_backend.py:209` (fix lands in GREEN)

### Minor (fixed): a design-notes figure did not match the measured value

`docs/prs/29.md`'s design notes originally stated "`powell`: 37 reported
vs. 336 real, for `lm`" for finite-difference mode; the real figure was
`67` (`37` was that problem's *analytic*-mode reported count). Corrected
in the PR doc's "Review response" section; did not affect any assertion.

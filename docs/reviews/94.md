# Review: Issue #94 - robust_loss.py still crashes against trust-apl

## Summary

Tests-only (RED) stage review. Both new/extended tests reproduce the issue's own exact error and fail for the stated reason. No issues found.

## Findings

### Note: reproduction confirmed exactly

Ran both new tests directly against the unmodified `robust_loss.py`. `test_scipy_loss_precision_skips_a_backend_it_has_no_method_mapping_for` fails because `_UnmappedBackend.solve` is actually called (`AssertionError: unmapped-backend should have been skipped, not solved`), proving the current code drives every backend through the hardcoded `_METHOD = "trf"` rather than skipping unmapped ones. `test_trust_apls_lm_is_exercised_for_every_loss_instead_of_raising` fails with `ValueError: trust-apl has no method 'trf'`, raised from `apl_backend.py:144` - the exact error text the issue and `docs/prs/94.md` cite.

Confirmed the root cause directly in `trust_bench/studies/robust_loss.py:12,25`: `_METHOD = "trf"` is a single module-level constant, unconditionally passed to `run()` for every backend in the sweep, with no per-backend mapping or capability check. `APLBackend.capabilities()` only declares `lm`, `BFGS`, `trust-exact` (`apl_backend.py:106-130`); `trf` is scipy-only. This confirms the PR doc's framing precisely.

### Note: loss vocabulary check holds up

Verified `APLBackend`'s `lm` method declares `losses=frozenset(_LOSS_TO_TRUST)` where `_LOSS_TO_TRUST` maps exactly `linear`/`soft_l1`/`huber`/`cauchy`/`arctan` (`apl_backend.py:27-33`) - the same five losses as `robust_loss.py`'s `SCIPY_LOSSES`. The PR doc's claim that no further loss-vocabulary gap exists once the method mismatch is fixed is correct; the new APL test asserting `distance < 1e-6` for every loss at zero contamination is a reasonable invariant, since with no outliers present, the choice of robust loss shouldn't affect recovery of the true parameters.

### Note: test design matches established precedent

`_UnmappedBackend`'s skip-guard mirrors `bounded.py`'s own pattern of never crashing on a backend/method combination a study doesn't know how to drive, and the removed stale comment ("the APL backend does not exist yet") is correctly deleted rather than left to rot, since `APLBackend` has existed since #78.

# Implementation review

## Summary

The fix replaces the single hardcoded `_METHOD = "trf"` with `_METHOD_FOR_BACKEND = {"scipy": "trf", "trust-apl": "lm"}`, looked up per backend and skipped (not raised) when absent, exactly matching the RED-stage design notes. Both previously-RED tests now pass, and a direct end-to-end run of `scipy_loss_precision` with both backends produces real `trust-apl` rows for every loss, matching the issue's acceptance criteria. One Minor gap: the change is uncommitted and `docs/prs/94.md` has no GREEN-cycle section yet.

### Note: fix addresses the root cause, not a workaround

The per-backend method mapping is looked up by `backend.name`, not special-cased around a particular error path, and an unmapped name is skipped rather than swallowed behind a try/except around the `ValueError` - consistent with `bounded.py`'s established skip-guard precedent and with the design notes' explicit rejection of an implicit capability-driven method search (ambiguous for scipy, which has more than one method satisfying the full loss vocabulary). This is a genuine fix to the routing gap identified in the tests-only review, not a narrower patch that only silences the crash.

Location: `trust_bench/studies/robust_loss.py:12-30`

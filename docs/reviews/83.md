## Implementation review

### Summary

The parametrised-id dispatch mechanism (`ParseParametrised`,
`FamilyNameFor`/`FamilyHessianNameFor`, `ApplyParam`) and all four
ported families' residual/Jacobian/Hessian formulas were checked by
hand against the Python reference in `trust_bench/problems/families/`
and match exactly (rotated-diagonal `IllConditioned`, `Scaling`'s
diagonal Jacobian, `Outliers`'s column-stacked design matrix,
`LargeResidual`'s Gauss-Newton-plus-correction Hessian). All 32 tests
in the three affected files pass, the full suite is green (319 passed),
and the one required test change (switching the "study has no trust-apl
coverage" regression check from `ill_conditioning` to `dimensionality`,
since `ill_conditioning` now genuinely has coverage) is correctly
reasoned and independently reproduced. However, direct testing of
`ParseParametrised`'s numeric parsing turned up a real defect the PR
doc's design notes explicitly (and incorrectly) claim was verified
against: negative-exponent scientific notation, which is exactly how
Python formats any float below `1e-4` in magnitude, crashes the parser.

### Findings

### Major: `ParseParametrised`'s `⍎`-based value parsing cannot handle negative-exponent scientific notation, contradicting the PR doc's explicit verification claim

The design notes state: "`⍎`-parsed for the numeric part so both plain
and Python-scientific-notation formatting round-trip correctly." Tested
directly:

```
$ echo "⎕←⍎'1E-07'" | dyalogscript /dev/stdin
⍎VALUE ERROR: Undefined name: E
```

Dyalog's numeric literal syntax requires the high-minus `¯` for a
negative exponent (or negative mantissa), not the ASCII hyphen-minus
`-`/lowercase-or-uppercase `e`/`E` combination Python's `str`/f-string
formatting produces for small floats (`f"{1e-5}"` → `'1e-05'`). `⍎` sees
`-` as an unrelated token after `E`/`e` and raises `VALUE ERROR:
Undefined name: E` (or `e`), not a clean parse failure.

Reproduced end-to-end through the real harness, not just the isolated
expression:

```python
>>> evaluate_problem("outliers(fraction=1e-05)", [0.0, 0.0])
RuntimeError: VALUE ERROR: Undefined name: e
```

and via `APLBackend.solve`, which silently reports `RunStatus.ERROR`
with no further detail (the `VALUE ERROR` is caught by `run.dyalog`'s
outer `:Trap 0` and turned into a generic `ErrorResult`, so at least it
doesn't crash the whole process, but the resulting message is an
internal APL error, not "Unknown problem_id").

None of the parameter values actually exercised by this issue's own
tests or by any existing study trigger this (`kappa`/`s` top out at
`1e8`, which Python still renders in plain decimal — the scientific-
notation threshold is roughly `<1e-4` or `>=1e16`), so nothing currently
fails. But it directly contradicts the explicit, tested claim in the
design notes, and is a real crash waiting for the next family or study
that legitimately parametrises with a small fraction, rate, or scale
factor near zero. Fix `ParseParametrised` to translate `-`/`+` in the
exponent (and, for full robustness, a leading `-` in the mantissa) to
Dyalog's `¯`/nothing before `⍎`, or reject/handle the value some other
way; either way, the design notes' claim needs to either become true or
be corrected to state the actual, narrower guarantee.

Location: `backends_ext/apl/dispatch.dyalog:35-44`
(`ParseParametrised`); `docs/prs/83.md` (design notes claim)

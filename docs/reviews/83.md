## Findings

### Minor: `ill_conditioned`'s chosen `kappa=100.0` does not match the precedent the PR doc cites

The PR doc and a code comment in both `test_apl_difficulty_families.py`
and `test_apl_parity.py` state the four representative parameter values
match "the values already used in `tests/problems/test_families.py`'s
own parity checks". Checked directly: `scaling`'s `s=10.0` (∈
`SCALING_S_PARITY`), `large_residual`'s `rho=10.0` (∈
`LARGE_RESIDUAL_RHOS`), and `outliers`'s `fraction=0.3` (∈
`OUTLIER_FRACTIONS`) all do. `ill_conditioned`'s `kappa=100.0` does not:
`ILL_CONDITIONED_KAPPAS` in that file is `[1.0, 10.0, 1e3, 1e6]`, which
does not include `100.0` (it matches `ill_conditioning.py`'s own study
`KAPPAS` list instead, which does include `1e2`). This is a narrow
documentation-accuracy issue, not a test-correctness one:
`ill_conditioned.make(kappa)`'s residual is exactly zero at `x_star`
for any valid `kappa` (the family is a consistent linear system by
construction), so `test_evaluate_mode_reports_residual_jacobian_and_
hessian_for_a_parametrised_problem`'s assertion holds regardless of
which `kappa` is used, and the parity test's second probe point
(`x_star + 1.0`) still exercises real `kappa`-dependent behaviour.
Correct the comment/PR-doc claim (or the chosen value) so the stated
provenance is accurate.

Location: `tests/backends/test_apl_difficulty_families.py:16`,
`tests/backends/test_apl_parity.py:14`, `docs/prs/83.md`

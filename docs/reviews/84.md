# Review: Issue #84 - APLBackend: expose BFGS and Newton methods

## Summary

Tests-only (RED) stage review. Most of the new test surface is well
grounded: the dispatch convention (2-tuple result to BFGS, 3-tuple to
true Newton, both via `Min`) is confirmed against `trust`'s own
`Test.aplf` (`RosenbrockBFGS`/`RosenbrockNewton`), and the Rosenbrock
Hessian value asserted in `test_evaluate_mode_also_reports_the_hessian`
(`[[665.0, 240.0], [240.0, 100.0]]` at `x=(-1.2, 1.0)`) checks out by
hand against the project's own cost convention (`cost = 0.5 * r @ r`,
`trust_bench/backends/scipy_backend.py:56`) and the existing
`rosenbrock._hessian` Gauss-Newton-plus-correction formula. Running the
full suite (including slow/APL tests) confirms 17 of the 18 newly
failing assertions are genuinely RED against the current
`Solve.dyalog`/`apl_backend.py` (method field ignored, Hessian never
computed, `capabilities()` only declares `lm`). One test in the new
surface, however, passes unchanged against the pre-fix code and does not
demonstrate the behaviour it is meant to protect (see Major finding
below).

## Findings

### Major: `test_bounds_work_for_the_trust_exact_method` is not RED

`tests/backends_ext/apl/test_harness.py::test_bounds_work_for_the_trust_exact_method`
requests `method: "trust-exact"` against the bounded quadratic scenario
and asserts only `status`/`x_final`. Running it against the unmodified
harness (`Solve.dyalog` currently ignores `req.method` entirely and
always dispatches through the residual/Jacobian `lower` closure, i.e.
always LM) passes today: LM converges to the same boundary point
`(0.5, 0.0)` as `trust-exact` would for this scenario, so the assertion
holds whether or not the method is actually honoured.

Confirmed directly: `pytest tests/backends_ext/apl/test_harness.py -q`
shows 6 failures out of 7 changed/new tests in that file; this is the
one that passes. The PR doc's RED account ("17 new/changed assertions
fail... test_harness.py") does not mention this test as an exception,
so its pass is easy to misread as part of the confirmed RED set.

As written, this test cannot catch a regression where bounds handling
is broken specifically for `trust-exact` (e.g. `Bounded` wrapping the
Newton-shaped evaluator incorrectly) while LM's bounds handling remains
fine, since the harness would silently fall back to LM and the test
would still report success. Assert something that only holds when
`trust-exact` is genuinely exercised, e.g. `n_heval > 0`, before this
test is trusted as coverage for the bounds/Newton interaction.

Location: `tests/backends_ext/apl/test_harness.py:117-131`

## Implementation review

### Summary

`solve.dyalog` now dispatches on `req.method` (`lm`/`BFGS`/`trust-exact`),
each of the seven canonical problems gains a ported `*Hessian` function,
and `evaluate.dyalog`'s probe mode returns the Hessian too;
`apl_backend.py` forwards `method` to the harness and declares the two
new capabilities. The dispatch logic, Hessian ports, and
`n_heval`-never-null behaviour all check out against the RED test
surface and direct verification (dispatch guard, cost convention, dyalog
semantics). However, running the full suite surfaces a real regression
the PR doc does not account for, and a second, narrower correctness gap
in the new `n_heval` counter.

### Findings

### Critical: this change breaks `test_run_report_raises_clearly_when_a_study_does_not_support_the_selected_backend`

`ill_conditioning.sweep`'s existing skip-guard (`if method not in
supported: continue`) only skips a method for a backend that doesn't
declare it. Before this change, `trust-apl` declared only `"lm"`, and
`ill_conditioning.METHODS` is `["trust-exact", "Newton-CG", "BFGS"]`, so
every method was skipped for `trust-apl` and the study produced zero
`trust-apl` rows; `cli.py`'s `_check_backend_coverage` (from #77) then
raised `ValueError`, which is exactly what
`tests/integration/test_cli.py::test_run_report_raises_clearly_when_a_study_does_not_support_the_selected_backend`
asserts.

This PR makes `trust-apl` declare `"BFGS"` and `"trust-exact"`. Neither
is skipped by `ill_conditioning.sweep` any more, so `run()` is now
called for them; `ill_conditioning`'s problem ids
(`"ill_conditioned(kappa=...)"`) are not known to the APL harness's
`NameFor`, so each call now returns a real `RunResult`-shaped `ERROR`
row instead of being skipped. `_check_backend_coverage` only checks that
a backend's name appears at least once in the output, which an `ERROR`
row satisfies, so the `ValueError` this test expects is no longer
raised.

Confirmed directly:

- On the pre-implementation commit (`93126c6`), this test passes.
- On this commit (`5d5f5da`), it fails (`Failed: DID NOT RAISE
  ValueError`).
- `ill_conditioning.sweep(backends=[APLBackend()])` now returns 18
  `ERROR`-status rows for `trust-apl` (`kappa` x `{BFGS, trust-exact}`),
  confirmed by direct inspection.

The PR doc's verification section states "all 71 tests... pass" and
"no regressions", which is not accurate: the full suite (`.venv/bin/
pytest -q`, dyalogscript available in this environment) reports 298
passed, 1 failed, and the one failure is this pre-existing integration
test. This is exactly the kind of "silent empty/wrong result" #77 was
meant to guard against, now inverted: a backend that *shouldn't* be
credited with coverage for a study (because the study's problems are
still unported, per this issue's own stated scope) now passes the
coverage check by returning errors instead of being filtered out
upstream by `_check_backend_coverage`'s current all-or-nothing
"does the backend's name appear at all" test — that check cannot
distinguish "produced real data" from "produced only errors" for a
study using APL-unknown problem ids. Needs a decision before merge:
either the coverage check should not count all-`ERROR` coverage as
sufficient, or this specific test's premise (zero rows implies
`ValueError`) needs to be replaced with an assertion that matches the
new, `ERROR`-row-producing reality it no longer describes.

Location: `trust_bench/studies/ill_conditioning.py` (skip-guard now
inert for `trust-apl`); `trust_bench/cli.py:28-36`
(`_check_backend_coverage`); `tests/integration/test_cli.py:173-177`

### Minor: `n_heval`'s final increment does not correspond to a real Hessian evaluation

`solve.dyalog`'s post-convergence block recomputes `grad` via `Apply f
r.p` (residual/Jacobian only, to avoid a wasted Hessian evaluation
purely for `grad_norm_final`) but still does `hcalls←hcalls+1` for
`trust-exact` unconditionally:

```
finalpair←Apply f r.p
grad←(⍉2⊃finalpair)+.×1⊃finalpair
:If method≡'trust-exact'
    hcalls←hcalls+1
:EndIf
```

No `Apply hf` call happens here, so this increment does not reflect a
genuine Hessian evaluation; it mirrors the pre-existing `calls←calls+1`
pattern immediately above, but that one does correspond to a real
`Apply f` call. The result is that `res.n_heval` for `trust-exact` is
always exactly one higher than the number of times the Hessian was
actually computed, which cuts against the PR's own stated goal (a
"genuine", never-fabricated Hessian count, replacing the old hardcoded
`None`/`0` synthesis). No test asserts an exact `n_heval` value, so this
does not surface as a failure, but it is a real, if narrow, inaccuracy
in the reported metric. Either drop the final increment (accepting
`n_heval` under-reports by one relative to iterations, consistent with
not having recomputed the Hessian there) or compute `hf` at that point
too, whichever the metric is meant to mean.

Location: `backends_ext/apl/solve.dyalog:48-52`

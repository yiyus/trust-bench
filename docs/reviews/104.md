# Implementation review

## Summary

The four new problems (`noisy_expdec`, `logistic`, `michaelis_menten`, `gaussian_peak`) have correct, cross-verified analytic derivatives, a correctly-wired native APL port, and `typical.sweep()`/`cli.py` wiring exactly matching the design notes. All directly relevant tests pass (parity, problem invariants, study coverage, end-to-end CLI). One Major gap: despite this study's entire premise being a measured `trust-apl`-vs-`scipy` capability difference (`trust-exact` diverging on two of the four problems - independently reproduced below), no committed test exercises `typical.sweep()` or the `typical` CLI study with the `trust-apl` backend at all.

## Findings

### Major: no test coverage exercises trust-apl for the typical study, missing this project's own established per-study pattern

Independently ran `typical.sweep(backends=[SciPyBackend(), APLBackend()])` directly: `trust-apl`'s `trust-exact` returns `MAX_ITER` with `dist_to_opt` in the billions/thousands on `gaussian_peak` and `noisy_expdec`, while converging cleanly on `logistic` and `michaelis_menten` - reproducing the commit message's claim exactly. This is the specific, real capability difference this issue was filed to surface (per its own acceptance criteria and context section).

`tests/studies/test_typical_study.py`'s two tests both call `sweep()` with no `backends` argument, which resolves to the module default `BACKENDS = [SciPyBackend()]` - `trust-apl` is never included. `tests/integration/test_cli.py`'s `test_report_command_produces_every_milestone_artefact` likewise runs `main(["report", ...])` with no `--backends` flag, so the `typical` study's row in `typical.csv` is never checked against real `trust-apl` data either. This project has an established, consistent pattern for exactly this situation - a `pytest.mark.slow`, `dyalogscript`-gated companion file (`test_bounded_apl.py`, `test_robust_loss_apl.py`, `test_dimensionality_apl.py`) that exercises real APL-backend behaviour for a study - and `typical.py` has no such companion. Without it, nothing protects the measured divergence (or the convergence on the other two problems) against silent regression, and nothing confirms `typical.sweep()`/`_write_typical` even runs mechanically against the real harness without raising.

Recommend adding a `tests/studies/test_typical_study_apl.py` (or extending an existing slow-marked file) that runs `sweep(backends=[SciPyBackend(), APLBackend()])` and pins the current, real behaviour: `trust-exact` converges on `logistic`/`michaelis_menten` and does not spuriously converge on `gaussian_peak`/`noisy_expdec` (or, at minimum, that every `(problem, method)` pair dispatches to a real status without raising), mirroring the precedent in the three existing `_apl.py` study test files.

Location: `tests/studies/test_typical_study.py`, `tests/integration/test_cli.py` (missing: a `trust-apl`-inclusive test)

### Note: fix addresses the root cause, not a workaround

The four new problems are ordinary curve-fitting problems (not adversarial constructions), each with a native APL port and a numerically-computed optimum, exactly matching the design notes. `typical.sweep()` has no skip-guard and iterates each backend's own declared methods directly, correctly reflecting the stated design rationale (there is no "unsupported method" case to skip here). `_check_backend_coverage` is applied in `_write_typical` the same as every other study, so a backend producing zero rows is still caught even though a backend producing rows with a non-`CONVERGED` status is not (and, per the finding above, isn't tested for either way with `trust-apl`).

Location: `trust_bench/studies/typical.py`, `trust_bench/cli.py:152-155`

### Note: analytic derivatives independently verified

Ran the full cross-language parity suite (`tests/backends/test_apl_parity.py -m slow`): 16 passed, including all four new problems at both their standard start and computed optimum. Independently confirmed the embedded APL data literals (`_NoisyExpDecY`, `_MichaelisMentenY`) match the Python-side fixed-seed-generated arrays value-for-value, ruling out a copy-paste transcription error that the parity test's tolerance might have masked. `logistic`'s and `gaussian_peak`'s Hessians are the most involved (4x4 and 3x3 with `cosh`/`sinh` correction terms respectively) and both pass parity at the stated tolerances, consistent with the design notes' description of a sympy-derived, cross-checked derivation.

### Note: code quality and comment hygiene

No GitHub issue references in any new source file (problems, `typical.py`, `cli.py`, or the `.dyalog` files). The one substantive fix made during GREEN and not anticipated at RED - tightening `logistic`/`gaussian_peak`'s optimum computation (`ftol=xtol=gtol=1e-14`) rather than only loosening the known-optimum test tolerance - is the right call: the PR doc correctly attributes the discrepancy to the default solver's own numerical slack at a larger parameter scale, not a formula error, and fixes it at the source rather than papering over it in the test. The RED-stage test-basename collision (`tests/studies/test_typical.py` renamed to `test_typical_study.py`) is a mechanical, well-explained fix unrelated to any implementation logic.

# Review: Issue #95 - Document trust's BFGS robustness gap; stop trust-bench's status classification from conflating a stall with convergence

## Summary

Tests-only (RED) stage review of staged, uncommitted work. The new test fails for the stated reason (`RunStatus.STALLED` doesn't exist yet), and every empirical claim in the PR doc - the BFGS cost blow-up, the trust-exact stall-reported-as-`CONVERGED` pattern, and the root cause in `solve.dyalog`'s status classification - is independently reproduced exactly. No issues found.

## Findings

### Note: reproduction confirmed exactly

Ran the new test directly: `AttributeError: STALLED` from `RunStatus.__getattr__`, exactly as the PR doc states - `RunStatus` currently has `CONVERGED`/`MAX_ITER`/`FAILED`/`DIVERGED`/`ERROR` only, no `STALLED` member.

### Note: the root-cause diagnosis is precise, not a guess

Confirmed `backends_ext/apl/solve.dyalog:87-93` directly: the status classification is exactly the three-bucket `MAX_ITER` / `FAILED` (`r.dnorm>r.dmax`) / else-`CONVERGED` the PR doc describes, with no check against `r.cost`/`cfg.tolc` at all. Confirmed `Newton.aplo:18`'s own termination check directly: `(ti<i)∨(dx∧.=⍺ ⍵)∨(tc>c)∨(r>0)∧tr>r` - four distinct conditions, of which `tc>c` (cost genuinely below tolerance) and `(r>0)∧tr>r` (relative-change stalled) are exactly the two the PR doc says get collapsed into one `CONVERGED` label. This is the actual harness code, not an inferred description of it.

### Note: the empirical figures are reproduced almost to the last digit

Ran `APLBackend().solve(ill_conditioned.make(kappa=kappa), "trust-exact", "standard", RunConfig(max_iter=200))` directly for kappa in `{1e6, 1e7, 1e8}`: `CONVERGED` with `cost_final≈1.10e-15` at kappa=1e6 (genuine convergence), `CONVERGED` with `cost_final≈6.964` (`dist_to_opt≈3.732`) at both kappa=1e7 and 1e8 - matching the PR doc's cited `cost_final≈6.96`/`dist_to_opt≈3.73` exactly, and confirming the same wrong `CONVERGED` label at both kappa values, not a fluke at one point.

Ran the equivalent `BFGS` sweep against both `APLBackend` and `SciPyBackend` for kappa in `{1e4, 1e6, 1e7, 1e8}`: `trust-apl` reports `CONVERGED` with `cost_final≈1,097,297` at kappa=1e4 (matching the PR doc's cited figure exactly), climbing to `FAILED` with cost in the trillions by kappa=1e7/1e8, while `SciPyBackend`'s own `BFGS` stays at or near machine precision (`0.0`-`1e-25`) across the entire range - confirming the capability gap is real, large, and BFGS-specific (not shared by `trust-exact`, which tracks scipy's own numbers through kappa=1e6).

### Note: design rationale is sound

`STALLED` as a new member rather than repurposing the existing unused `DIVERGED` is the right call: "diverged" implies the solver moved away from a solution, while the relative-change criterion firing means the solver stopped moving with no guarantee about proximity to the optimum - a materially different claim. Scoping the fix to `solve.dyalog` alone (not `Newton.aplo`) matches the actual division of responsibility confirmed above: `Newton.aplo` returns raw state and never itself asserts convergence; the overclaim is introduced entirely by the caller's classification. The claim that `dimensionality`'s existing tests (`test_solve_bfgs_completes_without_a_timeout_at_a_practical_dimension`, `test_solve_trust_exact_converges_at_n_1000`) already cover the BFGS-specific pattern at increasing `n` is confirmed directly against the current test file - both exist with exactly the described assertions, so no new test is needed there.

### Note: comment hygiene

No GitHub issue references in the new test's comment; it states the measured invariant (the two Newton.aplo termination conditions, the specific cost values at kappa=1e6 vs 1e7) rather than narrating the investigation.

# Implementation review

## Summary

Implementation-stage review of `6c7dd44` (RED) and `5bf7bc7` (GREEN). The fix addresses the root cause identified in the tests-only review: `solve.dyalog`'s status classification now distinguishes a genuine cost-tolerance guarantee from a relative-change stall using a second, method-agnostic signal (the final gradient norm), rather than the cost-only gate the PR doc's design notes originally proposed. That first attempt broke two already-passing tests (`large_residual`/`outliers`, whose true optimum has nonzero residual by construction) and, separately, every bounded-problem test (an active-boundary optimum has a genuinely nonzero unconstrained gradient); both regressions were caught by running the full suite and fixed before this stage, and both fixes are documented with the measured evidence behind them. All 370 tests pass. No issues found.

## Findings

### Note: root cause and repro independently confirmed

Re-ran every case cited in the PR doc directly against the current commit: `trust-exact` at kappa=1e6 reports `CONVERGED` (`cost_final≈1.10e-15`, `grad_norm_final≈4.7e-8`); at kappa=1e7/1e8 it now reports `STALLED` (`cost_final≈6.96`, `grad_norm_final` 4.8 and 94.3 respectively, both far above the `1e-2` gate). `BFGS` at kappa=1e3 reports `CONVERGED` (`grad_norm_final≈1e-10`); at kappa=1e4 it now reports `STALLED` (`cost_final≈1.1e6`, `grad_norm_final≈1.48e7`) rather than the previous, wrong `CONVERGED`. Matches the PR doc's figures exactly.

### Note: the two regressions this cycle caught are real, and the fixes for both are correct

`large_residual(rho=10.0)` and `outliers(fraction=0.3)` solved with `lm`: reproduced `cost_final≈46.8`/`19.3` (large by construction, these families' true optimum has nonzero residual) alongside `dist_to_opt` of `~4e-8`/`~2.5e-15` (genuinely at the optimum) and `grad_norm_final` of `~1.6e-6`/`~1e-14` (small). A cost-only gate would misclassify these as `STALLED`; the gradient-norm addition correctly reports `CONVERGED`. Confirmed directly against the current commit: both report `CONVERGED`.

The bounded case: `quadratic` with a lower bound of `0.5` on `x1` (its unconstrained optimum is `(0,0)`, infeasible), solved with `BFGS`, reaches `x_final≈(0.5, ~0)` with `grad_norm_final≈0.5` - genuinely nonzero, since an active bound means the unconstrained gradient need not vanish at the constrained optimum (only a projected/KKT gradient would). The fix correctly special-cases `bounded` to keep the prior, unconditional `CONVERGED` classification, documented in both the code comment and `docs/methodology.md` as a known limitation (box-constrained near-optimality isn't checked), not a claim this fix resolves it. Confirmed directly: the same scenario now reports `CONVERGED` again.

### Note: the `1e-2` gradient-norm threshold is a harness heuristic, honestly labelled as one

`docs/methodology.md`'s new section states plainly that `trust` itself uses no such threshold anywhere, and gives the measured margin justifying it: every genuine convergence in this issue's dataset (including the large-residual case) leaves a gradient norm at or below `1.6e-6`; every genuine stall leaves one at `4.8` or higher. Six orders of magnitude of margin across every case measured so far is a comfortable choice, and the PR doc/methodology.md are candid that this is a heuristic disambiguator, not a value with a first-principles derivation - appropriate given trust's own engine has no equivalent concept to appeal to.

### Note: design conformity and comment hygiene

`RunStatus.STALLED` (not repurposing the unused `DIVERGED`) and confining the fix to `solve.dyalog` (not `Newton.aplo`) both match the tests-only review's already-approved design notes exactly. `trust_bench/backends/apl_backend.py`'s `_STATUS` dict update is the minimal corresponding change. No GitHub issue references in either commit's source comments; the `solve.dyalog` comment explaining the bounded exclusion states the invariant (unconstrained gradient genuinely nonzero at an active bound) rather than narrating the investigation.

### Note: capability-gap documentation matches the measured data

Re-ran the `ill_conditioning` sweep with `trust-apl` wired in: `BFGS`'s cost climbs from machine precision through `~1.1e6` at kappa=1e4 to `~2.7e14` at kappa=1e8 while scipy's own `BFGS` stays at `0.0` from kappa=1e6 onward - matches `docs/methodology.md`'s new "Capability boundary" section exactly. The claim that `trust-exact` tracks scipy's own trust-exact closely until the same stall pattern appears only at kappa=1e7/1e8 is consistent with the Note above. The dimensionality figures (`n=100` BFGS worse residual than scipy's, `n=1000` timeout) are cited from already-existing, already-passing tests rather than asserted fresh, and those tests still pass unchanged.

## Findings

### Minor: Fingerprint stability test cannot distinguish a genuinely deterministic fingerprint from a cached one

`test_machine_fingerprint_is_stable_across_calls` calls `capture()` twice in
the same process and compares `machine_fingerprint`. Section 4.3 documents
`machine_fingerprint` as "a stable hash to group runs by machine", which
implies stability across separate process invocations (a benchmark run
today must group with a run next week on the same machine). Two calls
within one test process pass this check even for an implementation that
computes the fingerprint once (or generates a random value) at import time
and caches it in a module-level variable; such a value would be stable
within a process but would differ on every fresh run, defeating the stated
purpose. The test cannot fully verify the acceptance-adjacent contract
without either asserting the fingerprint is a deterministic function of
observable machine properties (e.g. recomputing an expected hash from
`platform.node()`/CPU info independently) or exercising `capture()` across a
subprocess boundary. Not a blocker for this stage, since the acceptance
criteria only require a "populated" and "round-trippable" `EnvProvenance`,
but worth keeping in mind if `capture()`'s implementation caches rather than
deterministically derives the fingerprint.

Location: `tests/unit/test_provenance.py:24-25`

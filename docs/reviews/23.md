## Findings

### Minor: `backend_results()` solves the "standard" start twice per rho/backend

`backend_results()` (`trust_bench/studies/large_residual.py:60-76`) stores
`run(..., "standard", ...)` in `results`, then recomputes the same
`(problem, backend, "standard")` solve again inside the `distances`
comprehension, since `large_residual.make`'s only registered start is
`"standard"`. The double solve is currently free in practice (a single
`SciPyBackend` least-squares solve on a 2-parameter, 40-point problem), and
the structure mirrors `baseline.py`'s genuinely multi-start case (Rosenbrock's
`far`), so it is not a design defect, just a redundant call for this
particular family. Not blocking; worth a one-line fix (seed `distances` from
`results[(rho, backend.name)]` for the `"standard"` entry and only re-run for
any other registered starts) if `large_residual.make` ever grows a second
start.

Location: `trust_bench/studies/large_residual.py:70-75`

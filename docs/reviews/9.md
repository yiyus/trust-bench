# Review: Issue #9 - Canonical problem set: Rosenbrock, Beale, Powell, Helical, ExpDec, Quadratic, Linear

## Summary

Tests-only (RED) stage review. `tests/problems/numerics.py` implements a
standard 4th-order 5-point central-difference Jacobian and a central-
difference Hessian (verified against the textbook stencils), and
`tests/problems/test_canonical.py` parametrises both acceptance criteria
over all seven problems. The RED failure
(`ModuleNotFoundError: No module named 'trust_bench.problems'`) is
confirmed. One Major gap: the per-probe-point and per-optimum assertions
sit inside a bare `for` loop with no check that the iterable is non-empty,
so a problem module that fails to populate `probe_points` or `optima`
would pass both tests vacuously.

## Findings

### Major: an empty `probe_points` or `optima` list passes its test vacuously

`test_analytic_jacobian_matches_finite_differences_at_every_probe_point`,
`test_analytic_hessian_matches_finite_differences_at_every_probe_point`, and
`test_known_optimum_satisfies_gradient_and_cost_invariants` each iterate
`for point in problem.probe_points` / `for optimum in problem.optima` and
assert inside the loop body. If a problem module ships with an empty list
(a plausible slip when hand-transcribing seven problems' worth of
`Problem(...)` construction, e.g. forgetting to populate `probe_points` for
one of the seven), the loop body never runs, no assertion fires, and the
test reports green. The acceptance criterion says "at every `probe_point`,"
which an empty set satisfies only vacuously, not in the sense the criterion
intends.

Recommendation: assert non-emptiness before the loop (e.g.
`assert problem.probe_points` / `assert problem.optima`), or restructure so
each point/optimum is its own parametrised case (so a problem contributing
zero cases is visible in the test count rather than silently absorbed into
a single green result).

Location: `tests/problems/test_canonical.py:27-50`

### Minor: shared FD helper relies on implicit pytest import-mode, with a generic module name

`test_canonical.py:3` does `from numerics import fd_hessian, fd_jacobian`,
which resolves only because `tests/problems/` has no `__init__.py` and
pytest's default "prepend" import mode inserts that directory onto
`sys.path`. This is the first shared-test-helper module in the suite (prior
issues' tests only import from `trust_bench`), and the generic name
`numerics` is a plausible future collision point if another test directory
without an `__init__.py` also wants a module by that name. Not blocking;
either a more specific name (`fd_stencils.py`) or moving the helper into a
`conftest.py` fixture would make the dependency on pytest's import
machinery explicit rather than incidental.

Location: `tests/problems/test_canonical.py:3`

## Verification

- APL semantics: not applicable; this module is pure Python/NumPy, not APL.
- Finite-difference formula check: `fd_jacobian`'s stencil
  (`(-rp2 + 8*rp1 - 8*rm1 + rm2) / (12*h)`) matches the standard 4th-order
  5-point central-difference first-derivative formula exactly. `fd_hessian`'s
  diagonal stencil (`(f(x+h)-2f0+f(x-h))/h^2`) and mixed-partial stencil
  (the 4-point cross-difference divided by `4h^2`) are the standard
  textbook second-derivative and cross-partial finite-difference formulas.
- Tolerance conformance: `test_canonical.py`'s `atol=1e-6` (Jacobian) and
  `atol=1e-2` (Hessian) match the PR doc's stated, empirically-derived
  margins exactly, and `fd_jacobian`/`fd_hessian`'s default `h` values
  (`1e-4`, `1e-3`) match what the PR doc says was used to derive those
  margins.
- Known-optimum invariant: `grad_star = problem.jacobian(x*).T @ r(x*)` is
  the correct gradient formula for `0.5*||r||^2`; `cost_star = 0.5*r@r`
  matches the registered `Optimum.cost_star` field. Both checks use
  `atol=1e-8`, tighter than the FD tolerances, correctly reflecting that
  this check compares analytic values to each other, not to an FD estimate.
- Comment hygiene: clean. No GitHub issue/PR IDs in either test file.
- Actual problem correctness (the seven analytic Jacobians/Hessians
  themselves): not independently verifiable at this stage since
  `trust_bench.problems` does not exist yet; the PR doc states this was
  checked numerically in a throwaway script excluded from the PR, to be
  re-verified against these tests once the implementation lands.
- Tests: RED confirmed. `pytest tests/problems/test_canonical.py` fails at
  collection with `ModuleNotFoundError: No module named
  'trust_bench.problems'`, the same failure class as issues #2, #3, #5, #6,
  and #8. Running the full suite together shows no collection collision
  from the bare `numerics` import.

## Recommendation

Request changes: close the vacuous-pass gap for empty `probe_points`/
`optima` before merge.
